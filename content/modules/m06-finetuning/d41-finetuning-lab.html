Module 6 · Fine-tuning & Model Adaptation Day 41 75 min

Lab — design a fine-tuning plan

No GPU today. The hard part of fine-tuning is the thinking, deciding whether to do it at all, then specifying the data, the config, the eval, and the rollout. You will write a complete, defensible plan for a realistic scenario, then check it against a model solution. This is exactly the artifact a senior interviewer asks you to talk through.

By the end you can
  • Decide whether fine-tuning is even the right tool, before designing it.
  • Write a data specification: example count, format, and how to build the pairs.
  • Sketch a LoRA config and justify each choice from the constraints.
  • Define an eval plan: a held-out set plus the Module 5 judge, with an explicit ship bar.
  • Lay out a safe rollout, and articulate the whole plan as an interview answer.

The scenario

Brief

You want MaiQ to always turn a free-text issue into a strict JSON ticket in a consistent, terse house tone. The required shape is fixed:

{
  "category": "bug" | "feature" | "question" | "infra",
  "priority": "low" | "medium" | "high" | "urgent",
  "summary": "<= 14 words, terse, no fluff",
  "component": "string | null"
}

Today, a strong general model with a careful prompt and a few examples gets this right most of the time, but it slips: occasional extra prose around the JSON, an invented enum value, an over-long summary, roughly a few percent of the time under load. Product wants the slip rate near zero, and wants to drop the long prompt to cut latency and cost. Your job: decide if fine-tuning is right, and if so, plan it end to end.

How to run this lab

Spend about 45 minutes writing your own plan first, in the five-part template below, before opening any model solution. Write real sentences and real numbers (ranges are fine), not bullets of intent. Then spend ~30 minutes comparing against the model solution under each section and noting where your reasoning was thinner. The gap is the lesson, exactly like the daily practice problems, but at system scale.

Part 1 — Is fine-tuning even right?

Your turn

Before any config, justify the lever. Run the scenario through Day 37's flowchart. Is this a FORM problem or a FACTS problem? Have prompting and few-shot been exhausted? Is there a latency/cost reason? Could structured output (tool/function calling, schema-constrained decoding) solve it without training? Write 4 to 6 sentences ending in a clear decision: fine-tune, or not, and why.

Show model solution — Part 1

This is a FORM problem (a fixed format and tone), not a FACTS problem, so RAG is not the lever. Two things justify fine-tuning here, and both are legitimate per the flowchart. First, the behavior is well-defined and demonstrable with examples, which is what SFT teaches best. Second, there is a real latency/cost driver: the current solution leans on a long prompt plus few-shot examples on a large model, and fine-tuning a smaller model lets you shed that overhead, the classic "distill the prompt into the weights." Before committing, the honest first move is to try harder cheap levers: enforce the shape with structured output (tool/function calling or constrained decoding) so the JSON is schema-valid by construction, which often eliminates the extra-prose and invalid-enum slips outright. The defensible decision: attempt structured output first; if the residual problem is purely cost/latency or the tone consistency at the last percent, fine-tune a small model so it produces the shape and tone natively with a short prompt. Fine-tuning is justified, but only as the rung above structured output, not instead of it.

Part 2 — Data specification

Your turn

Specify the dataset. How many examples to start? What is each example (input and ideal output)? How will you build them, and how do you guarantee quality and consistency? What format/wrapper do they need? Where does the loss apply? Write the spec a teammate could execute.

Show model solution — Part 2

Count: start with a few hundred clean pairs (say 300 to 500); this is a narrow form task, so quality and consistency matter far more than volume, and 400 hand-checked pairs beat thousands of noisy ones (Day 38). Each example: input = a real free-text issue; ideal output = the exact, valid JSON ticket in the house tone. Build: mine real MaiQ issues, draft each ticket (optionally bootstrap drafts with the current model, then human-correct every one), and reject any pair whose JSON is invalid, whose enums are off, or whose summary breaks the 14-word, terse rule, because the model copies exactly what you show it, mistakes included. Consistency: one rigid format, the same field order, the same tone, across every example; validate every "ideal" output against the JSON schema programmatically before it enters the set. Format: wrap each pair in the base model's own chat template via its tokenizer (never hand-rolled role markers), and mask the loss to the assistant (JSON) tokens only so the model learns to produce the ticket, not echo the issue. Hold out 10 to 20 percent as a validation split that never touches training.

Part 3 — LoRA config sketch

Your turn

Pick the adaptation method and a starting config. Full FT, LoRA, or QLoRA, and why? Choose a starting rank r and alpha, learning-rate posture, and epoch count, justifying each from the constraints (narrow task, small clean data, one GPU, forgetting risk). Decide whether you would merge the adapter for MaiQ. You are not expected to give exact optimal numbers, you are expected to reason.

Show model solution — Part 3

Method: LoRA. The data is small and the task narrow, so full fine-tuning is overkill and raises the forgetting risk; freezing the base and training a ~1% adapter is cheaper and structurally limits drift. Use QLoRA instead only if the chosen base will not fit in memory on the single GPU, accepting a small quantization fidelity cost. Rank r: start low, around 8 to 16, because a strict format/tone change is a low-complexity update; raise r only if the held-out eval shows underfitting. Alpha: set it to roughly r or 2r and tune the alpha/r ratio, not the raw numbers. Learning rate / epochs: low LR, 2 to 3 epochs, watching train-versus-validation loss for the overfit elbow, this is the Day 38 catastrophic-forgetting discipline, since over-tuning a small homogeneous set would erode general ability. Merge: if MaiQ serves this single fixed behavior from a dedicated model, merge the adapter for the leanest serving artifact; if instead MaiQ runs multi-tenant and other adapters share the base, keep it separate to preserve hot-swapping (and remember a QLoRA adapter cannot be cleanly merged into a 4-bit base without dequantizing).

Part 4 — Eval plan (hold-out + the Module 5 judge)

Your turn

Define how you decide the fine-tune worked, with an explicit ship bar. What do you measure on the held-out set? Which checks are programmatic versus judged? How does the Module 5 LLM-as-judge fit, and for what? How do you detect catastrophic forgetting? State the numeric gate you would ship on.

Show model solution — Part 4

Loss is necessary, not sufficient (Day 38), so evaluate the behavior on the held-out split with a layered eval. Programmatic checks (the bulk): JSON parses; schema conforms; category and priority are valid enums; summary is ≤ 14 words; no extra prose outside the JSON. These are deterministic ground truth, so they are graded by code, not a judge (Module 5's rule: do not use an LLM judge where a programmatic check exists). The Module 5 LLM-as-judge covers what code cannot score: tone/terseness and whether the category/priority/summary are semantically reasonable for the issue, scored against a numbered rubric with anchored levels, and ideally calibrated against a few human labels. Forgetting check: a regression probe of general questions (and a handful of out-of-distribution issues) to confirm the model did not get narrower or dumber. Ship bar (explicit): for example, ≥ 99% schema-and-enum conformance and 100% no-extra-prose on the held-out set, judge tone score above a set threshold, and no regression on the general probe; if it misses, the fix is more/better data, not more epochs.

Part 5 — Rollout

Your turn

You have a fine-tune that passes the bar. Now ship it safely. How do you roll out without risking production? Sketch the sequence and the guardrail/rollback. Keep it to 3 to 5 sentences.

Show model solution — Part 5

Roll out gradually, the same discipline as any model change. Start in shadow mode: run the fine-tuned model on real production traffic alongside the current solution without serving its output, and compare schema-conformance and judge scores on the live distribution (shadow catches real input variety and load that offline eval misses, Module 5). Then move to a small canary (single-digit percent of traffic) with a guardrail metric, the live schema-validation pass rate, and an automatic rollback if it dips below the bar. Ramp only as the canary holds, keep the previous version one switch away, and keep the schema validator in the serving path as a final net so a malformed ticket never reaches downstream code regardless of the model. Continue logging outputs to mine new failure cases into the next dataset iteration.

Put it together: the 90-second version

Interview-grade phrasing

"It is a FORM problem, so first I would try structured output to make the JSON valid by construction. If a fine-tune is still warranted for cost/latency or tone, I would collect a few hundred hand-checked (issue → ideal ticket) pairs in the model's chat template, mask loss to the response, and run a low-rank LoRA, small r, low LR, two to three epochs, watching the val elbow. I would gate on programmatic schema conformance plus an LLM-as-judge for tone, with a regression probe for forgetting, and ship only past an explicit bar. Rollout is shadow, then canary with a schema-validation guardrail and instant rollback, keeping the validator in the serving path." That is a complete, senior answer to "how would you fine-tune for X."

Module 6 review — rapid-fire

Close the module by drilling these. One-line keys; expand each into a 60 to 90 second answer with Claude using protocol prompt #2.

  • Prompting vs RAG vs fine-tuning — how do you choose?
    Key: cost ladder prompting < fine-tune < pretraining; start prompting + RAG + few-shot; fine-tune for form/style/latency/cost, never for new facts.
  • What does SFT actually do, and what does it teach poorly?
    Key: continues next-token training on (instruction → ideal response) pairs; teaches form/behavior well, recallable world facts poorly.
  • What is catastrophic forgetting and the four levers against it?
    Key: over-tuning erodes general ability; low LR, few epochs, mixed data, adapters (LoRA).
  • What is LoRA and why is it efficient?
    Key: freeze base, train low-rank adapter matrices A and B (~1% of params); cheap, swappable per task.
  • What is QLoRA and when do you use it over LoRA?
    Key: LoRA on a 4-bit quantized frozen base; use when memory-bound, accepting a small fidelity cost.
  • What is DPO and how does it differ from RLHF?
    Key: optimizes directly on preference pairs, no reward model or RL loop; simpler and more stable than RLHF.
  • When is fine-tuning the wrong answer?
    Key: when you need fresh/changing facts (use RAG), data is tiny, or prompting/structured output is not yet exhausted.
  • How do you prove a fine-tune is good enough to ship?
    Key: held-out behavioral eval (programmatic conformance + judge for tone) plus a forgetting regression probe, past an explicit bar; not loss alone.

Build it (the optional real run)

Hands-on (optional, needs a GPU or a managed endpoint)

If you want to make the plan real, the cheapest path is a managed fine-tune. Format ~20 of your hand-built pairs to the OpenAI fine-tuning data format (or run a tiny LoRA via Hugging Face TRL SFT trainer in Colab), launch one small job, and run your Part 4 programmatic eval on a held-out handful. The goal is not a production model; it is to feel that the dataset and the eval, not the training command, were the real work.

No GPU or budget? The lab is fully complete as a written plan, that is the interview-relevant artifact. See Tools & Claude for access.

Connect it to your work

Your MaindTec story

This lab is deliberately a MaiQ-shaped problem, so the plan you just wrote is one you could narrate as your own thinking. The evaluation half is genuinely yours: the layered held-out eval, programmatic checks plus an LLM-as-judge for what code cannot score, is the same framework you built and used to run 11-plus retrieval and chunking experiments, so you can speak to the eval and rollout with real authority. The honest line on the fine-tuning itself is "I have not run a production fine-tune at MaiQ, but here is exactly how I would scope one, and notice my first instinct is to try structured output before reaching for training," which is precisely the senior, cost-aware judgment that decided your 37% cost reduction. Pairing a strong eval story you have lived with a correctly-scoped fine-tuning plan is a very convincing answer to any model-adaptation question.

Today's takeaways

The fine-tuning skill is the plan, not the GPU: decide the lever (here, structured output first, then a small fine-tune for cost/tone), specify a few hundred clean pairs in the chat template with masked loss, sketch a low-rank LoRA with the forgetting discipline, gate on a layered behavioral eval with an explicit ship bar, and roll out via shadow then canary with a schema guardrail and instant rollback. You can now answer "how would you fine-tune for X" end to end, and you have closed Module 6 with the framework that makes the answer obvious.
