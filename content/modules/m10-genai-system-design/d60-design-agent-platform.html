Design an agent platform (MaiQ-style)

Yesterday's RAG assistant answered questions. Today's system lets users build and run their own agents inside a multi-tenant product. This is the design behind MaiQ's Skill Store and Apaleo's Agent Hub, and it is the one design in this module you have actually shipped. We take it through the full framework, with the security model front and center.

By the end you can
  • Design a multi-tenant agent / skill platform end to end using the seven-beat framework.
  • Build a tool registry and enforce per-tenant scoping in code, never trusting the model with isolation.
  • Specify conversational skill authoring plus a Skill Store / marketplace for sharing agents.
  • Lock down third-party skills: signed manifests, sandboxed per-invocation credentials, static analysis + eval + human review for high-risk scopes.
  • Operate it: circuit breakers per skill, prompt/version management, tracing, and cost budgets.

What an agent platform actually is

A single agent (Module 4) is an LLM that decides which tools to call in a loop. An agent platform is the infrastructure that lets many users, across many tenants, define new agents and new tools safely, run them, and share them. The hard part is not the ReAct loop. The hard part is that you are now executing user-authored behavior with access to tools that can read data and take actions, inside a building shared by other paying customers.

So the dominant concern flips from "is the answer good" (RAG) to "is this safe and isolated" (platform). When you frame the problem, lead with that, it signals you understand the real risk surface.

Interview-grade phrasing

"An agent platform is three layers: a tool registry of capabilities with declared scopes, a runtime that executes an agent's tool calls under tenant-scoped, sandboxed credentials enforced in code, and an authoring + distribution layer (conversational skill authoring plus a Skill Store) for users to build and share agents. Every security decision is made in code at the tool boundary; the model is never trusted to enforce isolation or permissions." That single sentence is the whole design.

1 · Clarify (functional + non-functional)

Pin scope before drawing. The questions that move this design:

  • Who authors agents? End users in chat (conversational authoring), developers via SDK, or both? MaiQ-style means non-technical users compose agents from existing tools in natural language.
  • What can tools do? Read-only data access, or write actions (send email, create ticket, charge)? Write actions raise the stakes and force approval gates and idempotency.
  • First-party vs third-party tools. Are all tools built by you, or can outside developers publish skills others install? Third-party code is the whole security problem, treat it as untrusted.
  • Tenancy + sharing. Multi-tenant, and can a skill authored in one tenant be published to a marketplace and installed by another? That crosses a trust boundary.
  • Latency, cost, reliability. Is a slow or failing skill allowed to degrade the whole product? (No, hence circuit breakers and budgets.)

Assume for this walkthrough: conversational authoring for end users plus an SDK for developers, tools that include both reads and writes, a marketplace where third-party skills can be published and installed across tenants, strict multi-tenant isolation, and a hard requirement that one bad skill cannot take down the platform or leak another tenant's data.

2 · Estimate (back of envelope)

Size the registry and the run volume so you pick the right datastore and know where the cost lives.

Tenants ~5,000 Users ~100,000 Tools in registry ~500 first-party + growing third-party → small table, fits anywhere Skills (agents) ~10 / active tenant ≈ tens of thousands of definitions Skill runs say 5 runs/user/day × 100k ≈ 500,000 runs/day each run = 3-8 LLM calls (agent loop) + several tool calls LLM calls ~2-4M / day → this is the cost center, budgets per tenant are mandatory Run records 500k/day × retained 90d ≈ 45M trace rows → time-series / OLAP-friendly store Concurrency peak maybe ~1-2k concurrent runs → runtime must isolate + cap per tenant

Headline decisions you verbalize: the tool/skill registry is tiny relational data (Postgres), but the run/trace volume is large and append-heavy (a tracing backend like Langfuse/LangSmith or a columnar store), and with millions of agent LLM calls per day, per-tenant cost budgets are a first-class feature, not an afterthought. You just justified two storage choices and a budget system from arithmetic.

3 · High-level architecture

Draw three planes: authoring (how agents get created), runtime (how they execute safely), and governance (review, tracing, budgets). The security boundary, where code enforces scope, runs right through the runtime.

Read it once: a user authors a skill in chat (or SDK), which compiles to a signed manifest declaring exactly which tools and scopes it needs; publishing to the marketplace runs static analysis, eval, and human review for risky scopes; at runtime the agent loop picks a tool, the registry validates the call, and tool execution enforces tenant scope and sandboxed credentials in code; governance wraps everything with tracing, per-skill circuit breakers, version pinning, and per-tenant budgets. 4 · Data model

Four entities carry the platform: the tool catalog, the skill (an agent) and its versions, the per-tenant installation/grant, and the run record. The grant table is where authorization lives, separate from the model.

-- the capability catalog: what a tool is and the scope it requires
CREATE TABLE tool (
  tool_id      uuid PRIMARY KEY,
  name         text,                    -- action-verb name
  json_schema  jsonb,                   -- typed params, validated before exec
  scopes       text[],                  -- e.g. {crm.read} or {billing.write}
  risk         text                     -- 'read' | 'write' | 'high'
);

-- a skill = an authored agent, versioned and signed
CREATE TABLE skill_version (
  skill_id     uuid,
  version      int,
  manifest     jsonb,                   -- tools used, scopes, prompt, author
  signature    text,                    -- signed: tamper-evident, verified at install
  status       text,                    -- draft | in_review | approved | deprecated
  PRIMARY KEY (skill_id, version)
);

-- the authorization record: which tenant granted which scopes to which skill
CREATE TABLE skill_grant (
  tenant_id    uuid,
  skill_id     uuid,
  version      int,                     -- pinned: installs a specific version
  granted_scopes text[],                -- the actual permission ceiling, in DATA not prompt
  PRIMARY KEY (tenant_id, skill_id)
);

-- append-heavy run log for tracing, cost, audit
CREATE TABLE skill_run (
  run_id       uuid PRIMARY KEY,
  tenant_id    uuid, skill_id uuid, version int,
  tool_calls   jsonb,                   -- name, args, result, latency per call
  tokens       int, cost_usd numeric, status text, started_at timestamptz
);

The load-bearing idea: skill_grant.granted_scopes is the permission ceiling and it lives in the database, checked in code on every tool call. The skill's prompt may request a tool, but if the grant does not include that scope for this tenant, the tool execution layer refuses, regardless of what the model produced. Authorization is data, enforced by code, never an instruction.

5 · Deep-dive: the tool registry and per-tenant scoping in code

This is the heart of the design and where you spend your time. The single rule, stated as bluntly as possible: never trust the model to enforce isolation, scope, or permissions. The LLM proposes a tool call; code disposes. The execution path on every single tool call:

  1. Validate the call against the schema. Registry checks the tool exists and the arguments match its typed JSON schema. A hallucinated tool name or schema-violating args is rejected before anything runs (bad args are more common than bad names).
  2. Check the scope against the grant, in code. Load granted_scopes for this (tenant_id, skill) and verify the tool's required scopes are a subset. No match, no execution. This is the line that prevents privilege escalation.
  3. Mint a sandboxed, per-invocation credential. The tool runs with a short-lived, narrowly-scoped credential issued for this call and this tenant, not a broad service key. The credential expires in seconds and can only touch this tenant's data. A leaked or misused credential has a tiny blast radius.
  4. Enforce idempotency and approval on writes. Write tools (create, charge, delete) require an idempotency key so an agent retry loop cannot fire the action twice, and high-risk writes route through an approval gate (the agent drafts the action, a human approves, then it executes via a framework interrupt).
# the tool boundary: code enforces what the model only requested
def execute_tool(call, ctx):              # ctx = {tenant_id, skill_id, run_id}
    tool = registry.get(call.name)
    if tool is None:
        raise ToolNotFound(call.name)         # hallucinated tool
    validate(call.args, tool.json_schema)    # schema gate

    grant = grants.get(ctx.tenant_id, ctx.skill_id)
    if not set(tool.scopes) <= set(grant.granted_scopes):
        raise ScopeDenied(tool.scopes)        # ★ authorization in CODE

    if tool.risk == "high" and not ctx.approved:
        return request_human_approval(call, ctx)  # approval gate

    cred = mint_scoped_credential(ctx.tenant_id, tool.scopes, ttl=30)  # sandboxed
    if breaker.is_open(call.name):
        raise SkillUnavailable(call.name)     # circuit breaker
    return tool.run(call.args, cred, idempotency_key=ctx.run_id)
The one mistake that fails the interview

Saying "the system prompt tells the agent only to use tools for its own tenant" is an instant red flag. Prompts are not a security boundary: the model is non-deterministic and prompt injection in a retrieved document or a tool result can redirect it. Every isolation and permission decision is enforced in code at the tool boundary, with the model treated as an untrusted caller. Say this explicitly and you flip a red flag into a green one.

6 · Conversational authoring, the Skill Store, and third-party safety

Conversational skill authoring. A user describes the agent they want ("watch this CRM list and draft a follow-up when a deal stalls"); the platform turns that into a structured skill manifest: the tools it will call, the scopes it needs, its prompt, and a version. The user composes from the existing tool catalog in natural language, so no code, but the output is a strict, validated artifact, not free text. That artifact is what gets reviewed, signed, and run.

The Skill Store / marketplace. Authored skills can be published so other tenants install them. Installing a third-party skill crosses a trust boundary (its author is not the installer), so the platform cannot run it on faith. The safeguards, in order of when they fire:

  • Signed manifests. The manifest is cryptographically signed so it is tamper-evident; the runtime verifies the signature at install and refuses anything altered after review. You install the exact bytes that were approved.
  • Static analysis at publish. Before a skill is listed, scan its manifest and any author-supplied code for dangerous patterns (unexpected scope requests, network egress, secret access). Cheap, automatic, catches the obvious.
  • Eval at publish. Run the skill against a behavioral test suite to check it does what it claims and does not misbehave, before it can reach other tenants.
  • Human review for high-risk scopes. A skill requesting read-only, low-risk scopes can auto-approve; a skill requesting billing.write or PII access goes to a human reviewer. Tier the gate by risk so you do not bottleneck safe skills, and you never auto-ship a dangerous one.

This exact pattern, signed manifests, sandboxed per-invocation credentials, static analysis plus eval plus human review for risky scopes, and circuit breakers per skill, is the marketplace security model. It is also literally what shipped in MaiQ's Skill Store.

8 · Operations: breakers, versioning, tracing, budgets

Circuit breakers per skill. Each skill (and each downstream tool) gets its own breaker: Closed (allow, count failures) → Open (fail fast for a cooldown when error rate spikes) → Half-open (let one probe through; success closes it). Per-skill isolation means a misbehaving third-party skill trips its own breaker without taking down the rest of the platform.

Prompt and version management. Every skill is pinned to a specific version at install (skill@version), so a tenant's running agents do not change underfoot when the author publishes an update; upgrades are an explicit, reviewable action. Prompts are versioned artifacts you can roll back, never edited live in production.

Tracing and audit. Emit a span tree per run: the agent's reasoning steps, every tool call with its arguments, result, tokens, cost, and latency, plus an immutable audit log of who ran what against which tenant's data. This is both your debugging surface (right answer on the wrong trajectory is still a bug) and your compliance surface.

Per-tenant cost budgets. With millions of agent LLM calls a day, an unbounded or runaway agent loop is a financial incident. Each tenant has a token/cost budget; the runtime meters spend per run and per tenant and cuts off or throttles when a budget is exhausted, so one looping skill cannot run up an unbounded bill.

Worked example: a third-party skill, authored to executed

Example

A developer publishes a "Stale Deal Nudger" skill: it reads a tenant's CRM and, when a deal has not moved in 14 days, drafts a follow-up email. Author: conversational/SDK authoring compiles it to a manifest declaring scopes {crm.read, email.draft}, signs it, sets status in_review. Publish gate: static analysis sees only read + draft (no email.send, no billing), eval confirms it drafts sensibly; because the scopes are low-risk, it auto-approves to the Store. Install: tenant acme installs skill@3; the runtime verifies the signature and writes a skill_grant with granted_scopes = {crm.read, email.draft}. Run: the agent loop tries to call email.send (the model improvised an escalation). Code checks the grant, sees email.send is not granted, and refuses, the model's overreach is denied at the tool boundary. The legitimate email.draft call runs under a 30-second credential scoped to acme's mailbox, the breaker is closed, the run logs 4 LLM calls and 6 tool calls with full cost, and the draft lands for a human to send. No prompt was trusted to keep acme's data or a dangerous action in check; code did.

Practice — do these before revealing

Answer each out loud as if whiteboarding, then compare.

Problem 1

An engineer proposes: "We give every agent the same broad service credential and put 'only access tenant X' in the system prompt. It is simpler." Tear this down and give the correct design in two sentences.

Show solution

It is unsafe on two axes. First, a prompt is not a security boundary: the model is non-deterministic and prompt injection (in a retrieved doc or a tool result) can redirect it to use that broad credential against another tenant, so isolation must be enforced in code at the tool boundary, not requested in text. Second, a broad service credential means any leak or misuse touches every tenant, a catastrophic blast radius. The correct design: authorization lives in a per-tenant grant checked in code on every tool call, and each call runs under a short-lived credential scoped to exactly that tenant and that tool, so a compromise affects one call for one tenant. The model proposes; code authorizes and scopes.

Problem 2

You open a marketplace so third-party developers can publish skills others install. Walk through everything that fires between "developer hits publish" and "another tenant's agent executes a tool from that skill."

Show solution

At publish: the skill compiles to a manifest (tools, scopes, prompt, version); static analysis scans for dangerous patterns (unexpected scopes, egress, secret access); an eval suite checks behavior; if the requested scopes are high-risk (writes, billing, PII) it routes to human review, otherwise it can auto-approve; on approval the manifest is cryptographically signed and listed. At install: the installing tenant's runtime verifies the signature (refusing anything tampered post-review), pins a specific version, and writes a skill_grant recording exactly which scopes that tenant authorizes. At execution: the agent proposes a tool call; code validates args against the schema, checks the tool's scopes are a subset of the grant, mints a per-invocation tenant-scoped credential, applies idempotency/approval on writes, and checks the per-skill circuit breaker before running. Every step is code-enforced; the model is treated as untrusted throughout.

Problem 3

One popular third-party skill starts erroring intermittently and its retries are spiking latency and cost for everyone. What in your design contains the damage, and what is the longer-term fix?

Show solution

Containment is the per-skill circuit breaker: that skill's breaker trips Open on the error-rate spike and fails its calls fast for a cooldown, so the misbehaving skill stops consuming runtime and dragging down latency for unrelated skills and tenants. The per-tenant cost budget also caps the financial bleed from retry loops. Half-open then lets a single probe through to detect recovery before fully reopening traffic. Longer term: pin and freeze the bad version, alert the author, surface the failure in tracing (the span tree shows which tool call fails and why), and require the fixed version to pass eval again before it can be re-published, since a new version is a reviewable, re-evaluated artifact, not a silent live edit.

Build it (20–25 min)

Hands-on

Build the tool boundary in miniature. Define two tools as Python functions with typed argument models and a scopes list each (one read, one high-risk write). Build a tiny registry and an execute_tool wrapper that, on every call, validates args against the schema, checks the tool's scopes are a subset of a hardcoded per-tenant grant, and rejects otherwise. Then drive it with a Claude tool-use loop (or LangGraph with an interrupt) where the model picks tools, and deliberately prompt it to call the write tool without the grant, confirm your code, not the model, blocks it. Add a per-skill failure counter that opens a breaker after three errors. You have just built the security core of an agent platform.

Stretch: add an approval interrupt on the write tool so high-risk actions pause for a y/n before executing, the human-in-the-loop gate.
