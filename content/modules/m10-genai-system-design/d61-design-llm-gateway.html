Design an LLM gateway at scale

The RAG assistant and the agent platform both make LLM calls. Once you have more than a handful of services calling providers directly, you want one front door. An LLM gateway is the reverse proxy for all model traffic: auth, routing, rate limiting, caching, fallback, cost metering, and tracing in a single chokepoint. Today we design it end to end.

By the end you can
  • Justify a gateway: why one entry point for all LLM traffic beats per-service SDK calls.
  • Design the request path: auth → rate limit → cache → route → provider call with fallback.
  • Implement per-tenant rate limiting with a token bucket, and prompt + semantic caching.
  • Build a retry/fallback chain with a circuit breaker per provider, and stream responses through.
  • Add cost metering, per-tenant budgets, and full logging/tracing, and argue build vs buy.

Why a gateway at all

When two services call the model API directly it is fine. When twenty do, you have twenty places that each re-implement retries, twenty sets of API keys, no shared view of spend, no shared cache, and no single switch to flip when a provider has an outage or you want to swap models. A gateway pulls all of that into one component every request passes through, so cross-cutting concerns are solved once.

It is the same instinct as an API gateway in classic system design, specialized for LLM traffic: the things you centralize, caching, routing, rate limiting, fallback, cost, are exactly the things that are painful and inconsistent when scattered across services.

Interview-grade phrasing

"An LLM gateway is a single entry point for all model traffic that centralizes the cross-cutting concerns: authentication, model routing and selection, per-tenant rate limiting, prompt and semantic caching, a retry and fallback chain with a circuit breaker per provider, cost metering and budgets, and logging/tracing. It turns provider calls into one governed, observable, swappable chokepoint, and it streams." Lead with that and you have named every component before drawing one.

1 · Clarify (functional + non-functional)

Pin scope first. The questions that shape this design:

  • Who calls it? Internal services only, or external customers too? Multi-tenant changes rate limiting and cost attribution.
  • How many providers/models? One provider, or several (Anthropic, OpenAI, a self-hosted model)? Multiple providers is what makes routing and fallback worth building.
  • Latency overhead budget. The gateway sits in the hot path, so its own overhead must be tiny (single-digit milliseconds excluding the model call). Streaming must pass through with no buffering penalty.
  • Caching appetite + risk. Is prompt caching enough, or do you want semantic caching too (and is that safe for this traffic)?
  • Governance. Per-tenant budgets, audit logging, PII handling, who is allowed which models?

Assume: internal services plus some external API customers (multi-tenant), three providers/models behind it, sub-5ms gateway overhead, streaming required, prompt caching always-on and semantic caching opt-in per route, and hard per-tenant rate limits and cost budgets.

2 · Estimate (capacity numbers)

Size the throughput and the cost flowing through the chokepoint so you know what the gateway must survive.

Traffic peak ~500 LLM req/s across all tenants Tokens ~3k in + ~500 out per request (mixed RAG + chat + agent) Cost (raw) @ ~$5/1M in, ~$15/1M out → ~$0.015 + $0.0075 ≈ ~$0.0225/req 500 req/s ≈ 43M req/day → ~$970k/day uncached → caching is the headline Cache target ~90% hit on hot/repeat reads → ~10x cost + latency reduction on those Rate limits per tenant: e.g. 100 req/s burst, 60 req/s sustained (token bucket) Gateway state rate-limit counters + cache keys → Redis (atomic, fast, shared) Overhead gateway adds < 5 ms (excluding the upstream model call) → must be lean Logs/traces 43M spans/day → async export to a tracing/OLAP backend, never inline blocking

Two decisions fall out: the gateway's own work must be cheap and mostly I/O-bound (its latency is dominated by the upstream model, so it must add almost nothing), and shared mutable state, rate-limit counters and cache, belongs in Redis because it must be atomic and visible to every gateway instance. And again, caching is the cost story.

3 · High-level architecture

Draw the gateway as an ordered pipeline in the request path. Order matters: reject early (auth, rate limit), serve cheap (cache), then do the expensive thing (route to a provider) with safety nets (fallback, breaker), and always emit a trace.

clients (RAG svc, agent runtime, external API customers) │ request: {tenant, model_hint, messages, stream?} ▼ ┌────────────────────────────── LLM GATEWAY ──────────────────────────────────┐ │ │ │ ① AUTH verify API key / JWT → resolve tenant + allowed models │ │ │ │ │ ② RATE LIMIT token bucket per tenant (Redis, atomic) → 429 + Retry-After │ │ │ if empty │ │ ③ CACHE prompt-cache key (exact) → semantic-cache (cosine ≥ τ, opt-in) │ │ │ HIT ──────────────────────────────────────────────▶ return cached │ │ │ MISS │ │ ④ ROUTE choose model: hint / policy / cost-quality / health │ │ │ │ │ ⑤ CALL + FALLBACK CHAIN │ │ │ ┌─ provider A ─[breaker]─▶ 200 ✓ ────────────────────┐ │ │ │ ├─ 429/5xx? backoff+jitter, retry (idempotent only) │ │ │ │ └─ breaker OPEN / exhausted ▶ provider B ▶ provider C │ │ │ │ ▼ │ │ ⑥ STREAM pass tokens straight through (SSE), no buffering write cache │ │ │ │ │ ⑦ METER count tokens → cost → per-tenant budget; emit span (async) │ └────────────────────────────────────┬───────────────────────────────────────────┘ ▼ Redis (buckets + cache) · tracing/cost backend (async)

Walk it top to bottom: authenticate and resolve the tenant, check the token bucket (fail fast with a 429 and Retry-After if empty), check the prompt cache then the optional semantic cache (return immediately on a hit), route to a model, call it through the fallback chain with a per-provider breaker, stream tokens straight back while writing the cache, and meter the cost and emit a trace asynchronously. Each numbered stage is a small, independently testable middleware.

4 · Data / state model

The gateway is mostly stateless compute over a little shared hot state in Redis plus an async log sink. Three pieces of state:

# 1) per-tenant rate-limit bucket (Redis hash), refilled lazily on read
ratelimit:{tenant_id} = { tokens: 73.2, last_refill_ts: 1718900000.0 }
#    capacity = burst (e.g. 100), refill rate r = sustained (e.g. 60/s)

# 2) cache entries
promptcache:{sha256(model + normalized_messages)} = { response, ts }   # exact match
semcache:{route}  →  vector index of (query_embedding → cached_response) # cosine ≥ τ

# 3) per-tenant spend counter (rolling window), checked before/after a call
budget:{tenant_id}:{yyyymm} = { spent_usd: 412.55, limit_usd: 1000 }

And every completed request appends a span to the tracing/cost backend (tenant, model, tokens in/out, cost, latency, cache-hit flag, provider used, status), written asynchronously so logging never blocks the response. Note what you do not store: raw PII. You log a hashed user/session and the prompt version, not the user's raw text where avoidable.

5 · Deep-dive A: per-tenant rate limiting with a token bucket

Rate limiting protects providers (you have your own quotas to respect), protects tenants from each other (noisy-neighbor isolation), and protects you from runaway loops. The token bucket is the standard: a bucket holds up to capacity tokens (your burst allowance), refills at rate r per second (your sustained rate), and each request removes one token. Empty bucket means reject with 429 and a Retry-After telling the client when a token will be available.

The critical detail in a distributed gateway: the check-and-decrement must be atomic, or two gateway instances both see one token left and both let a request through. Do it in a Redis Lua script (or equivalent atomic primitive) so refill, check, and decrement happen as one operation.

-- token bucket as one atomic Redis Lua op (refill + check + take)
local b   = redis.call("HMGET", KEYS[1], "tokens", "ts")
local tok = tonumber(b[1]) or CAP
local ts  = tonumber(b[2]) or NOW
tok = math.min(CAP, tok + (NOW - ts) * RATE)   -- lazy refill
if tok < 1 then return 0 end                  -- empty → caller returns 429
redis.call("HMSET", KEYS[1], "tokens", tok - 1, "ts", NOW)
return 1                                       -- token taken → proceed

For LLM traffic you often rate-limit on tokens per minute, not just requests per second, because a 50-token request and a 50k-token request are not the same load. The same bucket math works, you just deduct estimated tokens instead of 1, and you can run two buckets (requests and tokens) in parallel.

6 · Deep-dive B: caching, the cost lever

From the estimate, caching is the difference between viable and bankrupt. Two complementary layers:

Prompt caching (exact, always safe). Two flavors. (1) The gateway's own response cache: identical (model + normalized messages) returns the stored completion, perfect for deterministic, repeated calls. (2) Provider-side prompt caching (Anthropic): you mark a stable cacheable prefix (system prompt, long shared context, tool definitions) so the provider bills that cached input at roughly a tenth and skips reprocessing it; the trick is to put static content first and the variable part last, with a short (~5 min) TTL. This is the single biggest cost lever and it is essentially risk-free because the cache key is exact.

Semantic caching (fuzzy, powerful, dangerous). Embed the incoming query, and if a previously-answered query sits within a cosine threshold, return its cached answer. This catches paraphrases (huge on FAQ-style traffic) but it is risky: it is wrong for personalized, time-sensitive, or high-stakes responses, where a near-duplicate is not actually the same question. So semantic caching is opt-in per route, with a conservative threshold and an exclusion list for sensitive intents.

Where semantic caching bites

"What is my account balance?" from two different users embeds nearly identically but must never share a cached answer. The fix is to scope the cache key by tenant/user where the response is personalized, and to disable semantic caching entirely on time-sensitive or high-stakes routes. Treat a too-eager semantic cache as a correctness bug, not a cost win.

7 · Deep-dive C: routing, retries, and the per-provider circuit breaker

Routing / model selection. The gateway decides which model serves a request: an explicit hint from the caller, a policy (this tenant may only use models X and Y), a cost-quality rule (cheap model for short/simple calls, strong model for hard ones), or provider health. Centralizing routing means you can swap or A/B models for every service by changing one rule, not twenty deployments.

Retries. Providers return 429s and 5xxs. Respect the provider's Retry-After header, use exponential backoff with jitter (jitter avoids a synchronized retry stampede), cap the attempts, and only retry idempotent calls, retrying a non-idempotent action can double-fire it.

Fallback chain + circuit breaker per provider. If provider A is erroring or its breaker is Open, fall through to provider B, then C. Each provider has its own circuit breaker: Closed (allow, count failures) → Open (fail fast for a cooldown after the error rate spikes, do not keep hammering a down provider) → Half-open (send one probe; success closes it). Per-provider isolation means an Anthropic outage trips only Anthropic's breaker and traffic flows to the fallback without manual intervention.

# fallback chain: try providers in order, skip any whose breaker is open
def call_with_fallback(req, chain):          # chain = [providerA, providerB, providerC]
    for p in chain:
        if breaker[p].is_open():
            continue                          # down → skip to next provider
        try:
            return retry_with_backoff(p.call, req, max_attempts=3)
        except ProviderError:
            breaker[p].record_failure()        # may trip Open
    raise AllProvidersDown()                   # last resort: clear error to caller

8 · Trade-offs, ops, and build vs buy

Cost metering + budgets (ops). Every call's tokens are counted, converted to cost, and added to a per-tenant rolling spend counter; when a tenant hits its budget the gateway throttles or rejects with a clear error. This is where the gateway pays for itself: one authoritative view of spend, enforced before the money is gone, instead of a surprise invoice.

Logging/tracing (ops). Emit a span per request (tenant, model, tokens, cost, latency, cache-hit, provider, status), exported asynchronously to a tracing backend so you get dashboards for cost-per-request, cache-hit rate, p95/p99 latency, and provider error rates, plus per-tenant attribution. Never log raw PII.

DecisionYou gainYou give up
Central gateway vs direct SDK callsOne place for cache, routing, limits, cost, fallbackA new hop, a potential SPOF, ~ms overhead (mitigate: HA, lean code)
Semantic cache onBig hit-rate on paraphrased trafficCorrectness risk on personalized/time-sensitive responses
Multi-provider fallbackSurvives a provider outage automaticallyBehavioral differences between models; more to test
Cost-quality routingCheaper at scaleMis-routing risk; needs a complexity/confidence signal
Token-based rate limitingFair load accounting for variable request sizesMust estimate tokens up front; more bookkeeping than req/s

Build vs buy. Mature open-source and managed gateways exist (the LiteLLM-style proxy is the common reference) and cover most of this. The senior answer is not "always build": buy or adopt open source for the commodity 80 percent (multi-provider routing, basic caching, cost logging) and build only the parts that are genuinely yours, custom per-tenant budgets, bespoke routing policy, or deep integration with your own tracing. Reach for a custom gateway when your routing/governance needs outgrow the off-the-shelf knobs, and say exactly that, naming what you would buy versus build is the mark of someone who has weighed it.

Worked example: one request through the gateway

Example

The agent runtime sends a chat completion for tenant acme, streaming, model hint "fast". ① Auth: API key resolves to acme, allowed models {fast, strong}. ② Rate limit: the Lua token bucket finds 41 tokens, takes one, proceeds (a burst tenant would get a 429 + Retry-After here). ③ Cache: exact prompt-cache miss; semantic caching is off on this route (it is a personalized agent reply), so no fuzzy lookup. ④ Route: hint "fast" plus policy selects provider A's small model. ⑤ Call: provider A returns a 529 overload; backoff+jitter retries once, fails; A's breaker trips toward Open; the chain falls through to provider B, which returns 200. ⑥ Stream: tokens pass straight through over SSE with no buffering; on completion the response is written to the prompt cache. ⑦ Meter: 3.1k in + 480 out → ~$0.022 added to acme's monthly spend (well under the $1,000 budget); an async span records tenant, model, tokens, cost, 1.8s latency, cache miss, provider B, status 200. One request, every cross-cutting concern handled in one place, and a provider outage absorbed without anyone paging an engineer.

The 11-point checklist, gateway flavor

1. Auth (key/JWT → tenant + allowed models) · 2. Rate limiting (per-tenant token bucket, atomic) · 3. Prompt caching (exact + provider prefix cache) · 4. Semantic caching (opt-in, threshold, sensitive-route exclusion) · 5. Routing / model selection (hint / policy / cost-quality / health) · 6. Retries (Retry-After, backoff+jitter, idempotent only) · 7. Fallback chain + per-provider circuit breaker · 8. Streaming pass-through · 9. Cost metering + per-tenant budgets · 10. Logging/tracing (span per request, async, no PII) · 11. HA + build-vs-buy (no SPOF, adopt OSS for the commodity 80%).

Practice — do these before revealing

Answer each out loud as if whiteboarding, then compare.

Problem 1

You run three gateway instances behind a load balancer. A tenant's limit is 60 req/s but they are getting through at ~180 req/s. What is the bug, and how do you fix it without making the gateway slow?

Show solution

Each instance is keeping its own local token bucket, so the tenant effectively gets 3 × 60 = 180 req/s, one limit per instance. Rate-limit state must be shared and atomic: move the bucket into Redis and do the refill-check-decrement as one atomic operation (a Lua script), so all three instances draw from the same bucket and the global limit holds. It stays fast because it is a single in-memory Redis round trip on the hot path. If even that round trip is too costly at extreme scale, a known refinement is local token leasing (each instance leases a small slice of the budget from the central bucket and refreshes periodically), trading a little precision for fewer central calls, but the default correct answer is the shared atomic bucket.

Problem 2

Turning on semantic caching cut your bill nicely, but support reports a few users getting answers meant for someone else's question. Diagnose and fix without disabling caching everywhere.

Show solution

Semantic caching is matching near-duplicate queries across users and returning a cached answer that was correct for a different context, classic on personalized or stateful questions ("my balance", "my last order") that embed almost identically. Fixes, layered: (1) scope the semantic cache key by tenant/user wherever the response depends on who is asking, so cross-user matches cannot happen; (2) exclude personalized, time-sensitive, and high-stakes routes from semantic caching entirely (keep exact prompt caching there, which is always safe); (3) raise the cosine threshold so only very close matches hit. The principle: semantic caching is a per-route opt-in, never a global default, because a fuzzy match is a correctness risk, not just a cost knob.

Problem 3

Your primary provider has a 30-minute partial outage (intermittent 5xx). Describe exactly what your gateway does, second by second, so that callers barely notice, and one thing that would make it worse.

Show solution

On the first failures, retries with exponential backoff and jitter absorb the transient errors; as the error rate climbs, the provider's circuit breaker trips Open, so the gateway stops sending traffic to the failing provider (no more hammering it, no more waiting on doomed calls) and the fallback chain routes requests to provider B automatically. Callers see slightly higher latency and a possible model swap, but not failures. Periodically the breaker goes Half-open and sends a single probe; when the provider recovers, that probe succeeds and the breaker closes, restoring primary traffic, all without a human in the loop. What makes it worse: aggressive retries with no breaker and no jitter, every instance synchronously hammering the down provider in lockstep, which amplifies the outage (a retry storm / thundering herd) and can take down the fallback too.

Interview questions for this topic

From your Notion bank's GenAI System Design and LLMOps slices. Drill the keys, then whiteboard each.

  • Design an LLM gateway.
    Key: one entry point, auth, routing/model selection, per-tenant rate limiting, prompt + semantic caching, retries/fallback chain + per-provider breaker, cost metering, logging/tracing, streaming.
  • How do you design a rate limiter for an LLM API client/gateway?
    Key: token bucket (refill rate r, capacity = burst), per tenant + global, atomic check-decrement in Redis Lua; rate-limit on tokens/min as well as req/s.
  • How do you handle 429s and provider outages?
    Key: respect Retry-After, exponential backoff + jitter, cap attempts, retry idempotent only; circuit breaker per provider + fallback chain to the next provider.
  • What is a circuit breaker and its states?
    Key: Closed (allow, count failures) → Open (fail fast, cooldown) → Half-open (one probe, success closes); per-provider isolation so one outage does not cascade.
  • What caching layers does a gateway use and where is each dangerous?
    Key: exact prompt cache (safe, biggest lever, provider prefix cache billed ~10%) + semantic cache (opt-in, risky for personalized/time-sensitive, scope by user, exclude sensitive routes).
  • How do you control and attribute LLM cost across many tenants?
    Key: meter tokens → cost per request, per-tenant rolling spend counter + budget enforced at the gateway, dashboards for cost/request and cache-hit rate.
  • What capacity numbers do you bring to a gateway design?
    Key: ~$5/1M in and ~$15/1M out for a GPT-4-class model, aim ~90% cache hit on hot reads, keep gateway overhead to single-digit ms, state in Redis.
  • Build vs buy an LLM gateway?
    Key: adopt OSS/managed (LiteLLM-style proxy) for the commodity 80% (routing, basic caching, cost logging); build only bespoke budgets/routing/governance and deep tracing integration.

Build it (20–25 min)

Hands-on

Build a minimal gateway in FastAPI. Add one endpoint that (1) checks a per-tenant token bucket in Redis via a Lua script and returns 429 + Retry-After when empty, (2) checks an exact prompt cache (hash of model + messages) and returns the cached completion on a hit, (3) on a miss calls the Claude streaming API and passes tokens straight through, and (4) writes the result to the cache and logs tokens, cost, and latency. Then wrap the provider call in a tiny circuit breaker (open after 3 failures, half-open after a cooldown) and a fallback to a second model. Hit it in a loop and watch the bucket throttle, the cache serve repeats for free, and the breaker fail fast when you point it at a dead endpoint. That is the gateway in 120 lines.

Stretch: compare your hand-rolled gateway against the open-source LiteLLM proxy and list which features you would buy versus build.

Today's takeaways

An LLM gateway is one front door for all model traffic that centralizes auth, per-tenant rate limiting, caching, routing, fallback, cost metering, and tracing. Order the request path to reject early and serve cheap: auth, token-bucket rate limit (atomic in Redis), prompt then optional semantic cache, route, call with backoff and a per-provider circuit breaker and fallback chain, stream through, then meter and trace asynchronously. Prompt caching is the safe headline cost lever; semantic caching is powerful but a per-route correctness risk. And the senior move on build-vs-buy is to adopt the commodity 80 percent and build only your bespoke governance.
