Design a semantic search / ranking system

Search and ranking is the most common ML-shaped design at product companies, eBay, Wolt, trivago, marketplaces and travel all run a hybrid retrieve-then-rank stack. It is also the engine the RAG assistant sits on. Today we design a semantic plus lexical search service end to end, with a learned ranker, a latency budget, and both offline and online evaluation.

By the end you can
  • Design a search/ranking service: hybrid retrieval → cross-encoder rerank → filters/tenant isolation.
  • Explain the candidate-generation / ranking / re-rank funnel and why each stage exists.
  • Use an embedding feature store and query understanding to feed the ranker.
  • Evaluate offline (NDCG, recall) and online (CTR, conversion) and connect the two.
  • Hold a latency budget with caching and explain freshness/indexing for changing inventory.

The funnel: why search is staged

You cannot run an expensive scoring model over a million items per query, and you cannot get good ordering from a cheap one. So search is a funnel: a cheap, high-recall stage narrows millions to hundreds, then progressively richer (and slower) stages refine the order of that short list. Each stage trades a bit of recall for a lot of precision, in the direction where it is affordable. This is the same shape as the recommendation/ranking framework from Module 9: candidate generation, then ranking, then re-rank.

Interview-grade phrasing

"A search service is a three-stage funnel: candidate generation (cheap, high-recall hybrid retrieval, BM25 plus dense, narrows millions to a few hundred), ranking (a learned model scores that short list on rich features), and re-ranking (a cross-encoder for precision, plus business rules, diversity, and freshness). Tenant and permission filters live in the retrieval query, and I tune it against offline NDCG/recall and online CTR." That sentence is the whole architecture.

1 · Clarify (functional + non-functional)

Pin scope first. The questions that move a search design:

  • What are we searching? Documents, products, restaurants, hotels? Corpus size, and how often items change (a marketplace's inventory churns constantly).
  • Query type. Short keyword queries, natural-language questions, or both? Are there strong exact-match needs (SKUs, brand names) that lexical must own?
  • Personalization + filters. Per-user ranking? Hard filters (price, location, in-stock, tenant)? Filters are constraints, not ranking signals.
  • Latency + scale. Search is interactive: a typical budget is well under ~200ms p95 end to end. Peak QPS?
  • What is "good"? The business metric, clicks, conversions, bookings, and the offline proxy (NDCG/recall) you will optimize between launches.

Assume: a multi-tenant product catalog of ~10M items that change often, mixed keyword and natural-language queries with real exact-match needs, hard filters plus light personalization, p95 under 150ms, and the north-star metric is conversion with CTR as the faster online signal.

2 · Estimate (capacity)

Catalog ~10M items × 1 embedding each (384-768 dim is common for search) Vector store 10M × ~1.5 KB (int8, 384-dim) ≈ 15 GB → fits a dedicated ANN index Traffic peak ~1,000 search QPS Latency budget ~150 ms p95 end-to-end, spent roughly: retrieval (ANN + BM25) ~20-40 ms ranking (GBDT/features) ~10-30 ms rerank (cross-encoder, top ~50-100) ~30-60 ms ← the costly stage filters/assembly/network remainder Funnel widths ~10M ──candidate gen──▶ ~500 ──rank──▶ ~50 ──rerank/business──▶ top 10 Freshness inventory changes/min → near-real-time index updates, not nightly rebuilds

Two decisions fall out: at 10M vectors with a tight latency budget you want a dedicated ANN index (HNSW/IVF) rather than vectors-beside-relational, because brute-force cosine is O(N·d) and hits a scaling cliff; and the cross-encoder reranker is your latency bottleneck, so it only ever runs over the top ~50 to 100, never the full candidate set. The funnel widths are the design.

3 · High-level architecture

Draw two lanes again: an indexing lane that keeps the search index fresh as items change, and a query lane that runs the funnel per request. Query understanding sits at the front of the query lane; the feature store feeds the ranker.

Walk it: a query is normalized and embedded (with intent and personalization signals), candidate generation runs hybrid retrieval with hard filters baked into the query to narrow ~10M to ~500, the learned ranker scores those ~500 on rich features pulled from the feature store down to ~50, the cross-encoder reranker plus business rules order the final top 10, and the whole thing is tuned offline on NDCG/recall and validated online on CTR/conversion. Hot queries short-circuit at the cache.

4 · Data model and the feature store

Search needs three data shapes: the inverted/ANN indexes for retrieval, the filterable item attributes, and the features the ranker consumes. The feature store is what keeps offline (training) and online (serving) features identical.

# item document in the search index (dense + lexical + filter fields)
{
  "item_id":   "sku-918273",
  "tenant_id": "acme",            # isolation filter
  "text":      "... title + description ...",  # → BM25 (lexical)
  "vector":    [0.02, ...],         # → ANN (dense, semantic)
  "price": 39.0, "in_stock": true, "geo": [..],  # → HARD filters
  "updated_at": "..."             # → freshness
}

# FEATURE STORE: features the ranker scores, served consistently train + serve
item_features[item_id]   = { ctr_7d, conversion_rate, popularity, recency, price_z, ... }
query_features[query]    = { length, intent, embedding, has_brand_token, ... }
user_features[user_id]   = { category_affinity, locale, recent_clicks, ... }
# ranker input per (query,item,user) = concat(query_f, item_f, user_f, retrieval_scores)

The point of the feature store: the exact same feature values and transforms are used to train the ranker offline and to serve it online, which is how you prevent training-serving skew (the silent killer where a feature is computed one way in the notebook and another way in production, so the live model underperforms its offline numbers). It also gives point-in-time correctness so you do not train on a feature value that did not exist yet at query time.

5 · Deep-dive: the three funnel stages

Stage 1, candidate generation (recall). Hybrid retrieval is the default: BM25 owns exact matches (brand names, model numbers, the literal token the user typed) and dense ANN owns semantic matches (synonyms, paraphrase, "running shoes" matching "trainers"). Fuse with RRF so a lexical rank and a vector rank combine without a score-scale fight. Hard filters, tenant, price range, in-stock, geo, are applied in the retrieval query, not as a post-filter, so excluded items never consume a candidate slot and isolation is guaranteed at the data layer. Goal: do not miss anything relevant; precision comes later.

Stage 2, ranking (the learned model). The ~500 candidates are scored by a learned ranker, typically a gradient-boosted tree model (e.g. XGBoost-style, which is strong and cheap for tabular ranking features) trained with a learning-to-rank objective. Its inputs are the rich features from the feature store: item signals (historical CTR, conversion, popularity, recency), query signals (intent, length), user signals (affinity, locale), and the retrieval scores themselves. This stage turns "topically relevant" into "actually good for this query and this user," and it is where most of the quality lives.

Stage 3, re-ranking (precision + policy). A cross-encoder reads the (query, item) pair jointly for the very top ~50 to 100 and gives the most accurate relevance signal, the same precision booster as in RAG. Then business logic shapes the final list: diversity (do not return ten near-identical items), freshness boosts, sponsored/promoted placement, and any hard policy. This is where product requirements that are not pure relevance get encoded, kept separate from the learned score so they are auditable.

Filters are constraints, not features

A hard filter (in-stock, tenant, price ≤ X, within delivery radius) must be enforced as a query predicate, never blended into a relevance score. If you let the ranker "mostly" respect a constraint, it will eventually surface an out-of-stock or out-of-area result that ranked highly on other signals, which is a correctness bug, not a ranking quirk. Filter first, rank within the filtered set.

6 · Query understanding

The query is a first-class input, not just a string. A small front-end stage improves everything downstream: normalize and spell-correct ("nike runing shoez"), detect intent (navigational vs broad), extract structured tokens (brand, category, price phrases that become filters), and produce the query embedding for dense retrieval. For ambiguous or multi-part queries you can rewrite or decompose before retrieval. The payoff is concrete: catching a brand token routes it to a lexical exact-match boost, and fixing a typo prevents a zero-result page. Keep it cheap, this runs on every keystroke-driven search, so it sits inside the latency budget.

7 · Evaluation: offline and online, connected

You optimize between launches on offline metrics and you decide ship/no-ship on online metrics, and you must keep the two correlated.

Offline (fast, cheap, run on every change). Build a judged set of (query, ranked results with graded relevance labels). Score with NDCG (graded, position-discounted, rewards putting the best results highest), recall@k (did candidate generation even retrieve the relevant items, the recall stage's own KPI), and MRR for known-item queries. Mine real query logs to build and refresh the judged set so it reflects actual traffic, not a stale benchmark.

Online (slow, true, decides launches). The metric that matters is behavior: CTR, dwell, and ultimately conversion/bookings. Run a per-user A/B test of the new ranker against the old, with guardrail metrics (latency, zero-result rate, revenue) so a CTR win that tanks latency or conversion is caught, and run to significance. For risky changes, shadow first to see real input distribution and latency before exposing users.

Connect them: pick the offline metric (NDCG) that best correlates with your online metric (conversion) on historical experiments, so an offline win is a trustworthy predictor of an online win. If they diverge, your judged labels or features are off, fix that before trusting offline numbers.

8 · Trade-offs, latency, freshness (ops)

DecisionYou gainYou give up
Hybrid vs pure-vectorExact-match recall (SKUs, brands) that dense alone missesTwo indexes + fusion to maintain
Hybrid vs pure-lexical (BM25 only)Semantic/synonym recall, NL queries workEmbedding infra, ANN index, drift to manage
Cross-encoder rerankBest precision at the topThe latency bottleneck, so cap it to top ~50-100
Learned ranker (GBDT)Big quality lift from rich featuresTraining pipeline, feature store, skew risk, retraining
Near-real-time indexingFresh inventory, no stale resultsMore complex pipeline than nightly batch rebuilds

Latency budget (ops). Spend it deliberately, the funnel exists to keep the expensive stage small. Cache hot queries and their results (a large share of search traffic is repeated head queries), cache item features in the online store, run BM25 and ANN in parallel, and keep the cross-encoder strictly on the short list. Aim for a high cache-hit rate on head queries and a tight p95.

Freshness / indexing (ops). A marketplace's inventory changes constantly, so the index cannot be a nightly rebuild. Drive updates from a change stream (CDC / events): when an item is created, updated, sold out, or removed, enrich, embed, and upsert it into the ANN and lexical indexes within seconds, and update its features. Stale results (showing an out-of-stock item, missing a just-listed one) are a real quality and trust problem, so freshness gets first-class treatment, not a cron job.

Worked example: one search through the funnel

Example

A user on tenant acme in Berlin searches "waterproof hiking boots under 100", in-stock only. Query understanding: normalizes the text, extracts the price phrase into a hard filter price ≤ 100, infers a broad product intent, and embeds the query. ① Candidate generation: BM25 matches "waterproof", "hiking", "boots" lexically while dense ANN pulls semantically close items ("trekking boots", "GORE-TEX footwear") that never use the exact words; the retrieval query carries tenant_id='acme' AND in_stock=true AND price≤100 AND within(geo, Berlin_radius), so out-of-stock, over-budget, or out-of-area items are never candidates; RRF fuses to ~500. ② Rank: the GBDT ranker scores those 500 using item CTR/conversion, recency, the user's category affinity, and the retrieval scores, narrowing to ~50. ③ Re-rank: a cross-encoder reads each (query, item) pair for true relevance, then diversity logic ensures the top 10 are not all the same brand and a couple of fresh new arrivals get a boost. The user sees ten in-stock, in-budget, in-area boots ordered by likelihood to convert, in under 150ms, and the click that follows feeds tomorrow's CTR feature and the next A/B.

The 11-point checklist, search flavor

1. Indexing (enrich, embed, upsert; near-real-time freshness) · 2. Query understanding (normalize, spellfix, intent, embed) · 3. Candidate generation (hybrid BM25 + dense, RRF) · 4. Hard filters + tenant isolation in the query · 5. Learned ranking (GBDT + feature store) · 6. Re-rank (cross-encoder + business rules + diversity) · 7. Feature store (no train/serve skew, point-in-time) · 8. Offline eval (NDCG, recall@k, MRR on a judged set) · 9. Online eval (CTR/conversion A/B + guardrails) · 10. Latency budget + caching · 11. Freshness/indexing pipeline (CDC-driven).

Practice — do these before revealing

Answer each out loud as if whiteboarding, then compare.

Problem 1

A teammate proposes "just embed everything and do pure vector search, it is simpler than maintaining BM25 too." On a product catalog, what breaks, and what do you propose?

Show solution

Pure vector search fails on exact-match queries: SKUs, model numbers, brand names, and rare tokens embed poorly and get drowned by semantically "near" but wrong items, so a user searching a precise product code may not find it. Dense retrieval is great at synonymy and paraphrase but weak where the literal token is the intent. Propose hybrid: run BM25 (which dominates exact-match via TF-IDF with length normalization and TF saturation) alongside dense ANN, and fuse with RRF so each contributes by rank without a score-scale clash. You keep dense's semantic recall for natural-language queries and lexical's precision for codes and names, which is exactly the mix a catalog needs. The maintenance cost of a second index is worth it; losing exact-match is not acceptable on a marketplace.

Problem 2

Your new ranker scores beautifully offline (NDCG up 8%) but in the A/B test conversion is flat. Give three plausible causes and how you would investigate.

Show solution

(1) Training-serving skew: a feature is computed differently offline than online, so the live model is not the one you evaluated, check that the feature store serves identical values and transforms in both paths, and log served features to compare against training. (2) Label/metric mismatch: your offline judged labels or NDCG do not correlate with conversion (maybe they reward relevance but ignore price/availability that drive purchase), re-pick the offline metric that historically tracks conversion and refresh labels from real logs. (3) A guardrail is masking the win: the new ranker is slower (latency up) or raised the zero-result rate, suppressing conversion despite better ordering, inspect the guardrail metrics in the A/B. Investigate by segmenting the A/B (does it win for some query types and lose for others?), confirming feature parity, and shadow-running to compare live vs offline scores on the same traffic. Offline is a proxy; the A/B is truth, so the bug is almost always in the gap between them.

Problem 3

Sellers complain their just-listed and just-restocked items take hours to appear, and out-of-stock items keep showing. What in your design is wrong, and how do you fix freshness without rebuilding the whole index constantly?

Show solution

The symptom says the index is being refreshed in slow batches (e.g. a nightly or hourly rebuild) while inventory changes by the minute, so the index is chronically stale. The fix is incremental, event-driven indexing: emit a change event (CDC or an app event) on every item create, update, sold-out, and delete, and have an indexing consumer enrich, embed, and upsert just that item into the ANN and lexical indexes within seconds, plus refresh its features and its in_stock filter field. You never rebuild the whole index for a single change. To keep out-of-stock items from surfacing immediately even before re-embedding, the in_stock hard filter in the retrieval query excludes them the moment the flag flips. Freshness is a first-class pipeline driven by change events, not a cron job.

    Key: CDC/event-driven incremental upserts (seconds, not nightly rebuilds); in-stock/price as hard filter fields flipped immediately.

Build it (20–25 min)

Hands-on

Build the funnel small. Index ~1,000 product records (use any open catalog) into both a lexical index and a dense ANN index, pgvector or an in-memory ANN library is fine. For a query, run BM25 and dense in parallel, fuse with RRF, and apply a hard in_stock + price filter inside the retrieval step. Then add a simple ranking step (sort the fused candidates by a blend of fused score and a popularity/CTR field) and, for the top ~50, a cross-encoder rerank (a small reranker model). Finally, write ten judged queries with relevance labels and compute NDCG@10 and recall@50 before and after adding the reranker, you will see the precision lift the funnel is built for. Reading: semantic vs lexical search and the reranking concept.

Stretch: deliberately add an exact-match query (a product code) and confirm BM25 finds it while pure dense ranks it poorly, the case that justifies hybrid.

Search is a funnel: cheap high-recall hybrid candidate generation, then a learned ranker on rich features, then a cross-encoder rerank plus business rules, each stage trading recall for precision where it is affordable. Hard filters and tenant isolation live in the retrieval query, never in the ranking score. A feature store keeps train and serve features identical so the live ranker matches its offline numbers. You optimize on offline NDCG/recall and decide launches on online CTR/conversion A/Bs, and you keep the two correlated. Freshness is an event-driven indexing pipeline, not a nightly rebuild.
