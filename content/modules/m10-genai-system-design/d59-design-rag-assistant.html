Design a production RAG assistant

This is the flagship GenAI design question and the model answer for the whole module. We take "design a RAG assistant over our company docs" from the opening clarifying question to operations, in the exact order an interviewer wants to hear it. Learn this walkthrough cold and the next four lessons become variations on a theme.

By the end you can
  • Drive a GenAI design from clarify → estimate → architecture → data model → deep-dive → trade-offs → ops without prompting.
  • Specify an ingestion pipeline: parse, parent-child chunk, embed, index, with incremental updates.
  • Design hybrid retrieval (BM25 + dense, RRF) with a reranker and a non-negotiable tenant filter.
  • Make generation grounded: citations, a citation guardrail, and refusal when context is weak.
  • Attach the eval, observability, and cost story, and recite the 11-point AI-design checklist.

The framework, made GenAI-specific

Module 8 gave you the generic system-design spine. For a GenAI system the same seven beats apply, but the bottleneck and the failure modes move. In a URL shortener the hard part is keyspace and read scale. In a RAG assistant the hard part is retrieval quality and grounding: roughly 70 to 75 percent of bad answers are bad retrieval, not bad generation. So when you reach the deep-dive, go deep on retrieval, not on the LLM.

Say the structure out loud at the start so the interviewer knows you have a plan: "I will clarify scope and non-functionals, do a back-of-envelope estimate, sketch the high-level architecture, define the data model, deep-dive retrieval since that is where RAG lives or dies, name the trade-offs I am making, then cover eval, observability, and cost." Then execute it.

Interview-grade phrasing

"A RAG assistant is two systems stitched together: an offline ingestion pipeline that turns documents into a searchable index, and an online query path that retrieves, reranks, and generates a grounded answer with citations. Most of the engineering risk is in retrieval quality and tenant isolation, not in the model call." Lead with that and you have framed the whole problem correctly.

1 · Clarify (functional + non-functional)

Never start drawing boxes. Spend the first two minutes pinning scope. The questions below are the ones that actually change the design, grouped so you can rattle them off:

  • Corpus. How many documents, what formats (PDF, HTML, Confluence, tickets), how fresh must answers be, how often do docs change?
  • Tenancy. Is this multi-tenant? Per-tenant data isolation is a hard requirement, not a nice-to-have, and it shapes the index design.
  • Access control. Within a tenant, do users have different document permissions (some docs are HR-only)? That pushes filtering into the data layer.
  • Latency + scale. Target p95? Expected queries per second at peak? Is streaming required?
  • Quality bar + risk. Is a wrong answer embarrassing or dangerous? Must every claim be cited? Is a confident-but-unsupported answer worse than "I do not know"?

For this walkthrough assume the answers: ~500k documents across one large org, multi-tenant SaaS, per-document ACLs exist, p95 under 3 seconds with streaming, citations mandatory, and "I do not know" is strongly preferred over a hallucinated answer. Those answers justify every later decision, so state them.

2 · Estimate (back of envelope)

Do the arithmetic out loud. It shows you can size a system and it tells you which datastore survives. Two numbers matter most: vector count and vector storage.

Corpus 500,000 docs × ~8 chunks/doc ≈ 4,000,000 chunks (child) Embedding 1536-dim float32 = 1536 × 4 bytes ≈ 6 KB per vector Vector store 4,000,000 × 6 KB ≈ 24 GB (raw, float32) with int8 quantization (~1.5 KB) ≈ 6 GB (fits in RAM easily) Traffic 50 QPS peak → retrieval + rerank + 1 LLM call per query LLM tokens ~4k in (context) + ~400 out per answer Cost/query @ ~$5 / 1M in, ~$15 / 1M out ≈ $0.02 + $0.006 ≈ ~$0.026 / query 50 QPS sustained ≈ 4.3M queries/day → order ~$100k/day IF uncached → caching + routing is not optional, it is the cost story (step 7)

The headline takeaways you verbalize: at 4M vectors a single Postgres + pgvector instance is comfortably in range (the practical pgvector ceiling is roughly 10M before you reach for a dedicated store), and uncached generation cost is the thing that will get the project killed, so cost controls earn a first-class slot. You just turned vague "it should scale" hand-waving into two decisions.

3 · High-level architecture

Now the picture. Draw it as two lanes: the offline ingestion lane that runs when documents change, and the online query lane that runs per request. Keep them visually separate, because they scale and fail independently.

Walk it left to right once: a query arrives, you check the semantic cache, you run hybrid retrieval with the tenant and ACL filter pushed into the query, rerank the union, assemble the top parents into a context with citation markers, generate a streamed grounded answer, run the citation guardrail, and emit a full trace. Everything else is detail hung on this skeleton.

4 · Data model

Two tables carry the system. Keep child chunks (small, good for embedding precision) separate from parents (larger, good context for the LLM). This is the parent-child pattern: index small, return large.

-- parents: the unit you actually feed the LLM (a section / page)
CREATE TABLE chunk_parent (
  parent_id   uuid PRIMARY KEY,
  doc_id      uuid NOT NULL,
  tenant_id   uuid NOT NULL,        -- isolation key, never optional
  acl         text[] NOT NULL,      -- groups allowed to see this
  source_uri  text,                     -- for the citation link
  title       text,
  content     text,                     -- full parent text
  updated_at  timestamptz,
  version     int
);

-- children: small, embedded, what retrieval searches over
CREATE TABLE chunk_child (
  child_id    uuid PRIMARY KEY,
  parent_id   uuid REFERENCES chunk_parent(parent_id) ON DELETE CASCADE,
  tenant_id   uuid NOT NULL,
  acl         text[] NOT NULL,      -- denormalized for fast filtering
  content     text,
  embedding   vector(1536),             -- pgvector column
  tsv         tsvector                  -- BM25-ish full-text search
);

-- the two indexes that make retrieval fast
CREATE INDEX ON chunk_child USING hnsw (embedding vector_cosine_ops);
CREATE INDEX ON chunk_child USING gin (tsv);
CREATE INDEX ON chunk_child (tenant_id);   -- filter before/with the ANN scan

Notice tenant_id and acl live on the child row even though they duplicate the parent. That denormalization lets the retrieval query filter without a join, which keeps the hot path fast. The cascade delete means removing a document cleans up its children automatically, which matters for the right-to-be-forgotten and re-index cases.

5 · Deep-dive: retrieval, because that is where RAG lives or dies

This is where you spend your time. A senior answer treats retrieval as four sequenced decisions.

Chunking. Parent-child by default. Embed child chunks of roughly 256 to 512 tokens with 10 to 20 percent overlap so the embedding is focused; return the parent so the LLM has enough surrounding context to answer. Fixed-size is cheap but slices through sentences; parent-child gives you precise retrieval and coherent context at once.

Hybrid retrieval. Run BM25 (lexical, owns exact matches like error codes, product names, SKUs) and dense embeddings (semantic, owns paraphrase and synonymy) in parallel, each returning ~50 candidates, then fuse with Reciprocal Rank Fusion. RRF combines by rank not raw score, so you sidestep the score-scale mismatch between a BM25 number and a cosine similarity. Pure dense misses "the exact token the user typed"; pure lexical misses "they asked it a different way." Hybrid covers both.

# RRF: rank-only fusion, scale-free, ~3 lines of the whole idea
def rrf(rankings, k=60):
    scores = {}
    for ranked_list in rankings:            # [bm25_ids, dense_ids]
        for rank, doc_id in enumerate(ranked_list):
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank)
    return sorted(scores, key=scores.get, reverse=True)

Reranking. Hybrid gives you ~50 to 100 candidates with decent recall but mediocre precision at the very top. A cross-encoder reads the (query, chunk) pair jointly and scores true relevance, so you rerank the union down to the 8 you will actually pass to the LLM. This is the single highest-leverage precision lever in RAG and it is cheap because N is small. Bi-encoders scale retrieval (you precompute embeddings); cross-encoders are too slow to run over the whole corpus but perfect for reranking a short list.

The tenant + ACL filter is non-negotiable and lives in the query. You never retrieve broadly and then ask the LLM to ignore other tenants' documents. The filter is a WHERE tenant_id = $1 AND acl && $user_groups predicate applied inside the retrieval query, so a document the user cannot see is never even a candidate. Isolation is a data-layer guarantee. An LLM-layer "please only use tenant X" instruction is a leak waiting to happen.

The classic failure

Dropping the tenant filter under load (an optimization that "broadens recall") is how cross-tenant leaks happen. Treat the tenant predicate like a foreign-key constraint: it is part of the query's contract, it is tested, and it is never removed for performance. If you must speed up, add a per-tenant namespace or index, do not remove the filter.

6 · Grounded generation: citations and the citation guardrail

Retrieval found the right passages. Generation must now answer only from them and prove it. Three mechanisms:

  • Numbered context. Assemble the reranked parents into the prompt with explicit source markers, for example [1], [2], each tied to a source_uri. Instruct the model to cite the marker after each claim and to answer only from the provided sources.
  • Refusal when context is weak. If the top reranker score is below a threshold, or no chunk is relevant, the model should say "I do not have enough information in the documents to answer that." A grounded "I do not know" beats a confident fabrication, which you said in step 1 was the quality bar.
  • Citation guardrail (post-generation check). After generation, verify the answer's claims are actually supported by the cited chunks. The cheap version checks that every citation marker the model used maps to a real retrieved source; the stronger version runs a fast faithfulness check (an NLI or LLM-as-judge pass) that flags any sentence not entailed by its cited context, and you either strip it or return a low-confidence warning.
# context assembly with citation markers (sketch)
def build_context(reranked):
    blocks, sources = [], []
    for i, ch in enumerate(reranked, 1):
        blocks.append(f"[{i}] {ch.content}")
        sources.append({"marker": i, "uri": ch.source_uri})
    system = ("Answer ONLY from the numbered sources. Cite the [n] "
              "after each claim. If the sources do not contain the "
              "answer, say you do not know.")
    return system, "\n\n".join(blocks), sources

7 · Trade-offs (name what you give up)

Seniors are graded on naming the cost of their choices, not on having a "best" answer. Say these unprompted:

DecisionYou gainYou give up
Add a rerankerBig precision jump at top-k~50-200ms latency, an extra model dependency
Hybrid over pure denseExact-match recall (codes, names)Two indexes to maintain, fusion tuning
pgvector over a dedicated storeVectors next to relational data + ACLs, one DB, joins are freeScaling ceiling ~10M; dedicated stores win past that
Parent-child chunkingPrecise retrieval + coherent contextMore moving parts, larger context tokens (cost)
Strict refusal thresholdFewer hallucinations, higher trustMore "I do not know", some answerable queries declined

8 · Eval, observability, cost (operations)

Eval. Two layers. Retrieval: a golden set of (question, ground-truth chunk) pairs scored by recall@k and MRR. Generation: faithfulness (does every claim trace to the context, the metric you trust most), answer relevancy, and contextual precision/recall, scored RAGAS-style by an LLM judge calibrated against human labels. Gate releases on these offline, then ship behind shadow mode and a per-user A/B with a faithfulness floor as a guardrail metric.

Observability. Emit a span tree per request: a retrieval span, a rerank span, and an LLM span, each carrying tokens, cost, and latency, exported over OTLP to LangSmith or Langfuse. Log the prompt version, model, temperature, retrieved doc IDs, and a hashed user/session, never raw PII. On a rolling sample, score live faithfulness so quality regressions surface before users complain.

Cost. From step 2 this is the existential risk, so attack it deliberately: embed once and cache (never re-embed unchanged chunks), prompt-cache the stable system + instruction prefix (Anthropic prompt caching bills cached input at roughly a tenth), semantic-cache near-duplicate queries, route easy queries to a cheaper model, and keep top-k tight. This is exactly the lever set that cuts RAG cost ~30 percent without a quality hit.

Worked example: one query end to end

Example

User in tenant acme (groups: eng, all) asks: "What is the retry policy for error code E-4021 in the billing service?"

1. Cache. Semantic cache miss (novel query). 2. Retrieve. BM25 nails E-4021 (exact token, dense alone would have struggled); dense pulls passages about "retry" and "backoff" that never say the code. The query carries WHERE tenant_id='acme' AND acl && '{eng,all}', so an HR-only doc that mentions the code is never a candidate. RRF fuses both lists. 3. Rerank. A cross-encoder pushes the one paragraph that actually states "E-4021 retries 3x with exponential backoff" to rank 1. 4. Assemble. Top 8 parents with markers [1]..[8]. 5. Generate. Streamed answer: "E-4021 triggers up to three retries with exponential backoff [3]." 6. Guardrail. The cited [3] chunk entails that sentence, so it passes; the trace records 3.9k input tokens, 1 LLM call, 2.1s, and a faithfulness sample of 1.0.

The 11-point AI-design checklist (recite at the end of any GenAI design)

Run this list to prove you covered the surface. It is the checklist behind every Module 10 answer:

1. Ingestion (parse, chunk, embed, incremental update) · 2. Retrieval (hybrid + fusion) · 3. Reranking · 4. Tenant / ACL isolation in the data layer · 5. Grounded generation + citations · 6. Guardrails (citation / faithfulness / refusal) · 7. Eval (offline golden set + online A/B) · 8. Observability (span tree, tokens, cost, latency) · 9. Cost controls (caching, routing, top-k) · 10. Latency budget (streaming, parallel retrieval) · 11. Failure handling (provider retry/fallback, empty-retrieval path).

Practice — do these before revealing

Answer each out loud as if whiteboarding, then compare. The gap is your lesson.

Problem 1

Your RAG assistant returns fluent but wrong answers about 20 percent of the time. Before touching the prompt or swapping models, how do you localize whether the problem is retrieval or generation, and what is your first fix?

Show solution

Split the pipeline and measure each half. Take the failing questions, look at the retrieved context for each: if the correct passage was never retrieved, it is a retrieval problem (most likely, ~70-75 percent of cases); if the right passage was in context but the answer ignored or contradicted it, it is a generation/grounding problem. For retrieval failures, check recall@k on a golden set first, then the usual suspects in order: chunking (are answers split across chunk boundaries?), missing reranker, and top-k too low. The first fix is almost always adding or improving the reranker and verifying chunk boundaries, because that lifts precision at the top where the LLM reads. Only after retrieval is healthy do you tighten the grounding prompt and add the citation guardrail. Fixing generation while retrieval is broken is treating a symptom.

Problem 2

Product wants the assistant to support 5,000 tenants, each with strict data isolation. A teammate proposes one shared index with a tenant_id field in the prompt so the LLM only uses the right docs. What is wrong, and what do you propose instead?

Show solution

Putting isolation in the prompt is unsafe: the LLM is non-deterministic and an attacker (or a bad retrieval) can surface another tenant's chunk that the model then summarizes, leaking data across a trust boundary. Isolation must be a data-layer constraint. Progression by scale: (1) a mandatory metadata filter, WHERE tenant_id = $1, applied inside the retrieval query so other tenants' rows are never candidates, this covers most cases; (2) per-tenant namespaces / partitions for stronger blast-radius control and faster filtered scans; (3) per-tenant indexes for the largest or most sensitive tenants. The model never decides who sees what; the query does. And the tenant predicate is tested like a constraint and never dropped for performance.

Problem 3

Finance flags that the assistant's monthly LLM bill is unsustainable. List four levers in priority order and the one quality risk to watch on each.

Show solution

In rough priority: (1) Prompt caching of the stable system + instruction prefix, biggest lever, billed at roughly a tenth of normal input, risk is almost none if the prefix is genuinely static. (2) Semantic caching of near-duplicate queries, big on FAQ-style traffic, risk is serving a stale or wrong answer for personalized or time-sensitive queries, so set a conservative cosine threshold and exclude high-stakes intents. (3) Model routing, send easy/short queries to a cheaper model and reserve the strong model for hard ones, risk is mis-routing a hard query, so gate routing on a confidence or complexity signal. (4) Tighter top-k and smaller context, fewer tokens per call, risk is dropping a chunk that contained the answer, so validate recall@k stays flat after the cut. You did exactly this kind of measured trimming at MaiQ.

Build it (20–25 min)

Hands-on

Sketch this design on paper, then make it concrete. Stand up pgvector locally, create the two-table schema above, and ingest five of your own PDFs with parent-child chunking. Write one retrieval query that applies a tenant_id filter and does a hybrid BM25 + vector lookup fused by RRF, then pass the top 8 parents to a Claude call that answers with [n] citations. Finally, add the cheap citation guardrail: assert every [n] the model emitted maps to a retrieved source, and have the model refuse when nothing relevant comes back. You now have the spine of the model answer running on your laptop.

Stretch: add a cross-encoder reranker between fusion and assembly (e.g. a small reranker model) and measure recall@5 before and after on ten hand-written questions.

Today's takeaways

A RAG assistant is an offline ingestion lane plus an online query lane, and the deep-dive always belongs to retrieval because that is where most answers go wrong. Hybrid retrieval plus a reranker gives recall and precision; the tenant and ACL filter lives in the data layer and is never dropped. Grounded generation means numbered citations, refusal on weak context, and a post-generation faithfulness guardrail. Close every GenAI design with the 11-point checklist and the cost story, and you sound like someone who has run one in production.
