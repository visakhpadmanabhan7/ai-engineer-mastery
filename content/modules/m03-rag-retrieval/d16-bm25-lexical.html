BM25 & lexical search

Dense embeddings are not the whole story. The decades-old keyword retriever BM25 still beats them flat on exact matches: product codes, error strings, names, acronyms. Today you learn why, and why every serious RAG system keeps it.

By the end you can
  • Explain TF-IDF, then how BM25 improves it with length normalization and term-frequency saturation.
  • Reason about the k1 and b knobs without memorizing the formula.
  • Name what BM25+ and BM25F add.
  • Say precisely why lexical search still beats dense on exact-match queries.
  • Decide when keyword search alone is the right call, and set up tomorrow's hybrid lesson.

The case for keyword search

Yesterday's dense retrieval finds passages that mean the same thing. That is exactly wrong for a class of queries where you do not want semantic similarity, you want the literal token. If a user searches for error code ERR_4042, part number X-7781-B, the function name parse_invoice, or a surname like "Nakamura," there is no useful "meaning" to embed. An embedder will happily return semantically nearby chunks (other error codes, other part numbers) and miss the one exact string the user actually needs.

Lexical (keyword) search inverts the model: it scores documents by how well their words match the query's words. The workhorse is BM25 (Best Matching 25), a probabilistic ranking function that has been the backbone of search engines for decades and remains the strong baseline every new retriever is measured against. It is fast, interpretable, needs no GPU, and on exact-match queries it is simply better than dense retrieval.

1 · From TF-IDF to BM25

BM25 is best understood as a refinement of TF-IDF, so start there. TF-IDF scores a query term in a document by multiplying two intuitions:

  • Term Frequency (TF): a term that appears more often in a document is more relevant to it. "Invoice" appearing 8 times signals the doc is about invoices.
  • Inverse Document Frequency (IDF): a term that appears in few documents is more discriminating. "Invoice" is informative; "the" appears everywhere and tells you nothing, so IDF down-weights common words and up-weights rare ones.

TF-IDF works but has two flaws BM25 fixes, and these two fixes are the entire point of BM25:

Interview-grade phrasing

"BM25 is TF-IDF plus two corrections: term-frequency saturation, so the tenth occurrence of a word adds far less than the second, and document-length normalization, so a long document does not win just by being long. That is why it has stayed the lexical baseline for decades." Say that and you have shown you understand it, not just named it.

  • Term-frequency saturation. In raw TF-IDF, a word appearing 100 times scores 100× a word appearing once, which over-rewards keyword stuffing. BM25 makes TF saturate: the contribution rises quickly for the first few occurrences then flattens, so going from 1 to 2 mentions matters a lot but 50 to 51 barely moves the score. Relevance is not linear in count, and BM25 models that.
  • Document-length normalization. A long document naturally contains more of every word, so raw TF unfairly favors long docs. BM25 normalizes by length relative to the average document, so a short, tightly-on-topic passage can outscore a long one that merely mentions the term in passing.

2 · The two knobs: k1 and b

BM25 exposes exactly two parameters, and you can reason about both from the intuitions above, no formula needed:

  • k1 (term-frequency saturation): controls how fast the TF contribution flattens. Higher k1 means repeated terms keep helping for longer (closer to linear TF); lower k1 means the score saturates almost immediately after the first occurrence. Typical values sit around 1.2 to 2.0.
  • b (length normalization strength): controls how much document length is penalized, on a 0-to-1 dial. b = 1 fully normalizes by length; b = 0 ignores length entirely. The common default is around 0.75, a strong but not total length penalty.

You rarely need to tune these, the defaults are robust, which is part of BM25's appeal. But an interviewer may ask what they do, and "k1 is the saturation knob, b is the length-normalization knob" is the answer.

3 · BM25+ and BM25F

Two named variants you should recognize:

  • BM25+ adds a small lower-bound correction so that very long documents containing a query term are never scored essentially zero just because length normalization pushed them down. It patches an edge case where standard BM25 can over-penalize long docs.
  • BM25F ("F" for fields) handles documents with multiple fields of differing importance, a title, an abstract, a body, and lets you weight a match in the title more heavily than a match buried in the body. Useful when structure carries signal (a query word in the title is a stronger relevance cue than the same word on page 9).

You do not need to implement these. Knowing they exist, and what each fixes, is enough to sound fluent.

4 · Why lexical still beats dense on exact match

This is the heart of the lesson, and the reason BM25 is not going anywhere. Dense embeddings compress text into a smooth semantic space, and that compression is lossy on exactly the things that must match literally:

  • Rare tokens and out-of-vocabulary strings. A part number X-7781-B or a fresh error code carries no learned semantics; the embedder has nothing meaningful to place it near, so it gets a vague vector. BM25 just matches the literal token and ranks it first.
  • Names, codes, acronyms, identifiers. "Nakamura," SKU-42, NullPointerException need exact equality, not nearness. Dense search will surface other names or other exceptions; lexical search surfaces the one.
  • Precision when the user knows the exact word. When someone types a precise term, they want that term, not its semantic cousins. Lexical search respects that intent; dense search may "helpfully" generalize away from it.

The flip side, and why you do not use BM25 alone, is that lexical search is blind to synonyms and paraphrase: "reset my password" and "recover my login" share no tokens, so BM25 scores them as unrelated, while a dense model nails it. The two methods fail on opposite queries. That complementarity is the whole reason for tomorrow's lesson.

Concept

Lexical and dense retrieval have complementary failure modes. BM25 misses synonyms and paraphrase but nails exact tokens (codes, names, acronyms, rare strings). Dense nails meaning but blurs exact identifiers. Neither alone is enough, which is precisely why production RAG fuses both.

5 · When keyword search alone is the right call

Sometimes you do not need embeddings at all, and reaching for them is over-engineering. BM25 alone is the right call when:

  • The corpus is dominated by identifiers and structured tokens, log search, code search, SKU and part lookups, where queries are almost always exact strings.
  • You need full interpretability: BM25 can show exactly which query terms matched and why a document ranked where it did, which a vector score cannot.
  • You have no GPU budget or tight latency, BM25 is cheap, CPU-only, and battle-tested at massive scale (it is what classic search engines run on).
  • You are building a baseline: always measure BM25 first. A surprising number of "we need fancy embeddings" problems are solved by a well-tuned keyword index, and you cannot claim dense retrieval helped until you have beaten the BM25 baseline.

Worked example: BM25 intuition on a tiny corpus

Example

Three documents, query: invoice error. Watch how saturation, IDF, and length normalization combine, qualitatively.

D1 (short): "Invoice error: code 4042 on submit." D2 (long): "...invoice... [400 more words of unrelated text] ...invoice...invoice..." D3 (short): "Payment received; thank you for your order." term "invoice": appears in D1, D2 → moderate IDF (common-ish here) term "error": appears in D1 only → HIGH IDF (rare, discriminating) D1: has BOTH terms, short doc → "error" (high IDF) scores strongly, length normalization rewards the tight match → RANK 1 D2: has "invoice" 3× but TF SATURATES (3rd mention ≈ 2nd), missing "error" entirely, and it is LONG (penalized) → RANK 2 D3: has neither query term → score ~0, not returned

Read what each correction did. IDF made "error" worth more than "invoice" because it is rarer and more discriminating. Saturation stopped D2 from winning on three repetitions of "invoice." Length normalization rewarded short, on-topic D1 over long, padded D2. The document with both terms, tightly, ranks first, which is exactly what a human would pick. No neural network, no GPU, fully explainable.

Why this matters for you as an AI engineer

Three consequences, derived not memorized

Keep BM25, always ← it nails exact tokens (codes, names, acronyms) that dense retrieval blurs, so it is not legacy, it is the half of retrieval embeddings cannot do.
Measure the baseline first ← BM25 is the bar every retriever must clear, so you cannot claim dense or hybrid helped until you have beaten a tuned keyword index.
Failure modes are complementary ← BM25 misses paraphrase, dense misses identifiers, which is the entire argument for the hybrid fusion you build tomorrow.

Problem 1

Your pure-dense RAG system is great at conceptual questions but users complain it "can't find" specific error codes and order numbers they paste in. Explain mechanically why, and what you would add.

Show solution

An error code like ERR_4042 or an order number is a rare, essentially out-of-vocabulary token with no learned semantics, so the embedder produces a vague, uninformative vector and dense search returns semantically nearby chunks (other codes, other orders) rather than the exact-string match the user wants. Dense retrieval is lossy on precisely the literal identifiers these queries hinge on. The fix is to add a lexical retriever (BM25) that matches the exact token, then fuse it with dense results (tomorrow's hybrid search), so conceptual queries are served by embeddings and exact-token queries are served by BM25. The two cover each other's blind spots.

Problem 2

A teammate says "BM25 is obsolete, embeddings replaced it." Give the two-sentence correction, naming one query type where BM25 wins decisively and one where it fails.

Show solution

BM25 is not obsolete; it has complementary strengths to dense retrieval and remains the baseline every new retriever is benchmarked against, which is why production systems run both. It wins decisively on exact-match queries, product codes, error strings, names, acronyms, rare tokens, where the user wants the literal token and embeddings only return semantic neighbors. It fails on synonym and paraphrase queries ("reset password" vs "recover login"), where there is no token overlap and a dense model is needed. Because they fail on opposite query types, the right architecture fuses them rather than choosing one.

Problem 3

You set b = 0 in BM25 and suddenly long, rambling documents dominate the results for every query. Explain what b does and why this happened.

Show solution

The parameter b controls document-length normalization on a 0-to-1 dial: at b = 1 scores are fully normalized for length, at b = 0 length is ignored entirely. With b = 0 you removed the penalty that stops long documents from winning just by being long, a long, rambling document contains more occurrences of almost every term simply because it has more words, so its raw term-frequency contribution inflates and it outranks short, tightly-on-topic passages. Restore b toward the usual default of about 0.75 so length is penalized strongly (but not totally), letting a concise on-topic chunk outscore a long one that merely mentions the term in passing. This is exactly the flaw in plain TF-IDF that BM25's length normalization was designed to fix.

Build it (15–20 min)

Hands-on

Build a BM25 index over a small corpus and probe its strengths. Use the rank_bm25 Python library (or your search engine's built-in BM25): index 15 to 20 short passages, half conceptual prose and half containing identifiers like error codes or part numbers. Run two kinds of query, an exact identifier (e.g. the code itself) and a paraphrase of a concept, and note which BM25 nails and which it whiffs. Then re-run the same queries against the dense search you built on Day 15 and confirm the failure modes are opposite. You have just motivated tomorrow's hybrid lesson with your own eyes.

Today's takeaways

BM25 is a probabilistic bag-of-words ranker: TF-IDF plus term-frequency saturation and document-length normalization, with two robust knobs (k1 for saturation, b for length). It still beats dense retrieval decisively on exact-match queries, codes, names, acronyms, and rare tokens, because embeddings are lossy on the things that must match literally. It fails on synonyms and paraphrase, which is the exact opposite of where dense retrieval fails, so their strengths are complementary. Use BM25 alone for identifier-heavy corpora or as the baseline you must beat, and fuse it with dense retrieval everywhere else, the subject of tomorrow's hybrid search lesson.
