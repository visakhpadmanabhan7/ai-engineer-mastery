The build, end to end

Here is the pipeline you are assembling. Keep this picture open as you work the checkpoints, each checkpoint builds one box.

~20 docs │ CP1 parent-child chunk ▼ parents (big) ── children (small, embedded) │ CP2 embed + index (dense) AND build BM25 ▼ query ─┬─► dense top-N ─┐ └─► BM25 top-N ─┤ CP3 RRF fusion → fused top-N ▼ CP4 (optional) cross-encoder rerank → top-k │ return PARENT of each child chunk ▼ CP5 generate-with-citations [1][2]… │ golden set ──► CP6 recall@k / MRR, reranker OFF vs ON ← the whole point

Setup. A virtual environment and three small libraries. No GPU needed; the models below run on CPU.

import numpy as np
from sentence_transformers import SentenceTransformer, CrossEncoder
from rank_bm25 import BM25Okapi

embedder = SentenceTransformer("sentence-transformers/all-MiniLM-L6-v2")
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

Checkpoint 1 — parent-child chunking

Build

Take ~20 documents (paste docs, README sections, or wiki pages into a list of strings). Split each into parent blocks of a few paragraphs, and each parent into smaller child chunks (~2–3 sentences) that you will embed. Keep a mapping so that retrieving a child lets you return its parent. Why: you index small (precise matching) but return large (enough context for the generator), the parent-child idea from Day 14.

Show solution

A deliberately simple splitter, swap in a real sentence splitter later. The data model is the lesson: every child knows its parent.

def chunk_docs(docs, parent_chars=900, child_chars=300):
    parents, children = [], []
    for doc_id, text in enumerate(docs):
        for p_start in range(0, len(text), parent_chars):
            parent = text[p_start:p_start + parent_chars]
            pid = len(parents)
            parents.append({"pid": pid, "doc": doc_id, "text": parent})
            for c_start in range(0, len(parent), child_chars):
                children.append({
                    "cid": len(children),
                    "pid": pid,                       # link back to parent
                    "text": parent[c_start:c_start + child_chars],
                })
    return parents, children

parents, children = chunk_docs(docs)
child_texts = [c["text"] for c in children]

Real systems use token-aware, sentence-respecting splitters and overlap; character windows are fine to learn the structure. The non-negotiable part is the pid link: retrieval scores children, generation reads parents.

Checkpoint 2 — embed, index, and BM25

Build

Embed every child chunk into a matrix (your dense index). In parallel, build a BM25 index over the same child texts (tokenized). Two retrievers over the same units: dense for semantics, BM25 for exact terms (Day 15–16). If you have pgvector available, you may store the embeddings there instead of numpy; the lab stays in numpy so it runs anywhere.

Show solution 
# Dense index: one normalized vector per child (cosine == dot on unit vectors).
emb = embedder.encode(child_texts, normalize_embeddings=True)
emb = np.asarray(emb, dtype="float32")            # shape (n_children, dim)

# BM25 index over the same child texts (lowercased whitespace tokens).
tokenized = [t.lower().split() for t in child_texts]
bm25 = BM25Okapi(tokenized)

def dense_search(query, n=20):
    q = embedder.encode([query], normalize_embeddings=True)[0]
    scores = emb @ q                                  # cosine similarity
    return np.argsort(-scores)[:n].tolist()           # child indices, best first

def bm25_search(query, n=20):
    scores = bm25.get_scores(query.lower().split())
    return np.argsort(-scores)[:n].tolist()

Both functions return ranked child indices, a uniform shape, which is what makes the next step (fusion by rank) clean. Brute-force numpy cosine is fine at ~20 docs; at scale you would swap in an ANN index or pgvector (Day 16), but the interface stays the same.

Checkpoint 3 — RRF fusion

Build

Fuse the two ranked lists with Reciprocal Rank Fusion. RRF scores each item by sum over lists of 1/(k + rank) (a common k is 60), then re-sorts. Why RRF and not score-addition: BM25 and cosine live on different, incomparable scales, so adding raw scores is meaningless; RRF uses only ranks, so it is scale-free (Day 17).

Show solution 
def rrf_fuse(ranked_lists, k=60, n=20):
    scores = {}
    for lst in ranked_lists:
        for rank, cid in enumerate(lst):          # rank 0 = best
            scores[cid] = scores.get(cid, 0.0) + 1.0 / (k + rank + 1)
    fused = sorted(scores, key=scores.get, reverse=True)
    return fused[:n]

def first_stage(query, n=20):
    return rrf_fuse([dense_search(query, n), bm25_search(query, n)], n=n)

That is the entire hybrid retriever. An item ranked highly by either retriever floats up; an item ranked highly by both floats highest. No score normalization, no tuning of relative weights, which is exactly why RRF is the pragmatic default for combining heterogeneous rankers.

Checkpoint 4 — optional rerank, return parents

Build

Add a switchable reranker: when on, rescore the fused child candidates with the cross-encoder and keep the top-k; when off, just take the fused top-k. Either way, map the surviving children to their parents (dedup by pid) and return those for generation. The switch matters: Checkpoint 6 measures recall with it off vs on.

Show solution 
def retrieve(query, k=5, use_reranker=True, first_n=20):
    cand = first_stage(query, n=first_n)              # fused child indices
    if use_reranker:
        pairs  = [(query, child_texts[c]) for c in cand]
        scores = reranker.predict(pairs)              # joint (query, child) scoring
        cand   = [c for c, _ in
                  sorted(zip(cand, scores), key=lambda x: x[1], reverse=True)]
    # take top-k children, then dedupe to their parents (order preserved)
    seen, parents_out = set(), []
    for c in cand:
        pid = children[c]["pid"]
        if pid not in seen:
            seen.add(pid); parents_out.append(parents[pid])
        if len(parents_out) == k:
            break
    return parents_out

One function, one boolean, two behaviors, that boolean is your experiment. Returning parents (not children) is the parent-child payoff: precise matching on small units, full context for the generator.

Checkpoint 5 — generate with citations

Build

Feed the retrieved parents to an LLM with a strict instruction: answer only from the provided context, cite each claim with the chunk number it came from, and say "I don't know" if the context does not contain the answer. Citations make the answer auditable and force grounding, the anti-hallucination posture from Day 1.

Show solution 
def build_prompt(query, parents_out):
    ctx = "\n\n".join(f"[{i+1}] {p['text']}"
                       for i, p in enumerate(parents_out))
    return (
        "Answer the question using ONLY the context below. "
        "Cite sources inline like [1], [2]. If the answer is not in "
        "the context, say 'I don't know.'\n\n"
        f"Context:\n{ctx}\n\nQuestion: {query}\nAnswer:"
    )

Checkpoint 6 — the golden set and the measured delta

Build (the point of the whole lab)

Hand-label a tiny golden set: 8–12 questions, each with the set of ideal_pids (which parent chunks actually contain the answer). Compute recall@k and MRR over it, twice, once with use_reranker=False, once True, and print the delta. This is the Day 20 discipline made concrete: do not claim the reranker helps, show it.

Show solution 
golden = [
    {"q": "what is the webhook retry limit on Pro?", "ideal_pids": {12, 13}},
    {"q": "how do I rotate an API key?",           "ideal_pids": {4}},
    # … 8-12 total, ideal_pids = parents that truly answer it
]

def evaluate(golden, k=5, use_reranker=True):
    recalls, rrs = [], []
    for ex in golden:
        got = [p["pid"] for p in retrieve(ex["q"], k=k, use_reranker=use_reranker)]
        ideal = ex["ideal_pids"]
        recalls.append(len(ideal & set(got)) / len(ideal))   # recall@k
        rr = 0.0
        for rank, pid in enumerate(got):                  # first relevant's reciprocal rank
            if pid in ideal:
                rr = 1.0 / (rank + 1); break
        rrs.append(rr)
    return sum(recalls) / len(recalls), sum(rrs) / len(rrs)

r_off, mrr_off = evaluate(golden, k=5, use_reranker=False)
r_on,  mrr_on  = evaluate(golden, k=5, use_reranker=True)
print(f"reranker OFF  recall@5={r_off:.2f}  MRR={mrr_off:.2f}")
print(f"reranker ON   recall@5={r_on:.2f}  MRR={mrr_on:.2f}")

Read it honestly. The reranker typically lifts MRR most (it reorders, pulling the right parent higher) and lifts recall@5 when the relevant child was in the fused top-20 but ranked below 5. If recall@5 barely moves, that is a finding, not a failure: it tells you the fused first stage was already surfacing the chunk in the top-5, so the reranker's value here is ordering, not coverage, and you would widen first_n or revisit chunking to chase the remaining misses. Either way you now have a number, which is the entire habit this module exists to build.

Worked example: read one before/after run

Example

A run of the lab over 20 product-doc pages with a 10-question golden set might print:

reranker OFF recall@5 = 0.70 MRR = 0.58 reranker ON recall@5 = 0.80 MRR = 0.74 Δrecall@5 = +0.10 ΔMRR = +0.16 interpretation: • MRR jumped most → the reranker mainly REORDERED (right parent now higher). • recall@5 rose too → on 1 query the answer-parent was in the fused top-20 but below rank 5; the cross-encoder lifted it into the window. • the 2 still-failing queries: inspect them. Missing from fused top-20? → that's a FIRST-STAGE problem (chunking / embedder / widen first_n), NOT something more reranking can fix.

This is the loop in miniature. You did not ship the reranker because a blog said to; you measured a +0.10 recall and +0.16 MRR on your data, then used the residual failures to point at the next fix (first-stage coverage, not reranking). That is the difference between tuning and guessing, and it is the same loop that, run 11+ times with cost tracked alongside quality, produces a result like "37% cheaper at a 2% quality drop." You have now built, end to end, the machine that produces numbers like that.

Module 3 review — can you answer these cold?

Quick self-check across the whole module. One line each; if any is shaky, revisit that day before moving to agents.

  • Walk through a vanilla RAG pipeline, then say where most failures occur.
    Key: ingest → chunk → embed → index → retrieve → generate; ~70-75% of bad answers are retrieval failures.
  • Compare fixed / semantic / sentence-window / parent-child chunking.
    Key: fixed = N-token+overlap; semantic = similarity splits; sentence-window = index sentence, expand at retrieval; parent-child = index small, return parent (good default).
  • BM25 vs dense, and why combine them with RRF?
    Key: BM25 = exact-term bag-of-words (codes, names); dense = semantic vectors; RRF fuses by rank, scale-free, so neither score scale dominates.
  • What does a reranker do and where does it sit?
    Key: cross-encoder rescoring the fused top 50-100, joint (query, chunk) read, a precision stage before generation.
  • When do query transformations (decomposition / step-back / HyDE) help vs hurt?
    Key: help ambiguous/multi-hop/vocab-mismatch queries; hurt precise, well-specified ones; gate them, do not apply reflexively.
  • What is agentic RAG versus vanilla?
    Key: vanilla = fixed retrieve-then-generate; agentic = retrieval is a tool the agent calls with crafted args and iterates.
  • Which retrieval metric do you optimize first, and how do you prove a change?
    Key: recall@k (the downstream ceiling); prove via offline golden set → shadow/interleave → guarded online A/B.
  • How do you cut RAG cost ~30% without losing quality?
    Key: embed once, cache reranks, route, lower top-k, smaller generator, prompt caching, validated on the golden set (e.g. 37% cut at ~2% drop).
