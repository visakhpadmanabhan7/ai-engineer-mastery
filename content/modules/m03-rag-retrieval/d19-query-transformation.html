Query transformation

So far you improved retrieval by improving the index and the ranking. Today you improve the thing you actually search with: the query. A raw user question is often a bad search query, and rewriting it before retrieval, sometimes into several queries, is one of the cheapest ways to rescue multi-hop and ambiguous asks.

By the end you can
  • Explain why the user's literal question is frequently the wrong retrieval query.
  • Apply decomposition, step-back prompting, HyDE, and multi-query expansion, and say when each fits.
  • Name when query transformation hurts (precise, well-specified queries) and avoid it there.
  • See agentic RAG as the generalization: retrieval becomes a tool the model calls and iterates.
  • Describe CRAG and self-RAG as the self-correcting end of the spectrum.

1 · The query is not the question

Embedding-based retrieval matches the query vector against chunk vectors. But a user's question is written for a human, not for your index. It carries conversational framing ("hey, quick one, so for the new plan..."), pronouns that depend on chat history ("does it still apply to them?"), multiple sub-questions fused into one sentence, or a vocabulary that simply does not match how the answer is phrased in your docs. The fix is a small, cheap LLM call before retrieval that transforms the question into one or more queries optimized for the retriever.

Every technique below is a variation on that idea: change the query so the vectors land where the answer lives. They differ in how they change it, and therefore in which failure they fix.

user question │ ▼ ┌──────────────────────────┐ │ query transformation │ small LLM call(s), pre-retrieval │ (rewrite / decompose / │ │ step-back / HyDE) │ └────────────┬─────────────┘ one or more transformed queries │ ▼ retrieve (hybrid + rerank) → merge/dedupe → generate

2 · Decomposition: split a multi-hop question

Some questions require facts that live in different chunks, and no single chunk answers the whole thing. "Which of our enterprise plans launched after the SSO feature, and what is their webhook retry limit?" is two hops: first find when SSO launched, then find plans launched after that, then look up their retry limits. A single embedding of the whole sentence is a muddled average that retrieves neither hop cleanly.

Decomposition asks an LLM to break the question into atomic sub-questions, retrieves for each, and gives the generator the union of the results. It is the right tool whenever the answer is an and of facts from separate places.

# Decompose, retrieve per sub-question, then answer over the union.
sub_qs = llm("Break this into independent sub-questions, one per line:\n" + question)
chunks = []
for q in sub_qs.splitlines():
    chunks += retrieve(q, k=5)            # hybrid + rerank from earlier days
chunks = dedupe(chunks)
answer = llm("Answer using only these chunks:\n" + format(chunks) + question)

3 · Step-back: ask the broader question first

Sometimes the user's question is too specific to retrieve well. "Does the v3.2.1 webhook drop the amount_cents field on refund events specifically?" may not match any chunk verbatim, but a broader query, "how does the v3 webhook payload handle amount fields," will land on the changelog that contains the answer. Step-back prompting generates a more general version of the question, retrieves on that, and lets the generator zoom back in. It is the inverse of decomposition: instead of splitting into narrower pieces, you abstract upward to a query that actually has matching documents.

4 · HyDE: search with a hypothetical answer

Here is the asymmetry HyDE exploits: a question and its answer passage often look very different in embedding space (a question is short and interrogative; a doc passage is declarative and detailed), so question-to-chunk similarity can be weak. HyDE (Hypothetical Document Embeddings) sidesteps this. Ask the LLM to write a plausible answer to the question, hallucinations and all, then embed that and retrieve with it. You are no longer matching a question against documents; you are matching a document-shaped thing against documents, which is what embeddings are better at.

classic: embed("how do I rotate an API key?") ──► search short, interrogative, weak match to doc prose HyDE: llm → "To rotate an API key, open Settings → API, click Rotate, the old key stays valid for 24h…" embed(that hypothetical answer) ──► search doc-shaped, lands near the real doc, even if the made-up specifics are wrong

The made-up "24h" might be wrong, that does not matter, because you never show the hypothetical to the user. It exists only to produce a better search vector. The real, retrieved chunk is what grounds the final answer. HyDE shines on jargon-light or vaguely-worded questions where the user's phrasing does not match the corpus; it can mislead when the LLM's hypothetical drifts into a topic the corpus does not cover.

Interview-grade phrasing

"Query transformation rewrites the question before retrieval so the search vector lands where the answer lives. Decomposition splits multi-hop asks into atomic sub-queries, step-back abstracts an over-specific question into one with matching documents, and HyDE generates a hypothetical answer and embeds that to close the question-versus-passage gap. They help on ambiguous and multi-hop queries and can hurt precise, well-specified ones, so they are conditional, not default."

5 · Multi-query expansion

A single phrasing can miss the chunk that phrases the same idea differently. Multi-query expansion asks the LLM to generate several paraphrases of the question, retrieves for all of them in parallel, and fuses the results (RRF from Day 17 is the natural merge). It widens recall by covering more of the ways the answer might be worded, at the cost of more retrieval calls and a real risk of pulling in off-topic chunks if the paraphrases drift. Use it when recall is the bottleneck and vocabulary mismatch is the suspected cause.

TechniqueFixesCost / risk
DecompositionMulti-hop questions needing facts from separate chunks.N retrievals + a merge; over-splitting simple questions.
Step-backOver-specific queries with no verbatim match.One extra call; can retrieve too broadly, dilute precision.
HyDEQuestion-vs-passage embedding gap; vague phrasing.One generation; hypothetical can drift off-corpus.
Multi-queryVocabulary mismatch, single-phrasing misses.N retrievals + fusion; paraphrase drift adds noise.

6 · When transformation hurts

None of this is free, and worse, it is not always helpful. For a precise, well-specified query, "what is the rate limit for the /v3/charges endpoint", the literal question is already the perfect search string. HyDE can hallucinate a number that drags retrieval toward the wrong section; multi-query can paraphrase the exact endpoint name into something that no longer matches; decomposition splits a single-hop question into clumsy fragments. Every transformation also adds an LLM call before retrieval, raising latency and cost.

So the rule is: transform when the question is ambiguous, multi-hop, vaguely worded, or vocabulary-mismatched with the corpus, and pass the query through untouched when it is already specific and well-formed. The decision itself can be a cheap classifier (or a heuristic on query length and structure), and that "decide whether and how to transform" idea is the bridge to the next section.

7 · Agentic RAG: the generalization

Notice what every technique above shares: a small reasoning step decides how to query. Push that idea to its conclusion and retrieval stops being a fixed pre-step and becomes a tool the model chooses to call, with arguments it crafts, possibly several times, until it has what it needs. That is agentic RAG. Vanilla RAG is a straight line, retrieve then generate. Agentic RAG is a loop: the model reads the question, decides whether to search and with what query, inspects what came back, and either answers or searches again with a refined query.

VANILLA RAG (fixed) AGENTIC RAG (the model drives the loop) query → retrieve → generate query ─► [LLM] ─► need info? │ yes → call retrieve(crafted query) │ observe results │ enough? ─ no ─┐ │ │ refine query │ no ◄────────────────┘ ▼ answer (grounded)

Decomposition, step-back, HyDE, and multi-query all become behaviors the agent can choose rather than a pipeline you hard-wire: it can decompose a hard question, retry a failed search with a broader query, or decide retrieval was not needed at all. The cost is the cost of agents (latency, non-determinism, the need for tracing and evaluation), which is exactly what Module 4 is about. For now, hold the framing: query transformation is the static, dev-chosen version of a decision an agent can make dynamically.

8 · CRAG and self-RAG: grade and self-correct

One more rung up: what if the system checks its own retrieval and corrects course? Two named patterns:

  • CRAG (Corrective RAG) adds a lightweight grader that scores each retrieved chunk's relevance. Confident hits go straight to generation; clearly irrelevant chunks are dropped; and if retrieval looks weak overall, it falls back, typically a web or broader search, rather than answering from junk. The point is to never feed the generator irrelevant context, because grounded generation on bad chunks produces confident, wrong, well-cited answers.
  • Self-RAG trains the model to interleave special tokens that decide whether to retrieve, then critique whether each retrieved passage supports the answer and whether the answer is actually grounded in it. Retrieval and self-assessment become part of generation itself.

You will rarely implement self-RAG from scratch, but its idea, grade the chunks, drop the irrelevant, fall back when retrieval fails, is a pattern you can add to any pipeline with one extra grading call, and it connects directly to Module 5's evaluation work: a grader is just an inline, per-request faithfulness check.

Worked example: route one question through the right transform

Example

Question: For customers who upgraded to Enterprise after we shipped SSO, is the webhook retry limit different from the Pro plan?

A cheap classifier flags this as multi-hop, so the system decomposes it.

decomposed → (1) "When was SSO shipped?" (2) "Which plans are Enterprise tier?" (3) "Webhook retry limit on Enterprise plan" (4) "Webhook retry limit on Pro plan" retrieve each (hybrid + rerank) → dedupe → generate over the union sub-q (3) phrasing missed → step-back fallback: "how does webhook retry work across plans?" → lands the comparison table answer: grounded in the retry-limit table chunk + the SSO ship-date chunk

No single embedding of the original sentence would have cleanly retrieved both the SSO ship date and the per-plan retry table, they live in different chunks. Decomposition fetched each fact; when sub-question 3's literal phrasing missed, a step-back to the broader "retry across plans" query recovered the comparison table. Contrast this with "what is the Pro plan's webhook retry limit", a single-hop, specific question where you would skip all of it and retrieve the raw query directly. The skill is matching the transform to the question, not applying transforms reflexively.

Problem 1

HyDE retrieves using a hypothetical answer the LLM made up, which might contain false facts. Why is that not a grounding/hallucination risk, and where could HyDE genuinely go wrong?

Show solution

The hypothetical document is never shown to the user and never enters the final answer, it is used only as a search vector, so its made-up specifics cannot leak into the response; the actual retrieved chunk is what grounds generation. HyDE's real failure is retrieval drift: if the hypothetical answer wanders into a topic or framing the corpus does not contain, its embedding pulls retrieval toward the wrong region and you fetch confidently-irrelevant chunks. It helps most on vague or jargon-light questions where the user's phrasing does not match the docs, and least when the question is already precise or the corpus is narrow.

Problem 2

Your team enables multi-query expansion globally and average answer quality drops on a chunk of queries even though recall went up. What is happening, and how do you make the technique pay off?

Show solution

Applying multi-query to every query hurts the precise, well-specified ones: paraphrasing an exact endpoint name or identifier produces variants that match the wrong chunks, so you inject off-topic context and the generator's quality falls even as raw recall rises. The fix is to make transformation conditional: gate it behind a cheap classifier or heuristic that detects ambiguous or vocabulary-mismatched questions, and pass specific, well-formed queries through untouched. Fusing the multi-query results with RRF and then reranking also limits the damage by demoting chunks that only one drifted paraphrase surfaced.

Problem 3

Sketch how you would turn a fixed "decompose-then-retrieve" pipeline into an agentic-RAG version, and name what you give up by doing so.

Show solution

Expose retrieval as a tool the model can call with a crafted query, and let the model run a loop: read the question, decide whether and how to query (it can decompose, broaden via step-back, or skip retrieval), inspect the returned chunks, and either answer or call retrieve again with a refined query, optionally adding a CRAG-style grader that drops irrelevant chunks and triggers a fallback search. What you give up is the predictability of the fixed pipeline: you inherit non-determinism, higher and more variable latency from multiple LLM-plus-retrieval round trips, and the need for tracing and trajectory evaluation to know the agent actually queried sensibly, which is exactly the Module 4 and Module 5 cost. The upside is handling messy, multi-hop, or unexpected questions a static pipeline cannot.

Today's takeaways

The user's literal question is often a poor search query, so transform it first: decomposition for multi-hop, step-back for over-specific, HyDE to close the question-versus-passage gap, multi-query for vocabulary mismatch. Every transform costs an LLM call and can hurt precise queries, so gate them, do not apply reflexively. Agentic RAG is the generalization: retrieval becomes a tool the model chooses, crafts arguments for, and iterates on, and CRAG/self-RAG add a grading step that drops irrelevant chunks and falls back when retrieval fails. Match the technique to the question; that judgment is the senior signal.
