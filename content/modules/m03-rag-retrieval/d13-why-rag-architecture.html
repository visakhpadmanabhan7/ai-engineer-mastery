Why RAG & the full architecture

RAG is the single most common pattern you will build as an AI engineer, and the one interviewers probe hardest. Today you get the why, the full six-stage pipeline, and a map of where every later lesson in this module plugs in.

By the end you can
  • Say in two sentences why RAG beats fine-tuning for knowledge, and when it does not.
  • Draw the six-stage pipeline (ingest, chunk, embed, index, retrieve, generate) and split it into offline and online halves.
  • Name which stage each of the next eight lessons sharpens.
  • Distinguish vanilla, agentic, and hierarchical RAG at a glance.
  • Explain why roughly 70 to 75 percent of bad RAG answers are retrieval failures, not generation failures.

The problem RAG solves

From Module 1 you know an LLM's knowledge is parametric: lossy, frozen at the training cutoff, and blended with everything else it ever read. That is fine for general language, useless for "what does our refund policy say as of this morning." Two facts make this unavoidable: the model cannot have seen your private documents, and even public facts go stale the instant training stops.

Retrieval-Augmented Generation (RAG) fixes this without touching the weights. At query time you fetch the relevant passages from an external store and place them in the context window, then ask the model to answer using them. The model's job shifts from "recall this fact" to "read these passages and synthesize an answer," which is exactly what LLMs are good at.

Interview-grade phrasing

"RAG retrieves relevant context at query time and injects it into the prompt, so the model answers from supplied evidence instead of frozen parametric memory. You reach for it whenever knowledge is private, changing, or needs to be cited." Say that and the interviewer knows you have shipped one.

1 · Why RAG instead of fine-tuning

The most common interview trap is conflating the two. They solve different problems. Fine-tuning changes how the model behaves (form, style, format, tone); RAG changes what it knows at answer time. For injecting knowledge, RAG wins on three axes:

  • Dynamic knowledge. Update a database row and the next answer reflects it. Fine-tuning bakes facts into weights, so any change means re-running a training job. New document on Monday should not require a GPU on Tuesday.
  • Provenance and citations. Because you know exactly which passages you retrieved, you can show the source. A fine-tuned model cannot tell you which training example produced a claim, which is fatal in legal, medical, finance, and any audited setting.
  • No retrain, lower cost and risk. No training pipeline, no catastrophic forgetting, no eval-the-whole-model after every doc change. You add, edit, and delete knowledge with ordinary CRUD.

Fine-tuning still wins for behavior: a fixed output format, a house style, a narrow classification, or shaving latency and cost by teaching a small model one job. The senior answer is "RAG for knowledge, fine-tuning for form, and they compose." You will go deep on fine-tuning in Module 6.

QuestionReach for
Facts change, or are private to us?RAG (knowledge in a store, not weights)
Need to cite the source of every claim?RAG (you keep the retrieved passage)
Want a fixed JSON shape or house tone always?Fine-tuning (behavior, not facts)
Cut cost by teaching a small model one task?Fine-tuning (Module 6)
Both fresh knowledge and a strict format?Both (fine-tune the form, retrieve the facts)

2 · The six-stage pipeline

Every RAG system, however fancy, is a refinement of these six stages. The crucial split: the first four run offline (once per document, ahead of time) and the last two run online (per user query, on the hot path). Confusing the two is a classic mistake; embedding your whole corpus on every request would be absurd.

════════════ INGESTION (OFFLINE · once per document) ════════════ raw sources PDFs · web · Notion · DB rows · tickets │ ① INGEST / PARSE load + extract clean text (layout, tables) ← Day 14 (parsing) │ ② CHUNK split into retrievable units + overlap ← Day 14 │ ③ EMBED each chunk → fixed-dim vector ← Day 15 │ ④ INDEX store vectors (+ text + metadata) in an ANN DB ← Day 15 (pgvector) │ ╰───────────────────────► [ VECTOR + KEYWORD STORE ] ◄───────────────╮ │ ════════════ RETRIEVAL (ONLINE · per user query) ════════════ │ user query │ │ │ ⑤ RETRIEVE embed query · BM25 lexical · fuse · rerank · top-k ──────╯ │ (Day 15 dense · Day 16 BM25 · Day 17 fusion · Day 18 rerank) │ ⑥ GENERATE stuff top-k chunks into the prompt → LLM → grounded, cited answer │ answer + citations (eval: Day 19-21)

Read the diagram as a contract. Ingestion builds the store; do it well once and the rest gets easy. Retrieval queries the store; do it badly and the best model in the world produces garbage because it never sees the right evidence. Generation is the last and usually least fragile step, a point we return to below.

3 · Where every later lesson plugs in

This module is the pipeline, lesson by lesson. Keep this map; each later day is a deep dive into one stage:

  • Day 14 — Chunking & parsing (stages ①②). The highest-leverage decision in the whole system. How you split documents largely sets the ceiling on quality.
  • Day 15 — Embeddings & vector search, pgvector (stages ③④). Turning chunks into vectors and finding nearest neighbors fast.
  • Day 16 — BM25 & lexical search (stage ⑤, lexical half). The classic keyword retriever that still beats dense models on exact matches.
  • Day 17 — Hybrid search & fusion (stage ⑤, fuse). Run BM25 and dense together and merge the rankings.
  • Day 18 — Rerankers (stage ⑤, refine). A precision pass over the top candidates before they reach the model.
  • Days 19 to 21 — Evaluation, query transforms, advanced RAG. Measuring retrieval and generation, rewriting queries, and patterns like GraphRAG and agentic RAG.

Notice how much of the module lives inside stage ⑤. That is not an accident, and it leads to the most important rule in RAG.

Concept

RAG quality is dominated by retrieval, not generation. Across production systems, on the order of 70 to 75 percent of wrong answers trace to retrieval: bad chunking, weak embeddings, no reranker, or the wrong top-k. The model can only be as good as the passages you hand it. "Garbage in, confident garbage out."

4 · Vanilla vs agentic vs hierarchical RAG

"RAG" is a family, not one design. You will meet all three; know the difference and, more importantly, when each earns its complexity. Start vanilla, and add structure only when a measured failure demands it.

FlavorHow it worksUse when
Vanilla
retrieve then generate		One fixed pass: embed query, retrieve top-k, generate. No loops, no decisions.The default. Most questions are answerable from a single retrieval. Always start here.
Agentic
retrieval as a tool		The LLM decides whether and how to retrieve, crafts the query, can call retrieval multiple times, and reason across results (Module 4).Multi-hop questions, ambiguous queries, or when the model should choose among several sources.
Hierarchical
multi-level structure		Index at multiple granularities (summaries over sections over chunks) or build a tree of summaries (RAPTOR-style) and route top-down.Large corpora needing both global synthesis and local detail; "summarize across 200 docs."

You will build vanilla first because it is debuggable and frequently good enough. Agentic and hierarchical RAG arrive in Day 21 and Module 4, once you can measure whether the extra moving parts actually helped. Complexity you cannot measure is complexity you cannot defend.

Worked example: trace one question through the pipeline

Example

A support assistant over a product knowledge base. A user asks: Does the Pro plan support SSO, and from which version? Watch the two halves run.

OFFLINE (already done, last night's ingestion): docs.pdf → parsed → chunked into ~400-token pieces with 50-token overlap chunk #418 "...Single Sign-On (SSO) is available on Pro and Enterprise plans starting in release 3.2..." → embedded → stored (id, vec, text, {plan:pro}) ONLINE (this request): ① embed the query → q-vector ② dense search → chunk #418 (cos 0.83), #073 (0.79), #511 (0.74) ... BM25 on "SSO" "Pro" "version" → also surfaces #418 strongly (exact tokens) ③ fuse the two ranked lists (RRF) → #418 rises to rank 1 ④ rerank top 50 with a cross-encoder → #418 confirmed most relevant ⑤ top-3 chunks → prompt → LLM ⑥ answer: "Yes. SSO is supported on the Pro plan starting in release 3.2." + citation → chunk #418

Two things to internalize. First, the answer is grounded: every claim came from a retrieved passage, so it is citable and checkable. Second, the model did almost nothing clever, the win was getting chunk #418 to the top. That is why the rest of this module obsesses over retrieval. If chunk #418 had been split across two chunks, or buried at rank 40, the model would have guessed, and "starting in release 3.2" is exactly the kind of specific detail it would hallucinate.

Why this matters for you as an AI engineer

Three consequences, derived not memorized

Knowledge belongs in a store, not weights ← parametric memory is lossy and frozen, so you add, edit, and delete facts with CRUD and never retrain to update a policy.
Debug retrieval before generation ← ~70 to 75 percent of bad answers are retrieval failures, so when an answer is wrong, first ask "did the right chunk get retrieved at all?"
Start vanilla, earn complexity ← agentic and hierarchical RAG add latency, cost, and failure surface, so add them only when an evaluation shows vanilla falling short.

Practice — do these before revealing

Say each answer out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

A product manager says: "Our docs change weekly. Let's just fine-tune the model on them every Friday so it always knows the latest." Give the two-sentence pushback and the better design.

Show solution

Fine-tuning bakes facts into weights, so weekly retraining is slow, expensive, risks catastrophic forgetting, and still cannot cite which source a claim came from. RAG solves the same need by retrieving the current passage at query time: a doc change is a database upsert, not a training run, and every answer is grounded and citable. Senior add-on: "we can re-embed only the changed documents in seconds, and fine-tune separately, just for output format if we need it."

Problem 2

A RAG assistant gives a confidently wrong answer. You have a fixed budget to investigate one stage first. Which stage, and what is the single fastest check?

Show solution

Investigate retrieval first, because roughly 70 to 75 percent of wrong RAG answers are retrieval failures, not generation failures. The fastest check is to log the chunks that were actually retrieved for that query and read them: if the passage that contains the correct answer is not in the retrieved set (or is buried below the top-k), the problem is upstream (chunking, embeddings, top-k, or no reranker), and no prompt tweak will fix it. Only if the correct passage was retrieved and the model still answered wrong do you move on to suspect generation or the prompt.

Problem 3

Sort these four operations into offline vs online, and say why putting any one of them on the wrong side would hurt: (a) embedding the corpus, (b) embedding the user query, (c) building the ANN index, (d) reranking the top candidates.

Show solution

Offline: (a) embedding the corpus and (c) building the ANN index, both done once per document at ingestion. Doing them per request would re-encode and re-index the entire knowledge base on every query, which is wildly expensive and slow. Online: (b) embedding the user query and (d) reranking the top candidates, because the query is unknown until the user arrives and reranking operates on the candidates that query retrieved. The mental model: build the store offline, query the store online. The only embedding on the hot path is the single short query, everything heavy was precomputed.

Interview questions for this topic

From your Notion bank's RAG & Retrieval slice and common 2026 screens. Drill the one-line keys, then expand each into a 90-second answer with Claude using protocol prompt #2.

  • What is RAG and why use it instead of fine-tuning?
    Key: retrieves context at query time; wins on dynamic knowledge, provenance/citations, no retrain. Fine-tune for form, not facts.
  • Walk through a vanilla RAG pipeline end to end.
    Key: ingest → chunk → embed → index → retrieve → generate; first four offline, last two online.
  • What is the most common RAG failure mode?
    Key: retrieval (~70 to 75 percent of bad answers): bad chunking, bad embeddings, no reranker, wrong top-k.
  • What is agentic RAG versus vanilla RAG?
    Key: vanilla = fixed retrieve-then-generate; agentic = retrieval is a tool the agent decides to call, with crafted args, and can iterate.
  • Which RAG stages run offline and which run online, and why does it matter?
    Key: ingest/chunk/embed/index offline (once per doc); retrieve/generate online (per query). Never re-embed the corpus per request.
  • When would you NOT use RAG?
    Key: when the need is behavior/format/latency (use fine-tuning), or the knowledge already fits cheaply in context, or there is no external corpus to retrieve from.
  • How does RAG help with hallucination and auditability?
    Key: grounds answers in retrieved passages you can cite; checkable provenance and an evaluable faithfulness signal, instead of opaque parametric recall.

Today's takeaways

RAG injects retrieved context at query time so the model answers from evidence, not frozen memory, and it beats fine-tuning for knowledge on dynamic facts, citations, and cost. The pipeline is six stages: ingest, chunk, embed, index (offline) then retrieve, generate (online). Quality is dominated by retrieval, so roughly 70 to 75 percent of bad answers are retrieval failures and you debug there first. Start with vanilla RAG and add agentic or hierarchical structure only when an evaluation proves you need it. This map is the spine of the entire module.
