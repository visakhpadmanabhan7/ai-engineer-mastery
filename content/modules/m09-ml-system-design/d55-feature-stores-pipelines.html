Feature stores & training pipelines

Models get the glory, but features decide whether the system works. Today is about the unglamorous infrastructure that keeps the features your model trained on identical to the features it serves on, because the gap between those two is the single most common silent ML bug in production.

By the end you can
  • Explain why features, not models, are usually the bottleneck in ML systems.
  • State the two jobs a feature store does: consistency and point-in-time correctness.
  • Define training-serving skew, give three concrete causes, and prevent each.
  • Choose between batch, streaming, and on-demand feature computation.
  • Explain point-in-time joins and why a naive join leaks the future into training.

1 · Features are the heart of the system

A model is a function from features to a prediction. Swap a mediocre model for a great one and you might gain a few points; feed either one better features and you can move the needle far more. In practice most of the engineering effort, and most of the bugs, live in the feature layer, not the model layer. So in a system design interview, when you reach stage 3 of the framework, do not wave it away with "and then I compute some features." That is where a senior engineer slows down.

The hard part is not computing a feature once. It is computing the same feature, the same way, in two very different worlds: an offline world where you process millions of rows in a batch job to build a training set, and an online world where you must produce that feature for one user in a few milliseconds at request time. Keeping those two worlds in agreement is the entire problem.

2 · The two worlds, and the store between them

A feature store is the system that sits between feature producers and both consumers (training and serving), so that a feature is defined once and read consistently in both places.

RAW SOURCES event logs · transactional DB · streams · 3rd-party APIs │ ▼ ┌─────────────────── FEATURE PIPELINES ───────────────────┐ │ batch job (nightly Spark) · streaming job (Flink) │ │ ── one transform definition, reused on both sides ── │ └───────────────┬───────────────────────────┬─────────────┘ │ │ OFFLINE PATH ONLINE PATH ▼ ▼ ┌──────────────────┐ ┌──────────────────┐ │ OFFLINE STORE │ │ ONLINE STORE │ │ (warehouse / │ │ (Redis / KV, │ │ parquet, big, │ │ low-latency, │ │ historical) │ │ latest value) │ └────────┬─────────┘ └────────┬─────────┘ │ │ ▼ point-in-time join ▼ key lookup, <10 ms ┌───────────────┐ ┌───────────────┐ │ TRAINING SET │ (x, y) pairs │ SERVING │ │ build → train│ │ fetch features│ └───────────────┘ │ → model → pred│ └───────────────┘ ▲ │ └───── SAME feature definition ──────┘ (this equality is the whole point)

Two stores, one definition. The offline store is big, historical, and read in bulk to build training sets. The online store is a low-latency key-value store (Redis is the classic choice) holding the latest feature value per entity, read by key in single-digit milliseconds at request time. The feature store's promise is that the value you trained on and the value you serve on were produced by the same transform, so the model sees in production exactly the kind of input it learned from.

Concept

A feature store solves two problems. First, consistency: the offline (training) and online (serving) features come from one definition, which prevents training-serving skew. Second, point-in-time correctness: when building a training row for an event at time t, it joins only feature values that were known at t, never later, so you do not leak the future into the past.

3 · Training-serving skew: the #1 silent bug

Training-serving skew is when the features (or the transforms that produce them) differ between training and serving, so the model receives subtly different inputs in production than it learned from. It is dangerous precisely because nothing crashes. The pipeline runs, the API returns 200s, and the model just quietly performs worse than your offline numbers promised. Three common causes:

  • Two implementations of one transform. The classic. A data scientist writes the training features in pandas; an engineer reimplements them in Java for the serving path. One uses mean to fill missing values, the other uses zero. One normalizes by the training-set mean, the other by a hardcoded constant. The model sees different distributions and degrades.
  • Different data freshness. Training uses a feature aggregated over a full clean day; serving reads a value that is several hours stale because the online store has not refreshed. "Average spend last 7 days" means something different if one side is current and the other lags.
  • Time leakage at training time. You accidentally compute a feature using data from after the prediction timestamp (e.g. "total order value" that includes orders placed after the moment you were predicting). Offline it looks brilliant; in production that future information does not exist, so the model collapses.
Watch out

Skew does not raise an exception. Your offline AUC says 0.91, production behaves like 0.78, and the logs are clean. If a model "tested great but underperforms live," suspect training-serving skew before you suspect the model. The fix is structural, not a tuning knob.

4 · Preventing skew: share the transform, log what you served

There are two structural defenses, and a strong answer names both:

  • Share the transform. Define each feature exactly once, in one piece of code, and call that same code in both the batch pipeline and the serving path. A feature store enforces this; without one you can still do it by extracting transforms into a shared library that both paths import. The rule: never let two implementations of one feature exist.
  • Log the features you actually served. At inference time, write the exact feature vector you fed the model (alongside the prediction and a request id). This does two things: it lets you train on real served features next time (closing the loop so train and serve are identical by construction), and it lets you compare served-versus-recomputed values to detect skew when it sneaks in. Logging served features is one of the highest-value, lowest-effort moves in production ML.

5 · Batch vs streaming vs on-demand features

Not every feature is computed the same way. Match the computation style to how fast the feature must reflect reality:

StyleFreshnessExampleHow
BatchHours / daily"User's avg order value over 90 days"Scheduled job (Spark) writes to offline + online store overnight.
StreamingSeconds"Clicks in the last 5 minutes"Stream processor (Flink/Kafka) updates the online store in near real time.
On-demandRequest time"Distance between user and this restaurant right now"Computed live from request inputs; cannot be precomputed (depends on the query).

Most real systems mix all three: slow-moving aggregates from a nightly batch job, fast-moving counts from a streaming job, and a few request-specific features computed on the fly. The interview signal is matching freshness to need, do not pay for a streaming pipeline to keep a 90-day average "fresh," and do not serve a 12-hour-stale value for "items viewed this session."

6 · Point-in-time joins and data versioning

To build a training row for a label that occurred at time t, you must attach the feature values as they were at t. A naive join ("join the user's current features to every historical label") attaches today's values to a year-old event, which leaks information the model could never have had. That is a point-in-time (or "as-of") join: for each label timestamp, take the latest feature value with an effective time at or before that timestamp. This is the single most error-prone step in building training data, and it is exactly what a feature store automates for you.

Data versioning closes the loop on reproducibility. A model artifact is only meaningful alongside the exact training data and feature definitions that produced it. Version your datasets and feature transforms (tools like DVC or LakeFS, or simply immutable, timestamped snapshots), so that when a model misbehaves six weeks later you can reconstruct precisely what it was trained on. "We cannot reproduce the training set" is a sentence you never want to say in an incident review.

Worked example: the skew that ate the fraud model

Example

A fraud team trains a model. A key feature is avg_txn_amount_30d, the user's average transaction amount over the last 30 days. Offline, computed in pandas over the warehouse, AUC is 0.93. In production, the serving service (written in Go) recomputes the same feature by querying recent transactions, but to stay fast it caps the lookback at 1000 most-recent rows and, for new users with no history, returns 0.0 instead of the training pipeline's global mean.

Result: for high-volume merchants the cap silently changes the average; for new users the model sees 0.0 where it learned to see the mean. Two different transforms, one feature name. Live AUC behaves like 0.84, fraud slips through, and the logs are spotless. The fix is not to retune, it is to delete the Go reimplementation and have both paths call one shared feature definition, then log the served feature vector so the next training set is built from exactly what production saw.

Practice — do these before revealing

Say each answer out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

A teammate proposes: "Let's not bother with a feature store. The data scientists write features in their training notebook, and we'll just have backend re-implement the important ones in the API." Name the bug class this guarantees and the two structural fixes that avoid it.

Show solution

It guarantees training-serving skew: two independent implementations of each feature will drift in missing-value handling, normalization constants, lookback windows, and rounding, so the model receives different inputs in production than it trained on, and nothing will error, the model just silently underperforms. Two structural fixes: (1) share the transform, define each feature once in shared code that both the batch pipeline and the serving path import, so there is literally one implementation; and (2) log the served feature vector at inference time, so you can train on real served features (closing the loop) and detect skew by comparing served versus recomputed values. A feature store packages both, but a shared transform library plus served-feature logging gets you most of the way without one.

Problem 2

You are building a training set to predict whether a user will upgrade. For each historical "did upgrade?" label you join the user's total_lifetime_spend from your current production table. Offline accuracy is suspiciously high. What did you do wrong, and how do you fix it?

Show solution

You leaked the future. total_lifetime_spend read from the current table includes spend that happened after the upgrade decision you are predicting, so the feature is partly a consequence of the label, not a cause, classic time leakage, and it inflates offline accuracy while being impossible to reproduce at serving time. The fix is a point-in-time (as-of) join: for each label at time t, attach the feature value as it was at or before t (the user's lifetime spend up to that moment), never the value as of today. This is exactly the correctness guarantee a feature store provides, and it usually drops your "suspiciously high" offline number down to something honest that actually holds in production.

Problem 3

For a food-delivery ranking model, you have three candidate features: (a) the restaurant's average rating over its lifetime, (b) how many times the restaurant has been ordered in the last 10 minutes, (c) the walking distance between this user's current location and this restaurant. Assign each to batch, streaming, or on-demand, and justify.

Show solution

(a) Batch, lifetime average rating moves slowly, recomputing it nightly is plenty fresh and cheap; a streaming pipeline here would be wasted money. (b) Streaming, "orders in the last 10 minutes" is a fast-moving signal (it captures a sudden surge or a kitchen going down) and must update in near real time, so a stream processor maintains it in the online store. (c) On-demand, distance depends on the user's current location, which only exists at request time, so it cannot be precomputed and must be calculated live from the request inputs. The point: match computation style to freshness need; most systems use all three at once.

Interview questions for this topic

From your Notion bank's Module 9 slice. Drill the one-line keys, then expand each into a 90-second answer with Claude using protocol prompt #2.

  • What is a feature store and what problem does it solve?
    Key: a system that defines features once and serves them consistently to both training (offline) and serving (online), preventing training-serving skew and guaranteeing point-in-time correctness.
  • What is training-serving skew and how do you prevent it?
    Key: features/transforms differ between train and serve (two implementations, freshness gaps, leakage); prevent by sharing one transform definition across both paths and logging the served feature vector.
  • What is point-in-time correctness and why does a naive join break it?
    Key: build each training row with feature values known at the label's timestamp; a naive join attaches today's values to old labels, leaking the future and inflating offline metrics.
  • Why are features often the bottleneck rather than the model?
    Key: most effort and bugs live in computing the same feature consistently across the offline and online worlds; better features usually beat a better model, and the feature layer is where skew and leakage hide.
  • When do you use batch, streaming, or on-demand feature computation?
    Key: match computation to freshness need, batch for slow aggregates (90-day average), streaming for fast counts (clicks last 5 min), on-demand for request-specific values (live distance).
  • Why log the features you served at inference time?
    Key: lets you train on exactly what production saw (closing the loop, eliminating skew by construction) and detect skew by comparing served vs recomputed values; cheap, high value.
  • How do you make a model's training reproducible six weeks later?
    Key: version the dataset and feature definitions (DVC/LakeFS or immutable timestamped snapshots) alongside the model artifact, so you can reconstruct the exact training set during an incident.

Build it (15–20 min)

Hands-on

Reproduce skew on your own machine to feel it. In a notebook, make a tiny dataset of users with a few transaction amounts. Write a "training" feature function avg_amount(df) that fills missing users with the global mean. Then write a separate "serving" feature function that fills missing users with 0.0 and caps the lookback at the 2 most recent rows. Compute both on the same users and print the differences, this is skew, made concrete in ten lines. Now refactor: extract one shared compute_avg_amount(rows) and call it from both, and watch the differences vanish. For the production version of this pattern, skim the Feast feature-store docs at docs.feast.dev to see how offline and online stores are wired in real life.

Connect it to your work

Your MaindTec story

You already run the "define the transform once, on both sides" discipline, just for an LLM stack. Your FastAPI ingestion service writes documents into pgvector, and the embedding step is effectively a feature pipeline: the text must be chunked and embedded with the same transform at ingestion (the "offline" path that fills the index) and at query time (the "online" path that embeds the user's question). If those two used different chunking or a different embedding model, your retrieval would silently degrade, the RAG analogue of training-serving skew. So when an interviewer asks about feature stores, you can ground it: "consistency between the offline-built index and the online query path is the same problem I solve in retrieval, where the ingestion and query embeddings have to come from one definition or recall quietly drops."

Today's takeaways

Features, not models, are usually the bottleneck, and the hard part is producing the same feature consistently in two worlds: bulk offline for training and low-latency online for serving. A feature store does two jobs, consistency (one definition feeds both sides) and point-in-time correctness (join only what was known at the label's timestamp). Training-serving skew is the #1 silent bug, it never crashes, it just makes production worse than your offline numbers, and you prevent it by sharing one transform and logging the features you served. Match feature computation to freshness need (batch / streaming / on-demand), and version your data so a model's training is always reproducible.
