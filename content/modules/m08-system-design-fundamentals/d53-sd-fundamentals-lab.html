Lab — design a rate limiter

The distributed rate limiter is the classic system design warm-up: small enough to finish in one interview, deep enough to test algorithms, distributed coordination, and failure thinking. Today you drive the full Module 8 methodology end to end on it, and walk away with a reference design and working token-bucket code.

By the end you can
  • Run the seven-step framework from Day 48 on a concrete problem, start to finish.
  • Choose between token bucket and sliding window and justify it.
  • Explain why the limiter check must be atomic, and implement it as a single Redis operation.
  • Reason about distributed coordination: shared state vs per-node limits and their trade-offs.
  • Decide fail-open vs fail-closed when the limiter's own backing store is down.
  • Sketch a token bucket in Python and read the Lua-style atomic check-and-decrement idea.
How to run this lab

Spend the first 35 minutes attempting the design yourself on a blank page using the seven-heading template from Day 48 (Clarify, Estimate, Architecture, Data model, Deep-dive, Trade-offs, Ops), before reading the reference. Then compare against the worked design below and close the gaps. Finish with the code sketch and the review questions. Resist scrolling to the reference first, the value is in the gap between your design and this one.

The prompt

"Design a rate limiter for our public API. It must cap how many requests a client can make in a time window, work across our fleet of many API servers, add negligible latency, and degrade gracefully if its own infrastructure has problems." That is the whole prompt, deliberately open. Your job is to drive it.

Step 1 · Clarify

Pin scope and the non-functional axes out loud before designing anything.

  • Granularity: what is the limit keyed on? Per-user (API key), per-IP, per-endpoint, or a combination? Assume per-API-key, with the ability to also limit per-endpoint (a tier you can layer).
  • The limit: e.g. 100 requests per minute per key, with a burst allowance. Confirm whether short bursts above the steady rate are acceptable (they usually are).
  • Scope: distributed across many stateless API servers (from Day 49), so the count cannot live in one server's memory, two servers must share a view of a key's usage.
  • Latency: the limiter is on the hot path of every request, so its overhead must be ~1ms or less. This pushes the check into a fast in-memory store and forces it to be a single round trip.
  • Behavior on limit: reject over-limit requests with HTTP 429 and a Retry-After header. Confirm whether to hard-reject or queue (reject is standard for an API limiter).
  • Failure behavior: if the limiter's backing store is unavailable, do we fail-open (allow traffic, protect availability) or fail-closed (block, protect the backend)? This is a real decision, name it now and resolve it in the trade-offs.

Step 2 · Estimate

Size it so the storage and throughput choices are grounded. Say the API serves 50,000 requests per second at peak across the fleet, with, say, 1 million distinct active API keys in a window. The limiter must therefore do ~50,000 check-and-update operations per second, each in about a millisecond, and hold counter state for ~1M keys. Per-key state is tiny (a count and a timestamp, tens of bytes), so 1M keys is on the order of tens of megabytes, trivially in-memory. The throughput number is the binding constraint: 50k ops/s of atomic counter updates is exactly what an in-memory store like Redis is built for, and it tells you a relational database on the hot path would be a mistake.

Step 3 · Choose the algorithm

This is the deep-dive the problem is really about. Several algorithms exist; you should know the trade-offs and pick one with a reason.

AlgorithmHow it worksTrade-off
Fixed windowCount requests per fixed clock window (e.g. each minute); reset at the boundary.Simple and cheap, but allows a 2x burst straddling the boundary (full quota at :59 and again at :00).
Sliding window logStore a timestamp per request; count those within the trailing window.Exact and smooth, but stores every request timestamp, so memory grows with traffic. Expensive at scale.
Sliding window counterWeighted blend of the current and previous fixed window's counts.Approximates the sliding log cheaply (two counters), smooths the boundary burst. A common production pick.
Token bucketA bucket refills at rate r tokens/sec up to capacity C; each request spends a token, reject if empty.Allows controlled bursts up to C, smooth average rate, tiny state (tokens + last-refill timestamp). The default.
Leaky bucketRequests enter a queue drained at a fixed rate; overflow is dropped.Enforces a perfectly smooth output rate, but adds queueing/latency and no burst allowance.

Pick token bucket for this API. It allows the controlled bursts real clients need (a client that was idle can briefly exceed the steady rate, which is friendly and realistic), it enforces the correct long-run average, and its state is just two numbers per key, the token count and the last-refill time, which is cheap to store and cheap to update atomically. The sliding window counter is the strong runner-up if you want to forbid bursts and keep it smooth; mention it as the alternative and say what each costs (token bucket allows bursts up to C; sliding window counter smooths them away).

Interview-grade phrasing

"I would use a token bucket: it allows controlled bursts up to the bucket capacity, enforces the right average rate via the refill rate, and needs only two values per key, tokens and last-refill timestamp. If the requirement is to forbid bursts entirely, I would switch to a sliding window counter, which approximates an exact sliding window with two counters and smooths the boundary burst that fixed window suffers." Naming the alternative and its cost is the senior move.

Step 4 · Architecture and the atomicity problem

Because the fleet is stateless and many servers serve one key's traffic, the bucket state must live in a shared store every server reaches. Redis is the natural choice: in-memory (sub-millisecond), supports atomic operations, and is already in the stack. Each API server, on each request, asks Redis "does this key have a token? if so, take one," and allows or rejects based on the answer.

The subtle, interview-critical part is atomicity. The check ("are there tokens?") and the update ("decrement") must happen as one indivisible operation, because if they are two separate round trips, two concurrent requests can both read "1 token left," both decide to allow, and both decrement, letting through more than the limit. That is a classic read-modify-write race, and under 50k ops/s it will happen constantly. The fix is to make the whole refill-check-decrement a single atomic step in Redis, either a Lua script (Redis runs it atomically server-side, no interleaving) or Redis's built-in atomic commands. The principle is identical to "keep transactions short and avoid races" from Day 50: the limiter's correctness rests entirely on the check-and-decrement being atomic.

request ─▶ API server (one of many, stateless) │ key = api_key ▼ Redis: ATOMIC refill + check + decrement ← ONE Lua script / atomic op │ (never two round trips — that race-conditions the count) ┌───────┴────────┐ tokens ≥ 1? tokens = 0? │ │ allow reject 429 + Retry-After state per key: { tokens, last_refill_ts } ~tens of bytes, ~1M keys → tens of MB

Step 5 · Distributed coordination

The shared-Redis design above is the clean answer: one source of truth, exact global limit, every server consistent. Its costs are a network hop per request (kept tiny by Redis being in-memory and the op being single round trip) and Redis being a critical dependency (which feeds the failure discussion). Make Redis itself redundant (replicated, multi-AZ) so it is not a single point of failure.

Know the alternative because interviewers probe it: local (per-node) counters, where each server enforces limit÷N locally with no shared store. This is faster (no network hop) and has no shared dependency, but it is only approximate, because the load balancer does not split a key's traffic perfectly evenly across nodes, so the effective global limit drifts, and it breaks down as nodes are added or removed. The middle ground is a local token bucket with periodic async reconciliation against a shared store, trading exactness for speed. State the trade clearly: centralized Redis gives an exact limit at the cost of a hop and a dependency; per-node gives speed and independence at the cost of an approximate, drifting global limit. For a public API where the limit is a contract, exactness usually wins, so centralized Redis.

Step 6 · Failure modes: fail-open vs fail-closed

The decision you flagged in step 1 must now be resolved: what happens when Redis (the limiter's backing store) is down or unreachable? There is no universally correct answer, only a trade you must name.

  • Fail-open (allow requests when the limiter cannot decide): protects availability, your API keeps serving even if the limiter is down, but it removes the protection, so a traffic surge during the outage can overwhelm the backend. Choose fail-open when the limiter exists mainly for fairness/abuse-prevention and the backend can tolerate a temporary unguarded burst.
  • Fail-closed (reject requests when the limiter cannot decide): protects the backend, but it converts a limiter outage into an API outage, which is often a worse user-facing failure than the thing you were guarding against. Choose fail-closed when the limit is protecting something fragile or expensive that absolutely must not be overrun.

The common production stance is fail-open with a local fallback: if shared Redis is unreachable, fall back to a coarse per-node local limit so you are not totally unprotected, and keep serving. That preserves availability while still applying a backstop. Whichever you choose, say why and tie it to what the limiter is actually protecting, that reasoning is the point of the question. Pair this with the resilience patterns you will see in LLMOps (circuit breakers, retries with backoff) so a flapping Redis does not amplify into cascading failure.

Step 7 · Wrap: ops

Close on operations, as always. Observability: emit metrics for allow vs reject rate per key and overall, current Redis latency on the limiter op, and an alert if reject rate spikes (possible attack or misconfigured limit) or if the limiter is failing over to local fallback. Config: limits should be data, not code, stored so they can change per key/tier without a deploy, and surfaced to clients via response headers (X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After) so well-behaved clients self-throttle. Rollout: ship in "monitor-only" mode first (count what would be rejected without rejecting), verify the limits are sane against real traffic, then enforce, the canary discipline from Day 48. Cost: dominated by the Redis op volume, which the estimate already sized at ~50k ops/s, comfortably within a small replicated Redis.

Show solution — token bucket sketch (Python) + the atomic Redis idea

A single-process token bucket. The key trick is lazy refill: you do not run a timer adding tokens, you compute how many tokens should have accrued since the last check, based on elapsed time and the refill rate, capped at capacity.

import time

class TokenBucket:
    def __init__(self, rate, capacity):
        self.rate = rate            # tokens added per second
        self.capacity = capacity    # max tokens (the burst allowance)
        self.tokens = capacity      # start full
        self.last = time.monotonic()

    def allow(self, cost=1):
        now = time.monotonic()
        elapsed = now - self.last
        # lazy refill: accrue tokens for the time that passed, cap at capacity
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last = now
        if self.tokens >= cost:
            self.tokens -= cost
            return True          # allow, spend a token
        return False             # reject → caller returns 429

# bucket = TokenBucket(rate=100/60, capacity=100)  # ~100/min, burst 100
# if not bucket.allow(): return Response(status=429, headers={"Retry-After": "1"})

This is correct for one process, but in the distributed design the state lives in Redis and the read-modify-write must be atomic, otherwise two servers both read the same token count and both allow. You make the refill-check-decrement one indivisible server-side step with a Lua script (Redis executes a script atomically, nothing interleaves):

-- KEYS[1] = bucket key; ARGV: rate, capacity, now, cost
-- stored hash: { tokens, last_refill_ts }; returns 1 (allow) or 0 (reject)
local rate     = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now      = tonumber(ARGV[3])
local cost     = tonumber(ARGV[4])

local b = redis.call('HMGET', KEYS[1], 'tokens', 'ts')
local tokens = tonumber(b[1]) or capacity
local ts     = tonumber(b[2]) or now

tokens = math.min(capacity, tokens + (now - ts) * rate)  -- lazy refill
local allowed = 0
if tokens >= cost then
  tokens = tokens - cost
  allowed = 1
end
redis.call('HMSET', KEYS[1], 'tokens', tokens, 'ts', now)
redis.call('EXPIRE', KEYS[1], 3600)   -- evict idle keys
return allowed

The Python version teaches the algorithm; the Lua version is what makes it correct in production. Same logic, but the script's atomicity is the whole reason it holds the global limit under 50k concurrent ops/s. (Numbers and the script structure are illustrative, not a copy of any specific API's parameters.)

Stretch variations

Once the base design is solid, interviewers extend it. Have a one-line answer for each:

  • Tiered limits (free vs paid keys): limits are config keyed by tier, the algorithm is unchanged, you just look up rate and capacity per key.
  • Limit by cost, not count (an expensive endpoint costs more tokens): spend cost > 1 tokens per request, which the bucket already supports, this is exactly how you would rate-limit an LLM API by token spend rather than request count.
  • Multiple limits at once (per-second and per-day): run independent buckets and require all to allow.
  • Global limit across regions: exact global limits across regions are expensive (cross-region coordination latency), so usually you accept per-region limits or eventual reconciliation, and you name that trade.

Module 8 review — the questions to have cold

A quick self-test across the whole module. Say each key out loud; if any feels shaky, reopen that day before moving to ML system design.

  • Walk through your system design methodology. (Day 48)
    Key: clarify (functional + non-functional) → estimate (QPS, storage) → high-level arch → data model → deep-dive the bottleneck → trade-offs (name what you give up) → ops (rollout, observability, cost).
  • Vertical vs horizontal scaling, and why statelessness? (Day 49)
    Key: vertical = bigger box (ceiling + SPOF); horizontal = more boxes behind a load balancer; stateless app servers so any node serves any request, state in DB/cache.
  • SQL vs NoSQL default, and what breaks when you shard? (Day 50)
    Key: default Postgres (relational + JSONB + FTS + pgvector); sharding by key spreads writes but cross-shard joins/transactions and hot keys hurt, pick the shard key carefully.
  • Cache strategies, and the two hard problems? (Day 51)
    Key: cache-aside (default) vs write-through/back, LRU + TTL; invalidation is hard (freshness vs hit ratio) and the stampede floods the DB, fix with single-flight + jittered TTL.
  • When do you go async, and queue vs pub/sub vs log? (Day 52)
    Key: decouple, smooth spikes, background work; queue = one consumer per message, pub/sub = all subscribers, log (Kafka) = durable/ordered/replayable; idempotent consumers + backpressure + outbox.
  • Design a rate limiter. (Day 53)
    Key: token bucket (or sliding window), shared Redis with an atomic check-decrement (Lua), per-user + global, distributed coordination, fail-open vs fail-closed on store failure.
  • Explain CAP/PACELC in practice. (Day 50)
    Key: under a partition choose consistency or availability; PACELC adds that even without a partition you trade latency vs consistency.
  • How do you name a trade-off like a senior? (Day 48)
    Key: "I chose X, which buys A but costs B, acceptable here because requirement R", never present a design as all upside.

Build it (the lab deliverable)

Hands-on

Make the limiter real against Redis. Implement the token-bucket logic as a Lua script and load it with SCRIPT LOAD / EVALSHA via redis-py, then write a tiny client loop that fires, say, 200 requests as fast as it can against a key configured for 100/min with a burst of 100, and confirm roughly the first 100 are allowed and the rest get rejected, then watch tokens trickle back as time passes. Then prove the race matters: implement a deliberately non-atomic version (separate GET then SET) and hammer it from several threads, you will see it allow more than the limit. Switch to the atomic Lua version and watch the over-allow vanish. That contrast is the whole lesson. Docs: Redis scripting (EVAL) and redis-py. Today's takeaways

You drove the full seven-step framework on a concrete problem and produced a defensible reference design. Token bucket is the default rate-limiting algorithm because it allows controlled bursts, enforces the right average, and needs only two values per key. In a distributed fleet the bucket state lives in a shared store (Redis) and the check-and-decrement must be atomic, a single Lua script, or concurrent requests race and over-allow. Distributed coordination is a trade between an exact centralized limit and a fast-but-approximate per-node one, and the failure question, fail-open vs fail-closed, is answered by what the limiter is actually protecting (commonly fail-open with a local backstop).
