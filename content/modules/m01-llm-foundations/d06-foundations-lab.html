Lab — inspect a real model

Five days of mechanism. Today you make it physical. You will call a real model and watch temperature change its behavior, split text with a real tokenizer and price it, embed sentences and compute similarity by hand, then write up what you saw. Concepts you can only recite are fragile; concepts you have measured are yours.

By the end you will have
  • Called the Claude API at temperature 0 versus 1 and observed determinism versus diversity firsthand.
  • Tokenized text, counted tokens, and converted a prompt into an actual dollar cost.
  • Embedded several sentences and computed cosine similarity in NumPy, confirming meaning is geometry.
  • Written a short reflection connecting each observation back to the Module 1 mechanism.
Lab format

This is a doing lesson, not a reading one. Each task below ends in a "ship this" checkpoint: a small, concrete artifact you should produce before moving on. Reference solution code sits in the collapsible blocks, but run yours first, the gap between your output and the solution is the actual learning. Budget about 90 minutes.

Task 1 · Temperature: determinism vs diversity (20 min)

Day 3 claimed that low temperature concentrates the distribution (near-deterministic) and high temperature flattens it (diverse). Now watch it. Call the same prompt several times at temperature 0, then several times at temperature 1, on both a creative prompt and a factual one, and observe the contrast.

import os
from anthropic import Anthropic

client = Anthropic()  # reads ANTHROPIC_API_KEY from the environment
MODEL  = "claude-sonnet-4-5"  # use whatever current model id you have access to

def ask(prompt, temperature, n=3):
    outs = []
    for _ in range(n):
        msg = client.messages.create(
            model=MODEL,
            max_tokens=60,
            temperature=temperature,
            messages=[{"role": "user", "content": prompt}],
        )
        outs.append(msg.content[0].text.strip())
    return outs

creative = "Write a one-line slogan for an independent coffee shop."
factual  = "What is the capital of France? Answer in one word."

for label, temp in [("T=0", 0.0), ("T=1", 1.0)]:
    print(f"\n=== creative @ {label} ===")
    for o in ask(creative, temp): print(" -", o)

print("\n=== factual @ T=0 vs T=1 ===")
print("T=0:", ask(factual, 0.0))
print("T=1:", ask(factual, 1.0))

Predict before you run: which block will show near-identical lines, and which will diverge? Then run it and check yourself.

Ship this

Produce the printed output and write one sentence answering: did the factual prompt diverge much between T=0 and T=1, and why is that different from the creative prompt? (Hint: a peaked distribution barely moves under temperature; a flat one moves a lot.)

Show solution

Expected behavior: at T=0 the creative slogans cluster tightly (often identical or near-identical), and at T=1 they spread into clearly different lines. The factual prompt stays "Paris" at both temperatures, because its distribution is so sharply peaked on the correct token that even flattening it leaves "Paris" dominant, there is simply no plausible alternative for the sampler to reach. That contrast is the whole point: temperature does not invent options that were not there, it only changes how far the sampler explores the mass the model already assigned. Where the model is confident, temperature barely matters; where it is genuinely uncertain (open-ended creative tasks), temperature is the diversity dial. No-key fallback: do the same in three fresh Claude web chats, the creative prompt diverges, the factual one does not.

# one-sentence writeup (example):
# "Factual barely moved because its next-token distribution is
#  peaked on 'Paris', so flattening it changes nothing; the
#  creative prompt has a flat distribution, so temperature 1
#  let the sampler reach many different valid continuations."

Task 2 · Tokenize and price a prompt (20 min)

Day 2 said tokens are the unit of cost and context. Make that a number. Use a real tokenizer to split text, count tokens, watch a rare word shatter, then turn a realistic RAG prompt into a per-call dollar figure.

import tiktoken

enc = tiktoken.get_encoding("cl100k_base")  # a widely-used BPE vocab

for word in ["the cat", "strawberry", "antidisestablishmentarianism", "SKU-4471-XQ"]:
    ids = enc.encode(word)
    pieces = [enc.decode([i]) for i in ids]
    print(f"{word!r:35} -> {len(ids)} tokens  {pieces}")

# --- price a realistic RAG call ---
def ntok(text): return len(enc.encode(text))

system_prompt = "You are a precise support assistant. Cite sources." * 20
one_chunk     = "Lorem ipsum dolor sit amet, " * 60   # stand-in for a retrieved chunk
question      = "How do I reset my password on the mobile app?"

in_tokens  = ntok(system_prompt) + 6 * ntok(one_chunk) + ntok(question)
out_tokens = 300

PRICE_IN, PRICE_OUT = 3.0, 15.0   # $ per 1M tokens (illustrative)
cost = in_tokens / 1e6 * PRICE_IN + out_tokens / 1e6 * PRICE_OUT
print(f"\ninput={in_tokens} out={out_tokens}  cost/call=${cost:.4f}")
print(f"at 1M calls/month: ${cost * 1_000_000:,.0f}")

Note: cl100k_base is a convenient generic tokenizer for the exercise; exact counts differ per model family, and for true Claude counts you would use the Anthropic token counting endpoint. The lesson, that retrieved context dominates input cost, holds regardless.

Ship this

Report the token count for each of the four strings (which one shattered most, and why), and the cost-per-call plus the monthly cost at a million calls. Then answer: which single slice of the input would you cut first to reduce cost, and what does that imply for your RAG design?

Show solution

Expected: "the cat" stays small (about 2 tokens), "strawberry" splits into a couple of subwords, the long rare word and the SKU code fragment into many tokens each, the exact code especially, which is precisely why lexical retrieval (BM25) beats embeddings on codes and acronyms (Module 3). On cost, the six retrieved chunks vastly outweigh the system prompt and question, so they are the dominant input cost, and the monthly figure at a million calls turns a "just add more chunks" decision into a five-figure line item. The slice to cut first is the retrieved context: lower top-k and add a reranker so each surviving chunk earns its place, which reduces input tokens directly. The implication is the Module 3 thesis in one line: fewer, better chunks is simultaneously a quality move (less lost-in-the-middle) and a cost move (fewer billed input tokens). No-key needed, this task is fully offline.

Task 3 · Embed and compare by hand (25 min)

Day 2 said meaning is geometry and cosine similarity is the workhorse comparison. Prove it: embed several sentences, compute the full similarity matrix in NumPy, and confirm that semantically related sentences score high and unrelated ones score near zero. This is dense retrieval in miniature.

import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")  # small, local, no API key

sentences = [
    "How do I reset my password?",
    "I forgot my login credentials and cannot sign in.",
    "What time does the cafe open on Sundays?",
    "The mitochondria is the powerhouse of the cell.",
]

V = model.encode(sentences, normalize_embeddings=True)  # unit vectors
# with unit vectors, cosine similarity == dot product == V @ V.T
S = V @ V.T

print("     " + " ".join(f"s{j}" for j in range(len(sentences))))
for i, row in enumerate(S):
    print(f"s{i}  " + " ".join(f"{x:4.2f}" for x in row))

# sanity-check the formula by hand on one pair (no normalize):
a, b = model.encode([sentences[0], sentences[1]])
cos = float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))
print("\nmanual cosine(s0, s1) =", round(cos, 3))

Read the matrix as a "find the nearest stored point to my query" table: if sentence 0 were a user query, which stored sentence would dense retrieval return? Confirm it is the paraphrase (sentence 1), not the cafe-hours or biology line.

Ship this

Produce the similarity matrix and the manual cosine value (it should match S[0][1]). Then answer: which two sentences are most similar and what is their score, which pair is least similar, and how does this matrix literally is a tiny dense-retrieval index?

Show solution

Expected: the diagonal is 1.00 (each sentence is identical to itself). Sentences 0 and 1 are the password-reset query and its paraphrase, so they score the highest off-diagonal value (typically well above 0.5), even though they share few exact words, which is exactly the strength of dense retrieval over keyword matching. The biology sentence (3) is unrelated to all the others and scores near zero against them, the noise floor. The manual cosine of s0 and s1 should equal S[0][1] to rounding, confirming that with unit-normalized vectors, dot product and cosine are the same operation. Why it is a retrieval index: if s0 is the query, you rank the stored sentences by their similarity column and return the top one, which is s1, that ranking-by-cosine is precisely what a vector database does at scale, just with an approximate-nearest-neighbor structure instead of a full matrix. You have built dense semantic search with four rows. This task needs no API key.

# expected shape (values illustrative):
#       s0    s1    s2    s3
# s0  1.00  0.62  0.18  0.04   <- s1 is the nearest neighbor of s0
# s1  0.62  1.00  0.16  0.05
# s2  0.18  0.16  1.00  0.07
# s3  0.04  0.05  0.07  1.00   <- unrelated to everything

Task 4 · Reflection writeup (15 min)

The most valuable part of any lab is the synthesis. In your own words (a short paragraph each, no more), write up four observations, tying each measurement back to the mechanism from Days 1 to 5:

  • Temperature. What you saw at T=0 vs T=1 on the two prompt types, and why a peaked distribution resists temperature while a flat one does not.
  • Tokens. Which strings fragmented and why, and the dollar figure you computed, plus what it implies for context budgeting.
  • Embeddings. The most and least similar pairs, and why "meaning is geometry" is more than a slogan after seeing the matrix.
  • One surprise. The single thing that did not match your prediction, and what it taught you.
Ship this

A four-paragraph reflection (one per bullet). Keep it concrete and mechanism-linked, not generic. This is the artifact that turns "I read about LLMs" into "I have measured how LLMs behave," which is exactly the framing that lands in interviews.

Show solution

A strong reflection is specific and traces every observation to a cause. Example shape: "Temperature 0 made the slogans identical and T=1 made them diverge, but the capital-of-France answer stayed 'Paris' at both, because its next-token distribution is so peaked that flattening it changes nothing, temperature only matters where the model is genuinely uncertain. The SKU code fragmented into many tokens while 'the cat' stayed at two, which is why exact codes are both expensive and better served by lexical retrieval, and the six retrieved chunks dominated my per-call cost, so cutting top-k is the first cost lever, not a bigger model. The password-reset query and its paraphrase scored highest despite sharing almost no words, which made 'meaning is geometry' concrete: cosine over embeddings, not keyword overlap, is what links them, and that matrix is literally a four-row dense index. My surprise was how little temperature affected the factual prompt, I had expected some drift, and it taught me that temperature acts on the distribution the model already produced, not on its knowledge." Yours should be in your own voice and reference your actual numbers.

Module 1 review questions

A quick self-check spanning the whole module. If any key feels shaky, revisit that day before starting Module 2. Expand each into a 90-second answer with Claude using protocol prompt #2.

  • Explain how an LLM generates text, to a smart non-engineer.
    Key: next-token prediction over a vocabulary, run in a loop; context steers the distribution.
  • What is self-attention and why was it the breakthrough?
    Key: content-based weighted mixing of tokens; lets context reshape meaning and is parallelizable across the sequence, unlike RNNs.
  • Why are price and context window both measured in tokens?
    Key: billing is per token (output costs more) and the window is a fixed token budget, not a word budget; so you count tokens and trim context.
  • What does temperature do, and why is temperature 0 not byte-identical?
    Key: scales logits before softmax (sharper toward greedy at 0, flatter above 1); float non-associativity, possible expert routing, and model updates mean "stable," not "identical."
  • What is the KV cache and why does it matter?
    Key: caches per-layer Key/Value tensors for prior tokens so each decode step only computes the new Query and attends the cache; makes generation append-cheap, costs GPU memory.
  • Why do LLMs hallucinate, and how do you mitigate it?
    Key: the objective rewards plausible continuations and knowledge is lossy parametric memory; ground it with RAG and check it with evaluation.
  • Foundation models vs traditional ML, when does each win?
    Key: FM for language-rich, ambiguous, multi-task, low-label, fast-iteration work; traditional ML for narrow, labeled, low-latency, regulated, cheap-at-scale prediction.
  • How do you decide what to delegate to an LLM versus not?
    Key: delegate language-shaped, ambiguity-tolerant, verifiable parts; route exact, fresh-fact, long-horizon, or high-stakes-unverifiable work to tools, retrieval, structure, or traditional ML.

Today's takeaways

You have now measured the mechanism, not just read it: temperature reshapes the distribution but cannot move a confident model off its answer; tokens are real money and real context budget, with rare strings and exact codes fragmenting most; cosine similarity over embeddings turns meaning into a rankable number, which is dense retrieval in four rows; and the reflection turned all of it into your own words. Module 1 is the spine, next-token prediction, attention, tokens, sampling, context and the KV cache, capabilities and limits, that every later module hangs on. You are ready to build on it.
