Module 1 · LLM Foundations Day 1 50 min

How LLMs actually work

Before you can design RAG pipelines, agents, or evaluation harnesses, you need a crisp mental model of the thing underneath all of it. Today you build the model that makes every later topic obvious.

By the end you can
  • Explain an LLM as a next-token predictor in one sentence, then defend it.
  • Sketch the Transformer from the outside in: embeddings → attention → MLP → unembed.
  • Describe what attention computes and why it is the key idea.
  • Separate the three training stages (pretraining, SFT, preference tuning) and say what each buys.
  • Derive three engineering consequences — hallucination, knowledge cutoff, non-determinism — directly from the mechanism.

The one-sentence mental model

A large language model is a function that, given a sequence of tokens, returns a probability distribution over the next token. That is the entire job. Everything else — chat, reasoning, code, "agents" — is that one operation, run in a loop, dressed up.

Hold onto that. When a model hallucinates, when it ignores half your prompt, when temperature changes the answer — you will explain all of it by returning to "it is predicting the next token from a distribution."

1 · Next-token prediction is the whole game

During training the model sees trillions of tokens of text and plays one game over and over: cover the next token, predict it, check, adjust. Formally it learns to maximize the probability it assigns to the token that actually came next, across the whole corpus. No labels, no humans — the text labels itself. This is why it is called self-supervised pretraining.

To predict the next token well across all of human text, the model is forced to internalize grammar, facts, formats, styles, a bit of arithmetic, and rough world knowledge — not because anyone taught those, but because they are useful for the prediction game. Capability is a side effect of compression.

Concept

An LLM does not "look things up." Its knowledge is parametric: baked into billions of weights as a lossy compression of its training data. That single fact explains hallucination and the knowledge cutoff, and it is the reason RAG exists.

2 · The Transformer, from the outside in

You do not need the math today. You need the data flow. A decoder-only Transformer (the GPT / Claude / Llama family) processes a sequence like this:

tokens: "The" "capital" "of" "France" "is" │ │ │ │ │ [ tokenizer → integer IDs ] ← Day 2 │ [ embedding lookup ] + [ positional info ] turns each ID into a vector │ ┌──────────▼───────────────────────────────────┐ │ Transformer block × N (e.g. 32, 80, …) │ │ ├─ multi-head SELF-ATTENTION (mix tokens) │ ← the only place tokens "talk" │ ├─ feed-forward MLP (think) │ │ └─ residual + layer-norm around each │ └──────────┬───────────────────────────────────┘ │ one rich vector per position [ unembedding / linear ] → logits over the whole vocabulary │ [ softmax ] → P(next token) → e.g. "Paris" 0.91, "the" 0.02, …

Three things to notice, because interviewers probe each:

  • Attention is the only operation that moves information between positions. The MLP processes each position independently. If two tokens need to interact, attention is where it happens.
  • Depth is repetition. The same block shape stacks N times. Early layers tend to capture syntax and local patterns; later layers capture more abstract, task-level structure.
  • The output is a distribution, every step. The final layer projects each position's vector onto the vocabulary to get a logit per possible token, then softmax turns logits into probabilities.

3 · Attention: the key idea

Attention lets every token decide which other tokens to pull information from. Each token emits three vectors: a Query ("what am I looking for?"), a Key ("what do I offer?"), and a Value ("here is my content"). A token's new representation is a weighted sum of all Values, where the weight of token j is how well i's Query matches j's Key.

Intuition: in "The animal didn't cross the street because it was too tired," the token "it" sends out a Query that matches the Key of "animal" more than "street," so it pulls in meaning from "animal." That is coreference, learned, with no rule written for it.

Multi-head means several of these attention patterns run in parallel, each free to specialize (one head tracks subjects, another tracks syntax, etc.). Causal masking means a token can only attend to earlier tokens, never future ones — mandatory for next-token prediction, and the reason generation is left-to-right.

Interview-grade phrasing

"Attention computes, for each token, a content-based weighted average over the values of the tokens it is allowed to see, with weights given by a softmax over query·key similarity. Self-attention is the mechanism that lets context change a token's meaning." Say that and you sound like you understand it, because you do.

4 · How it actually learned to be useful

A raw pretrained model is a brilliant autocomplete, not an assistant. Ask it a question and it might continue with more questions, because that is what the internet looks like. Three stages turn it into something you can ship:

StageWhat happensWhat it buys
1. PretrainingNext-token prediction on web-scale text. Months, huge GPU clusters.Raw knowledge & language ability → a base model.
2. SFT
supervised fine-tuning		Train on curated (instruction → good response) pairs written/curated by humans.It learns the format of being a helpful assistant.
3. Preference tuning
RLHF / DPO		Humans rank responses; the model is optimized to prefer the better ones.Tone, safety, helpfulness, refusal behavior → a chat model.

You will revisit SFT and DPO properly in Module 6. For now, the takeaway is that "the model" you call via an API is a base model wrapped in two layers of human-preference shaping — which is exactly why prompting style and system messages have such leverage (Module 2).

5 · Generation is a loop

The model predicts one token. To write a paragraph it runs autoregressively: predict a token, append it to the input, predict the next, and so on until a stop token or limit.

"The capital of France is" → P(next) → pick "Paris" "The capital of France is Paris" → P(next) → pick "." "The capital of France is Paris." → P(next) → pick <end> ■ stop

Two consequences you will use for the rest of your career: output is generated serially, so latency grows with the number of output tokens (Module 7); and which token gets "picked" depends on a sampling rule (Day 3), which is where non-determinism enters.

Worked example: trace one completion

Example

Prompt: Translate to French: "cheese" →

Step 1 — tokenize the prompt into IDs. Step 2 — one forward pass produces, at the final position, a logit for every token in the vocabulary. After softmax the top of the distribution might be:

token logit P "from" 14.2 0.74 ← "fromage" often starts here as a sub-word "from" "age" (next step continues the word) " from" 9.1 0.004 "cheese" 8.7 0.003 "Kä" 7.0 0.0005 ← German leaked in; low probability …

The model picks (say) from, appends it, runs again, gets age, appends, and converges on fromage. Notice: it never "knew" the French word as a fact it retrieved; it assigned high probability to the sub-word continuation because that pattern dominated its training data. Correctness and confidence are not the same thing — a wrong but common pattern can also get high probability. That gap is hallucination.

Why this matters for you as an AI engineer

Three consequences, derived not memorized

Hallucination ← knowledge is lossy parametric compression, and the objective rewards plausible tokens, not true ones. Fix: ground it (RAG, Module 3) and check it (eval, Module 5).
Knowledge cutoff ← weights freeze at the end of training. Fix: put fresh facts in the context, don't expect them in the weights.
Non-determinism ← generation samples from a distribution. Fix: lower temperature for reproducibility; never assume byte-identical outputs (Day 3).

Practice — do these before revealing

Say each answer out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

A colleague says: "We don't need RAG, the model already knows our product docs because they're on the internet." Give the two-sentence rebuttal grounded in how LLMs store knowledge.

Show solution

Even if the docs were in pretraining, the model stores them as lossy parametric memory blended with everything else, so it will confidently paraphrase, conflate versions, and invent specifics — and anything updated after the cutoff is simply absent. RAG puts the exact, current passage in the context at query time, which is both more accurate and auditable (you can cite the source). Bonus senior signal: "and it lets us update knowledge by changing a database row instead of retraining."

Problem 2

Why does an LLM's response latency depend much more on the length of the output than the length of the input?

Show solution

The input prompt is processed in a single parallel forward pass (the "prefill"). Output, by contrast, is autoregressive: each new token needs its own forward pass and cannot start until the previous token exists. So output tokens are produced serially, one network pass each, while input tokens are consumed roughly all at once. N output tokens ≈ N sequential steps. (We will quantify this and meet the KV cache on Day 4 and in Module 7.)

Problem 3

In one sentence each, name what pretraining, SFT, and RLHF/DPO contribute. Then: which stage most explains why a good system prompt changes behavior so much?

Show solution

Pretraining = raw knowledge and language from next-token prediction; SFT = the format of following instructions, learned from curated instruction/response pairs; preference tuning (RLHF/DPO) = alignment to human preference for tone, helpfulness and safety. System prompts have leverage chiefly because of SFT and preference tuning: the model was explicitly shaped to treat instruction-style and system-style context as something to obey, so steering it through that channel is using a behavior it was trained into.

Interview questions for this topic

From your Notion bank's Foundations deep-dive and common 2026 screens. Drill the one-line keys, then expand each into a 90-second answer with Claude using protocol prompt #2.

  • Explain how an LLM generates text, to a smart non-engineer.
    Key: next-token prediction over a vocabulary, run in a loop; context steers the distribution.
  • What is self-attention and why was it the breakthrough?
    Key: content-based weighted mixing of tokens; lets context reshape meaning, parallelizable across the sequence (unlike RNNs).
  • What is the difference between a base model and an instruct/chat model?
    Key: base = pure next-token predictor; chat = base + SFT + preference tuning, so it follows instructions and is aligned.
  • Why do LLMs hallucinate, mechanistically?
    Key: objective rewards plausible continuations, knowledge is lossy & parametric; high probability ≠ true.
  • What does "knowledge cutoff" mean and how do you work around it in production?
    Key: weights freeze at training end; supply fresh facts via context/RAG or tools, don't rely on the weights.
  • Why might the same prompt give different answers on two calls?
    Key: sampling from a distribution (temperature/top-p); set temperature low/0 for near-determinism, but caveats remain.

Today's takeaways

LLM = next-token predictor over a vocabulary. Attention is the only cross-token operation and is what makes context matter. Knowledge is lossy and parametric, which directly forces RAG, grounding, and evaluation. Generation is a serial loop, which directly forces caching and latency work.
