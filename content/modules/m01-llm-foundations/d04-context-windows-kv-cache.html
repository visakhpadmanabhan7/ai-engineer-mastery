Context windows & the KV cache

"Just put it all in the context" sounds free. It is not. Today you learn what the context window really is, the two-phase shape of inference (prefill then decode), the KV cache that makes generation tractable, why long context is expensive and fallible, and how to budget a window like an engineer.

By the end you can
  • Define the context window precisely and say what counts against it.
  • Separate the prefill phase from the decode phase and explain why their costs differ.
  • Explain the KV cache: what it stores, why it exists, and what it saves.
  • Explain why attention cost grows roughly quadratically with sequence length, and what that means for price and latency.
  • Recognize "lost in the middle" and budget a context window deliberately.

What the context window actually is

The context window is the maximum number of tokens the model can attend to at once: everything in a single forward pass, the system prompt, the full conversation history, retrieved chunks, tool definitions and tool results, and the tokens it is currently generating, all share one fixed budget. It is not memory in any persistent sense. The model has no recall of a previous request; each call starts blank and sees only what you place inside this window. When people say a model "remembers" earlier in a chat, what is happening is that the application re-sends the prior turns inside the window every time.

Two consequences follow immediately. First, the window is a hard ceiling: exceed it and content must be dropped or the request is rejected, which is why context budgeting (below) is a real engineering task. Second, because there is no persistence, anything the model should "know" for this call must be assembled and supplied as context every call, which is the entire premise of RAG (Module 3) and of context engineering (Module 2).

1 · Inference has two phases: prefill and decode

A single generation splits into two very different phases, and conflating them is a classic mistake.

  • Prefill processes your entire input prompt in one parallel forward pass. All input tokens are available at once, so the hardware crunches them together. This phase is compute-heavy but fast in wall-clock terms, and it produces the "time to first token." Its cost scales with input length.
  • Decode generates the output one token at a time, autoregressively (Day 1). Each new token requires its own forward pass, and token n+1 cannot begin until token n exists. This phase is serial, so its wall-clock cost scales with the number of output tokens, which is why long answers feel slow even on short prompts.

This is the precise reason, promised on Day 1, that latency depends far more on output length than input length. Prefill consumes the prompt roughly in parallel; decode pays one sequential step per generated token. Hold this two-phase picture, because both prompt caching and the KV cache are optimizations of these two phases respectively.

── PREFILL (parallel, one pass over the whole prompt) ── "Summarize this 4,000-token document: …………………" │ all input tokens processed together ▼ → time-to-first-token, and the KV cache is filled here ── DECODE (serial, one forward pass PER output token) ── step 1: …prompt… → "The" step 2: …prompt… "The" → " report" step 3: …prompt… "The" " report" → " finds" step 4: …prompt… "The" " report" " finds" → " that" │ each step attends back over everything before it ▼ output latency grows with the NUMBER of output tokens

2 · The KV cache: why decode does not recompute everything

Look at the decode diagram again. Naively, every step re-reads the entire growing sequence, so step 1000 would re-process 999 earlier tokens through every layer. That would make generation catastrophically slow, quadratic in the worst sense, recomputing the same work over and over.

The KV cache removes that waste. Recall attention from Day 1: each token produces a Query, a Key, and a Value. The crucial observation is that the Keys and Values of already-processed tokens never change as new tokens arrive (causal masking means earlier tokens cannot see later ones). So the model computes each token's Key and Value once and caches them, per layer. When generating the next token, the model only computes the new token's Query and attends it against the cached Keys and Values of all prior tokens. It does not recompute the past; it appends to the cache and reads it.

Interview-grade phrasing

"The KV cache stores the per-layer Key and Value tensors for every token already in the sequence, so each decode step only computes the new token's Query and attends against the cache instead of re-encoding the whole prefix. It turns generation from re-doing O(n) work each step into appending one token's worth, trading GPU memory for a large speed-up." That is the correct, complete statement.

The trade-off is memory: the cache grows with the sequence length and must live in fast GPU memory, and for long contexts and many concurrent requests the KV cache, not the model weights, becomes the binding memory constraint. That single fact drives much of LLM serving (Module 7): continuous batching, paged attention, and quantized caches are all about fitting more KV cache into limited memory. The KV cache also explains why agent loops and long shared prefixes are where caching pays off most: the same prefix is attended again and again.

3 · Why long context is expensive: the quadratic problem

The reason a 200K-token window is not simply "10x better than 20K" is the cost shape of attention itself. In self-attention, every token attends to every other token it is allowed to see, so the number of pairwise interactions scales with the square of the sequence length. Double the context and the attention computation roughly quadruples. This is the famous "attention is quadratic in sequence length" cost, and although modern systems soften it with optimized kernels (FlashAttention) and architectural tricks, the underlying scaling pressure is real.

Concretely, three costs rise as you grow the context:

  • Money. Every token in the window is billed (Day 2), so a longer prompt is a directly larger input bill, repeated on every call.
  • Latency. Prefill has to process more tokens, pushing out time-to-first-token; the KV cache also grows, pressuring memory and throughput.
  • Quality. More context is not monotonically better, which brings us to the next point.

The senior instinct is therefore the opposite of "stuff the window." It is "supply the smallest set of most relevant tokens that lets the model answer correctly." Relevance beats volume, on cost, speed, and accuracy at once.

4 · "Lost in the middle": more context can hurt

A robust, repeatedly observed finding: when relevant information sits in the middle of a long context, models use it less reliably than when the same information sits near the beginning or the end. Performance on "find and use this fact" tends to follow a U-shape across position. Burying the one passage that matters in the middle of 50 mediocre chunks can actively degrade the answer, even though the fact is technically "in context."

This reframes long context from a capability you should max out into a resource you should curate. Practical implications you will use constantly:

  • Position matters. Put the most important instructions and the most relevant retrieved content near the start or the end of the prompt, not buried in the middle.
  • Fewer, better chunks win. A reranker that surfaces the top 5 strong chunks usually beats dumping 50, both because of cost and because of lost-in-the-middle.
  • A big window is not a substitute for retrieval. Retrieval and reranking (Module 3) exist precisely to keep the context short, ordered, and relevant.
Concept

A large context window is permission, not a strategy. The costs (quadratic attention, billed tokens, larger KV cache, lost-in-the-middle) all push the same direction: curate the smallest, most relevant, best-ordered context that answers the question. "Bigger window" and "better retrieval" are not interchangeable.

5 · Budgeting a context window in practice

Treat the window as a fixed token account and allocate it on purpose. A workable mental template:

SliceRough shareNotes
System prompt / instructionssmall, fixedHigh leverage per token; a stable prefix, so it is the prime candidate for prompt caching.
Reserved output spaceset max_tokens for the real answerThe window must hold the generated tokens too; never let output and input together exceed the limit.
Retrieved context (RAG)the largest, most compressible sliceTune top-k and chunk size; rerank so each chunk earns its place. Cut here first under pressure.
Conversation historygrows over timeSummarize or truncate old turns rather than carrying everything verbatim.
Tool definitions / resultsvariesIn agent loops this accumulates fast; prune stale tool outputs.

Two habits separate engineers here. First, count, do not estimate: measure each slice with a tokenizer (Day 2), because the model truncates silently and a dropped instruction is a real bug. Second, order for the U-shape: keep the must-use content at the edges. The forward link is direct: a stable, well-ordered prefix is exactly what prompt caching (Module 7) rewards, because a cached prefix is billed at a fraction of the price and re-served fast. Today you learned why the prefix is special (prefill plus the KV cache); Module 7 turns that into the single biggest cost lever in LLM serving.

Worked example: trace prefill, decode, and the cache

Example

A request sends a 2,000-token prompt and the model generates a 200-token answer. Trace where the time and memory go, and why a follow-up question in the same chat is cheaper to start.

CALL 1 prefill: process 2,000 input tokens in ONE parallel pass → fills the KV cache with K,V for all 2,000 tokens (per layer) → emits the first output token (this is time-to-first-token) decode: 200 serial steps; step i computes only the NEW token's Query, attends it against the cached 2,000+i−1 prior K,V, appends K,V → output latency ≈ 200 sequential passes, NOT 200×2,000 recomputed end of call: KV cache now holds ~2,200 tokens of K,V CALL 2 (same conversation, +50-token question) the 2,200-token prefix is unchanged → if its KV/prefix is cached (prompt caching, Module 7), prefill mostly reuses it and only the new 50 tokens need processing → cheaper, faster first token

The lesson in one line: prefill cost tracks input length, decode cost tracks output length, and the KV cache is what makes decode append-cheap instead of recompute-expensive. Because the prefix is stable across a conversation, it is the natural unit to cache, which is why short, well-structured, front-loaded prompts are not just tidy, they are cheaper and faster on every subsequent turn.

Problem 1

A teammate proposes: "Our model has a 200K context window, so let's skip retrieval and just paste all 800 company documents into every prompt." Give the three-part rebuttal (cost, latency, quality).

Show solution

On cost, every token in the window is billed on every call, so pasting 800 documents means paying for an enormous prompt repeatedly, when only a few passages are ever relevant to a given question. On latency, prefill must process all those tokens, pushing time-to-first-token way up, and the KV cache balloons, pressuring memory and throughput. On quality, it is not even more accurate: attention cost is roughly quadratic in sequence length, and "lost in the middle" means the model uses information buried in a huge context less reliably than the same fact in a short, focused one, so drowning the relevant passage among hundreds of irrelevant ones can lower the answer quality. The right move is retrieval plus reranking to supply the smallest, most relevant, best-ordered set of chunks; a big window is permission to be sloppy, not a strategy.

Problem 2

Explain, to a backend engineer who knows caching but not ML, what the KV cache is and why generation would be unusably slow without it. Then state the cost of having it.

Show solution

When the model generates token by token, each new token needs to "look back" at every earlier token through the attention mechanism. For each earlier token, attention needs two derived tensors, a Key and a Value, and crucially those never change once the token is fixed, because a token can only attend to things before it. So instead of recomputing the Keys and Values for the entire growing prefix at every step (which would make step n re-do roughly n tokens of work through every layer, an explosive cost), the model computes each token's Key and Value once and caches them per layer. Generating the next token then only computes that one new token's Query and reads against the cache, so each step appends rather than recomputes. Without it, generation would be quadratic and effectively unusable on long outputs. The cost is memory: the cache grows with sequence length and must sit in fast GPU memory, and for long contexts and many concurrent users it, not the weights, becomes the limiting resource, which is why serving systems work hard to pack and shrink it.

Problem 3

You are designing a RAG prompt that must fit a 16K-token window: a 1,000-token system prompt, conversation history, retrieved chunks, and room for a 1,500-token answer. History has grown to 9,000 tokens and you are overflowing. Lay out a budgeting plan and the order in which you cut.

Show solution

Start by reserving the non-negotiable slices: about 1,500 tokens for the output and 1,000 for the system prompt, which leaves roughly 13,500 for history plus retrieved context. Then measure each slice with a tokenizer rather than guessing. The history at 9,000 tokens is both the largest and the most compressible, so cut there first: summarize older turns into a compact running summary and keep only the most recent turns verbatim, which can reclaim most of that 9,000. Next, size the retrieved context to fit what remains by lowering top-k and reranking so the few chunks that survive are the strongest, and place them near the end (or start) of the prompt to respect the lost-in-the-middle U-shape, with the system instructions at the very front. Keep the system prompt as a stable prefix so prompt caching can later make it cheap. The discipline throughout is count, do not estimate, because the model will silently drop tokens and a dropped instruction or the one relevant chunk is a real failure, not a cosmetic one.

Today's takeaways

The context window is one fixed token budget shared by prompt, history, retrieved content, tools, and output, and it is not persistent memory. Inference is prefill (parallel, scales with input, fills the KV cache) then decode (serial, scales with output), which is why latency tracks output length. The KV cache makes decode append-cheap by reusing prior tokens' Keys and Values, trading GPU memory for speed. Long context is expensive because attention is roughly quadratic and because "lost in the middle" means more context is not always better. So curate the smallest, most relevant, best-ordered context, and keep a stable front-loaded prefix that prompt caching can later reward.
