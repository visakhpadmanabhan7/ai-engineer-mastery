Agent memory

An LLM is stateless: every call starts from nothing. "Memory" is the engineering you bolt on to make an agent remember across turns and across sessions, without drowning its context window. Today you learn the three kinds and when each applies.

By the end you can
  • Distinguish short-term, long-term, and episodic memory and what each stores.
  • Manage short-term memory with message pruning and summarization.
  • Persist long-term user facts keyed by user_id in a vector or key-value store.
  • Explain why a checkpointer is short-term durability, not long-term memory.
  • Diagnose and mitigate context-window blow-up: lost-in-the-middle, cost, and latency.

1 · Why "memory" is engineering, not a model feature

The model has no memory. Each API call is a fresh forward pass over whatever tokens you send; nothing carries over unless you carry it. So "agent memory" is not something the LLM does, it is a system you build around it: deciding what to keep, where to store it, and what to put back into the context on the next call. The whole topic is really context management over time. Get it right and the agent feels like it knows the user; get it wrong and it either forgets everything or chokes on a bloated prompt.

There are three distinct kinds, and conflating them is a classic interview stumble. They differ on scope (this conversation vs forever) and what they hold (the raw recent exchange vs durable facts vs summaries of past sessions).

2 · Short-term memory: the working context

Short-term memory is just the conversation in the prompt right now, the recent user/assistant messages plus tool results that the model needs to keep the current task coherent. It lives in the context window and dies when the session ends. The engineering problem is that a long conversation grows past what you want to send every turn (cost, latency, and the lost-in-the-middle effect below). Two techniques manage it:

  • Pruning / windowing. Keep the last N turns (or last N tokens) and drop older ones. Simple and cheap; the risk is dropping something that still mattered. Often you keep the system prompt and the most recent turns, and trim the middle.
  • Summarization. When the history gets long, replace older turns with an LLM-generated summary ("the user is debugging a checkout bug; we ruled out the payment provider"). This preserves the gist at a fraction of the tokens. The trade-off is that summarization is lossy and costs an extra call, so you do it on a rolling basis, not every turn.

In practice you combine them: keep a verbatim recent window, and a running summary of everything before it. That keeps the prompt bounded while retaining continuity.

Concept

A checkpointer (Day 25) is short-term-memory durability, not long-term memory. It persists this run's working state so the agent can pause, crash, and resume the same conversation. It does not make the agent remember the user in a brand-new session next week. Persisting a session ≠ learning durable facts about a user. Mixing these up is a common interview slip.

3 · Long-term memory: durable user facts

Long-term memory holds stable facts about a user that should survive across all sessions: preferences ("always reply in metric units"), profile facts ("their company is MaindTec"), settings, prior decisions. Unlike short-term, it is not in the prompt by default, it is stored externally and keyed by user_id, then retrieved into context only when relevant. Two storage shapes:

  • Key-value, by user_id. Best for a known, structured set of attributes (units preference, plan tier, language). You fetch the user's record and inject the relevant fields. Fast, exact, easy to update.
  • Vector store, by user_id namespace. Best for open-ended remembered facts in natural language ("mentioned they manage a fleet team," "dislikes long emails"). You embed the new fact, store it under the user's namespace, and at query time retrieve the top-k facts relevant to the current message. This is RAG over the user's own history.

The deciding questions are what to remember and what to forget. Not every utterance is a durable fact, "what's the weather" is ephemeral; "I always want responses in German" is durable. A good system extracts and writes only salient, lasting facts (often via an LLM "is this worth remembering?" step), updates facts that change, and forgets the stale or one-off. Otherwise long-term memory becomes a junk drawer that pollutes retrieval. Then at query time you retrieve the relevant memories (KV lookup or vector top-k by user_id) and inject just those into the prompt, never the whole store.

Interview-grade phrasing

"Short-term memory is the working conversation in the context window, managed by pruning and summarization, and it dies with the session. Long-term memory is durable user facts stored externally by user_id in a KV or vector store and retrieved when relevant. Episodic memory is summaries of past sessions, indexed for retrieval. The hard parts are deciding what to remember versus forget, and retrieving only the relevant memories so the context does not blow up." That hits scope, storage, and the real engineering tension in one breath.

4 · Episodic memory: summaries of past sessions

Episodic memory sits between the other two: it stores summaries of what happened in earlier sessions, indexed so they can be retrieved later. Where long-term memory holds atomic facts ("prefers metric"), episodic memory holds the narrative of past interactions ("last Tuesday we debugged the RAG eval drop; the fix was adding a reranker"). At the start of a new session, or when the current task resembles a past one, you retrieve the relevant episode summaries and seed them into context, so the agent can say "last time we tried X, it did not work, let's try Y."

You build it by summarizing each completed session (or meaningful chunk) into a compact record, embedding it, and indexing it by user/topic. It is essentially RAG over the agent's own history of interactions. The reason it is its own category: it is neither the live conversation (short-term) nor a timeless fact (long-term), it is "what we did together, compressed and recallable." For assistants that work with the same user repeatedly, episodic memory is what makes the experience feel continuous rather than amnesiac.

5 · The context-window blow-up problem

Here is the failure that ties memory to everything else. The naive instinct, "just put more in the context", backfires three ways, and a senior engineer names all three:

  • Lost-in-the-middle. Models attend most reliably to the start and end of a long context and reliably least to the middle. Stuff 50 messages in and the crucial fact buried at position 27 gets effectively ignored. More context can lower accuracy. Placement matters: put the most important material near the top or bottom.
  • Cost. Every token in the prompt is billed on every call. An agent loop that keeps appending the full history pays for the entire growing transcript on each of its many turns, costs balloon superlinearly with conversation length.
  • Latency. Longer input means a larger prefill and slower responses. A bloated context makes every turn sluggish, which on a multi-turn agent compounds.

The mitigations are exactly the memory techniques above, used deliberately: prune and summarize short-term so the working window stays bounded; retrieve only the relevant long-term facts and episode summaries instead of dumping the whole store; and place the most important content at the edges of the context, not the middle. The goal is not "remember everything", it is "have the right small set of things in context at the moment of the call." Memory done well is selective, not exhaustive.

Anti-pattern

"We have a 200k context, so just keep the entire conversation and all the user's history in the prompt." This invites lost-in-the-middle accuracy loss, bills you for the full transcript on every turn, and slows each response. A big context window is a budget to spend wisely, not an excuse to skip memory management.

Worked example: a returning user, three memories at once

Example

A user opens a new session: "Can you pick up where we left off on the eval problem?" Watch which memory each piece comes from.

New session starts → assemble context from THREE sources: LONG-TERM (KV by user_id): "prefers concise answers; works on RAG systems" └─► inject as a short profile line EPISODIC (vector, by user): retrieve top episode for "eval problem" → "Session 4: eval faithfulness dropped; we suspected chunking, ran experiments, last step was testing a reranker." └─► inject the summary so the agent has the thread SHORT-TERM (this session): just the new message so far (will grow; prune + summarize as the conversation lengthens) Result prompt = system + [profile fact] + [relevant episode summary] + [current turn] ← small, relevant, edges-loaded NOT = the entire transcript of all 4 past sessions (that would blow up).

Three different mechanisms, one coherent context. The agent "remembers" the user's preference (long-term fact), recalls the storyline of the eval work (episodic summary), and tracks the live exchange (short-term), yet the assembled prompt is small because each source contributed only its relevant slice. Crucially, the past sessions were not dumped in verbatim, they were summarized into episodic memory and retrieved on demand. That is the difference between an assistant that feels continuous and one that either forgets you or buries the model in a wall of stale text.

Problem 1

A teammate says "we added a LangGraph checkpointer, so the agent now has long-term memory of the user." Correct them precisely, and say what they would actually need to build.

Show solution

The checkpointer gives short-term-memory durability, not long-term memory. It persists this run's working state (the messages and intermediate results of the current conversation) so the agent can pause, survive a crash, and resume the same session. It does nothing to make the agent recall the user in a brand-new session next week, persisting a session is not the same as learning durable facts about a person. To get actual long-term memory, they need to extract salient, lasting facts and store them externally, keyed by user_id (a key-value record for structured attributes like preferences/plan, and/or a vector store namespace for open-ended remembered facts), then retrieve the relevant ones into context at the start of future sessions. The defining difference: a checkpointer is scoped to a single run; long-term memory is cross-session and keyed to the user.

Problem 2

Your assistant has a 200k-token context, so an engineer just appends every message and every tool result and never trims. Users report it is getting slower, more expensive, and oddly worse at recalling details from earlier in long chats. Explain all three symptoms with the same root cause and give the fixes.

Show solution

All three stem from context-window blow-up, the "just keep everything" anti-pattern. Slower: a longer prompt means a bigger prefill, so each turn's latency grows with the transcript. More expensive: every token is billed on every call, and an agent re-sends the whole growing history each turn, so cost balloons superlinearly with conversation length. Oddly worse at recall: the lost-in-the-middle effect, models attend most to the start and end of a long context and least to the middle, so details buried mid-transcript get effectively ignored; more context can reduce accuracy. The fixes are deliberate memory management, not a bigger window: prune to a recent window and summarize older turns so short-term stays bounded; retrieve only the relevant long-term facts / episode summaries instead of dumping the whole history; and place the most important content at the top or bottom of the context, never the middle. The principle: keep the right small set of things in context, not everything.

Problem 3

For a personal-finance assistant, classify each of these into short-term, long-term, or episodic memory, and justify: (a) "I prefer all amounts in EUR," (b) the last six messages of the current chat, (c) "two months ago we set up a savings plan and you advised cutting subscriptions."

Show solution

(a) "I prefer amounts in EUR" → long-term memory. It is a stable, durable preference that should apply across all future sessions, so it belongs in external storage keyed by user_id (a KV attribute is ideal) and is retrieved/injected when relevant. (b) The last six messages of the current chat → short-term memory. It is the live working context for the conversation in progress; it stays in the prompt now and dies when the session ends (managed by pruning/summarization if the chat grows). (c) "Two months ago we set up a savings plan and advised cutting subscriptions" → episodic memory. It is a summary of what happened in a past session, a narrative of a prior interaction, not a timeless atomic fact and not the current conversation; you store it as an indexed episode summary and retrieve it when the current task relates to that savings plan. The tells: durability + atomic preference → long-term; live conversation → short-term; "what we did in a past session, summarized" → episodic.

Interview questions for this topic

From your Notion bank's Agents module. Drill the one-line keys, then expand each into a 90-second answer.

  • Short-term vs long-term vs episodic memory?
    Key: short = working context, pruned/summarized, dies with session; long = durable user facts in vector/KV by user_id; episodic = summaries of past sessions, indexed.
  • How do you manage short-term memory as a conversation grows?
    Key: pruning/windowing (keep last N turns/tokens) plus rolling summarization of older turns; combine a verbatim recent window with a running summary.
  • How do you store and retrieve long-term memory?
    Key: extract salient durable facts, store keyed by user_id (KV for structured, vector namespace for open-ended), retrieve only the relevant top-k into context; decide what to remember vs forget.
  • Is a checkpointer the same as long-term memory?
    Key: no, a checkpointer is short-term durability (resume the same run); long-term is cross-session facts keyed by user_id. Persisting a session ≠ learning durable user facts.
  • What is the context-window blow-up problem and how do you mitigate it?
    Key: lost-in-the-middle (accuracy), cost (billed per token per call), latency (prefill); mitigate by prune+summarize, retrieve only relevant memories, place key content at the edges.
  • What is lost-in-the-middle and why does it matter for memory?
    Key: models attend most to the start/end of long context, least to the middle, so dumping everything buries key facts; placement and selectivity beat raw context size.

Build it (15–25 min)

Hands-on

Build a tiny memory layer around the Claude API. (1) Short-term: hold a conversation; once it passes ~10 turns, replace the oldest turns with a one-paragraph summary you generate with a quick model call, and confirm the prompt stops growing. (2) Long-term: after each user message, run a cheap "is there a durable fact to remember here?" classification; if yes, write it to a dict keyed by a fake user_id (your stand-in for a KV/vector store), and at the start of a fresh conversation inject those facts and watch the assistant "remember" the user. Skim the LangGraph memory concepts and Pydantic for typing the stored facts. The lesson lands when you compare two new sessions, one with retrieved memory injected, one without, and feel the difference.

No API key? Design it on paper: list five user utterances and mark each as remember-as-fact, summarize, or forget, then write the retrieval rule for a new session. See Tools & Claude.

Today's takeaways

The model is stateless, so memory is engineering: context management over time. Short-term memory is the working conversation, kept bounded by pruning and summarization, and it dies with the session. Long-term memory is durable user facts stored externally by user_id (KV or vector) and retrieved when relevant, with the hard part being what to remember versus forget. Episodic memory is indexed summaries of past sessions, RAG over the agent's own history. A checkpointer is short-term durability, not long-term memory. And the unifying discipline is fighting context-window blow-up, lost-in-the-middle, cost, and latency, by keeping the right small set of things in context, not everything.
