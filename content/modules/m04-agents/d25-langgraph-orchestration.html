LangGraph & orchestration

Hand-rolled while loops are fine for a demo agent and a nightmare in production. LangGraph gives agent control flow a real runtime: a stateful graph with cycles, interrupts, and checkpoints. Today you learn what it buys and when the upgrade pays off.

By the end you can
  • Separate LangChain (integrations and chains) from LangGraph (a stateful graph runtime).
  • Name the LangGraph primitives: nodes, edges, state, cycles, interrupts, checkpointers.
  • Explain why a graph beats an ad-hoc loop once control flow gets complex.
  • Implement human-in-the-loop with an interrupt, and fan-out with Send.
  • Decide when to combine PydanticAI (typed I/O) with a graph rather than choosing one.

1 · LangChain is not LangGraph

People conflate these constantly, and the distinction matters in interviews. LangChain is the integrations-and-chains library: a huge set of adapters (model providers, vector stores, document loaders, retrievers) plus the abstractions to wire them into chains, the fixed pipelines from Day 22. It is excellent for "retrieve, then prompt, then parse." It is weaker when the control flow stops being a straight line.

LangGraph is a different thing: a stateful graph runtime for agentic control flow. You describe your agent as a graph of nodes (units of work) connected by edges (what runs next), all operating on a shared state object, and crucially the graph can contain cycles (loop back to a node), interrupts (pause for a human), and checkpointers (persist state so you can resume). LangChain answers "how do I call these components"; LangGraph answers "how do I orchestrate a stateful, looping, interruptible agent reliably." They are complementary: you often use LangChain's integrations inside LangGraph nodes.

Interview-grade phrasing

"LangChain is integrations and chains, great for fixed pipelines. LangGraph is a stateful graph runtime for agents: nodes, edges, a shared state, plus cycles, interrupts, and checkpointers. I reach for LangGraph the moment control flow gets complex, loops, branching, human-in-the-loop, durable resume, because a graph makes that explicit and recoverable where an ad-hoc while-loop makes it implicit and fragile." That cleanly answers "when does the LangGraph upgrade pay off."

2 · The primitives

Five concepts carry almost everything:

  • State. A single typed object (often a TypedDict or Pydantic model) threaded through the whole graph. Each node reads it and returns updates that get merged in. This is the spine, every node sees and contributes to one shared state, so there is no tangle of variables passed around by hand.
  • Nodes. Units of work: a function (or an LLM call, a tool call, a retrieval step). A node takes the state and returns a partial update.
  • Edges. What runs next. A normal edge always goes A → B. A conditional edge routes based on the state (a function inspects the state and returns the name of the next node), which is how you branch and how you loop.
  • Cycles. Edges can point backward, so the agent loop (call model → if it wants a tool, run the tool → go back to the model) is just a cycle in the graph. The ReAct loop from Day 22 becomes two nodes and a conditional edge.
  • Interrupts & checkpointers. An interrupt pauses the graph at a chosen point and hands control back to your application (for human approval, say). A checkpointer persists the state at each step to a store, so a run can be paused, resumed later, recover after a crash, or be inspected. Together they are what make the difference between a script and a durable system.
A LangGraph agent = a graph over ONE shared state object ───────────────────────────────────────────────────────── ┌─────────────┐ START ─► │ agent node │ ◄───────────────┐ (cycle: loop back) │ (LLM call) │ │ └──────┬──────┘ │ │ conditional edge: │ wants a tool? ──── yes ──► ┌─────┴─────┐ │ │ tool node │ no └───────────┘ ▼ ┌─────────────┐ │ END │ checkpointer: state saved at every step └─────────────┘ interrupt: pause here for a human, resume later STATE: { messages, tool_results, plan, … } threaded through all

3 · Why a graph beats an ad-hoc loop

You can absolutely build an agent with a hand-written while True: loop, a list of messages, and some if statements. For a small agent it is even the right call. The trouble is that as control flow grows, the loop accumulates implicit, hard-to-see structure: state lives in scattered local variables, branching hides in nested conditionals, and there is no clean place to pause, persist, or resume. Three things break first:

  • Persistence and resume. If your process dies mid-run, an ad-hoc loop loses everything. A checkpointed graph resumes from the last saved step. For long or expensive agent runs, this is the difference between "retry from scratch" and "continue."
  • Human-in-the-loop. Pausing a raw loop to wait for human approval, then resuming exactly where it stopped, is genuinely awkward to write correctly. An interrupt makes it a first-class operation.
  • Observability and structure. A graph is an explicit, inspectable object, you can see the nodes, the edges, the branches, and trace which path a run took. An ad-hoc loop's structure is implicit in the code, so debugging means reading control flow instead of reading a trace.

The honest version of the trade-off: a graph adds upfront structure (boilerplate) you do not need for a tiny agent. The payoff arrives precisely when control flow gets complex, multiple branches, loops, approvals, durable runs, which is exactly when an ad-hoc loop turns into spaghetti. So the senior heuristic is: simple loop while it is simple; move to a graph the moment you need cycles plus persistence plus human-in-the-loop together.

# A minimal LangGraph-style state machine for a ReAct agent.
from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated
import operator

class State(TypedDict):
    messages: Annotated[list, operator.add]   # nodes append to shared state

def agent(state):                                # node: call the model
    reply = llm.invoke(state["messages"])
    return {"messages": [reply]}

def tools(state):                                # node: run the requested tool
    result = run_tool(state["messages"][-1])
    return {"messages": [result]}

def route(state):                                # conditional edge: loop or stop
    last = state["messages"][-1]
    return "tools" if wants_tool(last) else END

g = StateGraph(State)
g.add_node("agent", agent)
g.add_node("tools", tools)
g.add_edge(START, "agent")
g.add_conditional_edges("agent", route)    # branch on state
g.add_edge("tools", "agent")              # CYCLE: tool → back to agent

app = g.compile(checkpointer=saver)             # durable: resume after crash

That is the entire ReAct loop expressed as a graph: two nodes, one conditional edge that either routes to the tool node or ends, and one backward edge that closes the cycle. The checkpointer makes it resumable. Compare that to the same behavior buried in a while loop and you can see why the structure is worth it once things grow.

4 · Human-in-the-loop via interrupt

This is LangGraph's standout feature and a direct callback to Day 23's approval gates. Some actions (charge a card, send to a customer, delete data) must not fire without a human saying yes. With a graph, you place an interrupt just before the risky tool node: the graph runs up to that point, persists its state via the checkpointer, and pauses, handing control back to your application. Your app shows the proposed action to a human; on approval, you resume the graph from exactly where it stopped, and the tool executes. On rejection, you route elsewhere.

The reason this is clean in a graph and painful in a raw loop is the checkpointer: because state is already persisted at every step, "pause and resume later" is free, the human can take an hour, the process can even restart, and the run continues. This is the textbook implementation of "classify read vs write, gate the write" from Day 23, now with a runtime that actually supports the pause.

... → [draft action node] → ⟨ INTERRUPT ⟩ ── state checkpointed, graph pauses │ your app shows the human: "Agent wants to: refund €240 to ORD-004812. Approve?" │ ┌── approve ────┴──── reject ──┐ ▼ ▼ resume → [execute tool] route → [ask for clarification]

5 · Send: parallel fan-out

Day 24's Plan-Execute needs to run independent steps in parallel. LangGraph expresses that with Send: from one node you can dispatch the same downstream node multiple times with different inputs, and they run as parallel branches whose results merge back into the shared state. If a planner produces eight independent lookups, a node emits eight Send calls to the "lookup" node, the runtime fans them out, and a downstream node synthesizes the merged results. This is how the parallelism from the previous lesson becomes a concrete graph operation instead of manual thread management.

6 · PydanticAI: combine, do not pick

A common false choice is "LangGraph or PydanticAI." They solve different layers. PydanticAI is about typed inputs and outputs: it leans on Pydantic to give each agent or tool a validated, strongly-typed interface, so what goes in and what comes out are guaranteed shapes (exactly the schema-enforcement and validation theme from Day 23). LangGraph is about orchestration: the stateful graph that decides what runs when.

The right move is usually to combine them: use LangGraph for the control flow (nodes, cycles, interrupts, checkpoints) and PydanticAI (or plain Pydantic) for the typed I/O of the nodes and tools inside it. You get a durable, interruptible graph whose every step has a validated contract. Picking one and forgoing the other usually means either an orchestrator with loose, error-prone I/O, or beautifully typed pieces with no robust runtime to coordinate them. The senior framing: orchestration and typed I/O are orthogonal concerns, pick the best tool for each layer.

Worked example: a refund agent with an approval gate

Example

Goal: an agent that looks up an order, decides whether a refund is warranted, and, if so, issues it, but no refund fires without human approval, and the run must survive a process restart while it waits.

START → lookup_order ──► decide ──┬── no refund ──► END │ └── refund warranted ▼ draft_refund (typed: {order_id, amount}) ▼ ⟨ INTERRUPT ⟩ ← state checkpointed; pause ▼ (human approves hours later) issue_refund (idempotent, key=order_id) ▼ END

Every primitive from today shows up. State threads the order and decision through the nodes. The conditional edge after decide branches on "refund warranted?". The interrupt before issue_refund implements Day 23's write-tool approval gate, and the checkpointer is what lets the human take their time without losing the run. The refund tool is idempotent (keyed on the order ID) so a resumed-or-retried graph cannot double-refund, and the draft_refund payload is a typed object so the amount and ID are validated before a human ever sees them. A raw while-loop could fake some of this, but the durable pause-and-resume around a human decision is exactly the thing it cannot do cleanly, which is the whole reason to use the graph.

Problem 1

A teammate asks: "We already use LangChain. Why would I add LangGraph, isn't that the same project?" Give the distinction and a concrete trigger that would make you actually switch.

Show solution

They are complementary, not the same. LangChain is integrations and chains, the adapters (models, vector stores, loaders, retrievers) and the glue to build fixed, linear pipelines; it is great for "retrieve → prompt → parse." LangGraph is a stateful graph runtime for agentic control flow: nodes, edges, a shared state, and, decisively, cycles, interrupts, and checkpointers. You keep using LangChain's integrations inside LangGraph nodes, so it is "add," not "replace." A concrete trigger to actually switch: the moment you need a real loop with persistence and human-in-the-loop together, for example, an agent that iterates over tool calls, must pause for human approval before a risky action, and must survive a process restart while it waits. A linear LangChain chain cannot express the cycle, the durable pause, or the resume cleanly; that combination is exactly what LangGraph's checkpointer and interrupt exist for. If you only have a straight pipeline, you do not need LangGraph yet.

Problem 2

Your agent runs a 12-step research task that takes four minutes and costs real tokens. Occasionally the server redeploys mid-run and the user has to start over, furious. You are using a hand-written while-loop. What is the fix, and which single primitive does the heavy lifting?

Show solution

Move the agent to a graph with a checkpointer; the checkpointer is the primitive doing the heavy lifting. The problem is that a hand-written while-loop keeps all its state in process memory, so a redeploy wipes it and the run restarts from zero, wasting four minutes and the tokens already spent. A checkpointer persists the graph's shared state to a durable store at every step, so after a restart the run resumes from the last saved step instead of from scratch. Modeling the 12 steps as nodes/edges also makes the structure explicit and traceable, but persistence-and-resume is the specific capability that fixes the furious-user symptom. (Bonus senior point: this is also what later enables human-in-the-loop and inspection, because the state is already durably saved at each step.)

Problem 3

Someone frames a decision as "LangGraph vs PydanticAI, which should we standardize on?" Explain why the framing is wrong and what you would actually do.

Show solution

The framing is wrong because the two solve orthogonal layers, so it is not an either/or. PydanticAI governs typed inputs and outputs: validated, strongly-typed interfaces for agents and tools (the schema-enforcement / validation concern). LangGraph governs orchestration: the stateful graph that decides what runs when, with cycles, interrupts, and checkpoints. Standardizing on one and dropping the other means either a robust orchestrator with loose, error-prone I/O, or nicely typed components with no durable runtime to coordinate them. What I would actually do is combine them: use LangGraph for control flow and PydanticAI (or plain Pydantic) for the typed I/O of the nodes and tools inside the graph, yielding a durable, interruptible graph where every step has a validated contract. Choose the best tool per layer rather than forcing a single winner.

Today's takeaways

LangChain is integrations and chains (fixed pipelines); LangGraph is a stateful graph runtime for agents, nodes, edges, shared state, plus cycles, interrupts, and checkpointers. A graph beats an ad-hoc loop precisely when control flow gets complex, because it makes structure explicit and traceable, persists state for crash-safe resume, and supports human-in-the-loop as a first-class pause. The ReAct loop is just two nodes and a conditional edge with a cycle; an approval gate is an interrupt before a write node, made durable by the checkpointer; and Send fans independent steps out in parallel. Finally, LangGraph and PydanticAI are orthogonal (orchestration vs typed I/O), so combine them rather than choosing, every node durable, every contract validated.
