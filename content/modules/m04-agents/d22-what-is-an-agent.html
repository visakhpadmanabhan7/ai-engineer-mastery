Module 4 · Agents & Tool Use Day 22 50 min

What is an agent, really?

"Agent" is the most overloaded word in AI right now. Today you get a definition precise enough to draw on a whiteboard, defend in an interview, and (the senior move) argue against when an agent is the wrong tool.

By the end you can
  • Define an agent in one sentence and contrast it cleanly with a chain.
  • Place any system on the agent spectrum, from chatbot to autonomous loop.
  • Draw the ReAct loop (Thought, Action, Observation) from memory.
  • Name the four situations where you should not reach for an agent.
  • Explain why "autonomy over control flow" is the line that actually matters.

The one-sentence definition

An agent is an LLM that has autonomy over its own control flow: at each step it decides what to do next, which tool to call, and when it is done, by looping over its own outputs until the task is finished or a budget runs out. The model is not just generating text; it is steering the program.

Contrast that with a chain: a fixed pipeline that you wired up in advance. "Retrieve, then summarize, then translate" is a chain. The path is decided by the developer, the LLM just fills in each box. The difference is not how smart the model is; it is who decides the next step. In a chain, the code decides. In an agent, the model decides.

Interview-grade phrasing

"A chain has a control flow the developer fixed at build time; an agent has a control flow the model chooses at run time. The defining property of an agent is autonomy over control flow: it decides the next action and which tool, in a loop, rather than running a path I hard-wired." Say that and the rest of the conversation gets easier.

1 · The agent spectrum

"Agent" is not binary, it is a dial. Most production "AI features" sit lower on this dial than the marketing suggests, and that is usually correct engineering. Here is the spectrum, from least to most autonomy:

LESS autonomy ───────────────────────────────────────────────► MORE autonomy ┌──────────┐ ┌──────────┐ ┌──────────────┐ ┌────────┐ ┌────────────┐ ┌─────────────┐ │ chatbot │ │ workflow │ │ tool-using │ │ agent │ │ multi-agent│ │ autonomous │ │ │ │ / chain │ │ LLM │ │ (loop) │ │ │ │ open-ended │ └──────────┘ └──────────┘ └──────────────┘ └────────┘ └────────────┘ └─────────────┘ reply to dev-wired calls ONE tool LOOP: many agents no fixed goal, one message fixed steps when asked, then decides coordinate, sets its own no tools (you decide returns (you tool+step handoffs subgoals, no loop the path) still decide itself, (Day 24) runs for hours the loop) until done └── code decides path ──┘ └────────── model decides path ──────────────┘

The single most important boundary on that line is between tool-using LLM and agent. A tool-using LLM calls a tool when prompted and hands control straight back to your code: one hop, deterministic shape. An agent owns the loop: it can call a tool, read the result, decide it needs another tool, call that, and only stop when it judges the task done. The loop is the agent. Everything to the right (multi-agent, autonomous) is "more loops, more autonomy, more ways to go wrong."

A useful litmus test: if you can draw the full execution path as a flowchart before running it, it is a chain or a tool-using LLM. If the path depends on what the model decides mid-run, it is an agent.

2 · The ReAct loop, the beating heart

Almost every single agent you will build or read about is some version of ReAct: Reason + Act. The model interleaves three moves in a loop: a private Thought (reasoning about what to do), an Action (a tool call with arguments), and an Observation (the tool's result, fed back into context). Then it thinks again. The loop ends when the model emits a final answer instead of another action.

┌─────────────────────────────────────┐ │ AGENT LOOP │ ▼ │ user goal ──► ┌─────────┐ │ │ THOUGHT │ "I need today's revenue, I'll │ │ │ query the sales DB first." │ └────┬────┘ │ ▼ │ ┌─────────┐ │ │ ACTION │ query_sales(date="today") │ └────┬────┘ │ ▼ │ [ runtime executes the tool ] │ ▼ │ ┌──────────────┐ │ │ OBSERVATION │ → {"revenue": 41200} │ └──────┬───────┘ (appended to context) ───┘ │ ▼ model decides: enough? ── no ──► loop again │ yes ▼ ┌──────────────┐ │ FINAL ANSWER │ "Revenue today is €41,200." ■ stop └──────────────┘

Why is ReAct still the default after years of fancier proposals? Because it is simple and debuggable from the trace. The full reasoning is right there: every Thought, every tool call, every Observation, in order. When an agent does something dumb, you read the trace and see exactly where it went wrong (usually a bad Thought or a tool returning garbage). It is excellent for tasks needing a handful of tool calls. Its weakness shows up when there are many steps that could run in parallel, which is what pushes you toward Plan-Execute (Day 24).

Concept

The "Observation" step is the part beginners forget. An agent is only useful because the result of an action re-enters its context and changes the next decision. An LLM that calls a tool but never reads the result back is not an agent, it is a one-shot function call. Feedback closing the loop is what makes it agentic.

3 · The analogy that makes it click

Think of an agent as an intern with API access. You give the intern a goal ("find out why checkout conversion dropped this week") and a set of tools they are allowed to use (the analytics dashboard, the deploy log, Slack). The intern decides the order: maybe check the deploy log first, see a release went out Tuesday, then pull conversion by day, notice the drop lines up, then post a summary. You did not script those steps. You scoped the goal, handed over the tools, and trusted the loop.

The analogy also predicts the failure modes, which is why it is worth keeping: an intern with vague instructions wanders; an intern with a dangerous tool and no oversight can do real damage; an intern who never stops asks the same question forever. Those map exactly to under-specified goals, missing approval gates, and runaway loops. Hold the intern picture and most agent design decisions become common sense.

4 · When NOT to use an agent

This is the section that separates a senior engineer from someone who just read an agent tutorial. Agents are slower, more expensive, less predictable, and harder to debug than a chain. You pay all of that for one thing: the ability to handle problems whose steps you cannot enumerate in advance. If you can enumerate the steps, an agent is the wrong tool. Reach for the simplest thing that works, then add autonomy only when a real requirement forces it.

Do not use an agent when

The task is deterministic and the steps are known. "Parse this invoice into fields X, Y, Z" is a chain (or a single structured-output call). Wrapping it in an agent adds latency, cost, and a new way to fail, for zero benefit.
Latency is tight. An agent loop is several sequential LLM calls plus tool round-trips. If you need a sub-second reply, the loop is dead on arrival; use a single call or a cached path.
The whole thing fits in one prompt. If a single well-built prompt (maybe with one tool call) solves it, you do not need a loop. Adding a loop is gold-plating.
It is safety-critical or irreversible. Anything that moves money, deletes data, or sends to customers should be a constrained workflow with explicit approval gates, not a model freely choosing actions. Non-determinism plus irreversibility is a bad combination.

Flip side, agents earn their cost when: the number and order of steps genuinely depend on intermediate results, the task spans several distinct tool domains, or a human would naturally "figure it out as they go." Triage, research, multi-step data lookups, and open-ended assistants are good fits. "Convert A to B" is not.

Worked example: same task, chain vs agent

Example

Task: "What was our most-used feature last month, and is it mentioned in the latest release notes?" Watch how the two designs differ.

CHAIN (you must know the path): get_feature_usage(month="last") → pick top feature → search_release_notes(feature) → format answer Problem: what if "most-used" needs disambiguation? What if the feature has two names? The fixed path cannot adapt; you'd hard-code every branch yourself. AGENT (model chooses the path): Thought: I need usage data first. Action: get_feature_usage(month="last") Observ.: top = "Skill Store", 9,400 invocations Thought: Now check if it's in the latest release notes. Action: search_release_notes(query="Skill Store") Observ.: 1 hit, v2.4 notes Thought: I have both facts. Final: "Skill Store was most-used (9.4k uses); yes, it's in v2.4."

The agent shines because step two's argument ("Skill Store") was not knowable until step one ran, and the model adapts if usage data comes back ambiguous. But notice the agent did exactly two tool calls in a fixed-looking order, so a sharp reviewer would ask: did this really need an agent, or would a two-step chain have done? Often the honest answer is "a chain", and saying so is the senior signal. Reach for the agent only when the branching is real, not hypothetical.

Practice — do these before revealing

Answer each out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

A teammate built a "support agent" that takes a ticket, retrieves the relevant doc, and drafts a reply, always in that exact order, every time. Is it an agent? Justify your answer using the definition, and say what it actually is.

Show solution

It is not an agent, it is a chain (or a tool-using LLM if the retrieval is exposed as a tool the model calls). The control flow is fixed by the developer: ticket → retrieve → draft, the same path on every ticket, with no point where the model decides the next step. The defining property of an agent, autonomy over control flow, is absent. Calling it an "agent" is a naming inflation that hides the real (and good) design. The senior reframe: "It is a deterministic chain, which is the right call here because the steps are always the same. If we later needed it to decide whether to escalate, query a second system, or ask a clarifying question based on the ticket, that branching is when it would become a genuine agent."

Problem 2

Your PM wants to "make the invoice parser an agent so it is smarter." The parser today takes a PDF and returns five fields with high accuracy. Talk them out of it (or into it) on engineering grounds.

Show solution

Push back, on grounds, not vibes. Invoice parsing is a deterministic, scoped task with known output: it is the textbook case for not using an agent. A single structured-output call (schema-enforced) already does it accurately, cheaply, and predictably. Wrapping it in an agent loop would add multiple sequential LLM calls (higher latency and cost), introduce non-determinism into something that currently has a clean contract, and create new failure modes (the model "deciding" to do something other than extract). "Smarter" is not a requirement; it is a feeling. The right question is "what can the current parser not do that the business needs?" If the answer is "handle invoices that require looking up a vendor in another system, or routing exceptions to a human," that is real branching and would justify a small agent. Absent a concrete capability gap, keep the deterministic parser.

Problem 3

Draw the ReAct loop from memory and label the three repeating steps. Then explain, in one sentence, what would happen if you removed the Observation step.

Show solution

The loop is Thought → Action → Observation → (back to Thought), repeating until the model emits a final answer instead of an action. Thought = the model's private reasoning about what to do next; Action = a tool call with arguments; Observation = the tool's result fed back into the model's context. If you removed the Observation step, the model would call a tool but never see the result, so it could not use the outcome to inform its next decision, collapsing the agent into a sequence of blind, disconnected one-shot calls. Closing the loop with the Observation is exactly what makes it agentic rather than a fixed pipeline.

Interview questions for this topic

From your Notion bank's Agents module. Drill the one-line keys, then expand each into a 90-second answer with Claude using protocol prompt #2.

  • What is an agent, and how is it different from a chain?
    Key: agent = LLM has autonomy over control flow (decides next step + tool in a loop); chain = fixed, developer-wired pipeline.
  • When should you NOT use an agent?
    Key: small/scoped/deterministic task, tight latency, expressible in a single prompt, or safety-critical. Knowing when not to is senior signal.
  • What is the ReAct loop and why is it still the single-agent default?
    Key: interleaves Thought → Action → Observation; simple and debuggable from the trace; good for a handful of tool calls.
  • Place a chatbot, a RAG pipeline, and a research assistant on the agent spectrum.
    Key: chatbot = no tools/loop; RAG pipeline = workflow/tool-using (dev-wired path); research assistant = agent (model owns the loop and tool order).
  • What single property most cleanly separates "agentic" from "not"?
    Key: who decides the next step. Code decides → not an agent; the model decides at run time → agent. Autonomy over control flow is the line.
  • Why is "the model is very capable" not enough to call something an agent?
    Key: capability is orthogonal; a brilliant model in a fixed pipeline is still a chain. It is the run-time control over the loop, not intelligence, that defines an agent.

Build it (15–20 min)

Hands-on

Using the Anthropic tool-use docs, define one tool (e.g. get_weather(city) that returns a hard-coded dict) and give Claude a goal that needs it. Run the call and print the full message list after each turn. You will literally see the loop: the model emits a tool_use block (Action), you execute it and append a tool_result (Observation), then the model produces the final text. Now ask a question that needs two tool calls in sequence and watch the loop run twice. You have just built a minimal ReAct agent by hand and seen exactly where the autonomy lives.

No API key yet? Trace it on paper: write a goal, decide which tool the model would pick, invent the Observation, and continue until "final answer." See Tools & Claude to set up access.

Today's takeaways

An agent is an LLM with autonomy over its control flow: it decides the next step and tool, in a loop, until done. A chain has a path the developer fixed; an agent has a path the model chooses. The spectrum runs chatbot → workflow → tool-using → agent → multi-agent → autonomous, and the agent boundary is "the model owns the loop." ReAct (Thought, Action, Observation) is the default because the trace is readable. The senior skill is knowing when not to use an agent: deterministic, scoped, tight-latency, or safety-critical work should stay a workflow. Think of an agent as an intern with API access, and most design choices fall out naturally.
