Tool / function calling

Tool calling is the single mechanism that turns a text predictor into something that can act on the world. Get the tool definitions right and your agent feels reliable; get them wrong and it picks the wrong tool, hallucinates arguments, and you blame the model.

By the end you can
  • Explain the tool-calling round-trip: model emits a call, runtime executes, result is appended, loop continues.
  • Write a tool definition with an action-verb name, a "use when / do NOT use when" docstring, tight typed params, and a return description.
  • Decide when tools should run in parallel versus serially.
  • Diagnose and prevent hallucinated tool calls (bad name vs bad arguments).
  • Make every write tool idempotent so retries are safe.

1 · The model does not run your code

The first thing to internalize: the LLM never executes anything. When you give a model a set of tools, you are handing it a menu of function signatures, not the functions. The model's only power is to emit a structured request that says "I would like to call get_weather with {"city": "Berlin"}." That request comes back to your runtime as data. Your code decides whether and how to run it, executes the real function, and appends the result. Then the model gets to look at the result and decide what to do next.

This separation is the whole safety story of agents. The model proposes; your code disposes. You can validate arguments, refuse a call, require human approval, rate-limit, or log, all before a single line of the actual tool runs. Tool calling is not "the model running functions"; it is "the model asking, your runtime answering."

┌──────────┐ 1. you send: prompt + tool SCHEMAS (names, params, docs) │ YOUR │ ───────────────────────────────────────────────────► ┌───────┐ │ RUNTIME │ │ MODEL │ │ │ ◄─────────────────────────────────────────────────── │ │ └────┬─────┘ 2. model returns a STRUCTURED tool call: └───────┘ │ { name: "get_weather", args: { city: "Berlin" } } │ │ 3. YOUR code validates the args, then executes the real function │ result = get_weather(city="Berlin") → {"tempC": 7} │ │ 4. you APPEND the result as a tool_result message and send back ▼ ┌──────────┐ 5. model reads the observation, decides: another tool? or final answer? │ MODEL │ ──► "It's 7°C in Berlin." ■ (or emits another tool call → loop) └──────────┘
Interview-grade phrasing

"Tool calling is a structured request/response: the model emits a schema-validated call, my runtime executes the real function, I append the result to the conversation, and the model continues. The model never runs code, which is precisely why I can validate, gate, and log every action before it executes." That sentence answers half the agent-safety follow-ups before they are asked.

2 · Anatomy of a tool definition the model picks correctly

The model decides which tool to call almost entirely from the tool's name and description. It cannot read your implementation. So a tool definition is really a piece of prompt engineering aimed at the router. Four parts carry the weight:

  • Name: an action verb. search_orders, create_ticket, get_user_balance. Verbs tell the model what the tool does. Avoid noun-blobs like orders or vague names like helper.
  • Description: "use this when / do NOT use this when." The positive case tells the model when to reach for it; the negative scoping is what stops it from over-firing or confusing two similar tools. Negative scoping is the highest-leverage sentence in the whole definition.
  • Parameters: tight and typed. Use enums instead of free strings where the value set is known, mark required vs optional honestly, give each field a one-line description, and constrain formats (dates as ISO, IDs as the real pattern). Loose params are where hallucinated arguments breed.
  • Return description: what comes back. Tell the model the shape and meaning of the result so it can use it in the next step instead of guessing.
# A tool definition is prompt engineering for the router.
# JSON-schema shape (works with Anthropic, OpenAI, and most frameworks).
{
  "name": "search_orders",                # action verb, not "orders"
  "description": (
    "Look up a customer's past orders by email or order ID. "
    "USE WHEN the user asks about an existing order's status, items, or date. "
    "Do NOT use to CREATE or CANCEL an order (use create_order / cancel_order), "
    "and do NOT use for product catalog questions (use search_catalog)."
  ),
  "input_schema": {
    "type": "object",
    "properties": {
      "email":    {"type": "string", "format": "email",
                    "description": "Customer email; provide this OR order_id."},
      "order_id": {"type": "string", "pattern": "^ORD-[0-9]{6}$",
                    "description": "Order ID like ORD-004812."},
      "status":   {"type": "string",
                    "enum": ["any", "open", "shipped", "delivered"],   # enum, not free text
                    "description": "Filter by status; default 'any'."}
    },
    "required": []   # neither is hard-required: email OR id (enforce in code)
  }
  # returns: list of {order_id, date, items[], status, total} — say so in the description.
}

Notice the description does three jobs at once: states the positive trigger, names the sibling tools it must not be confused with, and rules out an adjacent domain. That negative scoping is what makes a 10-tool agent pick correctly instead of flailing. Write tool descriptions as if the only reader is a slightly-too-eager intern who will use the tool the moment it seems vaguely relevant.

3 · Parallel vs serial calls

Modern models can emit several tool calls in one turn when the calls are independent. If the user asks "compare the weather in Berlin and Munich," the model can request get_weather("Berlin") and get_weather("Munich") together; you execute both (ideally concurrently) and return both results in one batch. That is a latency win: two round-trips collapse into one.

But parallel only works when the calls do not depend on each other's output. If step two needs step one's result as an argument ("find the top customer, then look up their orders"), the calls must be serial: the model has to see the first Observation before it can fill in the second call's arguments. The rule is simple: independent calls can fan out in parallel; dependent calls must wait. A good agent runtime executes the independent batch concurrently and only loops back to the model when a result is needed to decide the next move.

PARALLEL (independent) SERIAL (dependent) ────────────────────── ────────────────── one turn: turn 1: ├─ get_weather("Berlin") └─ get_top_customer() → "Acme" └─ get_weather("Munich") turn 2: (now it knows the arg) run both at once, 1 round-trip └─ get_orders(customer="Acme") 2 round-trips, unavoidable

4 · Hallucinated tool calls and how to stop them

A hallucinated tool call is when the model emits a call that is wrong in one of two ways. Knowing which kind you have determines the fix:

  • Wrong name: the model invents a tool that does not exist (refund_customer when you only have create_refund). Rarer.
  • Wrong arguments: the model calls a real tool with arguments that violate the schema, a made-up ID, a date in the wrong format, a missing required field, or a plausible-but-fabricated value. This is the common case, and the dangerous one, because the call looks valid.

The defenses stack, cheapest first:

Concept

Schema enforcement / structured tool calling. Use the provider's native tool-calling format (not "ask for JSON in the prompt"). The model is constrained to emit a call matching one of your schemas, which kills most bad-name and malformed-shape errors at the source.
Registry validation. Before executing, look the tool name up in your registry and validate the arguments against the schema (Pydantic is ideal). Reject unknown names and schema violations; never execute on trust.
Tight typed args. Enums, regex patterns, and required-field rules turn "plausible garbage" into "rejected at the door." A free-text status invites invention; a four-value enum does not.
Self-correction. When validation fails, do not crash, feed the validation error back as the Observation. The model usually fixes its own argument on the next turn. A reliable agent treats a bad tool call as a recoverable step, not a fatal one.

from pydantic import BaseModel, EmailStr, field_validator

class SearchOrdersArgs(BaseModel):
    email: EmailStr | None = None
    order_id: str | None = None

    @field_validator("order_id")
    def check_id(cls, v):
        if v and not v.startswith("ORD-"):
            raise ValueError("order_id must look like ORD-004812")
        return v

def dispatch(name, raw_args, registry):
    if name not in registry:                       # bad NAME → reject
        return tool_error(f"No tool named {name}.")
    try:
        args = registry[name].schema(**raw_args)     # bad ARGS → validate
    except ValidationError as e:
        return tool_error(str(e))                  # feed error back → self-correct
    return registry[name].fn(args)                   # only now do we execute

5 · Idempotency: the rule that lets agents retry safely

Agents loop and retry: a tool times out, the runtime retries; the model re-issues a call it thinks failed; a network blip duplicates a request. For a read tool (get_balance), running twice is harmless. For a write tool (charge_card, create_order, send_email, delete_record), running twice is a double charge, a duplicate order, a second email to an annoyed customer. This is not hypothetical; it is the most common way agents cause real-world damage.

The fix is idempotency: design write tools so that calling them twice with the same logical intent has the same effect as calling them once. The standard mechanism is an idempotency key: the caller (or runtime) generates a stable key for the operation, and the tool guarantees that two calls with the same key produce a single effect, returning the original result on the duplicate. Payment APIs do exactly this. For an agent, treat idempotency keys on every write tool as non-negotiable, because you cannot promise the loop will only fire once.

def create_order(items, idempotency_key):
    # 2nd call with the same key returns the SAME order, no duplicate.
    existing = orders.get_by_key(idempotency_key)
    if existing:
        return existing                       # safe replay
    order = orders.create(items=items, key=idempotency_key)
    return order

Worked example: a bad argument, recovered

Example

User: "What's the status of my order from Tuesday?" The agent has search_orders but the user gave no ID. Watch the recovery loop.

Thought: I'll search by the order they mean. Action: search_orders(order_id="ORD-TUESDAY") ← hallucinated arg Runtime: validate → fails regex ^ORD-[0-9]{6}$ Observ.: ERROR: order_id must look like ORD-004812; I have no ID for "Tuesday". Thought: I have no valid ID. The user gave their email earlier. Action: search_orders(email="jo@acme.com", status="any") ← corrected Observ.: [ {ORD-004812, Tue, "shipped"} ] Final: "Your Tuesday order ORD-004812 has shipped."

The hallucinated ORD-TUESDAY never touched the database, the schema validator caught it. Feeding the validation error back as the Observation let the model self-correct on the very next turn. This is the difference between a brittle agent (crashes on the bad call) and a reliable one (treats a bad call as a recoverable step). The schema did the catching; the feedback loop did the fixing.

Practice — do these before revealing

Answer each out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

Your agent has two tools, get_invoice and get_receipt, and it keeps calling the wrong one because users say "bill," "invoice," and "receipt" interchangeably. You cannot retrain the model. What is the highest-leverage fix, and where exactly does it go?

Show solution

Fix it in the tool descriptions with explicit positive and negative scoping, because the model routes almost entirely off the description. Give each tool a "USE WHEN / do NOT use when" line that disambiguates the overlapping vocabulary: e.g. get_invoice → "USE WHEN the user wants the amount owed or a pre-payment document (also called a 'bill'). Do NOT use for proof of a completed payment, that is get_receipt." and the mirror image on get_receipt. The negative scoping ("do NOT use for X, use the other tool") is the single most effective sentence because it directly resolves the confusion between siblings. Optionally tighten further by naming the trigger words in each description. No retraining required: the routing decision is prompt-shaped, so you fix it in the prompt-shaped surface, the tool definition.

Problem 2

A reviewer says "just retry every failed tool call three times, problem solved." For which tools is this safe, for which is it dangerous, and what one design change makes the dangerous ones safe to retry?

Show solution

Blanket retries are safe for read / query tools (get_balance, search_orders), which have no side effects, running them N times yields the same answer. They are dangerous for write tools (charge_card, create_order, send_email, delete_record), where each retry can produce a duplicate charge, order, or email. The one design change that makes writes safe to retry is idempotency, typically via an idempotency key: the tool deduplicates on the key so a second call with the same key returns the original result instead of repeating the effect. With idempotency keys in place, the reviewer's "retry three times" becomes safe across the board; without them, it is a recipe for double-charging customers. So the correct answer is "retries are fine, but only after every write tool is made idempotent."

Problem 3

You give a model a delete_user(user_id: str) tool with a free-text user_id and no validation. List two distinct things that can go wrong, and the specific guardrail for each.

Show solution

(1) Hallucinated argument: the model can fabricate a plausible-looking user_id that does not correspond to the user it means (or to anyone), and a free-text field gives it no friction. Guardrail: tight typed args + registry/Pydantic validation, constrain the format (e.g. UUID or a known pattern) and verify the ID exists before executing, returning a validation error to the model on failure so it can self-correct. (2) Irreversible action with no oversight: deletion is destructive and a non-deterministic agent should not fire it unattended. Guardrail: classify it as a write/high-risk tool and put an approval gate (a framework interrupt) in front of it, draft the delete, pause for a human to confirm, then execute. Bonus: make it idempotent so a retried delete does not error or cascade, and log every call for audit. Together: schema validation stops the wrong target, the approval gate stops the unattended destructive action.

Interview questions for this topic

From your Notion bank's Agents and Context-Engineering modules. Drill the one-line keys, then expand each into a 90-second answer.

  • What makes a good tool definition?
    Key: action-verb name, "use this when / do NOT use when" docstring (negative scoping), tight typed params, return description.
  • What is a hallucinated tool call and how do you prevent it?
    Key: nonexistent name or schema-violating args (bad args more common); prevent via structured tool calling, registry validation, tight typed args, self-correction from the error.
  • Walk through the tool-calling round-trip.
    Key: model emits a structured call → runtime validates + executes the real function → result appended as observation → model continues or finishes. The model never runs code.
  • When can tool calls run in parallel versus serially?
    Key: independent calls fan out in one turn (latency win); dependent calls must be serial because step two needs step one's result as an argument.
  • What is idempotency in tool design and why care?
    Key: agents retry loops; create/charge/delete firing twice is dangerous; idempotency keys make duplicate calls a single effect, non-negotiable on write tools.
  • Why is enforcing structured output / tool schemas better than asking for JSON in the prompt?
    Key: native tool calling constrains generation to the schema (strongest), killing malformed shapes and most bad names at the source; prompt-asked JSON is unconstrained and breaks.
  • The 3-layer constraint model for tool/structured outputs?
    Key: L1 output format (tool calling) / L2 schema (Pydantic types + required) / L3 semantic ("does this ID/date actually make sense"). Pair calling with post-validation.

Hands-on

From the Anthropic tool-use guide, define one write tool, say book_meeting(title, date, attendee), with an action-verb name, a "USE WHEN / do NOT use when" description, an enum or pattern on at least one field, and an idempotency_key param. Then do three experiments: (1) ask a question that needs it and confirm the call's arguments are well-formed; (2) deliberately ask something ambiguous and watch whether the model invents a bad argument, then add a validation error path and see it self-correct; (3) issue the same booking request twice and verify your idempotency check returns one booking, not two. Today's takeaways

Tool calling is a structured request/response: the model emits a call, your runtime validates and executes it, the result is appended, the loop continues, and the model never runs code itself. The model routes almost entirely off the tool's name and description, so a tool definition is prompt engineering: action-verb name, "use when / do NOT use when" scoping, tight typed params, return description. Independent calls fan out in parallel; dependent calls go serial. Hallucinated calls are usually bad arguments, not bad names, and you defend with schema enforcement, registry validation, typed args, and self-correction from the error. Finally, make every write tool idempotent, because the loop will retry, and an idempotency key is what keeps a retry from becoming a double charge.
