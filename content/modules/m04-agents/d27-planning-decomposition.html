Module 4 · Agents & Tool Use Day 27 55 min

Planning & decomposition

An agent is only as good as the plan it follows. Today you learn how to turn a fuzzy goal into executable sub-tasks, when an explicit planner beats a reactive loop, and how reflection makes an agent better only when a cheap, reliable error signal exists. You also learn the failure that quietly kills agents in production: planning where errors compound.

By the end you can
  • Decompose a goal into a dependency graph (DAG) of sub-tasks and explain what the edges mean.
  • Contrast planner-executor with reactive ReAct and say which one a given task wants.
  • State the one precondition under which a reflection or Reflexion loop actually helps.
  • Distinguish self-correction (fix a known error) from reflection (critique without a signal).
  • Recognize a loop-detection failure (the model repeats the same tool call) and fix it.
  • Decide when planning helps and when it hurts, by reasoning about error compounding.

1 · Decomposition: a goal becomes a graph

A task like "draft a competitor analysis for our pricing page" is not one action. It is a small set of sub-tasks with dependencies: gather competitor prices, gather our prices, compute the deltas, then write the summary. The write step cannot start until the three gather/compute steps are done, but the gather steps are independent of each other. That structure is a directed acyclic graph (DAG): nodes are sub-tasks, edges are "must finish before," and the absence of cycles is what lets you actually run it to completion.

Why bother making the graph explicit instead of letting the model "figure it out" turn by turn? Two reasons. First, parallelism: independent nodes can run at once, which a strictly sequential loop never exploits. Second, legibility: a plan you can print is a plan you can inspect, cache, partially re-run, and debug. The cost is rigidity, which is exactly the trade-off the rest of this lesson is about.

GOAL: "Competitor analysis for our pricing page" ┌─ gather competitor prices ─┐ │ │ ├─ gather our prices ────────┼──► compute deltas ──► write summary ──► ■ │ │ └─ (independent, run in ─────┘ parallel) sequential: each needs the prior nodes = sub-tasks · edges = "must finish before" · no cycles = it terminates
Concept

Decomposition is turning a goal into a DAG of sub-tasks: nodes are units of work, edges are ordering constraints, and parallelizable nodes share no path. The acyclicity is not cosmetic. It is the guarantee that the plan has a finite execution.

2 · Planner-executor vs reactive ReAct

You met the ReAct loop on Day 24: the model interleaves Thought, Action, and Observation, deciding the very next step each turn from what it just saw. It plans implicitly and incrementally. That is ideal when the path is short and unpredictable, because the model adapts to each observation and you never pay to plan steps you will not take.

A planner-executor splits the two jobs. A planner LLM emits the whole plan up front (often the DAG). An executor runs each node, frequently with no LLM call at all if the node is a deterministic tool. A re-planner (optional) inspects results and revises the remaining plan when reality diverges. This pays off when there are many steps, several of them parallel, and the structure is roughly knowable before you start. You spend one expensive planning call to unlock parallel execution and a stable, inspectable trajectory.

DimensionReAct (reactive)Planner-Executor
When the plan is decidedOne step at a time, mid-loopUp front, then revised
Best forFew steps, unpredictable path>6 steps, many parallel, knowable
ParallelismNone (strictly serial)Independent nodes run together
Adapts to surprisesImmediately, every turnOnly at re-plan points
DebuggabilityRead the traceRead the plan, then the trace
Interview-grade phrasing

"ReAct plans implicitly, one step at a time, and wins when the path is short and unpredictable. Plan-Execute plans explicitly, emits a DAG up front, and wins when there are more than a handful of mostly-parallel steps you can foresee. The planner trades adaptivity for parallelism and inspectability." That answer maps cleanly onto the Notion bank's Plan-Execute question.

3 · Tree-of-thought, at a glance

ReAct and a single plan both commit to one line of reasoning. Tree-of-thought generalizes this: at each step the model proposes several candidate next thoughts, scores them (self-evaluation or a heuristic), keeps the promising branches, and explores, more like search than a single chain. It can solve puzzles a linear chain fails on, because it can back out of a dead end.

The catch is cost. Exploring a tree multiplies LLM calls by the branching factor at every level, so it is reserved for hard reasoning problems where a wrong single chain reliably fails (constraint puzzles, some planning and math). For most production agent work it is overkill. Know it exists, know the shape, and reach for it only when a linear approach demonstrably cannot find the answer.

4 · Reflection and Reflexion: only with a cheap, reliable signal

Reflection means the agent critiques its own output and tries again. Reflexion is the named pattern: act, observe an outcome, write a short verbal self-critique, store it in memory, and let that critique condition the next attempt. It sounds universally good. It is not.

The whole value of a reflection loop depends on the quality of the error signal the critique is built on. When the signal is cheap and reliable, reflection is a near-free accuracy boost: the agent gets told concretely what was wrong and fixes it. When the signal is just the model grading itself with no external ground truth, reflection often launders a wrong answer into a confidently wrong answer, and you have paid double the tokens for it.

Where the signal is cheap and reliable (reflect)

The agent's output is verifiable by something other than the model:

  • Code runs: execute it, feed the stack trace back, retry. The interpreter is ground truth.
  • JSON parses: validate against a schema; a parse or validation error is a precise, free critique.
  • SQL returns: run the query; an error or an empty/obviously-wrong result is a concrete signal.
  • A test suite: pass/fail per case is the cleanest signal there is.

In every case the feedback is external, deterministic, and nearly free. That is the precondition.

Where reflection backfires

"Is this essay good?" "Is this strategy sound?" "Did I answer the user's intent?" with no rubric, no execution, and no external check. The model critiques itself from the same parametric beliefs that produced the answer, so it tends to either rubber-stamp or hallucinate a problem. Without a cheap reliable error signal, a reflection loop mostly adds latency and cost. Add a real check (a judge with a rubric, a schema, an executor) first, or skip the loop.

Interview-grade phrasing

"Reflexion only pays off when there is a cheap, reliable error signal to reflect on, code that runs, JSON that parses, SQL that returns, a test that passes. Then it is self-correction and it works. Without an external signal, the model just grades itself and you get a confidently wrong answer for twice the tokens." Saying the precondition out loud is the senior move.

5 · Self-correction vs reflection

The two terms get blurred, so separate them. Self-correction is the agent reacting to a concrete, observed error, a stack trace, a 422 validation response, a failing assertion, and producing a fix. It is tight, grounded, and high-value precisely because the error is real. Reflection is broader: the agent critiquing an output that may have nothing visibly wrong with it. Self-correction is reflection with a guaranteed signal; that is why it is the reliable half of the family.

Practically: build self-correction whenever your tools can fail loudly (validators, executors, type checks). Feed the error text straight back into the next turn so the model corrects against the real message rather than a guess about what might be wrong. You will implement exactly this in Module 4's lab.

6 · Loop detection: the model repeats the same tool call

A very common agent failure: the model calls the same tool with the same (or nearly the same) arguments, gets the same unhelpful result, and calls it again, forever, until your max-steps cap fires. Understanding why tells you how to fix it.

Why it happens. The tool returned something the model cannot use (an empty result, an error it does not know how to act on, a result that does not advance the goal), but nothing in the context tells the model "you already tried this and it did not work." Each turn looks fresh to a stateless predictor, so it makes the same locally-plausible choice again. It is a missing-memory problem, not a stupidity problem.

turn 1: search("Q3 revenue") → [] (empty) turn 2: search("Q3 revenue") → [] (same call, same empty result) turn 3: search("Q3 revenue") → [] ... loop until max_steps ■ FIX: (1) detect the repeat (2) surface it to the model (3) cap as a backstop

How to fix it. Three complementary moves:

  • Detect the repeat. Hash each (tool name, arguments) call. If the same hash appears N times in a row (often 2 to 3), you have a loop.
  • Surface it to the model. Inject an observation like "You already called search('Q3 revenue') twice and it returned nothing. Try a different query or a different tool." Now the context carries the memory the model lacked, and it can break out.
  • Cap as a backstop. A hard max_steps (and a per-tool call limit) guarantees termination even if detection misses an edge case. The cap is a safety net, not the primary fix.

Note how this connects to the previous lesson: loop detection is a short-term-memory feature. The repeated calls are invisible to the model precisely because nothing summarized "we are stuck."

Worked example: when planning helps, and when it hurts

Example

Task A (planning helps). "Build a slide deck section comparing our four products on price, latency, and support tier." The structure is fully knowable: for each of 4 products, look up 3 attributes (12 independent lookups), then assemble one summary. A planner emits a 12-node parallel layer plus one assembly node. Twelve lookups run together, the deck section is assembled once. Few decision points, many parallel steps, knowable shape: a textbook planner win.

Task B (planning hurts). "Debug why this user's checkout failed." You cannot plan step 5 until you see the result of step 1, because step 1's log might send you to the payment provider, the inventory service, or the user's own bad input. A rigid up-front plan would be wrong by step 2, and a re-planner firing every step is just ReAct with extra overhead. Here a reactive loop is correct: decide each next step from the last observation.

Why planning can hurt: errors compound

The compounding-error argument, derived not memorized

A plan with k steps that each succeed with probability p succeeds end to end with probability roughly pk. At p = 0.95 and k = 10 that is about 60%; at k = 20 it is about 36%. Long autonomous plans are multiplicatively fragile, because one early wrong sub-task poisons everything downstream and the agent keeps marching. So planning helps when steps are reliable and largely independent (a failure is local), and hurts when steps are interdependent and error-prone (a failure cascades). The senior instinct: keep plans short, make each step verifiable, and re-plan or fall back to a reactive loop the moment reality diverges from the plan.

Practice — do these before revealing

Say each answer out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

A teammate wraps every agent output in a Reflexion loop ("critique your answer, then improve it") and reports that quality went down on a customer-support summarizer. The same loop clearly helped on a code-generation agent. Explain the difference and what you would change for the summarizer.

Show solution

Reflection only helps when there is a cheap, reliable external error signal to critique against. The code agent has one: the code runs (or does not), so the critique is grounded in a real stack trace or failing test and the retry is a true self-correction. The summarizer has none: "is this summary good?" is the model grading itself from the same beliefs that wrote it, so the loop mostly rubber-stamps or invents a problem and burns double the tokens. The fix is to give the summarizer a real signal before reflecting: an LLM-as-judge with an explicit rubric (faithfulness to source, coverage of key points), a check that the summary only contains claims present in the source, or simply drop the loop. Add the check, then reflect on the check's verdict, not on the model's vibe.

Problem 2

Your agent intermittently spins: it calls get_order(id=null) three times in a row, then hits the step cap and fails. Walk through why this happens and design a fix that does more than just raise the cap.

Show solution

It is a loop, caused by missing memory plus a bad argument. The tool returns an error for id=null, but nothing in the context tells the model "you already tried this and it failed," so each turn it makes the same locally-plausible call again. Raising the cap just delays the failure. The real fix is three layered moves: (1) detect the repeat by hashing (tool name, args) and flagging when the same hash recurs 2 to 3 times; (2) surface it by injecting an observation such as "get_order with id=null has failed twice; the id is missing, ask the user or look it up first," so the context now carries the memory the model lacked and it can correct the bad argument; (3) keep the max_steps cap as a backstop so termination is still guaranteed if detection misses. Note the root cause is the null id, a self-correctable error if the failure message is fed back, which is a cleaner fix than fighting the loop.

Problem 3

A PM wants to "make the agent smarter" by having it generate a full 15-step plan for every request before acting. Most requests are 2 to 3 steps and unpredictable. Argue for or against, using the compounding-error idea, and propose a default.

Show solution

Argue against as the default. For 2 to 3 step, unpredictable requests, an up-front 15-step plan is wrong by step 2 because later steps depend on results you have not seen yet, and you pay an expensive planning call to produce a plan you will mostly discard. The compounding-error point makes it worse: a long plan succeeds end to end only at roughly pk, so a 15-step plan of even 95%-reliable steps lands near 46%, and an early mistake cascades through every downstream step while the agent keeps marching. The right default is a reactive ReAct loop for these short, unpredictable tasks, and reserve explicit planning for the minority of requests with many mostly-parallel, knowable steps. If you want a single policy, use ReAct by default and escalate to a planner only when the task is detected to have greater than ~6 foreseeable, largely independent steps.

Interview questions for this topic

From your Notion bank's Agents slice and common 2026 screens. Drill the one-line keys, then expand each into a 90-second answer with Claude.

  • Plan-Execute vs ReAct — when does Plan-Execute win?
    Key: >6 mostly-parallel steps knowable upfront; planner emits a DAG, executor runs, re-planner inspects and revises.
  • What is the ReAct loop and why is it still the single-agent default?
    Key: interleaves Thought → Action → Observation; simple, debuggable from the trace, good for a handful of tool calls and unpredictable paths.
  • When does reflection (Reflexion) actually help an agent?
    Key: only with a cheap, reliable error signal, code runs, JSON parses, SQL returns, a test passes; otherwise the model just grades itself.
  • An agent keeps repeating the same tool call. Why, and how do you fix it?
    Key: missing memory, the result was unusable and nothing said "already tried"; detect repeats by hashing (tool,args), surface it back, cap max_steps as a backstop.
  • When does explicit planning hurt rather than help?
    Key: interdependent, error-prone, unpredictable steps; errors compound (~p^k), a rigid plan is wrong early; prefer a reactive loop and re-plan on divergence.
  • Self-correction vs reflection — what is the difference?
    Key: self-correction reacts to a concrete observed error (stack trace, 422, failing test); reflection critiques without a guaranteed signal. Self-correction is the reliable half.
  • What is tree-of-thought and when is it worth the cost?
    Key: branch and score multiple candidate thoughts, search rather than a single chain; reserved for hard reasoning where a linear chain reliably fails, expensive otherwise.

Build it (15–25 min)

Hands-on

Write a tiny self-correcting loop with the Claude API and a JSON schema. Ask the model to produce a JSON object matching a Pydantic model with a couple of required fields and an enum. Validate the output; on a ValidationError, feed the exact error string back into the next turn as a message ("Your output failed validation: <error>. Fix it and return only valid JSON.") and retry, capping at 3 attempts. Log each attempt. You have now built self-correction against a cheap, reliable signal (the validator), the exact precondition from section 4. Then deliberately break it: remove the validation step and instead ask the model to "reflect on whether the JSON is good" with no schema, and watch the loop add cost without adding correctness. See the tool use docs and LangGraph docs for stateful loop patterns.

Stretch: add loop detection. Hash each retry's output; if two consecutive attempts are byte-identical, stop early and report "model is stuck" instead of burning the remaining retries.

Today's takeaways

Decomposition turns a goal into a DAG: nodes are sub-tasks, edges are ordering constraints. ReAct plans implicitly and wins on short, unpredictable paths; a planner-executor plans explicitly and wins on many knowable, parallel steps. Reflection helps only when a cheap, reliable error signal exists (code runs, JSON parses, SQL returns), otherwise it launders confident wrongness. Loop detection is a memory fix: detect repeated (tool, args), surface it, and cap as a backstop. And planning hurts whenever errors compound, so keep plans short, verifiable, and ready to fall back to a reactive loop.
