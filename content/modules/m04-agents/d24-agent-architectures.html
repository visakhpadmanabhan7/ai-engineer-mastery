Agent architectures

Once one ReAct loop is not enough, you have choices: plan ahead, or split the work across multiple agents. Today you learn which architecture fits which problem, so you can answer "would you use one agent or several here?" with a reason instead of a guess.

By the end you can
  • Contrast single-agent ReAct with Plan-Execute and say when each wins.
  • Name the three multi-agent topologies, supervisor, swarm, hierarchical, and their defaults.
  • Decide when to split into multiple agents versus one bigger agent.
  • Explain filtered handoffs and why they prevent state leakage.
  • Handle large tool spaces with tool retrieval / routing instead of dumping 50 tools in one prompt.

1 · ReAct vs Plan-Execute

You met ReAct on Day 22: one loop, Thought → Action → Observation, the model decides the next step after seeing the last result. It is the right default for tasks with a handful of mostly-sequential steps, because it is simple and the trace is readable. Its weakness: it decides one step at a time, so it cannot exploit steps that could all run at once, and on long tasks it can drift (lose the thread of the original goal after many turns).

Plan-Execute separates thinking from doing. A planner reads the goal and emits a plan upfront, often a DAG (directed acyclic graph) of steps with dependencies. An executor runs the steps, fanning out the independent ones in parallel. A re-planner inspects the results and revises the plan if reality diverged (a step failed, a result changed the picture). The win is when there are more than roughly six steps that are mostly knowable upfront and largely parallel: the planner sees the whole shape at once and the executor exploits the parallelism, where ReAct would plod through them one Observation at a time.

ReAct (decide-as-you-go) Plan-Execute (decide-then-do) ────────────────────────── ───────────────────────────────── Thought→Action→Obs PLAN: step1 ─┐ ↓ (then decide next) step2 ─┼─► (independent, parallel) Thought→Action→Obs step3 ─┘ ↓ step4 (needs step1) ← DAG edges Thought→Action→Obs EXECUTE: run 1,2,3 at once → then 4 ↓ RE-PLAN: did anything fail / change? Final if so, revise plan and rerun best for ≤ ~handful of best for >6 mostly-parallel, sequential, adaptive steps upfront-knowable steps
Interview-grade phrasing

"ReAct decides one step at a time off the last observation, simple and debuggable, ideal for a handful of sequential steps. Plan-Execute commits a DAG upfront, runs independent steps in parallel, and re-plans on divergence, which pays off once there are more than about six mostly-parallel steps. I start with ReAct and only reach for Plan-Execute when the parallelism and step count justify the added machinery." That answer shows you choose architecture by problem shape, not by hype.

2 · One bigger agent, or several?

The instinct to "add another agent" is usually premature. More agents means more coordination, more places for context to get lost in handoffs, more latency, and a harder-to-trace system. The default is one agent until a concrete pressure forces a split. Two pressures genuinely justify splitting:

  • Distinct tool domains. When the task spans clearly separate areas of expertise with their own tools (a "research" set, a "coding" set, a "billing" set), one agent juggling all of them confuses the routing and bloats its context. Splitting by domain gives each sub-agent a small, coherent tool set it uses well.
  • Too many tools. Past roughly 30 tools, a single agent's tool-selection accuracy degrades, the model has too many similar options and picks wrong. Partitioning the tools across specialized agents keeps each agent inside the ~10-20-tool sweet spot where routing is reliable.

Critically, splitting is not about making each agent "smarter." It is about keeping each agent's decision space small so it routes accurately. If your reason for adding an agent is not "this shrinks some agent's tool space or separates distinct domains," you probably do not need it. A focused single agent beats a sprawling multi-agent system that leaks context at every handoff.

3 · The three multi-agent topologies

When you do split, the agents have to coordinate, and there are three standard shapes. Pick by how much you value debuggability versus speed, and by how many agents you have.

SUPERVISOR (default) SWARM HIERARCHICAL (>20 agents) ──────────────────── ───── ───────────────────────── ┌───────────┐ A ⇄ B ┌─────────┐ │ SUPERVISOR│ ↑ ╲ ↓ │ top sup │ │ (router) │ │ ╲│ └────┬────┘ └─┬──┬───┬──┘ D ⇄ C ┌───────┼───────┐ │ │ │ ┌──┴──┐ ┌──┴──┐ ┌──┴──┐ ┌─┘ │ └─┐ agents hand │ sub │ │ sub │ │ sub │ ┌─┴─┐┌─┴─┐ ┌─┴─┐ off DIRECTLY │ sup │ │ sup │ │ sup │ │ A ││ B │ │ C │ to each other └──┬──┘ └──┬──┘ └──┬──┘ └───┘└───┘ └───┘ (no central workers workers workers one router decides router) teams of agents, who runs; results routed in layers return to supervisor
  • Supervisor (the default). One routing LLM receives the task, decides which specialist agent runs, collects the result, and decides what is next. Every decision flows through one place, so it is the most debuggable: the trace is a clean sequence of "supervisor picked A, A returned X, supervisor picked B." The cost is latency (the supervisor is a bottleneck and adds a hop per delegation) and a single point of routing failure. Choose this unless you have a specific reason not to.
  • Swarm (direct handoffs). Agents hand control directly to each other (agent A decides "this is really a billing question" and hands off to the billing agent) with no central router. It is faster (no supervisor hop) but harder to trace, because control bounces peer-to-peer and there is no single timeline. Use it when low latency matters more than a clean audit trail and the handoff logic is simple.
  • Hierarchical. Supervisors of supervisors, teams of agents routed in layers. This only earns its complexity at large scale (>20 agents), where a single flat supervisor would itself be choosing among too many options. Below that, it is over-engineering.

4 · Filtered handoffs (stop the context leak)

The quiet killer of multi-agent systems is state leakage: when one agent hands off to another, the naive move is to pass the entire conversation and scratchpad along. Now the billing agent is reasoning over the research agent's half-finished thoughts, irrelevant tool outputs, and dead-end attempts. Its context is polluted, its routing degrades, and its token bill climbs. The more agents in the chain, the worse the accumulation.

The fix is a filtered handoff: pass only what the receiving agent needs, a clean task statement and the specific facts relevant to it, not the whole history. Think of it like delegating at work: you do not forward an employee every email you have ever read, you give them the task and the few inputs they need. Filtered handoffs keep each agent's context tight and on-topic, which is exactly what keeps routing accurate. A clean handoff interface (task + scoped inputs + expected output) is the difference between a multi-agent system that scales and one that drowns in its own history.

Anti-pattern

Passing the full message history and every prior tool result on every handoff. It feels safe ("more context can't hurt") but it leaks irrelevant state into each agent, degrades routing, inflates cost, and compounds with each hop. Hand off a scoped task and only the inputs that agent needs.

5 · Large tool spaces: do not dump 50 tools in one prompt

The single most common multi-agent mistake at the architecture level is treating "we have 50 tools" as "give one agent 50 tools." Tool-selection accuracy falls off well before that, the model faces too many similar descriptions and picks wrong (the bad-argument and wrong-tool problems from Day 23 get worse with scale). The sweet spot for one agent is roughly 10-20 tools. Past that, restructure rather than overload. Three standard techniques:

  • Tool retrieval (semantic routing). Embed every tool's description, and at each step retrieve only the top 5-10 tools relevant to the current query. The agent only ever sees a small, relevant menu, so it routes accurately even though the catalog is huge. This is RAG applied to the tool list.
  • Categorical / hierarchical routing. Group tools into categories (a "calendar" group, a "CRM" group); a first decision picks the category, a second picks the tool within it. Two small choices beat one giant one.
  • Supervisor over specialist agents. Partition the tools across domain agents (section two) and let a supervisor route to the right agent, who then chooses among its small tool set. This is the multi-agent answer to a large tool space.

All three share one principle: keep the number of options in front of the model small at the moment of choice. Whether you achieve that by retrieving tools, nesting categories, or splitting agents, the goal is the same, accurate routing through a small decision space.

Worked example: design a customer-ops assistant

Example

Requirement: an internal assistant that can (a) answer billing questions, (b) look up and update CRM records, (c) search the docs/knowledge base, and (d) file engineering tickets. About 35 tools total across these areas. What architecture?

35 tools across 4 distinct domains → ONE agent would (a) exceed ~30 tools and (b) blur 4 separate tool sets. Split by DOMAIN, route via SUPERVISOR: ┌──────────────────┐ │ SUPERVISOR │ routes by intent, debuggable trace └──┬────┬────┬───┬─┘ ┌───────────┘ │ │ └───────────┐ ┌─────┴─────┐ ┌───────┴──┐ ┌─┴────────┐ ┌──┴────────┐ │ BILLING │ │ CRM │ │ DOCS │ │ TICKETS │ │ agent │ │ agent │ │ agent │ │ agent │ │ ~8 tools │ │ ~10 tools│ │ ~4 tools │ │ ~6 tools │ ← each in the sweet spot └───────────┘ └──────────┘ └──────────┘ └───────────┘ handoffs are FILTERED: supervisor passes the scoped task + needed IDs only, not the whole history. Write tools (update CRM, charge) are idempotent + gated.

Why this and not one big ReAct agent? Two of the split triggers fire at once: distinct tool domains (billing vs CRM vs docs vs tickets) and >30 tools total. A supervisor is the default topology (debuggable, and we are nowhere near the 20-agent threshold that would call for hierarchical). Filtered handoffs keep the CRM agent from drowning in billing context. And because some tools write (update a record, issue a credit), they stay idempotent with an approval gate, the Day 23 rules ride along into the architecture. Note what we did not do: no swarm (we want the clean audit trail), no hierarchy (too few agents). Every choice traces to a stated pressure.

Problem 1

A task always needs the same eight independent lookups (eight different data sources), then one synthesis step that uses all of them. An engineer built it as a ReAct loop and it is slow. What architecture fits better and why, specifically?

Show solution

Plan-Execute fits better. The key facts: the steps are (a) knowable upfront (always the same eight lookups), (b) mostly independent (the eight do not depend on each other), and (c) there are more than six of them, exactly the regime where Plan-Execute beats ReAct. ReAct is slow here because it decides one step at a time off the last observation, so it runs the eight lookups sequentially even though they could all run at once. A planner emits a DAG (eight parallel leaves feeding one synthesis node), the executor fans the eight out concurrently in a single wave, and the synthesis step runs once they return. That collapses eight sequential round-trips into roughly one parallel wave plus the synthesis, a large latency win. The re-planner can still handle a failed lookup. The whole reason it pays off is the parallelism the fixed, decide-as-you-go ReAct loop could not exploit.

Problem 2

Your multi-agent support system works but is "weirdly dumb": the billing agent sometimes answers as if it were mid-way through a research task. Diagnose the most likely cause and the fix.

Show solution

The most likely cause is state leakage through unfiltered handoffs: when control passes to the billing agent, the system is forwarding the entire conversation and scratchpad, including the research agent's half-finished thoughts and tool outputs, so the billing agent reasons over polluted, off-topic context and "acts like" it is still doing research. The fix is a filtered handoff: pass only a clean, scoped task statement plus the specific inputs the billing agent needs (the account ID, the question), not the whole history. Define a tight handoff interface (task + scoped inputs + expected output). This keeps each agent's context relevant, which restores accurate routing and behavior, and as a bonus cuts token cost. The general rule: more context is not safer in multi-agent systems; irrelevant context actively degrades each agent, and the damage compounds with every hop.

Problem 3

You inherit an agent with 60 tools crammed into one prompt; it frequently calls the wrong tool. You are told "do not just split it into agents, find the simplest fix first." What do you reach for, and what is the underlying principle shared by every valid fix?

Show solution

The simplest fix that does not introduce multiple agents is tool retrieval (semantic routing): embed all 60 tool descriptions, and at each step retrieve only the top 5-10 tools relevant to the current query, so the agent sees a small, relevant menu instead of all 60. (It is RAG applied to the tool list.) A second option short of full multi-agent is categorical / hierarchical routing: pick a tool category first, then a tool within it, two small choices instead of one 60-way choice. Full multi-agent (supervisor over specialists) is the heavier option you were told to avoid first. The underlying principle shared by all of them: keep the number of options in front of the model small at the moment of choice. Tool-selection accuracy collapses with too many similar options (well past the ~10-20 sweet spot), so every fix works by shrinking the decision space the model faces on any given step, whether by retrieving, nesting, or splitting.

Interview questions for this topic

From your Notion bank's Agents module. Drill the one-line keys, then expand each into a 90-second answer.

  • Plan-Execute vs ReAct, when does Plan-Execute win?
    Key: >6 mostly-parallel steps knowable upfront; planner emits a DAG, executor runs, re-planner inspects and revises. ReAct stays default for a handful of sequential steps.
  • Supervisor vs Swarm vs Hierarchical topology?
    Key: supervisor = one routing LLM (default, debuggable, slower); swarm = direct handoffs (faster, harder to trace); hierarchical for >20 agents.
  • When do you split into multiple agents instead of one bigger agent?
    Key: distinct tool domains, or >30 tools degrading selection. Goal is a small decision space per agent, not making each "smarter."
  • How do you handle a large tool space (50+) without wrong picks?
    Key: tool retrieval (embed descriptions, top 5-10), categorical/hierarchical routing, or a supervisor over specialists; ~10-20 tools is the sweet spot.
  • What is a filtered handoff and what does it prevent?
    Key: pass only the scoped task + needed inputs, not the full history; prevents state leakage that pollutes the receiving agent's context and degrades routing.
  • Why is the supervisor topology the default despite being slower?
    Key: every decision flows through one router, so the trace is clean and debuggable; you trade some latency for observability, which usually wins.

Today's takeaways

Start with single-agent ReAct; reach for Plan-Execute only when there are more than about six mostly-parallel, upfront-knowable steps, because that is where committing a DAG and running it in parallel beats deciding one step at a time. Stay with one agent until a concrete pressure, distinct tool domains or more than ~30 tools, forces a split, and remember that splitting is about shrinking each agent's decision space, not making it smarter. Of the topologies, supervisor is the debuggable default, swarm trades traceability for speed, and hierarchical only earns its keep past ~20 agents. Use filtered handoffs (scoped task + needed inputs, never the whole history) to stop state leakage. And for large tool spaces, keep the options small at the moment of choice via tool retrieval, categorical routing, or a supervisor over specialists.
