Module 4 · Agents & Tool Use Day 30 120 min

Lab — a tool-using agent

This is the Module 4 capstone. You will build a small ReAct agent from scratch: two or three typed tools (a calculator, a search/retrieval tool, and a write tool guarded by approval), a max-steps guard, structured tool calls validated with Pydantic, parse-error feedback fed into the next turn, and a trajectory log. Then you add one evaluation, tool-call accuracy on a handful of scripted tasks. Every concept from Days 22 to 29 shows up as a line of code you can run.

By the end you can
  • Implement a ReAct loop (Thought, Action, Observation) as a plain Python loop.
  • Define typed tools and validate the model's tool calls with Pydantic.
  • Feed a validation or parse error back into the next turn as self-correction.
  • Guard a write tool behind an approval gate and bound the loop with max steps.
  • Emit a trajectory log of every step, tool, args, and observation.
  • Write a tool-call-accuracy eval over scripted tasks and read its results.

What you are building

One file, runnable, no framework required (a LangGraph version is sketched at the end). The agent answers a request by looping: the model proposes a tool call, you validate and execute it, you feed the observation back, and it loops until it emits a final answer or hits the step cap. The architecture is exactly the ReAct default from Day 24, hardened with the reliability mechanisms from Day 28 and instrumented for the eval from Day 29.

user request │ ┌────▼─────────────────────────────────────────────────────┐ │ LOOP (max_steps cap — Day 28 hard cap) │ │ model → proposes {tool, args} │ Thought + Action │ Pydantic validate args ──fail──► feed error back ──┐ │ (self-correction, Day 27) │ │ ok │ │ │ dispatch: calc | search (read, run freely) │ │ │ write → APPROVAL GATE (Day 28) ─────────┘ │ Observation │ append observation → next turn │ │ log step → trajectory │ (trace, Day 28/29) └────┬─────────────────────────────────────────────────────┘ ▼ final answer + trajectory log ──► tool-call-accuracy eval (Day 29)

Checkpoint 0 · Setup and the tools

Define three typed tools. Each has a clear name, a tight Pydantic argument model, and a docstring stating when to use it. The write tool (save_note) is the one we will gate. We tag each tool read or write, exactly the classification from Day 28.

# pip install anthropic pydantic
from pydantic import BaseModel, Field, ValidationError
from typing import Literal
import json, hashlib

# ---- typed argument models (Day 23: tight, typed params) ----
class CalcArgs(BaseModel):
    expression: str = Field(description="a pure arithmetic expression, e.g. '(120*0.2)+5'")

class SearchArgs(BaseModel):
    query: str = Field(min_length=2, description="keywords to look up in the notes store")

class SaveArgs(BaseModel):
    title: str
    body: str

# ---- tool implementations ----
NOTES = {"return policy": "Returns accepted within 30 days with receipt.",
         "shipping": "Standard shipping is 3-5 business days."}
SAVED = {}

def calc(a: CalcArgs):
    return str(eval(a.expression, {"__builtins__": {}}, {}))   # sandboxed eval, demo only

def search(a: SearchArgs):
    hits = [v for k, v in NOTES.items() if a.query.lower() in k]
    return json.dumps(hits) if hits else "[]"

def save_note(a: SaveArgs, idem_key: str):
    if idem_key in SAVED:                 # Day 28: idempotency — retry is safe
        return f"already saved (key {idem_key[:8]})"
    SAVED[idem_key] = (a.title, a.body)
    return f"saved '{a.title}'"

# ---- registry: name, schema, fn, and READ/WRITE tag (Day 28) ----
TOOLS = {
    "calc":      {"schema": CalcArgs,   "fn": calc,      "kind": "read"},
    "search":    {"schema": SearchArgs, "fn": search,    "kind": "read"},
    "save_note": {"schema": SaveArgs,   "fn": save_note, "kind": "write"},
}
Lab caveat

The eval() calculator and in-memory dicts are for the lab only. In production a calculator tool uses a safe expression parser, not eval, and the notes store is a real database. The point here is the agent loop and its guards, not the tools themselves.

Checkpoint 1 · The ReAct loop with structured tool calls

Now the core. We use Claude's native tool calling so the model returns a structured tool name and a JSON args object (Day 21's "enforce structure with tool calling"). Each turn: send the conversation, read the model's chosen tool, validate the args with that tool's Pydantic schema, and, on failure, feed the exact error back so the next turn can self-correct (Day 27). A max_steps cap bounds the loop (Day 28). We log every step into a trajectory list (Day 29).

import anthropic
client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY

# Anthropic tool specs derived from the Pydantic schemas
TOOL_SPECS = [
    {"name": name,
     "description": spec["schema"].__doc__ or name,
     "input_schema": spec["schema"].model_json_schema()}
    for name, spec in TOOLS.items()
]

def run_agent(user_msg, approver, max_steps=8):
    messages = [{"role": "user", "content": user_msg}]
    trajectory = []                                  # Day 29: the trace

    for step in range(max_steps):                     # Day 28: hard cap
        resp = client.messages.create(
            model="claude-sonnet-4-5", max_tokens=1024,
            tools=TOOL_SPECS, messages=messages,
            system=("You are a careful assistant. Use search for facts in the "
                    "notes store, calc for arithmetic, save_note to persist a note. "
                    "When done, reply with a final answer and no tool call."))

        tool_uses = [b for b in resp.content if b.type == "tool_use"]
        if not tool_uses:                            # final answer → stop
            text = "".join(b.text for b in resp.content if b.type == "text")
            trajectory.append({"step": step, "final": text})
            return text, trajectory

        messages.append({"role": "assistant", "content": resp.content})
        results = []
        for tu in tool_uses:
            obs = dispatch(tu.name, tu.input, approver)   # validate + gate + run
            trajectory.append({"step": step, "tool": tu.name,
                               "args": tu.input, "obs": obs})
            results.append({"type": "tool_result", "tool_use_id": tu.id,
                            "content": obs})
        messages.append({"role": "user", "content": results})  # Observation

    return "stopped: hit max_steps", trajectory      # Day 28 backstop fired

Checkpoint 2 · dispatch: validation, approval gate, idempotency

The dispatch function is where the reliability layers live. It validates the args against the tool's Pydantic schema and, on ValidationError, returns the error text as the observation, that string becomes the model's next-turn signal and lets it self-correct (Day 27). If the tool is a write, it routes through the approval gate before executing and attaches an idempotency key so a retry cannot double-write (Day 28).

def dispatch(name, raw_args, approver):
    spec = TOOLS.get(name)
    if spec is None:
        return f"error: no such tool '{name}'"     # registry check (Day 28)
    try:
        args = spec["schema"](**raw_args)          # Pydantic validation (Day 21)
    except ValidationError as e:
        return f"validation error, fix the args and retry: {e}"   # self-correct

    if spec["kind"] == "write":               # Day 28: gate the write
        if not approver(name, args.model_dump()):
            return "action rejected by approver"
        idem = hashlib.sha256(json.dumps(args.model_dump(),
                              sort_keys=True).encode()).hexdigest()
        return spec["fn"](args, idem)         # idempotency key passed in
    return spec["fn"](args)                    # reads run freely

# approver: a policy or a human. Auto-approve, else ask on console.
def auto_approve(name, args): return True
def human_approve(name, args):
    return input(f"approve {name}({args})? [y/N] ").strip().lower() == "y"

# run it:
ans, traj = run_agent("What is our return window, and save a note titled "
                      "'policy' with the answer.", approver=human_approve)
print(ans)
for s in traj: print(s)
What you just built, in interview terms

"A ReAct agent with structured tool calls validated by Pydantic, parse errors fed back as self-correction, a read/write tool split with an approval gate and idempotency on the write, a max-steps backstop, and a trajectory log." That single sentence is most of Module 4, and every clause is a few lines you can point to.

Checkpoint 3 · One evaluation: tool-call accuracy

Finally, add the layer-2 eval from Day 29. Define a few scripted tasks, each with the tool you expect the agent to call. Run the agent (with auto_approve so it is non-interactive), pull the tool names from the trajectory, and score whether the expected tool was used. Print a pass/fail table. This is the smallest honest agent eval, and it already catches the most common failure: calling the wrong tool.

# scripted tasks: prompt → the tool we expect to see in the trajectory
CASES = [
    {"msg": "What is 12% of 250?",                 "expect": "calc"},
    {"msg": "What is our return policy?",            "expect": "search"},
    {"msg": "Save a note titled 'x' saying hello.",  "expect": "save_note"},
]

def eval_tool_accuracy(cases):
    passed = 0
    for c in cases:
        _, traj = run_agent(c["msg"], approver=auto_approve)
        used = {s["tool"] for s in traj if "tool" in s}
        ok = c["expect"] in used
        passed += ok
        print(f"{'PASS' if ok else 'FAIL'}  {c['msg'][:34]:34}  "
              f"expected={c['expect']:9} used={used}")
    print(f"\ntool-call accuracy: {passed}/{len(cases)}")

eval_tool_accuracy(CASES)

That is the full capstone: an agent, its guards, its trajectory, and one real eval. You can now extend it, more tools, a trajectory check that a read precedes a write, loop detection, without changing the shape.

Build checkpoints — do these, reveal the solution after

These are the lab's graded checkpoints. Attempt each before opening the solution; the solution code is what you should converge to.

Checkpoint A — parse-error feedback

Force the model to send a bad calc argument (e.g. by asking it to "compute the average of the team's feelings"), and make the agent recover rather than crash. What in the code above already handles this, and how would you confirm it worked from the trajectory?

Show solution

It is already handled in dispatch: when the Pydantic model rejects the args, the except ValidationError branch returns the error string as the observation instead of raising. That string becomes the model's next-turn input, so it can self-correct, the exact Day 27 pattern (a cheap, reliable signal: the validator). To confirm from the trajectory, look for a step whose obs starts with "validation error, fix the args and retry" followed by a later step where the same tool is called with corrected args (or the agent abandons that tool). The key property to articulate in an interview: the error is fed back into the loop, not swallowed or thrown, which is what turns a parse failure into a recoverable, observable event. If you want it stricter, cap the number of consecutive validation failures on the same tool so a model that cannot produce valid args eventually stops rather than burning all steps.

Checkpoint B — prove the approval gate and idempotency

Run the agent with human_approve and (1) reject a save_note, then (2) approve a save and trigger the same logical save twice. What should the trajectory show in each case, and which lines guarantee it?

Show solution

(1) On rejection, the gate in dispatch returns "action rejected by approver" as the observation and the write never executes, SAVED stays empty and the agent must continue or finalize without having persisted anything. The guaranteeing lines are the if spec["kind"] == "write" block calling approver(...) before spec["fn"], so no write runs without approval (Day 28's draft, pause, approve, execute). (2) On a repeated save, the idempotency key is a sha256 of the sorted args, so the same logical note produces the same key; the first call stores it, and the second hits the if idem_key in SAVED guard in save_note and returns "already saved" without writing again. The trajectory shows two save_note observations, one "saved 'x'" and one "already saved", proving the retry was safe. The senior point: you did not prevent the retry, you made it idempotent, which is the correct posture for any side-effecting tool an agent can re-issue.

Checkpoint C — extend the eval to a trajectory check

Layer-2 tool-call accuracy passes a task as long as the right tool appears anywhere. Add a minimal trajectory assertion (Day 29) for a task like "look up the return policy, then save it" that fails a run which saved without first searching. Sketch the check.

Show solution

Tool-call accuracy is necessary but not sufficient, it would pass a run that saved a hallucinated policy without ever searching, the classic "right answer, wrong trajectory" bug. The trajectory check asserts order: extract the ordered list of tool names from the trajectory and require that a search step appears before any save_note step. Sketch:

def policy_ok(traj):
    seq = [s["tool"] for s in traj if "tool" in s]
    if "save_note" in seq:
        return "search" in seq[:seq.index("save_note")]
    return True   # no write → nothing to verify

A run that saves before searching fails policy_ok even though it would pass tool-call accuracy, which is the whole point of trajectory eval: it scores the path, not just the presence of the right tool. In a real system this is where you encode the written policy ("verify before mutate"), and you would add the failing case to your golden set so it cannot regress.

LangGraph variant (optional)

The same agent in LangGraph trades the hand-rolled loop for a stateful graph runtime, which is the upgrade you make when control flow gets complex (Day 25): you get checkpoints, cycles, and interrupt-based approval for free. The plain loop above is the right teaching tool because every mechanism is visible; the framework version is what you reach for once the graph has many nodes, real human-in-the-loop pauses, or persisted state.

# sketch: LangGraph prebuilt ReAct agent + interrupt-based approval
from langgraph.prebuilt import create_react_agent
from langgraph.checkpoint.memory import MemorySaver

agent = create_react_agent(
    model="anthropic:claude-sonnet-4-5",
    tools=[calc_tool, search_tool, save_tool],     # same tools, LangChain-wrapped
    checkpointer=MemorySaver(),                     # enables interrupt + resume
)
# a write tool can call interrupt() internally to pause for approval (Day 28),
# then resume with agent.invoke(Command(resume=...), config).

Module 4 review — rapid q-list

The Agents & Tool Use slice in one pass. Say each key out loud; if any feels shaky, reread that day before moving to Module 5.

  • What is an agent and how is it different from a chain? (Day 22)
    Key: agent = LLM has autonomy over control flow (picks next step + tool in a loop); chain = fixed, dev-wired pipeline.
  • When should you NOT use an agent? (Day 22)
    Key: small/scoped/deterministic task, tight latency, single-prompt-expressible, safety-critical. Knowing when not to is senior signal.
  • What makes a good tool definition? (Day 23)
    Key: action-verb name, "use this when / do NOT use when" docstring, tight typed params, return description; negative scoping matters.
  • What is the ReAct loop and why is it the single-agent default? (Day 24)
    Key: interleaves Thought → Action → Observation; simple, debuggable from the trace, good for a handful of tool calls.
  • LangChain vs LangGraph — when does the upgrade pay off? (Day 25)
    Key: LangChain = integrations/chains; LangGraph = stateful graph runtime (cycles, interrupts, checkpoints); upgrade when control flow gets complex.
  • Short-term vs long-term vs episodic memory? (Day 26)
    Key: short = working context (pruned); long = durable user facts (vector/KV by user_id); episodic = summaries of past sessions.
  • When does reflection actually help, and how do you fix a tool-call loop? (Day 27)
    Key: reflect only with a cheap reliable signal (code runs / JSON parses / SQL returns); fix loops by detecting repeated (tool,args), surfacing it, capping max_steps.
  • How do you sleep at night with a non-deterministic agent? (Day 28)
    Key: five layers, constrain action space, trace every call, eval, guardrails/approval, canary + rollback; split read/write, idempotency on writes.
  • How do you evaluate an agent, and is "right answer, wrong tool" a bug? (Day 29)
    Key: final-answer + tool-call accuracy + trajectory + online/safety; wrong tool is a bug (luck breaks under data shift); τ-bench checks final state vs policy.
