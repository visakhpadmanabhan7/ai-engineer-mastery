Evaluating agents

Evaluating an agent is harder than evaluating a single completion, because a right answer reached the wrong way is not a success, it is luck that will break under data shift. Today you learn the four eval layers, the τ-bench way of testing an agent against a simulated user and a written policy, the metrics you actually report in production, and how to debug a flaky agent by diffing trajectories.

By the end you can
  • Name the four agent eval layers and say what each one measures.
  • Explain why "right answer, wrong tool" is a bug rather than a feature.
  • Describe a τ-bench style evaluation: simulated user, real tools, written policy, final-state success.
  • List the production agent metrics worth a dashboard and what each tells you.
  • Debug a flaky agent by replaying and diffing failing vs passing trajectories.
  • Connect agent eval to the broader evaluation discipline of Module 5.

1 · Why agent eval needs more than output scoring

For a single completion you score the output and you are done. An agent is a process: it chose tools, ordered them, reacted to results, maybe recovered from an error, and was supposed to follow a policy along the way. Two runs can produce the same final answer while one followed the rules and the other got lucky by skipping a required step. If you only score the final answer, you will ship the lucky one and it will fail the moment the inputs shift. So agent eval is layered: you measure the answer and the path.

Concept

Evaluating an agent means evaluating a trajectory, not just an output. The final answer tells you whether it succeeded; the trajectory tells you whether you can trust it to keep succeeding. A correct answer on a wrong path is not a pass, it is unmeasured risk.

2 · The four eval layers

Build them in order of effort. The first two are cheap and catch most regressions; the third is the senior layer; the fourth runs in production.

LAYER MEASURES TYPICAL METHOD ───────────────────────────────────────────────────────────────────────────────── 1. Final-answer correctness did the end result satisfy the goal? LLM-as-judge / exact match 2. Tool-call accuracy right tool, right args, each call? compare vs expected calls 3. Trajectory eval right tools, RIGHT ORDER, recovered, step-by-step trace check policy followed? (the whole path) 4. Online / safety real traffic: success, escalation, live signals + sampled judge guardrail hits, harmful outputs
  • Layer 1, final-answer correctness. Did the end state or response satisfy the user's goal? Use an LLM-as-judge with a rubric for open-ended answers, or exact/programmatic match when there is ground truth. Necessary but not sufficient on its own.
  • Layer 2, tool-call accuracy. For each step, did the agent pick the correct tool with correct, well-formed arguments? This catches hallucinated tool names, schema-violating args, and "used the wrong tool" mistakes. Cheap and high-signal; start here with layer 1.
  • Layer 3, trajectory eval. The senior layer. Score the whole path: were the right tools used in the right order, did the agent recover when a step failed, and did it follow the written policy throughout (e.g. "verify shipment before refunding," "never expose another tenant's data")? This is what separates a robust agent from a lucky one.
  • Layer 4, online and safety. In production you cannot run a golden set on live traffic, so you watch real signals: task success, human escalation rate, guardrail hits, and any harmful or off-policy outputs, with a judge sampling a slice. This catches the real input distribution your offline set never had.

3 · "Right answer, wrong tool" is a bug, not a feature

This is the line interviewers want to hear you say. If the agent produced the correct answer but used the wrong tool, queried the wrong data source, or skipped a required verification, that is a bug. The right answer on a wrong trajectory is luck: it happened to work on this input because the wrong path coincidentally landed somewhere correct, and it will break under the slightest data shift. A refund agent that returns the right amount without checking shipment status passed by accident; change the scenario to one where the item did arrive and the same path refunds incorrectly. Trajectory eval exists precisely to fail these "correct but lucky" runs.

Interview-grade phrasing

"A right answer on the wrong trajectory is not a feature, it is luck that breaks under data shift. That is exactly why I do not stop at final-answer correctness, I add trajectory eval: right tools, right order, recovered from failures, policy followed. Output eval tells me it worked this time; trajectory eval tells me it will keep working." That answer maps directly onto the Notion bank's "right answer but wrong tool" and "output vs trajectory eval" questions.

4 · τ-bench style evaluation

The hardest part of agent eval is realism: a static prompt-and-expected-answer dataset does not capture a multi-turn task where the agent has to ask for missing information, call real tools, and obey rules. The τ-bench (tau-bench) style addresses this with four ingredients:

  • An agent under test, the system you are evaluating.
  • A simulated user, another LLM playing a customer with a goal and partial information, so the agent has to converse, clarify, and handle a realistic back-and-forth instead of a single clean prompt.
  • Real tools against a sandboxed database or API, so tool calls have genuine effects and genuine failure modes.
  • A written policy, an explicit set of domain rules (e.g. airline or retail support rules) the agent is required to follow.

The crucial design choice is the success criterion: a run passes when the final state of the database matches the expected state, not when the agent's prose looks good. Did the right order actually get cancelled, the correct refund actually posted, and nothing forbidden touched? Checking the world's end state rather than the chat transcript is what makes this a real test of agent behavior, and it also naturally penalizes "right-sounding answer, wrong action."

┌─────────────┐ conversation ┌──────────────────┐ │ simulated │ ◄───────────────► │ agent under test │ │ user (LLM) │ (multi-turn) └────────┬─────────┘ └─────────────┘ │ real tool calls ▼ ┌────────────────────┐ │ sandboxed DB / API │ ◄── written policy governs allowed actions └────────┬───────────┘ │ SUCCESS = final DB state == expected state (not "the reply reads well")

5 · Production agent metrics

Offline eval gates releases; in production you watch a dashboard. The metrics that earn a place on it:

MetricWhat it tells you
Task success rateThe top-line, did the agent accomplish what the user wanted (end-state or judged).
Policy adherenceFraction of runs that followed the written rules, the trajectory signal in production.
Tool-call efficiencyTool calls (or steps) per task. Rising efficiency cost often means flailing or looping.
Recovery rateWhen a step failed, how often did the agent recover and still succeed (vs giving up or looping).
p95 step latencyTail latency per step. Agents multiply steps, so the tail dominates the user-felt wait.
Cost per taskTotal tokens and tool cost per completed task. The number finance and your roadmap care about.

Note the shape: task success and policy adherence are quality; tool-call efficiency, recovery rate, p95 step latency, and cost per task are health and economics. A senior dashboard shows both, because an agent that is getting more correct while quietly tripling its cost per task or its p95 latency is not actually improving.

6 · Debugging a flaky agent: replay and diff trajectories

A flaky agent succeeds on a task sometimes and fails other times on similar input. Because the run is non-deterministic, you cannot reason about it from the code, you have to look at what actually happened. This is where the tracing you built on Day 28 pays off: every run is a recorded span tree you can replay.

The technique is comparative. Collect several failing runs and several passing runs of the same (or near-identical) task, then diff the trajectories step by step. The point where they diverge is your bug. Maybe the failing runs all chose a different tool at step 2, or rephrased the query in a way that returned nothing, or skipped a verification the passing runs performed, or entered a loop the passing runs avoided. Diffing turns "it is flaky" into "it fails specifically when it picks tool B at step 2 after an empty search," which is a fixable statement.

PASSING runs FAILING runs step1 search(q) → hits step1 search(q) → hits step2 get(id=42) → ok step2 search(q2) → [] ◄── DIVERGENCE HERE step3 answer ✔ step3 search(q2) → [] (loop) step4 give up ✘ Diff the traces → the first divergent step is the bug. Replay it in isolation, then fix.
Interview-grade phrasing

"I debug a flaky agent by replaying it, not by re-reading the code, because the behavior is sampled. I pull several failing and several passing traces of the same task and diff the trajectories; the first divergent step is the root cause. That turns 'it is flaky' into a precise, reproducible statement I can fix." That is the senior, evidence-driven answer.

Worked example: catching a lucky pass

Example

A return-handling agent is evaluated on "the customer wants to return order 5567." Final-answer eval (layer 1) says PASS, the agent produced "Return authorized, $120 refunded," which matches the expected refund. But layer 3 trajectory eval looks at the path and finds the agent never called check_return_window, which the written policy requires (returns only allowed within 30 days). On this particular order the item happened to be within the window, so the answer was correct, by luck. Trajectory eval FAILS the run because the policy step was skipped. That is the whole value of the layer: the output looked perfect, but the agent would refund an out-of-window return on the next input. Had you shipped on layer 1 alone, you would have shipped a policy violation that only surfaces on the inputs your golden set did not contain. Fix: make the verification step mandatory and re-run; confirm via trajectory eval that check_return_window now precedes every refund.

Problem 1

Your agent passes 95% on final-answer eval, and the team wants to ship. You push back and ask for trajectory eval first. Justify the push-back concretely, what could a high final-answer score still be hiding?

Show solution

A 95% final-answer score only proves the agent reached a correct end state on the eval inputs; it says nothing about how. It can be hiding lucky passes: runs that produced the right answer on a wrong trajectory, skipping a required policy step, querying the wrong data source, or using the wrong tool that happened to land somewhere correct for this input. Those runs are correct by coincidence and break under data shift, which the golden set does not cover. It can also hide policy violations that did not change the answer (exposing data, taking an unauthorized action) and brittle recovery (the agent looped or nearly failed but scraped a pass). Trajectory eval, right tools, right order, recovered, policy followed, fails exactly these "correct but lucky" and "correct but non-compliant" runs, so it is the layer that tells you the 95% will hold in production rather than collapse on the first unusual input. Ship on the trajectory-gated number, not the output-only one.

Problem 2

Design a τ-bench style evaluation for a support agent that can look up orders, issue refunds, and update addresses, governed by a policy ("verify identity before any account change; refunds only within the return window"). What are the moving parts and what is the pass condition?

Show solution

Four moving parts. (1) The agent under test, with its three real tools wired to a sandboxed database, so calls have genuine effects and failures. (2) A simulated user, an LLM given a goal and partial info (e.g. "you moved and want your address updated, and you want a refund on order 5567"), so the agent must converse, ask for the missing identity verification, and handle a realistic multi-turn exchange rather than a single clean prompt. (3) The written policy, encoded as the rules the run is graded against: identity must be verified before any account change, refunds only inside the return window. (4) Seeded scenarios with a known correct end state per scenario. The pass condition is that the final database state matches the expected state, the address changed only if identity was verified, the refund posted only if within the window, and nothing forbidden was touched, not that the agent's replies read well. Checking the world's end state, plus a trajectory check that the policy steps actually happened, is what makes this a real behavior test and naturally fails right-sounding answers that took a forbidden or skipped action.

Problem 3

An agent succeeds on a task about 70% of the time and fails the rest, same kind of input. You have full tracing. Describe your debugging procedure and what a fix might look like.

Show solution

Because the behavior is sampled, do not re-read the code, replay and diff. Collect several failing traces and several passing traces of the same (or near-identical) task. Walk them step by step and find the first point where the trajectories diverge: that divergent step is the root cause. Typical findings: the failing runs picked a different tool at some step, rephrased the query into one that returned empty, skipped a verification the passing runs performed, or entered a loop (repeated the same tool call) the passing runs avoided. Replay that step in isolation to confirm it reproduces, which converts "it is flaky" into a precise statement like "it fails when it issues a second search after an empty first result and loops." The fix depends on the divergence: if it is a bad/looping query, add loop detection and feed the empty result back so the agent varies its approach (Day 27); if it is wrong tool selection, tighten the tool descriptions or constrain the action space; if it is a skipped policy step, make that step mandatory in the graph. Then re-run the eval and confirm the failing mode no longer appears, ideally adding the reproduced case to the golden set so it cannot regress.

Build it (15–25 min)

Hands-on

Write a tiny tool-call-accuracy eval (layer 2) for a two-tool agent. Define 4 to 5 scripted tasks, each with the expected sequence of (tool name, key args). Run the agent on each, capture its actual tool calls from the trace, and compute two numbers: tool-selection accuracy (did it call the expected tools) and argument validity (did the args parse and match). Print a per-task pass/fail table. Then add a minimal trajectory check on one task: assert that a required tool (e.g. a verification step) appears before the write tool, and watch it correctly fail a run that produced the right answer but skipped the step. See LangSmith's evaluation docs and the Anthropic tool use docs for structured call capture.

Stretch: add a layer-1 LLM-as-judge on the final answer and compare its verdict to your trajectory check on a deliberately "right answer, wrong path" task, see the judge pass it while the trajectory check fails it. That contrast is the whole lesson in one run.

Today's takeaways

Agent eval is layered because an agent is a process, not a single output: final-answer correctness, tool-call accuracy, trajectory eval, and online/safety. A right answer on a wrong trajectory is a bug, luck that breaks under data shift, which is why trajectory eval (right tools, right order, recovered, policy followed) is the senior layer. τ-bench style tests put the agent against a simulated user with real tools and a written policy, and judge success by the final database state, not the prose. In production you watch task success, policy adherence, tool-call efficiency, recovery rate, p95 step latency, and cost per task. And you debug flakiness by replaying and diffing trajectories.
