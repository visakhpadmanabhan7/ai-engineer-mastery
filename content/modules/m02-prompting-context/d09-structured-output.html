Structured output & schemas

The moment an LLM stops being a chatbot and starts being a component in a system, its output has to be parsed by code that does not tolerate "almost JSON." Today you learn the ladder of techniques that move you from hoping for valid output to guaranteeing it, and the three layers of constraint you still have to check even when the JSON parses.

By the end you can
  • Explain why structured output is the hinge between "demo" and "production".
  • Order the enforcement ladder from weakest to strongest and say what each guarantees.
  • Use Pydantic to validate, coerce, and reject model output.
  • Apply the three-layer constraint model: format, schema, semantics.
  • Predict where structured output breaks and design a validate-and-retry loop around it.

Why structured output is non-negotiable in production

A human reads prose. Code reads fields. The instant an LLM's output feeds a database write, a function argument, a UI component, or another LLM call, "the answer is roughly right" is not enough, the shape has to be exactly right or your program throws. A response that drifts from {"priority": "high"} to Priority: High (I'd estimate) is a production incident, not a style difference.

This is why structured output is one of the most-asked AI-engineering interview topics: it is the seam where probabilistic generation meets deterministic software, and the candidate who only knows "ask nicely for JSON in the prompt" is signaling they have not shipped. The senior move is to know there is a ladder of enforcement, increasing in strength, and to pick the right rung for the stakes.

The enforcement ladder

From weakest (just asking) to strongest (making invalid output impossible):

STRENGTH ↑ technique what it guarantees ───────────────────────────────────────────────────────────────────────────── weakest 1. PROMPT INSTRUCTION nothing: model usually complies, "respond in JSON, no prose" sometimes adds prose / drifts 2. JSON MODE valid JSON syntax, but NOT your (model constrained to emit JSON) keys, types, or required fields 3. TOOL / FUNCTION CALLING output matches a declared schema (schema attached to the call) (names, types, required): strong strongest 4. CONSTRAINED DECODING token sampling is masked to a (grammar / regex / JSON-schema) grammar, so invalid tokens cannot even be produced ───────────────────────────────────────────────────────────────────────────── ALWAYS, on top of any rung: Pydantic VALIDATION in your code (last line of defense)

Read the ladder as "how much can still go wrong":

  • Prompt instruction relies on the model choosing to comply. Fine for low stakes, but a distractor in the input or a long output can knock it off. Never trust it alone for a parser.
  • JSON mode forces syntactically valid JSON, which removes "trailing comma" and "prose wrapper" failures, but it does not know your keys exist or that priority must be one of three values. You can get valid JSON with the wrong shape.
  • Tool / function calling attaches a schema (field names, types, which are required) to the request, and the model is constrained to produce arguments matching it. This is the workhorse for production structured output: you get the right keys and types, not just valid syntax.
  • Constrained decoding is the strongest: the decoder masks the next-token distribution so only tokens consistent with a grammar or JSON schema can be sampled. Invalid output is impossible by construction. It is less universally available across providers, which is why tool calling is the common default.
Interview-grade phrasing

"I enforce structured output with the strongest mechanism available: tool or function calling so the model is constrained to a declared schema, or constrained decoding when I control the stack. Prompt-only JSON and JSON mode are weaker, JSON mode guarantees syntax, not my schema. Then I always validate the parsed object with Pydantic in code as the last line of defense, because even schema-constrained output can violate semantic rules." That answer walks the whole ladder and shows you know its limits.

Pydantic: your schema and your last line of defense

Even with tool calling, the object that lands in your code should pass through a validator before anything downstream touches it. Pydantic is the standard in Python: you declare the schema as a class, and parsing either returns a clean typed object or raises a precise error you can catch and act on. The same class can generate the JSON schema you hand to the model, so the contract is defined in exactly one place.

from pydantic import BaseModel, Field, field_validator
from typing import Optional, Literal

class Ticket(BaseModel):
    category: Literal["bug", "feature_request", "how_to", "billing"]
    priority: Literal["low", "medium", "high"]
    summary: str = Field(..., max_length=160)
    account_id: Optional[str] = None          # may legitimately be absent

    @field_validator("account_id")
    def digits_only(cls, v):
        if v is not None and not v.isdigit():
            raise ValueError("account_id must be digits or null")
        return v

# The schema you give the model is generated from the SAME class:
schema = Ticket.model_json_schema()          # feed this to tool/function calling

# After the model returns arguments, validate before trusting them:
try:
    ticket = Ticket.model_validate(raw_args)   # raw_args = dict from the tool call
except ValidationError as e:
    # e tells you exactly which field/rule failed → drives a retry (below)
    handle_invalid(e)

Three things this buys you for free: Literal rejects any category or priority outside the allowed set (the schema constraint), max_length catches an over-long summary, and the custom validator enforces a semantic rule (account IDs are digit strings) that no JSON schema expresses. One class is simultaneously your prompt's schema and your runtime guard.

The three-layer constraint model

"Valid output" is three separate questions, and each layer catches what the layer below cannot:

  • Layer 1 — Format. Is it syntactically parseable at all? Is it JSON, not prose? Enforced by tool calling / JSON mode / constrained decoding.
  • Layer 2 — Schema. Does it have the right keys, types, required fields, and enum values? Enforced by the declared schema and re-checked by Pydantic types and Literals.
  • Layer 3 — Semantics. Do the values make sense? A date that is valid JSON, a valid string, and parses as a date can still be in the year 1900 or after the resolution date. No schema catches this, only domain logic (a Pydantic validator, or a downstream check).

Junior engineers stop at Layer 1 ("it parsed, ship it"). The Layer 3 failures, a quantity of -5, an end date before a start date, a totals field that does not sum, are the ones that corrupt data silently and surface as a bug three weeks later.

Where structured output breaks

Failure modes to design against

Optional fields over-generated. The model invents a value for a field that should have been null or omitted, because "filling it in" looks more complete. Fix: make absence explicit in the prompt and allow null in the schema.
Similar enums confused. Closely worded categories ("error" vs "warning", "refund" vs "return") get swapped. Fix: distinct names, a one-line definition per value, and a few-shot example on the boundary (Day 8).
Truncation on long JSON. A large object hits the output-token limit mid-structure and the JSON is unparseable. Fix: cap or paginate the output, raise max_tokens, or stream and repair.
Deep nesting (> ~3 levels). Reliability drops as structures nest; the model loses track of which closing brace it owes. Fix: flatten the schema, or split into multiple calls each producing a shallow object.

Worked example: validate and retry

Example

Even strong enforcement occasionally returns an object that fails a semantic rule. The production pattern is not to crash, it is to feed the validation error back to the model and let it correct itself once or twice before giving up.

def extract_ticket(text, max_retries=2):
    messages = [{"role": "user", "content": text}]
    for attempt in range(max_retries + 1):
        raw_args = call_model_with_tool(messages, schema=Ticket.model_json_schema())
        try:
            return Ticket.model_validate(raw_args)        # L1+L2+L3 all pass → done
        except ValidationError as e:
            if attempt == max_retries:
                raise                                    # give up loudly, do not return junk
            # Show the model exactly what it got wrong, ask for a corrected object:
            messages += [
                {"role": "assistant", "content": str(raw_args)},
                {"role": "user",
                 "content": f"That failed validation: {e.errors()}. "
                            "Return a corrected object that satisfies every rule."},
            ]

Walk the trace. The model returns {"category":"bug","priority":"urgent","summary":"...","account_id":"acct-4471"}. Pydantic rejects it twice over: priority is not in the allowed Literal set, and account_id is not digits. The error object names both failures precisely. We append the bad output and the specific complaints, and call again. The model now returns priority:"high" and account_id:"4471", which validates, and we return a clean typed object. Two properties make this production-grade: it self-corrects on the exact error rather than re-rolling blindly, and it fails loudly after the cap instead of silently returning malformed data. This validate-and-retry loop is the same idea you will scale up into agent self-correction in Module 4.

Practice — do these before revealing

Answer out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

A teammate enables "JSON mode" and declares victory: "now the output is always valid, our parser is safe." Why is that confidence misplaced, and what would you add?

Show solution

JSON mode only guarantees Layer 1: the output is syntactically valid JSON. It says nothing about Layer 2 (your keys, types, required fields, enum values) or Layer 3 (whether the values make sense). You can get perfectly valid JSON with a missing required field, an extra hallucinated key, a string where you needed a number, or a priority value you never allowed, and your parser will then break or, worse, accept garbage. The fix is to go up the ladder to tool/function calling so a schema is enforced, and to validate the parsed object with Pydantic (types, Literal enums, and semantic validators) as the last line of defense. "Valid syntax" is necessary, not sufficient.

Problem 2

Your extraction returns a JSON object that parses and matches the schema, but a downstream report shows nonsense: an invoice where total does not equal the sum of line_items. Which layer failed, and where do you fix it?

Show solution

Layer 3, semantics, failed: the object is well-formed (Layer 1) and schema-correct (Layer 2), but the values are internally inconsistent. No format or schema mechanism catches a cross-field arithmetic rule, so the fix lives in your validation code, not in the prompt or the schema. Add a Pydantic model-level validator (e.g. a model_validator) that recomputes the sum of line items and rejects the object if total does not match within a tolerance, then route the failure into the retry loop or to a human queue. Two senior notes: (1) for anything this deterministic, consider computing total in code from the line items rather than trusting the model to add; (2) log the rejection so you can see how often the model gets it wrong, which is an evaluation signal (Module 5).

Problem 3

You must extract a deeply nested structure (document → sections → clauses → obligations, 4 levels) and reliability is poor: truncation and malformed nesting. Propose a redesign.

Show solution

Deep nesting past about three levels is a known break point, compounded here by output-length truncation on a large object. Two complementary fixes: (1) flatten the schema, instead of one giant nested object, emit a flat list of obligations where each row carries its section_id and clause_id as references, reconstructing the tree in code; this is shallow and far more reliable. (2) Decompose into multiple calls, first extract the sections, then for each section make a separate call to extract its clauses and obligations; each call produces a small, shallow object well under the token limit and is individually retryable. Both reduce per-call complexity, which is the actual cause of the malformed nesting. If output size is still a risk, also raise max_tokens and detect truncation (unbalanced braces) explicitly so you never hand a half-object downstream.

Interview questions for this topic

From your Notion bank's Prompting & Context slice and common 2026 screens. Drill the one-line keys, then expand each into a 90-second answer.

  • Why does structured output matter, and how do you enforce it?
    Key: downstream code parses it; enforce via tool/function calling (schema) or constrained decoding > JSON mode > prompt; validate with Pydantic on top.
  • Walk the enforcement ladder from weakest to strongest.
    Key: prompt instruction → JSON mode (syntax only) → tool/function calling (schema) → constrained decoding (invalid tokens impossible).
  • What is the three-layer constraint model for outputs?
    Key: L1 format (tool calling), L2 schema (Pydantic types/Literals), L3 semantics ("does this date/total make sense").
  • When does structured output break?
    Key: over-generated optional fields, confused similar enums, truncated long JSON, deep nesting (>3 levels); pair with post-validation.
  • JSON mode guarantees valid JSON — why is that not enough?
    Key: it secures only syntax (L1); wrong keys, wrong types, missing required fields, and bad enum values still pass.
  • How do you handle output that parses but fails a business rule?
    Key: a semantic validator rejects it, feed the specific error back for a bounded retry, fail loudly after the cap, log for eval.

Build it (15–20 min)

Hands-on

Define a small Pydantic model (3-4 fields, one Literal enum, one optional field, one custom validator). Read Anthropic's tool use docs, then make a tool/function call whose schema is YourModel.model_json_schema(), parse the returned arguments with model_validate, and deliberately give it a messy input that should trip the validator. Wrap it in the validate-and-retry loop from the worked example and confirm it self-corrects within two attempts and raises cleanly if it cannot.

Pydantic docs: docs.pydantic.dev. No API key yet? Model the schema and validator locally and feed it hand-written JSON, including a deliberately invalid one, to feel the three layers fire. See Tools & Claude for access.

Today's takeaways

Structured output is the seam between probabilistic generation and deterministic code, so getting the shape exactly right is non-negotiable in production. Climb the enforcement ladder: tool/function calling or constrained decoding beats JSON mode, which beats prompt-only, and Pydantic validation always sits on top. Check three layers, format, schema, and semantics, because valid JSON can still carry nonsense values. Design for the known break points (optional fields, similar enums, truncation, deep nesting) and wrap everything in a validate-and-retry loop that self-corrects on the exact error and fails loudly when it cannot.
