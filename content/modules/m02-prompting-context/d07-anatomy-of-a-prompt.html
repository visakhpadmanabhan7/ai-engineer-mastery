Anatomy of a great prompt

A prompt is not a wish you whisper at a model. It is a specification you hand to a system that will do exactly what you describe and nothing you forgot to say. Today you learn the parts, the roles they live in, and why the system message has more leverage than any other channel.

By the end you can
  • Name the three message roles (system, user, assistant) and say what each is for.
  • List the six components of a strong prompt and place each in the right role.
  • Explain why the system prompt has leverage, tracing it back to SFT and preference tuning from Day 1.
  • Turn a vague request into a specific, decomposed one and predict how the output changes.
  • Recognize the five mistakes that silently break otherwise reasonable prompts.

Three roles, and why the model treats them differently

Every chat-model API call is a list of messages, and each message carries a role. There are three you will use constantly:

  • system — the standing instructions: who the model is, how it behaves, what format it must produce, what it must never do. Written once, applies to the whole conversation.
  • user — the actual request and its inputs: the question, the document to summarize, the messy text to extract from.
  • assistant — the model's own turns. You can also write assistant turns yourself to provide examples, or to prefill the start of a response and steer its shape.

The roles are not just labels. The model was trained to weight them differently. Recall Day 1: a base model is pure autocomplete, then SFT teaches it the format of following instructions, then preference tuning shapes whose instructions it follows and how. During that shaping, system-role content is treated as the authority that governs the conversation, and user-role content as the request to be served within those rules. So putting a rule in the system message is not cosmetic, it is using a behavior the model was explicitly trained into.

Concept

The system prompt has leverage because the model's post-training (SFT + RLHF/DPO) taught it to treat system-role text as the governing contract for the whole exchange. A constraint in the system message is obeyed far more reliably than the same words buried in a user message, because you are steering through the channel the model was aligned to obey.

The six components of a strong prompt

A complete prompt answers six questions. You do not need all six every time, but when an output disappoints, the cause is almost always a missing or vague one of these.

  1. Role / persona — who is answering. "You are a senior technical writer" narrows tone, vocabulary, and assumptions. Goes in the system message.
  2. Task — the single action verb of what to do: classify, extract, summarize, rewrite, draft. State the one job plainly.
  3. Context — the inputs and background the model needs but does not have: the document, the user's situation, relevant facts. This is the part RAG fills at query time (Day 10, Module 3).
  4. Constraints — the boundaries: length, what to include, what to avoid, edge-case handling ("if the field is missing, return null, do not guess").
  5. Format — the exact shape of the output: JSON with these keys, a markdown table, three bullet points. Downstream code depends on this (Day 9).
  6. Examples — one or more demonstrations of input mapped to ideal output. The strongest way to pin a format or an edge case (Day 8).

Where each lives matters. Persona, constraints, and format are stable across requests, so they belong in the system message. Context and the specific task instance change per request, so they belong in the user message. Examples can sit in either, but framing them as prior user/assistant turns often reads most naturally to the model.

┌─────────────────────── SYSTEM (written once, governs all turns) ──────────────────┐ │ ROLE You are a support-ticket triage assistant for a B2B SaaS company. │ │ TASK Classify each ticket and extract the customer's core problem. │ │ CONSTRAINTS • Use ONLY the categories listed. • If unsure, use "other". │ │ • Never invent an account ID that is not present in the text. │ │ FORMAT Return JSON: {category, priority, summary, account_id|null}. │ │ FORMAT priority ∈ {low, medium, high}. summary ≤ 20 words. │ └────────────────────────────────────────────────────────────────────────────────────┘ ┌─────────────────────── USER (changes every request) ───────────────────────────────┐ │ CONTEXT Ticket text: │ │ "Hi, since the Tuesday update our exports to S3 are 502-ing for │ │ account 4471. This is blocking our nightly pipeline. Please help." │ └────────────────────────────────────────────────────────────────────────────────────┘ │ ▼ model fills the contract, not its imagination {"category":"integration_error","priority":"high", "summary":"S3 exports failing with 502 since Tuesday update, blocking nightly pipeline", "account_id":"4471"}

Notice what the structure buys you. The category is constrained to a closed set, so a parser downstream can trust it. The "never invent an account ID" line pre-empts the most likely hallucination. The format is explicit enough that the output drops straight into code. None of that happens if the prompt is just "triage this ticket."

Specificity: the single highest-return habit

The model cannot read your intent, only your words. Every place you are vague, it fills the gap with the most statistically average choice, which is rarely what you wanted. Compare:

  • Vague: "Summarize this." → You get a length, tone, and focus the model guessed.
  • Specific: "Summarize this incident report in exactly 3 bullets for an on-call engineer: what broke, blast radius, and the immediate next action. Omit background." → You get the artifact you actually need.

Specificity is not verbosity. A specific prompt names the audience, the length, the focus, and the failure behavior, then stops. Adding more words that do not constrain anything just dilutes the signal (an anti-pattern you will meet on Day 11).

Decomposition: when one prompt is the wrong unit

If a task has several distinct sub-jobs ("read this contract, extract the parties, then assess risk, then draft a summary email"), cramming them into one prompt forces the model to juggle goals and usually means it does each one a little worsely. Two cleaner moves:

  • Decompose into steps within one prompt — tell it to work through the sub-tasks in order before producing the final answer. This is the seed of chain-of-thought (Day 8).
  • Decompose into separate calls — one call extracts, its output feeds a second call that assesses, a third drafts. Each call is simpler, individually testable, and individually cacheable. This is the seed of chains and context engineering (Day 10).

The senior instinct is to ask "what is the smallest unit the model can do reliably?" and build up from there, rather than writing one heroic mega-prompt that fails in ways you cannot localize.

Worked example: from vague to specified

Example

Goal: extract structured data from a job posting. The first attempt is a single line.

Attempt 1 (user message only): Get the key info from this job ad: <text>

The model returns a prose paragraph, sometimes a bulleted list, with different keys each call. Unparseable. Now apply the six components, split across roles:

SYSTEM: You are a precise job-posting parser. Extract fields and return ONLY JSON, no prose, with these keys: {title, company, location, seniority, remote, must_have_skills[], salary|null} Rules: • seniority ∈ {junior, mid, senior, lead}; infer from wording, else "mid". • remote ∈ {onsite, hybrid, remote}. • must_have_skills: only skills explicitly required, max 8, lowercase. • salary: copy the stated range verbatim, or null if absent. Do NOT estimate. USER: Posting: "Senior ML Engineer, Acme GmbH (Munich, hybrid). You will own our retrieval stack. Required: Python, PyTorch, vector databases. €85k to €105k."

Now the output is deterministic in shape: seniority is "senior" (the word is present), remote is "hybrid", salary is copied verbatim rather than guessed, and the keys never drift because the contract is in the governing system message. The exact same model, the exact same input text, but the prompt did the engineering. We will make this output guaranteed parseable, not just usually parseable, on Day 9.

The five mistakes that quietly break good prompts

Failure modes, derived from the components

1. Rule in the wrong role. A hard constraint sitting in the user message gets overridden by later user text; move it to the system message.
2. Conflicting instructions. "Be concise" and "explain every step in detail" cancel out; the model picks one unpredictably. Audit for contradictions.
3. Vague constraints. "Keep it short" has no operational meaning; "max 20 words" does.
4. Unstated edge-case behavior. If you never say what to do when a field is missing, the model invents one. Always specify the null/"unknown"/"skip" path.
5. No format contract. Asking for data without naming the exact output shape guarantees drift that breaks downstream parsing.

Practice — do these before revealing

Answer out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

A teammate puts the instruction "Always respond in valid JSON" as the last sentence of a long user message, and it works most of the time but occasionally returns prose. What is the most likely cause, and what is the one-line fix?

Show solution

The rule is in the wrong role. User-message instructions compete with the rest of the user text and with any earlier turns, so a strong distractor in the input (a long document, an example that contains prose) can pull the model off the format. The fix is to move "respond in valid JSON, no prose" into the system message, where post-training taught the model to treat it as the governing contract for every turn. Senior bonus: for a hard guarantee, do not rely on instruction at all, use tool/function calling or constrained decoding (Day 9), which enforces the format mechanically rather than asking nicely.

Problem 2

You are asked to "improve" this prompt: "You are a helpful expert assistant. Please provide a comprehensive, detailed, and concise answer about our refund policy." Identify two problems and rewrite it.

Show solution

Two problems: (1) conflicting instructions — "comprehensive and detailed" fights "concise", so the model picks unpredictably; (2) missing context and format — there is no refund policy supplied and no output shape, so it will either hallucinate a policy or produce free-form prose. A rewrite separates roles and resolves the conflict: system = "You answer customer refund questions using ONLY the policy text provided. If the policy does not cover the question, say so and suggest contacting support. Answer in at most 4 sentences." user = the customer question plus the actual policy text as context. Now there is one clear length target, grounded context, and an explicit out-of-scope path.

Problem 3

You need the model to (a) read a customer email, (b) detect the sentiment, (c) extract any mentioned product names, and (d) draft a reply matching the sentiment. Argue for either one prompt or a decomposed pipeline, and say what you would measure to decide.

Show solution

Both are defensible; the senior answer names the trade-off. One prompt is simpler, fewer calls, lower latency and cost, and fine if the model handles all four reliably on your data. A decomposed pipeline (classify sentiment + extract products in one call, draft reply in a second that receives those as inputs) makes each step individually testable and lets you swap a cheaper model for the easy classification step while reserving the strong model for drafting. To decide, build a small labeled set and measure per-step accuracy: if the single prompt's extraction or sentiment degrades when bundled with drafting (a common "juggling goals" failure), split it. Start with one prompt, decompose only when measurement shows a step is being dragged down.

Interview questions for this topic

From your Notion bank's Prompting & Context slice and common 2026 screens. Drill the one-line keys, then expand each into a 90-second answer.

  • What are the components of a strong prompt, and where does each live?
    Key: role, task, context, constraints, format, examples; stable parts (persona/format/constraints) in system, per-request parts (context/task instance) in user.
  • Why does putting a rule in the system prompt have more leverage than putting it in the user message?
    Key: SFT + preference tuning trained the model to treat system-role text as the governing contract; user text competes with the request and distractors.
  • Where should a given rule live: system prompt, tool, few-shot, or retrieval?
    Key: persona/format → system prompt; hard-to-describe pattern → few-shot; external data/action → tool; per-tenant policy → retrieval/metadata, not the prompt.
  • What is the difference between making a prompt specific and making it longer?
    Key: specificity adds constraints (audience, length, focus, edge cases); length without constraint dilutes signal and reads as stuffing.
  • When do you decompose a task instead of writing one prompt?
    Key: when sub-jobs compete and degrade each other; split into ordered steps or separate testable/cacheable calls; build up from the smallest reliable unit.
  • Name three reasons a reasonable-looking prompt produces inconsistent output.
    Key: conflicting instructions, rule in the wrong role, no format contract / unstated edge-case behavior.

Build it (15–20 min)

Hands-on

Open the Anthropic prompt engineering docs or the Workbench. Take a one-line request you would actually make ("summarize this PR description", "classify this bug") and run it bare. Then rebuild it with all six components, putting persona + constraints + format in the system field and the input in the user field. Run both on three different inputs and compare consistency of shape, not just quality. Notice how the structured version's format stops drifting first.

Connect it to your work

Your MaindTec story

On MaiQ, the features you took from prototype to production are exactly these contracts hardened into code: a triage or extraction step is a system message defining the persona, the closed set of categories, and the JSON shape, with the per-document text arriving in the user message. When an interviewer asks "how do you write a reliable prompt," you can answer from production: "I treat the system message as the governing contract for persona, constraints, and output format, keep the changing context in the user message, and specify the edge-case path so the model never has to guess what to do with a missing field." That framing leads naturally into your structured-output and evaluation work in the next lessons.

Today's takeaways

A prompt is a specification, not a wish. Three roles exist and the model was trained to weight them differently, which is why the system message governs behavior. Six components answer the full spec: role, task, context, constraints, format, examples. Specificity beats verbosity, and decomposition beats one heroic mega-prompt when sub-jobs compete. Most "the model is dumb" complaints are really a missing constraint, a wrong-role rule, or an absent format contract.
