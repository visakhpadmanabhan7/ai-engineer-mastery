Building eval datasets

A judge and a metric are useless without something to measure. The dataset is the foundation of your whole evaluation stack, and the quiet truth is that most teams' eval sets are stale, unrepresentative, and slowly lying to them. Today you learn to build a golden set that stays honest.

By the end you can
  • Define a golden set and say what fields each item carries and why.
  • Seed an initial set by intent so it covers what users actually do, not just what is easy.
  • Mine production traces to keep the set growing toward real failures, so it does not rot.
  • Tag items so you can slice evaluation by intent, difficulty, and freshness.
  • Use synthetic generation and replay sets without poisoning your measurements.

The golden set

A golden set is a curated, version-controlled collection of test cases that represents what your system must get right. It is the fixed reference your offline evaluation runs against, the "expected" side of every comparison. For a RAG or retrieval system, each item typically carries three things:

  • query, the input, the actual question or request a user might send.
  • ideal_answer, the reference response a great system would produce. It anchors generation-quality scoring (faithfulness, relevance) and gives a human reviewer something to compare against.
  • ideal_chunks, the document passages that should be retrieved to answer correctly. This is what lets you measure retrieval directly (recall@k, MRR) and separate a retrieval failure from a generation failure, the single most important diagnostic split in RAG.

That third field is the one juniors forget and seniors insist on. Without ideal_chunks, when an answer is wrong you cannot tell whether retrieval failed to find the right passage or generation fumbled a passage it had. With it, you can score the two stages independently and fix the one that is actually broken. We meet recall@k and friends again in Day 34.

Interview-grade phrasing

"My golden set stores query, ideal answer, and ideal chunks per item, so I can score retrieval and generation independently and tell a retrieval miss apart from a generation error. I seed it by user intent, then continuously mine production traces, especially low-confidence cases and reported bugs, so it tracks the real input distribution instead of rotting. Every item is tagged with intent, difficulty, and a last-reviewed date so I can slice results and prune stale cases."

Seeding by intent (not by convenience)

The failure mode when you build the first version of a golden set is to write the cases that come easily to mind, which are almost always the clean, simple, happy-path questions. You end up with 40 variations of "what is the refund policy" and zero of the gnarly multi-part questions that actually break the system. Then your eval says everything is great while users suffer.

The fix is to seed by intent. Enumerate the distinct things users come to your product to do, the intents, and deliberately write cases for each, including the hard ones:

  • List the intents (policy lookup, troubleshooting, comparison, multi-step how-to, out-of-scope, ambiguous, follow-up referring to earlier turns).
  • Write several cases per intent, and inside each intent span easy to hard.
  • Deliberately include negative and edge cases: questions the system should refuse or say "I do not know" to, questions with no good answer in the corpus, adversarial phrasings. A set with only answerable questions cannot detect a system that hallucinates on the unanswerable ones.

Intent coverage, not case count, is what makes a small set powerful. Fifty well-chosen cases spanning every intent and difficulty beat five hundred near-duplicate easy ones.

Mining production so the set does not rot

Here is the single most important idea in this lesson: a static golden set decays. The day you write it, it reflects what you imagined users would ask. But real usage drifts, new features ship, new edge cases appear, the product changes, and a set that never updates slowly stops representing reality. An eval that does not track production becomes a comfortable lie.

The cure is to continuously mine production traces back into the set. Your observability pipeline (Day 35) is already logging every request. Pull the cases that are most informative:

  • Low-confidence interactions. Where the model hedged, retrieval scores were weak, or the judge scored the live answer poorly, these are the system's current weak spots and exactly what you want under test.
  • Bug reports and thumbs-down. Every user complaint or negative rating is a free, real-world failure case. Triage it, write the correct ideal_answer and ideal_chunks, and add it. Now that exact failure is regression-tested forever.
  • New-intent traffic. Queries that do not match any existing intent tag signal usage you did not anticipate, seed new cases for them.

This turns evaluation from a one-time chore into a living system that gets stronger every time the product fails, because each failure becomes a permanent test. Set a cadence: a quarterly refresh to prune dead cases and rebalance, plus continuous trickle from trace mining. That rhythm keeps the set both current and stable.

Tagging: turn one number into many

An aggregate score ("mean faithfulness 4.4") hides where you are strong and weak. Tags let you slice it. Tag every item with at least:

  • intent, so you can see "we are great at policy lookups but failing multi-step how-tos." That diagnostic is invisible in the average and obvious once you slice by intent.
  • difficulty (easy / medium / hard), so a regression on hard cases is not masked by a wall of easy ones. A mean can rise while your hardest, highest-value cases quietly get worse.
  • last_reviewed (a date), so you know which cases a human has actually verified recently and which may be stale. When you refresh, you revisit the oldest first.
  • source (seed / prod_trace / synthetic), so you can weight or audit by origin, useful when you suspect synthetic cases are skewing results.

With tags, one evaluation run becomes a breakdown: per-intent scores, hard-case scores, and a freshness audit, all from the same harness. That is how you find the specific thing to fix instead of staring at a single number that went down 0.1.

Synthetic data and its risks

You can use an LLM to generate eval cases: feed it a document and ask for plausible questions and ideal answers. This is genuinely useful for cold-starting a set before you have traffic, and for cheaply expanding coverage of an under-tested intent. But treat synthetic data with suspicion, because it carries three real risks:

  • Distribution mismatch. LLM-generated questions are often cleaner, more formal, and more "textbook" than what messy real users type. A set that is all synthetic measures the system on questions nobody actually asks.
  • Circularity. If the same model generates the cases and answers them and judges them, you are grading inside one model's worldview, easy questions for it, scored generously by it. The measurement can look great while meaning nothing.
  • Hidden errors. The generated ideal_answer can itself be wrong or hallucinated. An unverified synthetic "correct answer" silently corrupts every score computed against it.

The discipline: use synthetic data to bootstrap and to fill coverage gaps, tag it as synthetic, have a human spot-check a sample of the generated answers, and replace it with real mined cases as traffic arrives. Synthetic gets you off zero; production data gets you to truth.

Replay sets and frozen regressions

Two specialized cousins of the golden set earn their own names:

  • A replay set is a snapshot of real production inputs (the queries, not necessarily labeled answers) that you re-run a new system version against to see what changed. It answers "across real recent traffic, does the new version behave differently?" even where you have no ideal answer, you diff outputs old-vs-new and inspect the deltas. It captures the real input distribution in a way a hand-curated set never fully does.
  • A regression set is the discipline of freezing every past bug as a permanent test. When you fix a specific failure, you do not just patch it, you add a reproduction case to the set so the bug can never silently return. Over time this becomes your shield against the most demoralizing failure in AI systems: re-breaking something you already fixed, because a later prompt or model change quietly undid it. Each frozen repro is cheap insurance against regression.

Together with the intent-seeded core and the trace-mined additions, these give you a dataset that is representative (replay), durable (regression), and current (mining). That is a golden set that earns the name.

Worked example: build a 12-case set for a docs assistant

Example

You are starting a support-docs RAG assistant and have no traffic yet. Watch the set come together.

1. List intents. policy_lookup, troubleshooting, how_to_multistep, comparison, out_of_scope, ambiguous_followup. Six intents.

2. Seed two cases per intent (12 total), spanning difficulty. For each, write the query, the ideal_answer from the real docs, and the ideal_chunks (the exact doc passage IDs). Crucially, the two out_of_scope cases have ideal_answer = "I do not have information on that" and empty ideal_chunks, these test that the system refuses instead of hallucinating.

3. Tag everything. intent, difficulty, source=seed, last_reviewed=today.

4. Bootstrap a few more synthetically. Feed three docs to an LLM, ask for plausible questions plus answers, generate 6 candidates, personally verify 4 of them are correct and realistic, keep those 4 tagged source=synthetic, discard 2 that are too textbook.

5. Ship it as v1 and wire the flywheel. The day real traffic starts, begin mining: every thumbs-down and every low-retrieval-score query gets triaged and, if valid, labeled and added. By month two the set is half real production cases and the synthetic scaffolding has done its job.

Sixteen cases that span every intent, include refusals, and are already set up to grow beats a thousand auto-generated lookalikes. The set is small, honest, and built to stay honest.

Problem 1

Your team's golden set has scored a steady 4.6 for six months while user complaints keep climbing. Both observations are accurate. Diagnose the disconnect and prescribe a fix.

Show solution

The golden set has rotted: it is static, so it still measures the clean cases someone imagined six months ago, while real usage has drifted to inputs the set does not contain, new features, new intents, messier phrasings. A stable 4.6 on an unrepresentative set is meaningless; the complaints are the real signal. The fix is to make the set track production: stand up trace mining, pull the queries behind the complaints plus low-confidence and low-retrieval-score interactions, triage them, write correct ideal_answer and ideal_chunks, and add them, ideally tagged by the new intents you discover. Add a quarterly refresh to prune dead cases and rebalance toward current intent distribution. Within a cycle the set's score will drop to reflect reality, and now it is a number worth optimizing. The lesson: an eval set is a living asset, not a one-time artifact.

Problem 2

To save time, a colleague builds the entire eval set by asking an LLM to generate questions and ideal answers from the docs, no human review, then uses the same model to answer and judge. Scores are near-perfect. What is wrong, ranked by severity?

Show solution

Stacked problems, worst first. (1) Circularity: one model generates, answers, and judges, so it is grading questions it finds easy with a worldview it shares, near-perfect scores are an artifact of the closed loop, not evidence of quality. (2) Unverified ideal answers: with no human review, any hallucinated "correct" answer silently corrupts every score computed against it, you may be rewarding wrong outputs. (3) Distribution mismatch: LLM-generated questions are cleaner and more textbook than real user queries, so even if everything else were sound, the set measures questions nobody asks. Fixes: break the loop (different model family to judge, ideally to generate too), human-verify a sample of synthetic answers and tag source=synthetic, and treat synthetic only as a bootstrap to be progressively replaced by mined real traffic. Synthetic gets you off zero; it does not get you to truth.

Problem 3

You fix a nasty bug where the assistant confidently invented a refund amount. The fix works. What do you do with this incident so it pays off for years, and how is that different from just deploying the fix?

Show solution

Freeze it as a regression test. Deploying the fix solves the bug today; adding the exact reproduction to the eval set, the precise query, the correct ideal_answer ("the refund amount is not specified in the docs" / the true value), and the ideal_chunks, ensures the bug can never silently return. The difference matters because AI systems regress invisibly: a later prompt tweak, model upgrade, or retrieval change can quietly undo your fix, and without a frozen repro nobody notices until a user is harmed again. The regression set is your shield against re-breaking solved problems. Tag it (source=prod_trace, the intent, difficulty=hard) so it also enriches your slices. One incident, handled this way, becomes permanent protection rather than a one-time patch.

Build it (15–25 min)

Hands-on

Pick any small document you know well (a product README, a policy page, your own resume). By hand, build a 10-item golden set as JSON: each item has query, ideal_answer, ideal_chunks (the section it should pull from), and tags with intent / difficulty / last_reviewed. List at least four distinct intents and force yourself to include two cases the system should refuse (questions the document cannot answer). Then use the Anthropic API or the Claude web app to synthetically generate five more candidate items from the same doc, and personally verify each generated ideal_answer against the source, keep the correct ones tagged source: synthetic, discard the wrong or too-textbook ones. Notice how many synthetic answers needed correcting, that number is exactly why you never trust synthetic data blind. The RAGAS testset-generation docs (docs.ragas.io) show how a library does this at scale.

Today's takeaways

The golden set is the foundation of evaluation: for RAG, store query, ideal_answer, and ideal_chunks so you can split retrieval from generation. Seed by intent with edge and refusal cases, so coverage, not count, drives quality. A static set rots, so mine production (low-confidence, thumbs-down, bugs, new intents) continuously and refresh quarterly. Tag by intent, difficulty, freshness, and source to turn one number into a diagnostic breakdown. Use synthetic data only to bootstrap and verify it; freeze every fixed bug as a regression test. A living dataset is what keeps your whole eval honest.
