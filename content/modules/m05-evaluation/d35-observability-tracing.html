Observability & tracing

Evaluation tells you whether the system is good. Observability tells you what it actually did, request by request, when something goes wrong at 2 a.m. Without tracing, an LLM app is a black box that occasionally and inexplicably fails. Today you make it transparent.

By the end you can
  • Explain a trace as a span tree and read tokens, cost, and latency off every span.
  • Say what you log on every request and what you must never log.
  • Define SLOs for an LLM endpoint that capture quality, not just uptime.
  • Describe a cost dashboard and the levers it surfaces.
  • Write structured logs that are queryable instead of prose you grep in despair.

The unit of observability: the trace

An LLM request is rarely a single call. One user message can fan out into a retrieval, two tool calls, and three LLM invocations, each with its own latency and cost. When the final answer is wrong or slow, "the request failed" tells you nothing. You need to see inside it.

A trace is the full record of one request, structured as a span tree. A span is one unit of work, an LLM call, a tool call, a retrieval, with a start time, a duration, and attributes. Spans nest: the top-level request span contains child spans for each step, which can contain their own children. Each span carries the data that matters for that step:

  • LLM spans, model, prompt version, temperature, input and output token counts, computed cost, latency, and the stop reason.
  • Tool spans, tool name, arguments, result (or error), latency.
  • Retrieval spans, the query, the retrieved document IDs and scores, top-k, latency.

Read top to bottom, a trace is a complete, timed story of what the system did. You can see instantly that the slow request spent 1.9 of its 2.3 seconds in one tool call, or that the wrong answer came from a retrieval span that returned the wrong document. That is the difference between debugging and guessing.

ONE TRACE = span tree for a single user request (each span: name · latency · tokens/cost where applicable) ▸ request "What's our refund window and is item #A12 eligible?" 2.34s ├─ retrieval.hybrid 0.08s k=8, top doc_returns#p3 (0.81) ├─ llm.plan 0.41s in 1,240 tok · out 90 tok · $0.004 ├─ tool.lookup_order(A12) 1.52s ← 65% of total latency lives here │ └─ http GET /orders 1.49s 200 OK ├─ llm.answer 0.30s in 1,610 tok · out 140 tok · $0.006 └─ guardrail.citation_check 0.01s pass ─────────────────────────────────────────────── totals: 3,080 tok · $0.010 · 2.34s · stop=end_turn

This is standardized. The open standard is OpenTelemetry (OTel), with traces exported over OTLP to a backend. For LLM apps specifically, tools like LangSmith and Langfuse give you this span-tree view out of the box, with token and cost accounting per span and the prompt/response captured. You instrument once; you read traces forever.

Interview-grade phrasing

"My observability unit is a trace per request, a span tree where each LLM, tool, and retrieval span records its own tokens, cost, and latency. I export it via OpenTelemetry over OTLP, or use LangSmith or Langfuse for the LLM-native view. When a request is slow or wrong, the trace localizes it immediately, which span ate the latency, which retrieval returned the wrong document, so I debug from evidence instead of guessing."

What to log every request, and what NEVER to log

Logging is a deliberate choice, not "log everything." You want enough to debug and to reconstruct any decision, without creating a privacy liability or a cost explosion. Per request, log:

  • Versioning, prompt version, model name, temperature and other sampling params. Without the prompt version you cannot tell which prompt produced a bad answer after you have changed it twice.
  • Performance and cost, input and output token counts, per-span and total latency, computed cost.
  • The decision trail, tool calls with their arguments, retrieved document IDs and scores, the stop reason. This is what lets you replay why the system did what it did.
  • Correlation identifiers, a request ID, and a hashed user and session ID so you can group a user's interactions and trace a complaint to its request, without storing who they are.
Never log raw PII

Do not write raw personally identifiable information, names, emails, payment details, health data, into your logs or traces. Logs get shipped to third-party backends, retained for months, and read by many engineers; raw PII there is a compliance breach (GDPR and friends) waiting to happen. Hash or pseudonymize user and session identifiers, redact or omit sensitive fields, and log document IDs rather than raw document contents where you can. You can still debug, you group by the hashed ID and pull the doc by its ID, you just never persist the sensitive payload itself. Decide this before you ship, because retrofitting redaction after a leak is too late.

This is, incidentally, a place your Infosys background is directly relevant: named-entity redaction of unstructured text is exactly the technique you reach for to strip PII out of anything you must log. The instinct to redact before persisting is the senior move.

SLOs for an LLM endpoint

A Service Level Objective (SLO) is a target you commit to and alert on. For a normal web service these are latency and error rate. An LLM endpoint needs those plus quality and cost SLOs, because an LLM endpoint can be 100 percent "up", returning HTTP 200, while quietly returning garbage. Uptime is not quality. The set worth defining:

  • Latency, p95 and p99. Tail latency, not the average, because the average hides the slow requests that actually frustrate users. Set a budget (e.g. p95 under 2s) and alert when it is breached.
  • Error rate and refusal rate. Not just 5xx errors, but how often the model refuses or returns an empty/unusable answer. A climbing refusal rate is a real degradation that uptime monitoring will never see.
  • Cost per request budget. An SLO on spend. LLM costs scale with tokens and can silently explode when prompts grow or an agent loops; a cost-per-request alert catches the runaway before the invoice does.
  • A quality floor on a rolling sample. Continuously run your LLM-as-judge over a small rolling sample of live traffic and alert if, say, faithfulness on that sample drops below a threshold. This is the SLO that catches silent quality regressions, the system is up, fast, and cheap, but the answers got worse, and only a sampled quality metric will tell you.

The throughline: a healthy LLM endpoint is fast at the tail, rarely refuses, stays inside its cost budget, and holds a faithfulness floor. Monitor only uptime and you are blind to three of the four ways it can fail.

Say this in an interview

"For an LLM endpoint I set four SLOs: p95/p99 latency, error-plus-refusal rate, a cost-per-request budget, and a faithfulness floor measured by an LLM-as-judge over a rolling sample of live traffic. The last one matters most and people forget it, the endpoint can be up, fast, and cheap while the answers silently degrade, and only a sampled quality metric catches that."

Cost dashboards and their levers

Because cost is token-driven and can run away invisibly, you watch it on a dashboard built from the same trace data. The metrics that actually drive decisions:

  • Cost per request (and its trend), the headline number. A creeping rise usually means prompts are growing or context is bloating.
  • Cache-hit rate, the single biggest cost lever. Prompt caching and semantic caching only help if requests actually hit the cache; a low hit rate is money left on the table and a prompt for investigation (we cover caching properly in Module 7).
  • Model mix, what fraction of traffic goes to the expensive large model versus a cheaper one. Surfacing this is how you spot that 90 percent of traffic is paying for a frontier model when a smaller one would do, which is a routing opportunity.
  • Token distribution, input versus output tokens, since output tokens cost more and dominate latency. A shift here explains a cost or latency change.

A cost dashboard turns spend from a monthly surprise into a controllable, attributable metric. Every one of these levers, caching, routing, shorter context, comes back in Module 7; observability is what tells you which lever to pull.

Structured logging, so you can actually query it

One discipline underpins all of the above: log structured data (key-value JSON), not free-form prose. A log line like "the answer was slow for some user" is unsearchable; you cannot aggregate it, alert on it, or build a dashboard from it. A structured event can be queried, grouped, and graphed:

# BAD: prose log — ungreppable, unaggregatable
log.info(f"answered question for a user, took a while, used the big model")

# GOOD: structured event — queryable, alertable, dashboard-ready
log.info("llm_request", extra={
    "request_id":    req_id,
    "user_hash":     hash_user(user_id),   # hashed, never raw
    "prompt_version": "answer_v7",
    "model":         "claude-sonnet-4-5",
    "input_tokens":  1610,
    "output_tokens": 140,
    "cost_usd":      0.010,
    "latency_ms":    2340,
    "tool_calls":    ["lookup_order"],
    "doc_ids":       ["doc_returns#p3"],
    "stop_reason":   "end_turn",
    "refused":       False,
})

With structured events you can answer real questions in one query, "show p95 latency for prompt_version answer_v7", "what is the refusal rate by model this week", "which doc_ids appear in low-rated answers". That is observability you can operate, not logs you scroll through after the incident. Structured logging is the unglamorous foundation that makes the span tree, the SLOs, and the cost dashboard all possible.

Worked example: debug a latency spike from the trace

Example

Alert fires: p95 latency jumped from 2.1s to 4.8s. No errors. With observability, watch it resolve in minutes.

1. Slice the dashboard. Structured logs show the spike is isolated to prompt_version: answer_v8, shipped this morning. Other versions are flat.

2. Open a slow trace for v8. The span tree shows the retrieval.hybrid span ballooned from 0.08s to 2.6s, everything else is normal.

3. Read the retrieval span attributes. top-k jumped from 8 to 50, the v8 change raised top-k to "improve recall" and nobody load-tested it; at k=50 the reranker downstream is now scoring 50 candidates per query.

4. Cross-check cost. The cost dashboard confirms cost per request rose in lockstep (more reranking, more tokens). Two SLOs breached from one change.

5. Act. Roll back top-k to 12 (a middle ground), latency and cost return to budget; queue an offline experiment to find the real recall-versus-cost sweet spot (Module 5's whole method).

Without tracing this is hours of speculation. With it, the span tree pointed straight at the retrieval span, the structured logs tied it to a specific prompt version, and the cost dashboard corroborated the cause. Observability turned a mystery into a five-step diagnosis.

Problem 1

Your LLM endpoint reports 99.98 percent uptime and the on-call dashboard is all green, yet support tickets say the assistant has become useless this week. How is this possible, and what was missing from your monitoring?

Show solution

Uptime measures whether the service responds, not whether the answer is any good. An LLM endpoint can return HTTP 200 with a fast, well-formed, completely wrong or unhelpful answer, so it is "up" while quality has collapsed (a model change, a retrieval regression, a prompt edit). What was missing is a quality SLO: a faithfulness (or correctness) floor measured by an LLM-as-judge over a rolling sample of live traffic, alerting when it drops. Likely also missing is a refusal-rate metric, if the model started refusing or hedging more, that is degradation invisible to uptime. The fix is to add those two to the SLO set alongside latency and errors, so quality regressions page someone instead of waiting for tickets. The principle: for LLM systems, uptime is necessary but nowhere near sufficient, you must monitor quality directly.

Problem 2

A new engineer adds log.info(f"user {email} asked: {question}, we answered: {answer}") to help debugging. Name every problem with this line and rewrite the logging approach.

Show solution

Three problems. (1) Raw PII: it logs the user's email and likely their question/answer content verbatim, which ships to log backends, is retained for months, and is readable by many engineers, a GDPR-class compliance breach. (2) Unstructured prose: a single f-string is unsearchable and unaggregatable, you cannot compute p95 latency or refusal rate from it, so it is near-useless for actual observability. (3) Missing the data that matters: no prompt version, model, tokens, cost, latency, or request ID, the things you actually need to debug. Rewrite: emit a structured event with hashed user_hash (never raw email), redact or omit sensitive content (log document IDs, not raw text), and include prompt_version, model, input/output tokens, cost, latency, tool_calls, stop_reason, and refused. Now it is queryable and privacy-safe. Visakh's Infosys named-entity redaction is exactly the tool for stripping PII before anything is persisted.

Problem 3

Your monthly LLM bill doubled with no traffic increase. You have full tracing. Walk through how you would localize the cause, and name the two dashboard metrics most likely to reveal it.

Show solution

Cost is token-driven, so a flat-traffic doubling means tokens per request roughly doubled or the model mix shifted toward something expensive. Localize it with the cost dashboard and traces. The two metrics most likely to reveal it: (1) cost per request trend, confirms the rise is per-request, not volume, and dates it to a change; and (2) token distribution (input vs output) and/or model mix, shows whether prompts/context bloated (input tokens up, e.g. someone stuffed more history or larger chunks into context, or an agent started looping and re-sending context) or whether traffic started routing to a pricier model. A low cache-hit rate is the other prime suspect, if a change broke prompt-cache reuse, every request now pays full price. Then open representative traces from before and after to see exactly which span's tokens grew. The fix follows the cause: trim context, fix the routing, restore cache reuse, all Module 7 levers, but observability is what tells you which one.

Interview questions for this topic

From your Notion bank's Evaluation & Observability slice and common 2026 screens. Drill the one-line keys, then expand each into a 90-second answer with Claude using protocol prompt #2.

  • What does a production tracing pipeline look like for an LLM app?
    Key: a trace per request as a span tree (LLM / tool / retrieval spans), each span recording tokens, cost, and latency, exported via OpenTelemetry over OTLP, viewed in LangSmith or Langfuse.
  • What do you log every request, and what do you NOT log?
    Key: log prompt version, model, temp, input/output tokens, latency, cost, tool calls + args, doc IDs, stop reason, hashed user/session ID. Never log raw PII, hash IDs and redact sensitive content.
  • What SLOs would you set for an LLM endpoint?
    Key: p95/p99 latency, error + refusal rate, cost-per-request budget, and a faithfulness floor on a rolling sample. Quality SLOs matter because the endpoint can be up while answers silently degrade.
  • Why is uptime an insufficient health metric for an LLM service?
    Key: it can return 200 with a fast, well-formed, wrong answer, up but useless. You must monitor quality directly (sampled faithfulness) and refusal rate, not just availability.
  • How do you keep LLM cost under control with observability?
    Key: a cost dashboard, cost per request trend, cache-hit rate (biggest lever), model mix, input/output token distribution. Surfaces which lever (caching, routing, shorter context) to pull.
  • Why structured logging over prose logs?
    Key: structured key-value events are queryable, aggregatable, and alertable (compute p95, refusal rate, group by prompt version); prose f-strings are ungreppable and useless for real observability.
  • A request is slow but returns no error. How do you find the cause?
    Key: open its trace and read the span tree, find which span (often a tool call or oversized retrieval) ate the latency, then tie it to a prompt version via structured logs and cross-check the cost dashboard.

Today's takeaways

The unit of LLM observability is a trace per request, a span tree where every LLM, tool, and retrieval span carries its own tokens, cost, and latency (OpenTelemetry / OTLP, or LangSmith / Langfuse). Log the versioning, performance, decision trail, and hashed identifiers you need to debug, and never persist raw PII. Set SLOs that include quality and cost, p95/p99 latency, error and refusal rate, cost-per-request budget, and a faithfulness floor on a rolling sample, because uptime alone is blind to silent degradation.
