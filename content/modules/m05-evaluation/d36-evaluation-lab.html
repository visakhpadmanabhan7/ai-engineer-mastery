Lab — an LLM-as-judge harness

This is the module capstone. You will build a real, working evaluation harness end to end: a golden set, an LLM-as-judge with an anchored rubric, a run over two prompt variants, regression flagging, a cheap programmatic check alongside the judge, and finally calibration of the judge against your own human labels. By the end you will have the exact artifact behind your resume's evaluation story, in code, in your hands.

By the end you will have built
  • A 10-item golden set with queries, ideal answers, and ideal contexts.
  • An LLM-as-judge scoring faithfulness on an anchored 1 to 5 scale, reasoning before score.
  • A harness that runs two prompt variants over the set and reports per-variant scores.
  • A regression check that flags any case where variant B scores worse than variant A.
  • A cheap programmatic guardrail running alongside the judge.
  • A calibration step comparing the judge to 3 to 5 of your own human labels.
Setup (5 min)

You need Python and an Anthropic API key (pip install anthropic, set ANTHROPIC_API_KEY). See Tools & Claude for access. Total budget is small, this lab is a few dozen short LLM calls, cents of spend. Build it incrementally: do each checkpoint, run it, confirm the output, then move on. Do not paste the whole thing at once, the point is to understand each piece. Solution code is in the collapsible under each checkpoint, try it yourself first.

What you are building

The harness is the eval-driven loop from Day 31 made concrete. You are evaluating a tiny "answer from context" RAG generator: given a context passage and a question, produce a grounded answer. You will test two prompt variants of that generator, a plain one and one that explicitly instructs grounding, and use the judge to decide, with evidence, whether variant B is actually better. That is the whole job in miniature.

THE HARNESS, END TO END golden set (10 items) │ ├──► generator(variant A) ─┐ │ ├─► judge(faithfulness 1-5) ─┐ ├──► generator(variant B) ─┘ + programmatic guardrail │ │ ▼ │ per-variant mean score │ + per-item regression flags ▼ 3-5 human labels ──► calibrate judge (agreement) ──► trust? then ship the loop

Checkpoint 1 — the golden set (15 min)

Build this

Create a Python list of 10 golden items. Each is a dict with id, context, question, and ideal_answer. Cover a few intents and deliberately include at least two items the answer should refuse (the context does not contain the answer, ideal answer is "the context does not say"). This is your fixed reference, every later step measures against it.

Show solution

Keep it small, varied, and include the refusal cases, those are what catch a generator that invents facts.

Checkpoint 2 — the two generator variants (15 min)

Build this

Write a generate(context, question, system_prompt) function that calls Claude, plus two system prompts: variant A ("answer the question") and variant B ("answer ONLY from the context; if the context does not contain the answer, say so; do not use outside knowledge"). Variant B is the hypothesis: explicit grounding instructions should raise faithfulness, especially on the refusal cases. Run both over one item to confirm they work.

Show solution

Low temperature on the generator too, you want the comparison to reflect the prompt change, not sampling noise.

from anthropic import Anthropic
client = Anthropic()
GEN_MODEL = "claude-sonnet-4-5"

VARIANT_A = "You are a helpful assistant. Answer the user's question."

VARIANT_B = (
    "You are a helpful assistant. Answer the question using ONLY the provided context. "
    "If the context does not contain the answer, reply exactly: "
    "'The context does not say.' Do not use outside knowledge."
)

def generate(context: str, question: str, system_prompt: str) -> str:
    msg = client.messages.create(
        model=GEN_MODEL,
        max_tokens=200,
        temperature=0,
        system=system_prompt,
        messages=[{"role": "user",
                   "content": f"Context:\n{context}\n\nQuestion: {question}"}],
    )
    return msg.content[0].text.strip()

# smoke test on the refusal case
g = GOLDEN[3]
print("A:", generate(g["context"], g["question"], VARIANT_A))
print("B:", generate(g["context"], g["question"], VARIANT_B))

Checkpoint 3 — the LLM-as-judge (25 min)

Build this

Write the faithfulness judge. The rubric must have: a numbered 1 to 5 scale with explicit per-level anchors, an instruction to write reasoning first and the score last, and it must score only faithfulness (claims supported by context), ignoring style and length. Parse out an integer Score: N. Use a low temperature and, ideally, name a judge model from a different family than your generator to dodge self-preference (or at least be ready to defend the choice). Test it on one obviously-faithful and one obviously-fabricated answer.

Show solution

This is the heart of the harness. Note the anchors, the forced ordering, and the strict parse.

import re

JUDGE_MODEL = "claude-sonnet-4-5"   # in practice prefer a different family than the generator

JUDGE_RUBRIC = """You are a strict evaluator of FAITHFULNESS.
Score ONLY whether every claim in the ANSWER is supported by the CONTEXT.
Ignore style, tone, and length. Saying 'the context does not say' when the
context truly lacks the answer is FULLY FAITHFUL and should score 5.

Scale (integer 1-5):
5 = every claim supported by the context; nothing invented; correct refusal counts as 5.
4 = supported, with one trivial unsupported detail that does not mislead.
3 = mostly supported; one clear claim is unsupported by the context.
2 = several unsupported claims, or one important invented fact.
1 = key claims contradict or are absent from the context.

First write a brief REASONING naming the specific claim(s) you checked.
Then, on the FINAL line, output exactly: Score: N
"""

def judge(context: str, question: str, answer: str) -> dict:
    user = f"CONTEXT:\n{context}\n\nQUESTION:\n{question}\n\nANSWER:\n{answer}"
    msg = client.messages.create(
        model=JUDGE_MODEL, max_tokens=400, temperature=0,
        system=JUDGE_RUBRIC,
        messages=[{"role": "user", "content": user}],
    )
    text = msg.content[0].text
    m = re.search(r"Score:\s*([1-5])", text)
    if not m:
        raise ValueError(f"no score parsed:\n{text}")
    return {"score": int(m.group(1)), "reasoning": text.strip()}

# sanity test
ctx = "Returns are accepted within 30 days of delivery."
print(judge(ctx, "Return window?", "30 days from delivery.")["score"])      # expect 5
print(judge(ctx, "Return window?", "90 days, and shipping is free.")["score"]) # expect 1-2

Checkpoint 4 — run the harness over both variants (20 min)

Build this

Loop over the golden set, for each item generate with variant A and variant B, judge both, and collect per-item scores. Print the mean faithfulness per variant and a per-item table (id, score A, score B). This is the moment the eval-driven loop produces a number: is variant B actually better, on average, on the same fixed set?

Show solution

Store everything so the next checkpoints (regression flags, guardrail, calibration) can reuse it.

def run_variant(system_prompt: str) -> list:
    rows = []
    for g in GOLDEN:
        ans = generate(g["context"], g["question"], system_prompt)
        verdict = judge(g["context"], g["question"], ans)
        rows.append({"id": g["id"], "answer": ans,
                     "score": verdict["score"]})
    return rows

results_A = run_variant(VARIANT_A)
results_B = run_variant(VARIANT_B)

def mean(rows): return sum(r["score"] for r in rows) / len(rows)

print(f"\nmean faithfulness  A={mean(results_A):.2f}  B={mean(results_B):.2f}\n")
print(f"{'id':<5}{'A':>4}{'B':>4}")
for a, b in zip(results_A, results_B):
    print(f"{a['id']:<5}{a['score']:>4}{b['score']:>4}")

Expect variant B to win, mainly by getting the refusal cases (g4, g7) right where variant A may answer from outside knowledge. If B does not win, that is a real and interesting result, inspect the answers.

Checkpoint 5 — flag regressions (15 min)

Build this

A higher mean is not enough, the average can rise while specific cases get worse, and a worse case might be your most important one. Add a check that flags every item where variant B scored lower than variant A, and refuse to declare B a clean win if any regression exists. This is the guardrail discipline from Day 34 applied per-case.

Show solution

Always inspect the cases that got worse, not just the mean. That habit is the senior signal.

regressions = [
    {"id": a["id"], "A": a["score"], "B": b["score"]}
    for a, b in zip(results_A, results_B)
    if b["score"] < a["score"]
]

if not regressions and mean(results_B) > mean(results_A):
    print("CLEAN WIN: B beats A on the mean with no per-item regressions.")
elif regressions:
    print(f"HOLD: B regressed on {len(regressions)} item(s) — inspect before shipping:")
    for r in regressions:
        print(f"  {r['id']}: A={r['A']} -> B={r['B']}")
else:
    print("NO IMPROVEMENT: B did not beat A on the mean.")

Checkpoint 6 — a cheap programmatic check alongside the judge (15 min)

Build this

Not everything needs an LLM. Add a deterministic guardrail that runs for free next to the judge: for the refusal cases (ideal answer is "the context does not say"), programmatically assert the variant's answer actually signals refusal (contains a "does not"/"not in the context"/"cannot find" style phrase). Report a pass rate. This shows you reaching for the bottom layer of the pyramid where there is a deterministic ground truth, instead of paying a judge call for something a string check handles.

Show solution

The point: use the judge for subjective faithfulness, use a free assertion for the checkable structural property.

REFUSAL_MARKERS = ("does not say", "not in the context",
                   "cannot find", "no information", "not specified")

def looks_like_refusal(text: str) -> bool:
    t = text.lower()
    return any(m in t for m in REFUSAL_MARKERS)

def refusal_pass_rate(results) -> float:
    refusal_ids = {g["id"] for g in GOLDEN if "does not say" in g["ideal_answer"]}
    rel = [r for r in results if r["id"] in refusal_ids]
    passed = sum(1 for r in rel if looks_like_refusal(r["answer"]))
    return passed / len(rel)

print(f"refusal guardrail  A={refusal_pass_rate(results_A):.0%}  "
      f"B={refusal_pass_rate(results_B):.0%}")

You will typically see B near 100 percent and A much lower, concrete proof that the grounding instruction works, obtained for zero LLM cost.

Checkpoint 7 — calibrate the judge against your own labels (25 min)

Build this

This is the step that makes the judge trustworthy, and the one most people skip. Pick 5 of the variant-B answers, read each yourself, and assign your own faithfulness score 1 to 5 (you are the human ground truth). Then compare your 5 labels to the judge's scores on those same 5: report exact-match agreement and the mean absolute difference, and flag any case where you and the judge disagree by 2 or more. State whether you would trust this judge to gate decisions, and if not, what you would fix.

Show solution

You fill in HUMAN_LABELS with your own honest reads. The code measures agreement; the judgment is yours.

# YOU assign these after reading 5 of the variant-B answers yourself.
# (ground truth = your judgment, not the model's)
HUMAN_LABELS = {
    "g1": 5, "g4": 5, "g6": 5, "g7": 5, "g9": 5,
}

judged = {r["id"]: r["score"] for r in results_B}
pairs = [(hid, HUMAN_LABELS[hid], judged[hid]) for hid in HUMAN_LABELS]

exact = sum(1 for _, h, j in pairs if h == j) / len(pairs)
mad = sum(abs(h - j) for _, h, j in pairs) / len(pairs)
big_gaps = [(i, h, j) for i, h, j in pairs if abs(h - j) >= 2]

print(f"calibration on {len(pairs)} items:")
print(f"  exact agreement = {exact:.0%}")
print(f"  mean abs diff   = {mad:.2f}")
if big_gaps:
    print(f"  DISAGREEMENTS (>=2): {big_gaps}  -> sharpen rubric anchors here")
else:
    print("  no large disagreements — judge looks usable for gating")

Interpretation: high exact agreement and a mean absolute difference well under 1 means the judge tracks you and you can trust it to gate experiments. A 2-plus gap on any case means go back and sharpen the anchor (or add an in-context example) for that level, the disagreements tell you exactly which part of the rubric is ambiguous. With only 5 labels this is a directional check; in production you would use more labels and a proper statistic (Cohen's kappa for the discrete scores, Spearman for ranking) and want substantial agreement, roughly 0.6 or higher, before trusting the judge. The discipline is identical at any scale: never trust a judge you have not compared to a human.

Stretch goals (optional)

If you finish early and want to push the harness toward production quality:

  • Position-bias-proof pairwise judging. Instead of scoring each answer alone, judge A vs B head to head, then judge B vs A, and only count a winner if both orders agree. That neutralizes position bias from Day 32.
  • Add a second metric. Write an answer-relevancy judge (does the answer address the question asked) and report both faithfulness and relevancy, the start of the focused five-metric panel from Day 34.
  • Wire it as a CI gate. Make the script exit non-zero when variant B regresses on any item or falls below a faithfulness threshold, that is the prompt CI/CD gate, ready to drop into a workflow.
  • Cost and latency. Time each generate and judge call and tally token usage, the seed of the observability you built in Day 35.

Module 5 review — lock it in

You finished the evaluation module. Answer these out loud cold, then expand the weak ones with Claude. These are the questions a senior interview will actually press on.

  • Why is evaluation the most underrated AI engineering skill, and what is the eval-driven loop?
    Key: building a prototype is commoditized, proving a change is better on a non-deterministic system is the bottleneck. Loop: define metric + dataset, baseline, change one variable, re-measure, keep or revert.
  • Draw the three-layer eval pyramid and the offline/online split.
    Key: programmatic (cheap, every request) → LLM-judge (nightly, subjective quality) → human (weekly, ground truth). Offline = fixed set pre-deploy gate; online = live signals post-deploy truth.
  • What is LLM-as-judge, how do you make the rubric stable, and where does it break?
    Key: LLM scores outputs vs a rubric to scale human eval. Stable rubric: numbered scale, per-level anchors, in-context examples, reasoning before score, one dimension per call. Breaks on position/verbosity/self-preference bias and must be calibrated.
  • How do you build a golden set that does not rot, and what are replay and regression sets?
    Key: seed by intent, mine prod traces (low-confidence + bugs), tag, refresh quarterly. Replay = real inputs re-run to diff behavior; regression = every fixed bug frozen as a permanent test.
  • What RAG metrics do you report, which do you trust most, and what is the five-metric rule?
    Key: faithfulness (trust most), answer relevancy, contextual precision, contextual recall, plus one operational metric. Track a focused ~five so regressions do not hide in clutter.
  • How do you ship a change safely from offline to production?
    Key: offline gate → shadow (real traffic, no user risk) → canary (small growing per-user A/B) → full, watching guardrail metrics with a kill switch at every rung. Automate the offline gate with prompt CI.
  • What does an LLM observability pipeline look like and what SLOs do you set?
    Key: trace per request as a span tree (LLM/tool/retrieval spans with tokens, cost, latency), OTel/OTLP or LangSmith/Langfuse, structured logs, never raw PII. SLOs: p95/p99 latency, error+refusal rate, cost/request budget, faithfulness floor on a rolling sample.
