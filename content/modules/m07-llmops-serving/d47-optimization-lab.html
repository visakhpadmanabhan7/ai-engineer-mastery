Lab — cut cost & latency

Today you stop reading and start optimizing. You take a deliberately naive multi-call feature, apply the four levers from this module, and measure the before and after. By the end you will have a repeatable harness that proves a cost and latency win with numbers, exactly the artifact a senior interview wants you to describe.

By the end you can
  • Instrument an LLM feature with per-call token and cost accounting.
  • Apply prompt caching to a static prefix and measure the saving.
  • Add a semantic cache for repeat queries and reason about its risks.
  • Route easy turns to a smaller model and parallelize independent calls.
  • Report a clean before/after table of cost, latency, and any quality delta.

1 · The naive feature (the "before")

Our target is a small "document insights" feature. For each user question it makes three sequential calls against a large static reference document: classify the question's difficulty, draft an answer, then polish the wording. It is naive in five ways you will fix: the big reference doc is resent uncached every call, repeat questions are recomputed from scratch, every call uses the expensive model, the three calls run serially even though two are independent, and nothing is measured. Here is the instrumented baseline.

import asyncio, time, hashlib
from dataclasses import dataclass, field
from anthropic import AsyncAnthropic

client = AsyncAnthropic()  # reads ANTHROPIC_API_KEY from env

BIG = AsyncAnthropic  # placeholder import guard; real code uses the client above
REFERENCE_DOC = open("reference.md").read()   # a few thousand tokens, static

# --- price sheet (per token); update to the live numbers for your model ---
PRICES = {
  "big":   {"in": 5/1e6,  "out": 15/1e6, "cached_in": 0.5/1e6},
  "small": {"in": 0.8/1e6, "out": 4/1e6,  "cached_in": 0.08/1e6},
}
MODELS = {"big": "claude-big-model", "small": "claude-small-model"}  # put real ids here

@dataclass
class Meter:
    cost: float = 0.0
    calls: int = 0
    def add(self, usage, tier="big"):
        p = PRICES[tier]
        plain_in  = usage.input_tokens - getattr(usage, "cache_read_input_tokens", 0)
        cached_in = getattr(usage, "cache_read_input_tokens", 0)
        self.cost += plain_in*p["in"] + cached_in*p["cached_in"] + usage.output_tokens*p["out"]
        self.calls += 1

async def ask_naive(question, meter):
    # big doc resent uncached, expensive model, three SERIAL calls
    sys = [{"type": "text", "text": "You analyze the document."},
           {"type": "text", "text": REFERENCE_DOC}]
    diff   = await _call(MODELS["big"], sys, f"Rate difficulty (easy/hard): {question}", meter)
    draft  = await _call(MODELS["big"], sys, f"Answer: {question}", meter)
    polish = await _call(MODELS["big"], sys, f"Polish this answer: {draft}", meter)
    return polish

async def _call(model, system, user, meter, tier="big"):
    r = await client.messages.create(
        model=model, max_tokens=400, system=system,
        messages=[{"role": "user", "content": user}])
    meter.add(r.usage, tier)
    return r.content[0].text

Run a small workload (say 20 questions, some repeated) through ask_naive, timing the wall clock and reading meter.cost. That number and that latency are your baseline. Everything below is measured against it. Note: the model ids and exact prices above are placeholders, fill in your provider's live values before running, per the accuracy rule from the brief.

2 · The four levers

You will apply them in increasing order of effort, measuring after each so you can attribute the win:

  • Lever A — prompt caching the static reference doc, so it is billed at a fraction after the first call instead of full price three times per question.
  • Lever B — semantic cache for repeat questions, so a near-duplicate skips all three calls entirely.
  • Lever C — model routing, sending the easy turns (difficulty rating, polish) to a smaller model and reserving the big model for the hard drafting.
  • Lever D — parallelization, running the independent calls concurrently with asyncio so latency is the max of them, not the sum.
BEFORE AFTER ───────────────────────── ───────────────────────────── doc resent full ×3/question prompt-cached prefix (~10% billed) every repeat recomputed semantic cache short-circuits repeats big model on all 3 calls small model on classify + polish classify → draft → polish (serial) classify ∥ (draft → polish) concurrent nothing measured token + cost meter, before/after table

Build checkpoints — implement, then reveal

Do each checkpoint yourself first, run it, read the new cost and latency, then open the solution and compare. The measured delta is the lesson.

Checkpoint A — prompt caching

Add a cache_control breakpoint so the big static REFERENCE_DOC is cached. Re-run the workload and report the new cost. Why does the saving show up on the second call onward, not the first?

Show solution

Mark the reference block as cacheable and keep it first (static-first, Day 44):

sys = [{"type": "text", "text": "You analyze the document."},
       {"type": "text", "text": REFERENCE_DOC,
        "cache_control": {"type": "ephemeral"}}]   # <- breakpoint after the static doc

The first call writes the cache (a one-time write cost, slightly higher than a normal call); every call after, within the ~5 min TTL, reads the doc from cache and is billed roughly 10% on those tokens. Because the three calls per question all share the same prefix, calls two and three of the very first question already hit the cache, and all later questions do too. The Meter already separates cache_read_input_tokens, so the reported cost drops without any other change. This is usually the single biggest line-item win, because the reference doc was the largest input by far.

Checkpoint B — semantic cache

Add a semantic cache so a question that means the same as a recent one returns the cached final answer and skips all three calls. Implement a simple version and state where you would refuse to use it.

Show solution

Embed the question, compare to recent (embedding, answer) pairs, and return the stored answer above a conservative threshold:

class SemanticCache:
    def __init__(self, threshold=0.92):
        self.items = []          # (vector, answer)
        self.t = threshold
    async def get(self, q_vec):
        for vec, ans in self.items:
            if _cosine(q_vec, vec) >= self.t:
                return ans            # HIT: skip all LLM calls
        return None
    def put(self, q_vec, ans): self.items.append((q_vec, ans))

Wrap the feature: on a hit, return immediately and record zero LLM cost for that request; on a miss, run the (now cached + routed) pipeline and put the result. Refuse to use it for personalized, time-sensitive, or high-stakes queries (Day 44): two users' "summarize my document" embed alike but must not share an answer, and "what changed today" goes stale instantly. Gate those intents to bypass the cache, or include the user/tenant in the key. The cost win here is the largest per repeat (the whole request is free) but only materializes if your workload actually has repeats, so report hit rate alongside cost.

Checkpoint C — model routing

Route the difficulty-rating and polishing calls to the small model, keeping the big model only for the actual answer drafting. Re-measure. What is the quality risk and how do you check it?

Show solution

Pass the tier through and pick the model + price per call:

diff   = await _call(MODELS["small"], sys, f"Rate difficulty: {q}", meter, tier="small")
draft  = await _call(MODELS["big"],   sys, f"Answer: {q}",            meter, tier="big")
polish = await _call(MODELS["small"], sys, f"Polish: {draft}",        meter, tier="small")

Classification and rewriting are easy, format-bound tasks the small model handles well, and it can be 5-10x cheaper per token, so moving two of three calls off the big model is a large cost cut with little quality exposure. The risk is that the small model underperforms on the task you moved; you check it the same way you check any model change, with an eval (Module 5): run your golden set through both routings and confirm the routed version's answer quality stays within bounds. If difficulty is rated hard, you could also route the draft (and even keep polish on the big model) accordingly, routing is a policy you tune against measured quality, not a guess.

Checkpoint D — parallelize

The difficulty rating does not depend on the draft, so it should not block it. Restructure so independent calls run concurrently and report the latency change. Why does this change latency but not cost?

Show solution

Run the independent calls together with asyncio.gather; only chain what is truly dependent (polish needs the draft):

# difficulty rating and drafting are independent → run concurrently
diff, draft = await asyncio.gather(
    _call(MODELS["small"], sys, f"Rate difficulty: {q}", meter, "small"),
    _call(MODELS["big"],   sys, f"Answer: {q}",          meter, "big"),
)
polish = await _call(MODELS["small"], sys, f"Polish: {draft}", meter, "small")  # depends on draft

Latency for the two independent calls becomes the max of them instead of the sum, so wall-clock drops. Cost is unchanged because you still make the same number of calls on the same models with the same tokens, you only changed when they run, not whether they run. This is the key distinction: parallelization is a latency lever, caching and routing are cost levers. Combining all four gives the best of both.

Module 7 review — questions across all six days

A consolidated drill across the module. Say the one-line key, then expand the ones you stumble on with Claude.

  • Why are output tokens more expensive than input tokens?
    Key: input prefills in one parallel pass (cheap/token); output decodes serially, one pass per token, so it is priced 2-4x higher.
  • What are TTFT and TPS, and which does a long prompt hurt?
    Key: TTFT = prefill latency (a long prompt hurts this); TPS = decode speed; total ≈ TTFT + output/TPS.
  • Order the inference-optimization ladder.
    Key: prompt caching → KV caching → continuous batching → quantization → speculative decoding; cheapest/safest first.
  • Why is KV-cache memory the limit on concurrency and context length?
    Key: the cache scales with sequence length × batch and lives in VRAM, so memory, not compute, caps how much you serve at once.
  • What does continuous batching buy and does it change outputs?
    Key: 2-4x throughput by keeping the GPU saturated (token-level add/evict); outputs unchanged, it is a scheduling change.
  • Name the five cache layers.
    Key: prompt, semantic, tool-result, embedding, retrieval, each catching a different unit of repeated work.
  • How do you do Anthropic prompt caching correctly, and where is semantic caching dangerous?
    Key: static-first byte-identical prefix + breakpoint, cached input ~10%, ~5 min TTL; semantic cache is dangerous for personalized/time-sensitive/high-stakes intents.
  • What is the strongest agent safety control and why?
    Key: the read/write split with an approval gate, it contains even a hijacked model to a declined draft because it depends on irreversibility, not detection.
  • How do you ship a prompt change safely?
    Key: prompts as versioned code → golden-set CI gate that blocks regressions (incl. cost/format) → canary with auto-rollback; tag every call for attribution.
  • How do you handle a 429, and what is a circuit breaker?
    Key: respect Retry-After, backoff + jitter, cap attempts, retry only idempotent calls; breaker is Closed → Open → Half-open to fail fast during outages.
  • What are the four application-side levers to cut cost and latency?
    Key: prompt caching + model routing (cost), semantic cache + parallelization (latency); measure each and verify quality.
