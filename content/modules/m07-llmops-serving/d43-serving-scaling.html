Serving & scaling

When throughput, latency, or privacy push you off a hosted API, you reach for a serving stack. Today you learn the optimization ladder, which rung to climb when, and the one diagram (continuous batching) that interviewers love.

By the end you can
  • Decide between a hosted API and self-hosting on first principles, not vibes.
  • Order the inference-optimization ladder and say what each rung buys.
  • Explain continuous batching and why it gives 2-4x throughput over static batching.
  • Reason about KV cache memory as the real constraint on concurrency and context length.
  • Say when quantization and speculative decoding are worth it, and their risks.

1 · Self-host or use an API?

Start with the default: use a hosted API. You get frontier models, autoscaling, and zero GPU ops, and you pay per token. Self-hosting only earns its keep when one of a few specific pressures shows up:

  • Data residency / privacy: regulated data that cannot leave your boundary.
  • Cost at high, steady volume: when token spend on an open model would clearly beat renting GPUs that stay busy. Idle GPUs are pure loss, so this needs real utilization.
  • Latency / control: you need a fixed model version, custom quantization, or tight tail latency you cannot get from a shared endpoint.
  • A fine-tuned open model (Module 6) that you must serve yourself.

The honest trade is: API = speed-to-ship and no ops, higher per-token price, less control. Self-host = lower marginal cost at scale and full control, but you now own batching, GPU memory, autoscaling, and on-call. Most teams stay on the API far longer than they expect, and that is usually correct.

Interview-grade phrasing

"I default to a hosted API and only self-host for data residency, a fine-tuned open model, or genuine cost-at-scale where the GPUs stay busy. Self-hosting trades a higher engineering and ops burden, batching, KV-cache memory, autoscaling, for a lower marginal token cost and full control. The break-even is about utilization, not list price."

2 · The inference-optimization ladder

When you do serve a model, you climb a ladder of techniques, cheapest and safest first. Do not jump to the bottom rung; each higher rung adds complexity or risk:

rung what it does when to climb it ────────────────────────────────────────────────────────────────────────────────── 1 prompt caching reuse a shared prefix across calls long shared system prompts 2 KV caching reuse attention K/V within a request always on; agent loops 3 continuous batching pack many requests onto the GPU you have concurrency 4 quantization shrink weights (INT8/INT4) VRAM-bound, latency-sensitive 5 speculative decoding draft small + verify big decode latency is the pain

Rungs 1 and 2 are essentially free wins and should always be on. Rung 3 is where most serving throughput comes from. Rungs 4 and 5 are situational and carry quality or complexity costs, so you reach for them only when a measured bottleneck demands it. Prompt caching gets its own lesson tomorrow (Day 44); here we focus on the serving-side rungs.

3 · KV caching and why memory is the limit

Recall causal attention: token t attends to all earlier tokens. Naively, generating each new token would recompute the Keys and Values for every prior token, which is wasteful because those do not change. The KV cache stores the per-layer Key and Value tensors for every token already processed, so each new token only computes its own K/V and attends to the cached rest.

KV caching is what makes decode tractable, and it matters most when there is a long shared prefix reused many times: agent loops that keep re-reading the same growing context, or chat with a big system prompt. The catch is memory: the KV cache grows with (layers × heads × head_dim × sequence_length × batch) and lives in precious GPU VRAM. So the real ceiling on how many concurrent requests and how long a context you can serve is usually KV cache memory, not raw compute. This is why long context and high concurrency fight each other on a fixed GPU.

Interview-grade phrasing

"KV caching stores each layer's Key and Value tensors for tokens already seen, so each new token attends without recomputing the past. It is biggest on long shared prefixes and agent loops. The trade is VRAM: the cache scales with sequence length and batch size, so KV-cache memory, not FLOPs, is what caps concurrency and max context on a given GPU."

4 · Continuous batching: the throughput rung

GPUs are most efficient when processing many sequences at once, but LLM requests arrive at random times and finish at random lengths. Static batching waits to fill a batch, runs it, and cannot return any result until the longest sequence in the batch finishes, so short requests are held hostage by long ones and the GPU sits half-idle between batches.

Continuous batching (also called in-flight batching, the headline feature of vLLM) operates at the token level: it adds new requests into the running batch and evicts finished ones every step, so a slot freed by a completed sequence is immediately filled by a waiting one. The GPU stays saturated and short requests are not blocked. The payoff is on the order of 2-4x higher throughput versus static batching, which directly lowers cost-per-request because you serve more on the same hardware.

STATIC BATCHING (sequences locked together) step → 1 2 3 4 5 6 7 8 reqA [====================done] ← all wait for the longest reqB [======done] · · · · (idle, can't leave) reqC [===========done]· · · · (idle, can't leave) └ batch returns only after step 8; GPU underused ┘ CONTINUOUS BATCHING (token-level add/evict) step → 1 2 3 4 5 6 7 8 reqA [====================done] reqB [======done]→ reqD enters here [=========done] reqC [===========done]→ reqE enters [=====done] └ freed slots refill every step; GPU stays saturated ┘ 2-4x throughput

5 · Quantization: trade precision for VRAM and speed

Quantization stores weights (and sometimes activations) in fewer bits, typically INT8 or INT4 instead of 16-bit. This shrinks the memory footprint, which lets a model fit on a smaller GPU and frees VRAM for a bigger KV cache, and it can speed up memory-bound decode.

The cost is quality. INT8 is usually close to lossless for many models; INT4 saves more but can visibly degrade reasoning and edge-case behavior, so you must measure on your own eval set (Module 5) before shipping it. Rule of thumb: reach for quantization when you are VRAM-bound and latency-sensitive on self-hosted hardware, start at INT8, and only drop to INT4 if your evals say quality holds. Quantization is not a free win; it is a measured trade.

6 · Speculative decoding: same answer, faster

Decode is serial, one token per forward pass of a big model. Speculative decoding breaks that bottleneck with a draft-and-verify scheme: a small, fast draft model proposes the next N tokens, then the big target model verifies all N in a single parallel forward pass, accepting the longest prefix that matches what it would have produced and discarding the rest.

The key property for a skeptic: when done correctly, the output distribution is identical to running the big model alone. You are not approximating; you are using cheap guesses that the expensive model checks in parallel. When the draft is good, you get several tokens per expensive pass, so latency drops with no quality change. When the draft is bad, you fall back to roughly normal speed. The win depends on how well the draft model predicts the target, which is why drafts are usually a smaller member of the same family.

Interview-grade phrasing

"Speculative decoding has a small draft model propose N tokens and the big model verify them in one parallel pass, accepting the matching prefix. Crucially the output distribution is unchanged, so it is faster, not approximate. The speedup tracks the draft's acceptance rate, and it falls back to normal decode when the draft is wrong."

Worked example: pick the rung for three problems

Example

You self-host one open model and hit three separate issues. Match each to the right rung.

  • "Throughput is low even though each request is fast and the GPU is at 40% during traffic." The GPU is idling between requests, that is a batching problem. Turn on continuous batching (vLLM). Expect 2-4x throughput, no quality change.
  • "The model barely fits in VRAM and there is no room for a bigger batch." You are VRAM-bound. Quantize to INT8 to free memory for a larger KV cache and batch, then verify quality on your eval set before INT4.
  • "Per-token decode latency is the user complaint on long answers." Decode is the bottleneck. Add speculative decoding with a small same-family draft model; the output distribution is unchanged, so it is a safe latency win.

Notice the discipline: each symptom points to a specific rung, and you never reach past the rung the bottleneck demands. Climbing to INT4 or speculative decoding when the real fix was batching just adds risk for no reason.

Why this matters for you as an AI engineer

Serving is where cost becomes real

On a hosted API the provider has already climbed most of this ladder for you, which is why the API is cheap to ship on. The moment you self-host, every rung becomes your job, and getting them right is the difference between a serving cost that scales and one that bankrupts the feature. Even if you never run vLLM yourself, understanding KV-cache memory and continuous batching lets you reason about why long context is expensive and why concurrency has a ceiling, which shapes the systems you design in Module 10.

Practice — do these before revealing

Say each answer out loud in full sentences, as if to an interviewer. Then open the solution and compare. The gap is your lesson.

Problem 1

Your self-hosted endpoint serves long-context agent loops. Concurrency tops out far below what the GPU's compute suggests it should handle. What is the likely constraint, and name two levers to raise concurrency.

Show solution

The constraint is almost certainly KV-cache memory, not compute: long contexts make each request's cache large, and many concurrent long-context requests exhaust VRAM well before FLOPs become the limit. Levers: (1) quantize weights (e.g. INT8) to free VRAM for more KV cache, and (2) shorten the effective context, prune agent history, lower retrieved top-k, so each request's cache is smaller. Continuous batching helps utilization but does not create memory; the real ceiling here is bytes of KV cache, so you either free VRAM or shrink per-request context.

Problem 2

A colleague wants to switch from static to continuous batching but worries it will change model outputs. Reassure or correct them, and explain where the 2-4x actually comes from.

Show solution

Reassure them: continuous batching is a scheduling change, not a math change, so per-request outputs are unaffected. The speedup comes from GPU utilization. Static batching holds the whole batch until the longest sequence finishes and idles the GPU between batches, so short requests wait and capacity is wasted. Continuous batching adds and evicts sequences at the token level, refilling freed slots every step, so the GPU stays saturated and finished requests return immediately. Same outputs, far better hardware utilization, hence 2-4x throughput.

Problem 3

You quantize a model to INT4 and throughput improves, but a few hard reasoning cases now fail that passed before. What happened, and what is the disciplined response?

Show solution

INT4 quantization traded precision for memory and speed, and the lost precision degraded the model on exactly the hardest, most precision-sensitive cases, which is the classic INT4 failure mode. The disciplined response is to measure, not eyeball: run your eval set (Module 5) on INT4 vs INT8 vs full precision, and choose the lowest precision whose quality stays within your acceptable bound. If INT4 fails the bar, step back to INT8 (often near-lossless) and accept the smaller memory win. Never ship a quantization level you have not evaluated.

Today's takeaways

Default to a hosted API; self-host only for residency, a fine-tuned open model, or cost-at-scale with busy GPUs. The optimization ladder runs prompt caching, KV caching, continuous batching, quantization, speculative decoding, climbed cheapest-and-safest first. KV-cache memory, not compute, is the real ceiling on concurrency and context length. Continuous batching gives 2-4x throughput by keeping the GPU saturated, with no change to outputs. Quantization and speculative decoding are situational: one trades quality for VRAM and must be measured, the other keeps the distribution and only speeds decode.
