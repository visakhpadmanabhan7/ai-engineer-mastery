Caching strategies

Caching is the highest return-on-effort move in the entire LLM stack: the same work, never paid for twice. Today you learn the five layers, how to do Anthropic prompt caching right, and the one place caching quietly becomes dangerous.

By the end you can
  • Name the five cache layers in an LLM stack and what each one saves.
  • Implement Anthropic prompt caching correctly: stable prefix first, billing and TTL behavior.
  • Explain semantic caching and identify exactly where it is dangerous.
  • Key a tool-result cache and an embedding cache so hits actually land.
  • Pick the biggest cheap win for a given feature instead of caching blindly.

1 · Five cache layers, five different savings

"Caching" in an LLM app is not one thing. There are five distinct layers, each catching a different kind of repeated work. Knowing which one applies to a bottleneck is the senior skill:

LayerWhat it cachesWhat it saves
Prompt cacheA stable token prefix (system prompt, tools, big context) inside the providerRe-prefill cost + TTFT on the shared prefix
Semantic cacheThe full answer to a query, keyed by meaningThe entire LLM call for near-duplicate questions
Tool-result cacheThe result of a tool/API call, keyed by normalized argsRe-running expensive or rate-limited tools
Embedding cacheThe vector for a piece of text, keyed by text hashRe-embedding identical text (ingest + query)
Retrieval cacheThe retrieved chunk set for a queryRe-running the vector search + rerank

They stack. A single RAG request can hit four of them: embed the query (embedding cache miss the first time), search (retrieval cache), send the prompt with a cached system prefix (prompt cache), and if the same question was asked recently, skip all of it via the semantic cache. Each layer sits at a different point in the request path and saves a different unit of work.

user query │ ├─▶ [semantic cache] hit? → return cached answer (skips everything below) │ miss ↓ ├─▶ [embedding cache] vector for this query text? (hash key) │ ↓ ├─▶ [retrieval cache] chunks for this query? (search/rerank reuse) │ ↓ ├─▶ [tool-result cache] any tool calls? reuse by normalized args │ ↓ └─▶ LLM call with [prompt cache] on the stable prefix → answer ↓ (write answer back into semantic cache)
Interview-grade phrasing

"An LLM stack has five cache layers: prompt cache (reuse a stable prefix, saves re-prefill and TTFT), semantic cache (reuse a whole answer by meaning, saves the entire call), tool-result cache (reuse a tool's output by normalized args), embedding cache (reuse a vector by text hash), and retrieval cache (reuse a chunk set). The biggest cheap win for most apps is prompt caching the static system prefix."

2 · Prompt caching done right (Anthropic)

Prompt caching lets the provider reuse the computed state of a stable prefix of your prompt instead of re-prefilling it every call. The rules that make it actually work:

  • Put static content first. The cacheable prefix must be byte-identical across calls, so order your prompt so the unchanging parts (system instructions, tool definitions, a large reference document, few-shot examples) come before anything that varies (the user message, retrieved chunks for this query). A cache breaks at the first differing byte.
  • Mark the cache point. You signal where the cacheable prefix ends (Anthropic uses a cache_control breakpoint on the block); everything up to that point becomes the cached prefix.
  • Billing: cached input is much cheaper. On a cache hit, the cached input tokens are billed at roughly 10% of the normal input price. There is a modest one-time write cost to populate the cache, so caching pays off when the prefix is reused enough times to amortize the write.
  • TTL is short. The cache lives about 5 minutes and refreshes on use, so it shines for bursty, repeated traffic (an agent loop re-reading the same context, many users hitting the same big system prompt within minutes), not for once-a-day calls.

The structural mistake to avoid: interleaving variable content into the prefix. If you place the per-request user question before the big static document, nothing is cacheable because the prefix changes every call. Static-first is the whole game.

# prompt ORDER decides whether the cache works
messages = [
  {"role": "system", "content": [
     {"type": "text", "text": BIG_STATIC_INSTRUCTIONS},   # stable, goes first
     {"type": "text", "text": REFERENCE_DOC,
      "cache_control": {"type": "ephemeral"}},      # <- cache breakpoint here
  ]},
  {"role": "user", "content": user_question},          # varies per call, AFTER the cache point
]
# hit: the prefix above the breakpoint is billed ~10% of input price

3 · Semantic caching, and where it bites

A semantic cache skips the whole LLM call when a similar enough question was already answered. The mechanism: embed the incoming query, search a small store of past (query-embedding, answer) pairs, and if the nearest neighbor is above a cosine-similarity threshold, return its cached answer. It is the most powerful cache because it saves the entire call, and the most dangerous because "similar enough" is a judgment that can be wrong.

Where it is dangerous, three red zones:

  • Personalized answers. "What's my account balance?" from two different users embeds nearly identically but must never share an answer. Either do not semantically cache personalized queries, or include the user/tenant in the key.
  • Time-sensitive answers. "What's the status of order 1234?" or "today's price" goes stale immediately. A cache hit serves yesterday's truth. Use a short TTL or exclude these intents.
  • High-stakes answers. Medical, legal, or financial responses where a near-duplicate is not actually equivalent. A 0.95-similar question can have a materially different correct answer; serving the cached one is a real harm. Raise the threshold or do not cache.

The safe pattern: only semantic-cache stable, non-personalized, low-stakes, repeated queries (FAQs, general explanations), set a conservative threshold, and gate by intent so personalized and time-sensitive turns bypass the cache entirely.

Where semantic caching hurts

The failure is silent and confident: a wrong-but-plausible cached answer is returned with no error, because the system believes the queries are equivalent. Personalized, time-sensitive, and high-stakes intents are the three places where "near-duplicate query" does not imply "same correct answer." Gate those out by intent before they ever reach the cache.

4 · Keying the smaller caches

The tool-result, embedding, and retrieval caches are simpler but live or die on cache keys. A bad key means hits never land (key too specific) or wrong data is served (key too loose).

  • Tool-result cache: normalize the args before keying. get_weather(city="Munich") and get_weather(city=" munich ") should be the same key. Lowercase, trim whitespace, sort any list/dict args, drop fields that do not affect the result, then hash. Add a TTL that matches how fast the underlying data changes (a stock quote: seconds; a country's capital: forever).
  • Embedding cache: key on a hash of the exact text (and the embedding model id, since vectors are model-specific). Identical chunks across documents then embed once. This is a pure win with no staleness risk, because the same text always has the same vector for a given model.
  • Retrieval cache: key on the normalized query plus the filter set (tenant, metadata filters), because the same query under a different tenant filter must not collide. TTL it to your index's update cadence so freshly ingested docs are not hidden.

Worked example: cache a multi-tenant FAQ bot

Example

A docs assistant answers questions over a shared knowledge base for many tenants, with a 4000-token system prompt and 5 retrieved chunks per call. Where does each cache layer go?

  • Prompt cache on the 4000-token system prompt + tool definitions, marked static-first. Bursty traffic within 5 minutes is billed at ~10% on that prefix, big TTFT and cost win.
  • Embedding cache on query text (hash + model id). Repeated questions skip re-embedding for free.
  • Retrieval cache keyed on normalized query + tenant filter. Two tenants asking the same thing must not share results, so the tenant goes in the key.
  • Semantic cache only for general, non-personalized FAQs ("how do I reset my password?"), keyed with tenant if any answer differs per tenant, conservative threshold, short TTL. Account-specific or "current status" questions bypass it.
  • Tool-result cache if the bot calls a status API, keyed on normalized args with a short TTL because statuses change.

The dominant win here is the prompt cache (a 4000-token prefix re-prefilled on every call is the biggest waste), and the dangerous layer is the semantic cache, which is exactly why it is gated to safe intents and carries the tenant in its key.

Why this matters for you as an AI engineer

Caching is the cheapest lever you have

Of every cost and latency technique in Module 7, caching has the best effort-to-payoff ratio, no retraining, no quantization risk, no serving stack. The interview signal is not "caching is good" (everyone says that); it is knowing which of the five layers applies, that prompt caching needs a static-first prefix and bills cached input at ~10%, and crucially that semantic caching is dangerous for personalized, time-sensitive, and high-stakes intents. Naming that last hazard unprompted is what separates someone who has shipped caching from someone who has only read about it.

Problem 1

An engineer enabled prompt caching but the cache hit rate is near zero. Their prompt order is: retrieved chunks for this query, then the user question, then the big static system instructions at the end. What is wrong and how do you fix it?

Show solution

The prompt is in the wrong order. Prompt caching reuses a stable prefix, and the prefix here starts with the per-query retrieved chunks, which change every call, so the very first bytes differ and nothing is cacheable. Fix: reorder so the static content (system instructions, tool definitions, any fixed reference doc) comes first and is byte-identical across calls, place the cache breakpoint at the end of that static block, and put the variable parts (retrieved chunks, user question) after it. Static-first is mandatory; a cache breaks at the first differing byte.

Problem 2

Your semantic cache cut cost nicely, but support reports users occasionally getting answers about someone else's order. Diagnose the bug and give two fixes.

Show solution

The semantic cache is serving personalized and time-sensitive answers across users. "What's the status of my order?" from different users embeds almost identically, crosses the similarity threshold, and returns another user's cached answer, a classic dangerous-zone failure. Fixes: (1) gate by intent, classify order-status and account queries as personalized/time-sensitive and bypass the semantic cache entirely; and (2) include the user/tenant id in the cache key so a hit can only return that user's own prior answer, plus a short TTL so status answers do not go stale. The general fix is: never semantically cache personalized or time-sensitive intents.

Problem 3

Your tool-result cache for a search_products(query, filters) tool has a poor hit rate even though users ask very similar things. The key is the raw JSON string of the arguments. Why is the hit rate low, and what is the fix?

Show solution

Keying on the raw argument JSON means trivially different but semantically identical calls miss: {"query":"red shoes","filters":{"size":42,"color":"red"}} and {"query":"red shoes ","filters":{"color":"red","size":42}} produce different strings (trailing space, key order) and therefore different keys. Fix: normalize before keying, lowercase and trim the query, sort filter keys, drop fields that do not affect results, then hash the canonical form. Pair it with a TTL matched to how fast the catalog changes. Normalized keys are what make the smaller caches actually hit.

Today's takeaways

Five cache layers each save a different unit of work: prompt, semantic, tool-result, embedding, retrieval. Prompt caching is usually the biggest cheap win, and it requires a byte-identical static-first prefix; cached input bills around 10% with about a 5-minute TTL. Semantic caching is the most powerful and the most dangerous, never use it for personalized, time-sensitive, or high-stakes intents. The smaller caches live or die on normalized keys (and the right TTL). Caching is the best effort-to-payoff lever in the whole stack, as long as you cache the right thing in the right place.
