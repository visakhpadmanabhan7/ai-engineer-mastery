Deployment & CI/CD for LLM apps

An LLM feature is software, so it deserves containers, versioning, and a CI gate, plus one twist: a prompt change is a behavior change with no compiler to catch it. Today you build the pipeline and the reliability primitives that keep it from breaking in production.

By the end you can
  • Containerize an LLM app and manage config and secrets cleanly.
  • Treat prompts and tool definitions as versioned code (semver, tagged per call).
  • Design a CI/CD gate that runs a golden set and blocks on regression before canary.
  • Place the reliability primitives, timeout, retry with backoff and jitter, circuit breaker, fallback chain, in the call path.
  • Handle provider 429s correctly so a rate limit does not become an outage.

1 · Containerize the app

An LLM app is mostly ordinary backend code: an API server that orchestrates model calls, retrieval, and tools. So you ship it like any service, in a Docker container for reproducibility across machines. The LLM-specific notes are small but matter: the model weights are usually not in your image (you call a hosted API), so the image stays light; pin dependency versions so a silently updated SDK does not change behavior; and keep the container stateless so it scales horizontally (Module 8), with state in a database or cache.

Config and secrets are kept out of the image entirely. API keys, model ids, and tunables come from environment variables or a secrets manager injected at runtime, never baked into the build or committed to git. This lets you promote the same image through dev, staging, and prod by changing config, and rotate a leaked key without rebuilding.

# a light FastAPI image; no model weights, no secrets baked in
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt   # pinned versions
COPY . .
# ANTHROPIC_API_KEY, MODEL_ID, etc. injected at runtime, not here
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

2 · Prompts and tools are code

The defining discipline of LLMOps: a prompt is not a string buried in a function, it is a versioned artifact. Changing a prompt changes the system's behavior just as surely as changing code, but there is no type checker to catch a regression, so you impose the structure yourself:

  • Prompts as code. Store prompts (and tool definitions) in version control, reviewed via pull request like any change, not edited live in a console where no one sees the diff.
  • Semantic versioning. Give each prompt a version (e.g. summarize_v2.3). A wording tweak bumps the patch; a structural change bumps the minor or major.
  • Tag every call. Log the prompt version, model id, and temperature alongside each request (Module 5). When quality moves, you can answer "which prompt version and model produced this?" instead of guessing.

This is what makes the rest possible: if a prompt is a tagged, versioned artifact, you can run it against a test set in CI, roll it back like code, and attribute a production regression to a specific change.

Interview-grade phrasing

"I treat prompts and tool definitions as versioned code: in source control, changed via PR, semver-tagged, and logged on every call with the model id and temperature. A prompt edit is a behavior change with no compiler to catch regressions, so the discipline that replaces the compiler is a golden-set CI gate plus per-call version tagging for attribution."

3 · The prompt CI/CD gate

Because there is no compiler, the safety net is an automated evaluation gate in CI. When a pull request changes a prompt, a tool, or the model, the pipeline runs the change against a golden set (a curated set of representative inputs with expected outputs or rubric scores, Module 5) and blocks the merge if quality regresses past a threshold. Only after passing does the change roll out, and even then to a small slice first.

PR changes a prompt / tool / model │ ▼ ┌──────────── CI GATE ────────────┐ │ run golden set (N cases) │ │ score: correctness / faithfulness/│ │ format / cost / latency │ │ compare vs baseline (main) │ └───────────────┬─────────────────┘ regression?│ ┌──────┴───────┐ YES NO │ │ BLOCK merge merge → CANARY (e.g. 5% traffic) (report diff) │ watch online metrics + guardrails ├─ healthy → ramp to 100% └─ regression → auto-rollback

Two non-obvious points. First, the gate scores more than correctness: format adherence, faithfulness, and even cost and latency are regressions worth blocking on, a prompt that is 5% more accurate but doubles tokens may not be a win. Second, passing CI is necessary but not sufficient, because offline eval cannot fully predict live behavior (Module 5), so you still canary: release to a small percentage, watch online metrics and guardrails, and ramp up only if healthy, with automatic rollback if not.

4 · Reliability primitives in the call path

You are calling a remote service that can be slow, rate-limited, or briefly down. Four primitives wrap every external LLM (or tool) call so a hiccup does not cascade:

  • Timeout. Never wait forever. Bound each call so a hung request fails fast instead of holding a worker.
  • Retry with backoff and jitter. On a transient failure, retry, but wait exponentially longer each time (backoff) and add randomness (jitter) so many clients do not retry in lockstep and hammer the recovering service (the thundering herd). Cap the number of attempts.
  • Circuit breaker. If failures pile up, stop calling for a cooldown: Closed (normal, counting failures) trips to Open (fail fast, do not call) after a threshold, then Half-open (let one probe through; success closes the circuit, failure re-opens it). This protects both you and the struggling dependency.
  • Fallback chain. When the primary path fails or the breaker is open, degrade gracefully: try a secondary model or provider, serve a cached answer, or return a clear "temporarily unavailable" rather than a stack trace.

Crucially, only retry idempotent calls. Retrying a read is safe; blindly retrying a write can double-charge or duplicate a record (Module 4), so writes need idempotency keys before they are eligible for retry.

import random, time
from anthropic import Anthropic, APIStatusError

client = Anthropic()  # reads ANTHROPIC_API_KEY from env

def call_with_retry(make_call, attempts=4, base=0.5, timeout=30):
    for i in range(attempts):
        try:
            return make_call(timeout=timeout)
        except APIStatusError as e:
            if e.status_code == 429 and i < attempts - 1:
                retry_after = float(e.response.headers.get("retry-after", 0))
                wait = max(retry_after, base * 2**i) + random.uniform(0, 0.3)  # backoff + jitter
                time.sleep(wait)
                continue
            raise
    # on exhaustion: caller triggers the fallback chain (smaller model / cache)

5 · Handling provider 429s

A 429 Too Many Requests means you hit a rate limit, not that anything is broken. The correct response is specific: respect the Retry-After header if the provider sends one (it tells you how long to wait), otherwise fall back to exponential backoff with jitter; cap attempts so you do not retry forever; and only retry idempotent calls. If 429s persist, that is a capacity signal, route some traffic to a secondary model, smooth bursts with a client-side token-bucket limiter, or queue. Treating a 429 as a hard failure turns a recoverable throttle into a user-facing outage; treating it as "wait and retry sensibly" keeps the feature up.

Worked example: ship a prompt change safely

Example

You improve the system prompt for a summarization endpoint. Trace it from edit to production.

  • Edit as code: bump summarize_v2.2 → v2.3 in the repo, open a PR. A reviewer sees the prompt diff like any code change.
  • CI gate: the pipeline runs v2.3 against the golden set of 60 representative documents, scoring faithfulness, format, length, and cost vs the v2.2 baseline. v2.3 improves faithfulness but its summaries run 25% longer (cost regression). The gate flags it; you tighten the length instruction and rerun until cost is within bounds and faithfulness still wins.
  • Canary: merge, deploy the new container image (same image, version comes from config), route 5% of traffic to v2.3, and watch online faithfulness sampling and the cost-per-request guardrail.
  • Ramp or roll back: metrics hold, so ramp 5% → 25% → 100%. Every request is tagged prompt=summarize_v2.3, so if a problem surfaces later you can attribute and roll back to v2.2 instantly.
  • In the call path the whole time: each model call has a timeout, retries 429s with backoff and jitter, and falls back to a smaller model if the breaker trips, so a provider hiccup during the rollout does not look like a bad deploy.

This is the LLMOps loop: prompts as code, a golden-set gate that blocks regressions including cost, a canary because offline eval is not the last word, and reliability primitives that keep the rollout legible.

Why this matters for you as an AI engineer

This is the "Ops" in LLMOps

Plenty of people can write a good prompt once. The senior skill is shipping prompt changes repeatedly and safely: versioned artifacts, a CI gate that catches regressions before users do, a canary because the gate is not infallible, and a call path that survives 429s and brief outages. When an interviewer asks "how do you deploy LLM changes without breaking production?", this lesson is the whole answer, and it is exactly the operational maturity that separates a demo from a product.

Problem 1

A teammate edited the production system prompt directly in the provider console to fix a bug. It worked, but two days later quality dropped and no one can tell what changed. What practices were missing, and what do you put in place?

Show solution

Two practices were missing: prompts as versioned code and per-call version tagging. Editing live in a console leaves no diff, no review, and no record, so a later regression cannot be attributed or rolled back. Put in place: store prompts in version control with a semver, change them only via PR (so the diff is reviewed and runs the CI golden-set gate), and tag every request with the prompt version, model id, and temperature. Now a quality drop is traceable to a specific version and reversible by redeploying the previous one. The console edit also bypassed the regression gate entirely, which is how the bad change reached users.

Problem 2

Your service starts returning errors to users whenever the LLM provider throttles you with 429s. Walk through how you would make the service resilient to rate limits without dropping requests on the floor.

Show solution

Treat 429 as "wait and retry", not as a failure. Respect the Retry-After header when present, otherwise back off exponentially with jitter, and cap the attempts so you do not retry indefinitely; only retry idempotent calls. Add a client-side token-bucket limiter to smooth bursts so you stop exceeding the limit in the first place, and a circuit breaker so that if the provider is sustainedly overloaded you fail fast for a cooldown instead of piling on. Finally, a fallback chain, route to a secondary model or serve a cached answer when the primary is throttled, so users get a degraded-but-working response instead of an error. Persistent 429s are a capacity signal to add routing or queueing.

Problem 3

Your golden-set CI gate passes a new prompt with a higher correctness score, so you ship it to 100% immediately. Two failures follow: a tool-call format the golden set never covered breaks downstream, and cost jumps. What two process changes prevent this next time?

Show solution

First, do not ship straight to 100%, canary. Offline eval cannot cover every live input distribution (the uncovered tool-call format proves it), so release to a small slice, watch online metrics and guardrails, and ramp only if healthy with automatic rollback. Second, the gate must score more than correctness: add format adherence and a cost (and latency) check to the golden-set comparison, so a correctness gain that breaks structure or doubles tokens is flagged as a regression rather than passing. Together: a multi-dimensional gate catches the cost and format issues offline, and the canary catches whatever the gate's dataset missed before it hits everyone.

Build it (15–20 min)

Hands-on

Take a small FastAPI app with one Claude-powered endpoint and add the deployment scaffolding. (1) Write a slim Dockerfile that reads ANTHROPIC_API_KEY and MODEL_ID from the environment, never baked in. (2) Move your prompt into a prompts/ file with a version string, and log that version on every request. (3) Wrap the model call with a timeout and a retry that backs off with jitter on 429 (respect Retry-After), using the Anthropic errors reference for status codes. (4) Write a tiny "golden set" of 5 input/expected pairs and a script that fails (exit non-zero) if any case regresses, the seed of a CI gate. Run it locally as if it were a pre-merge check.

Wire the golden-set script into GitHub Actions later so it runs on every PR. See Tools & Claude for API setup.

Today's takeaways

An LLM app ships like any service, containerized, stateless, secrets injected at runtime, with one twist: prompts and tools are versioned code because a prompt edit is an uncaught behavior change. The compiler you do not have is replaced by a CI gate that runs a golden set and blocks on regression (including cost and format), followed by a canary because offline eval is not the last word. Wrap every external call with a timeout, retry with backoff and jitter, a circuit breaker, and a fallback chain, and retry only idempotent calls. Handle 429s by respecting Retry-After and backing off, so a rate limit degrades gracefully instead of becoming an outage.
