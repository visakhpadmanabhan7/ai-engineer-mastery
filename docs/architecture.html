Architecture · how it actually works

Architecture

A single async FastAPI service that ingests a bundled curriculum into a database on startup, schedules a question bank with the FSRS-6 spaced-repetition algorithm, and fronts a provider-agnostic AI tutor that streams over any LLM you plug in. One process, no build step on the frontend, and the same code path on SQLite or any managed Postgres.

FastAPI async SQLAlchemy 2.0 async Pydantic v2 Anthropic / Groq / OpenAI-compatible FSRS-6 pgvector → JSON fallback

The one-line mental model

Browser (vanilla JS) → api.js → FastAPI routers → thin services (ai, srs, ingest, activity) → the database and the chosen LLM. Routers own HTTP and auth; services own the logic; SQLAlchemy and the provider SDKs own I/O. Nothing in a router talks to a provider SDK directly.

1 · Request lifecycle and components

The whole stack is one uvicorn process. On boot, the lifespan context manager runs init_db(), ingests content/ into the database, and seeds the dev user; then the routers serve, and the frontend is mounted last as static files so every /api/* path wins over the SPA.

Why one process

No message broker, no separate worker, no frontend build pipeline. The frontend is static files served by the same app; the only optional external dependencies are a Postgres URL and one LLM key. That keeps the free-tier deploy to a single web service and makes the repo runnable with ./run.sh and zero configuration.

4 · The provider-agnostic AI tutor

All LLM access goes through services/ai.py. Routers call stream_chat(...) or grade(...) and never touch a provider SDK. The provider is resolved once from config: LLM_PROVIDER may be anthropic, groq, none, or auto, and auto picks Groq if its key is set, else Anthropic, else none.

Two providers, one interface

The clients are created lazily inside getters (_anthropic() / _openai()) so the SDK only needs to be installed for the provider you actually use. The OpenAI client is pointed at OPENAI_BASE_URL (default Groq), so Groq, Together, OpenRouter, and local Ollama all reuse the same code path.

Two capabilities

Streamed stream_chat for teaching, deepen, and mock interviews (async generator yielding text deltas). Structured grade for scoring: it requests JSON only, then defensively extracts and normalizes score / feedback / one_fix / model_answer even if the model wraps it in fences.

Tutor vs judge: a deliberate cost split

Every provider runs the same pattern: a strong model teaches and runs mock interviews, a small fast model grades. Quality where it matters, cost and latency where it does not.

Three modes, one system prompt builder

system_for(kind, context) assembles the persona. A shared VISAKH_CONTEXT grounds the tutor in the learner's real, citable work (so explanations anchor to concrete experience and never fabricate beyond it), a VOICE block enforces a precise senior register and American English, and the kind swaps in the right teaching contract: deepen teaches one level past a textbook then asks probing questions one at a time; mock runs an escalating interview and withholds model answers until you respond, then scores; the default chat answers rigorously with a hint before the full answer.

The streaming detail that matters

The tutor reply is an HTTP StreamingResponse of text/plain. The request-scoped DB session is closing while bytes are still flowing to the browser, so the assistant turn cannot be persisted on it. The generator accumulates the streamed chunks and, after the stream ends, opens a fresh SessionLocal() to write the assistant message:

    async def gen():
        acc: list[str] = []
        async for chunk in ai.stream_chat(system, api_messages):
            acc.append(chunk)
            yield chunk
        # persist with a fresh session — the request session is already closing
        async with SessionLocal() as s2:
            s2.add(TutorMessage(session_id=session_id, role="assistant", content="".join(acc)))
            await s2.commit()

The user turn and the streak bump are committed before streaming starts, and the new session id is returned to the client in an X-Session-Id header (with X-Accel-Buffering: no so proxies do not buffer the stream). Multi-turn coherence comes from replaying the session's prior messages on each call.

Graceful degradation

With no key, provider() resolves to none: stream_chat yields one friendly message explaining how to enable the tutor, and grade returns a null score with the same hint. Network or model errors are caught and surfaced inline rather than crashing the request. Everything else in the app (curriculum, progress, the full FSRS review loop, analytics) works with no LLM at all.

5 · Spaced repetition (FSRS-6)

Scheduling lives in services/srs.py behind a tiny public surface: new_card(), review(blob, rating, now), and engine_name(). On import it constructs an FSRS Scheduler; if that import fails or the library's API differs, it logs a warning and silently uses a compact SM-2 implementation, so the review loop never goes down.

The review queue

GET /api/review/queue returns, in order: due cards (ReviewCard.due ≤ now, joined to their question, ordered by due date), then backfills up to the limit with brand-new questions that have no card yet (capped at 15 new per call). The query uses the indexed due column so the hot path stays cheap as the deck grows.

4-button grading

POST /api/review/grade takes a rating of 1 to 4 (Again / Hard / Good / Easy), runs the scheduler, then writes the next due, the serialized fsrs_state, the new state, and increments reps (and lapses on an Again). Every grade is also appended to ReviewLog and bumps the daily streak.

FSRS state is stored as JSON, not as fixed columns. The library's full Card is dumped with to_dict() into fsrs_state and rebuilt with from_dict() on the next review, which keeps persistence tolerant of library version changes; only due and state are broken out as real columns (the former indexed for the queue). The 4-button rating maps cleanly onto the SM-2 quality scale in the fallback path, so a card's history survives an engine switch.

Why FSRS over SM-2

SM-2 (the classic Anki algorithm) tracks one ease factor and a fixed interval ladder. FSRS-6 models memory with separate difficulty and stability states and predicts the recall probability of each card, scheduling the next review at a target retention. In practice that means roughly 30% fewer reviews than SM-2 for the same retention, the right default for an interview-prep deck where review time is the scarce resource. The SM-2 fallback exists only so the feature degrades instead of failing.

6 · Database portability

The same models and the same engine code run on SQLite locally and on any managed Postgres in production. Two pieces make that work: a portable vector column type, and URL normalization for host-provided connection strings.

The portable Vector type

Vector is a SQLAlchemy TypeDecorator. At connection time it checks the dialect and a module-level PGVECTOR_OK flag: on Postgres-with-the-extension it becomes a real pgvector column (HNSW-indexable, 384-dim for all-MiniLM-L6-v2); everywhere else it becomes a Text column and the embedding is JSON-encoded on the way in and decoded on the way out. The Python interface is identical either way, so application code never branches on the backend.

    def load_dialect_impl(self, dialect):
        if self._use_pg(dialect):            # postgresql AND PGVECTOR_OK
            from pgvector.sqlalchemy import Vector as PGVector
            return dialect.type_descriptor(PGVector(self.dim))
        return dialect.type_descriptor(Text())   # JSON-encoded list elsewhere

The flag is set during init_db(): it runs CREATE EXTENSION IF NOT EXISTS vector in its own transaction (so a failure on a host that does not grant the extension cannot abort the subsequent create_all), and only sets PGVECTOR_OK = True if that succeeds. On any Postgres without pgvector, the app logs a warning and quietly uses JSON columns; it still deploys and runs.

postgres:// → asyncpg URL normalization

Managed hosts (Render, Neon, Supabase) hand out postgres://... URLs that the async driver will not accept as-is. The db_url property rewrites them so you can paste the host's URL verbatim:

StepTransform
1. Scheme aliaspostgres:// → postgresql://
2. Async driverpostgresql:// → postgresql+asyncpg:// (if not already)
3. Strip libpq paramsdrop the ?sslmode=require… query string (asyncpg rejects libpq query params)

SSL is handled separately and out of band: pg_ssl_enabled turns on when PG_SSL is set or the raw URL contained sslmode=require / ssl=true, and the engine is then created with connect_args={"ssl": True}. So the stripped sslmode is not lost; it is re-applied the way asyncpg expects. pool_pre_ping=True keeps idle connections healthy on hosts that drop them.

7 · Authentication flow

POST /api/auth/register|login password ──▶ sha256 digest ──▶ base64 (44 bytes, < bcrypt's 72) ──▶ bcrypt.hashpw / checkpw │ on success ▼ create_access_token(email) ──▶ JWT { sub: email, iat, exp:+7d } (HS256) │ every protected call: Authorization: Bearer <JWT> │ deps.get_current_user ──▶ decode_token ──▶ sub ──▶ SELECT User WHERE email = sub │ invalid/expired ──▶ 401 Bearer challenge ▼ User ──▶ router handler

Tokens are stateless: there is no session table, so the JWT secret and a 7-day expiry are the whole story. Passwords are sha256-prehashed then base64-encoded before bcrypt, which sidesteps bcrypt's silent 72-byte truncation (long passphrases stay fully significant) and avoids the brittle passlib/bcrypt version coupling. verify_password and decode_token both fail closed: any exception returns False / None rather than leaking a 500. Protected routes depend on get_current_user, which resolves the bearer token to a real User row or raises a 401 with a WWW-Authenticate: Bearer challenge.

8 · Design decisions

Build on the stack it teaches

The curriculum covers async Python services, RAG, evaluation, and system design, so the app is deliberately FastAPI + async SQLAlchemy + Postgres/pgvector + an LLM API. Running and extending it is the lab. That is also why pgvector, embeddings, and the LLM-as-judge grader are first-class even though the bundled deck would run on plain text.

Provider-agnostic, BYO key

No vendor lock-in and no required spend: one config switch chooses Anthropic, Groq (free tier), or any OpenAI-compatible host. The tutor/judge split is the same everywhere, and the app degrades to fully working with no key at all. SDK imports are lazy so you install only what you use.

Vanilla-JS frontend, no build

Plain HTML, one api.js fetch layer, two CSS files. No bundler, no node_modules, nothing to compile, audit, or break. The frontend is served as static files by the same FastAPI process, which keeps the deploy a single web service. The backend is the interesting part; the UI stays out of its way.

Bundle the content

The 68 lessons and the question bank (with answer keys) live in content/ and are ingested on startup, so the repo stands alone with no external CMS, network call, or dependency on the Resume-app it was extracted from. Self-contained content is also what makes the free one-click deploy possible.

SQLite default, Postgres-ready

Default to SQLite so the barrier to running it is ./run.sh, while the portable Vector type and URL normalization mean production Postgres needs only a connection string, with no schema changes and no pgvector requirement. One codebase, two deployment shapes.

Thin routers, thin services

Routers own HTTP, auth, and persistence; services (ai, srs, ingest, activity) own logic and are swappable in isolation. That boundary is what lets the SRS engine and the LLM provider each fall back or change without the routers knowing.
